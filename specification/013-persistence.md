# Data persistence

**Purpose.** This document specifies Redis as Heimdall's mandatory runtime data persistence backend, fixes the connection modes and the security requirements on those connections, and defines the Redis data model — including data structure selection and key namespace — for each data domain in scope. It does not restate zone-file parsing, query-resolution semantics, cache policy, or RPZ evaluation logic; those questions are settled in [`001-server-roles.md`](001-server-roles.md), [`004-cache-policy.md`](004-cache-policy.md), [`006-protocol-conformance.md`](006-protocol-conformance.md), and [`011-rpz.md`](011-rpz.md) respectively.

**Status.** Stable.

**Requirement category.** `STORE`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md). For the cache segregation model this document extends, see [`004-cache-policy.md`](004-cache-policy.md). For the zone-transfer requirements relevant to atomic zone replacement and IXFR, see [`006-protocol-conformance.md`](006-protocol-conformance.md). For the DNSSEC validation outcomes encoded in cache entries, see [`005-dnssec-policy.md`](005-dnssec-policy.md). For the RPZ policy whose durable trigger store is specified here, see [`011-rpz.md`](011-rpz.md). For the cryptographic policy whose TLS requirements apply to the optional TCP connection mode, see [`003-crypto-policy.md`](003-crypto-policy.md). For the runtime-operations policy whose atomic-swap and reload requirements this document satisfies, see [`012-runtime-operations.md`](012-runtime-operations.md). For the threat model whose security posture governs Redis connection security, see [`007-threat-model.md`](007-threat-model.md).

## 1. Scope

This document covers three data domains.

**Authoritative zone data.** The RRsets that constitute zones served by the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md), section 1.1. Zone files on the filesystem are the configuration import source; Redis is the runtime store from which queries are served and against which zone-lifecycle operations are applied. Zone files remain the canonical configuration input: on a full reload triggered by `OPS-001` in [`012-runtime-operations.md`](012-runtime-operations.md), zone files are re-parsed and Redis is updated to reflect the new zone content.

**Query-response cache.** The cached RRsets maintained separately by the recursive resolver role and by the forwarder role, as fixed by [`004-cache-policy.md`](004-cache-policy.md). Redis is the persistence medium for cache state across process restarts and for cache sharing across multiple Heimdall instances in a deployment. Within a running process, cache lookups on the hot path are served from in-process data structures that are populated from, and written back to, Redis; the concrete in-process cache structure is governed by the open question in section 4 of [`004-cache-policy.md`](004-cache-policy.md).

**Response Policy Zone data.** The trigger and action entries managed for RPZ evaluation on the recursive resolver role, as fixed by [`011-rpz.md`](011-rpz.md). Redis is the durable store for RPZ trigger and action data. In-process data structures used for sub-linear query-time trigger matching are populated from Redis on load; the concrete matching structures are tracked as open in section 4 of [`011-rpz.md`](011-rpz.md) and in section 7 of this document.

## 2. Redis as the mandatory persistence backend

### 2.1 Normative requirements

- **STORE-001.** Redis MUST be Heimdall's sole runtime persistence backend for the three data domains defined in section 1. No other external database, key-value store, or filesystem-based storage mechanism MUST be used as a substitute for, or supplement to, Redis for the runtime representation of zone data, cache entries, or RPZ data.
- **STORE-002.** Heimdall MUST support Redis version 7.0 or later. No Redis feature introduced after version 7.0 may be relied upon without an explicit requirement in this document and a corresponding open question being resolved.
- **STORE-003.** Heimdall MUST treat Redis as an external service with its own availability and durability posture. The persistence durability of data stored in Redis — whether Redis persists data to disk via RDB snapshots, AOF logging, or not at all — is an operator responsibility. Heimdall MUST behave correctly regardless of which Redis persistence mode the operator has configured, including the no-persistence mode.
- **STORE-004.** On every full configuration reload triggered by `SIGHUP` under `OPS-001` of [`012-runtime-operations.md`](012-runtime-operations.md), Heimdall MUST re-parse all zone files and update the corresponding zone data in Redis atomically per zone in accordance with `STORE-023`. Redis is updated to reflect the reloaded zone content; it is not read as the source of truth during the reload. Zone files remain the canonical configuration input for the authoritative server role.

## 3. Connection modes

### 3.1 Unix domain socket (primary connection mode)

- **STORE-005.** Heimdall MUST support connecting to Redis over a Unix domain socket. A Unix domain socket MUST be the default and preferred connection mode for all three data domains defined in section 1.
- **STORE-006.** The path of the Unix domain socket used to connect to Redis MUST be operator-configurable. There is no mandatory default socket path.
- **STORE-007.** Heimdall MUST verify at startup that the configured Unix domain socket path is accessible and that the process has read and write permission on the socket. If the socket is not accessible, Heimdall MUST refuse to start and MUST emit a structured error event identifying the socket path and the nature of the access failure.

### 3.2 TCP (secondary connection mode)

- **STORE-008.** Heimdall MUST support connecting to Redis over TCP as an alternative to the Unix domain socket. A TCP connection MUST require explicit operator configuration; it MUST NOT be activated by default.
- **STORE-009.** When the TCP connection mode is enabled and the configured Redis host resolves to a non-loopback address, Heimdall MUST require TLS on the TCP connection to Redis. The TLS configuration on this connection MUST comply with the cryptographic policy in [`003-crypto-policy.md`](003-crypto-policy.md). Connecting to a non-loopback Redis instance over plaintext TCP MUST NOT be permitted.
- **STORE-010.** When the TCP connection mode is enabled and the configured Redis host resolves exclusively to a loopback address, TLS on the TCP connection is RECOMMENDED but MAY be omitted by explicit operator configuration.
- **STORE-011.** The TCP host and port used to connect to Redis MUST be operator-configurable.

### 3.3 Authentication

- **STORE-012.** Heimdall MUST authenticate to Redis using the Redis ACL authentication mechanism (`AUTH` command) on both connection modes. Anonymous connections to Redis — connections that do not present an ACL username and password — MUST NOT be permitted, regardless of which connection mode is in use. The Redis ACL username and password MUST be operator-supplied via configuration and MUST NOT be embedded in the Heimdall binary or default configuration.
- **STORE-013.** The Redis ACL user under which Heimdall authenticates MUST be granted only the specific commands and key patterns required for the data domains it accesses. Configuring the Redis ACL user with least-privilege access is an operator responsibility; this requirement records the principle so that the Heimdall documentation and reference deployment guides reflect it.

### 3.4 Connection pool

- **STORE-014.** Heimdall MUST maintain a connection pool to Redis rather than opening a new connection per operation. The connection pool MUST be shared across all data domains.
- **STORE-015.** The minimum and maximum size of the connection pool MUST be operator-configurable. The concrete default values are tracked as open in section 7.
- **STORE-016.** Heimdall MUST reconnect to Redis automatically on connection loss, using an exponential backoff strategy with a bounded maximum delay. The concrete backoff parameters are tracked as open in section 7.
- **STORE-017.** While the connection to Redis is unavailable, Heimdall MUST continue serving queries from its in-process state — loaded zone data, in-process cache — for as long as that state remains valid. Redis unavailability MUST NOT cause an immediate process exit or listener shutdown. Heimdall MUST emit a structured error event on every failed Redis operation during the outage window so that the operator can detect and diagnose the condition.

## 4. Data model

### 4.1 Authoritative zone data

- **STORE-018.** Each authoritative zone MUST be stored in Redis as a single Hash (HSET), one Hash per zone. Every RRset that belongs to the zone MUST be stored as a distinct field within that Hash.
- **STORE-019.** The Redis key for an authoritative zone Hash MUST follow the pattern `heimdall:zone:auth:{fqdn}`, where `{fqdn}` is the zone apex in fully-qualified domain name form, normalised to lowercase ASCII in accordance with `PROTO-002` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
- **STORE-020.** Each field within the authoritative zone Hash MUST uniquely identify a single RRset by the tuple `(owner_name, qtype, qclass)`. The exact encoding of this tuple as a Hash field name — including separators, normalisation of the owner name, and encoding of the qtype and qclass — is tracked as open in section 7.
- **STORE-021.** The value associated with each Hash field MUST be the serialised representation of the corresponding RRset, including the TTL and the complete set of RDATA records. The exact serialisation format is tracked as open in section 7.
- **STORE-022.** Zone deletion MUST be implemented as a single `DEL` command on the zone's Hash key. This provides O(1) zone deletion from the Heimdall perspective.
- **STORE-023.** Atomic zone replacement — replacing the complete RRset set of a zone without exposing a partially-updated state to concurrent queries — MUST be implemented using the `RENAME` command: the new zone data MUST first be written in full to a staging key of the form `heimdall:zone:auth:{fqdn}:staging`, after which `RENAME heimdall:zone:auth:{fqdn}:staging heimdall:zone:auth:{fqdn}` MUST be issued to promote the staging key to the live key in a single atomic operation. The atomicity of `RENAME` satisfies the requirement of `OPS-002` in [`012-runtime-operations.md`](012-runtime-operations.md) for the zone data domain.
- **STORE-024.** RRset lookup for query serving — given `(owner_name, qtype, qclass)`, retrieve the RRset — MUST be implemented as `HGET heimdall:zone:auth:{fqdn} <field>`. This operation MUST be O(1) average complexity at the Redis level.
- **STORE-025.** Zone enumeration for AXFR output MUST be implemented using `HSCAN` with a configurable `COUNT` hint rather than `HGETALL`, so that the Redis server remains responsive to other clients during the enumeration of large zones. The concrete default `COUNT` hint value is tracked as open in section 7.

### 4.2 Query-response cache

- **STORE-026.** Each cached RRset MUST be stored in Redis as an individual String value with the cache entry expiry set using the single-command `SET ... EX` or `SET ... PX` form. A two-command sequence of `SET` followed by a separate `EXPIRE` MUST NOT be used: the two-command form creates a window during which the key exists without an expiry, which violates the TTL semantics required for DNS cache entries.
- **STORE-027.** Cached RRsets for the recursive resolver role and cached RRsets for the forwarder role MUST be stored under distinct key namespaces, consistent with the cache-segregation requirement of `CACHE-001` in [`004-cache-policy.md`](004-cache-policy.md). A cached entry in one namespace MUST NOT be readable from or confused with an entry in the other namespace at any layer.
- **STORE-028.** The Redis key for a recursive resolver cache entry MUST follow the pattern `heimdall:cache:recursive:{owner}|{qtype}|{qclass}`. The Redis key for a forwarder cache entry MUST follow the pattern `heimdall:cache:forwarder:{owner}|{qtype}|{qclass}`. The exact encoding of the owner name, qtype, and qclass within the key — including normalisation rules and handling of characters that require escaping in Redis key strings — is tracked as open in section 7.
- **STORE-029.** The value stored for each cache entry MUST include, at minimum, the DNS TTL, the DNSSEC validation outcome for the entry in accordance with `DNSSEC-010` in [`005-dnssec-policy.md`](005-dnssec-policy.md), and the wire-encoded RRset. The exact serialisation format is tracked as open in section 7.
- **STORE-030.** The Redis key expiry set on each cache entry MUST be derived from the remaining DNS TTL of the entry at the time of the write. The expiry MUST be set in the same `SET` command that writes the value, using `EX` for second-precision expiry or `PX` for millisecond-precision expiry.

### 4.3 Response Policy Zone data

- **STORE-031.** Each RPZ zone's trigger and action entries MUST be stored in Redis as a single Hash (HSET), one Hash per RPZ zone. The structure mirrors the authoritative zone data model of `STORE-018` through `STORE-025`, applied to the RPZ trigger namespace.
- **STORE-032.** The Redis key for an RPZ zone Hash MUST follow the pattern `heimdall:zone:rpz:{zone_fqdn}`, where `{zone_fqdn}` is the RPZ zone apex in fully-qualified domain name form, normalised to lowercase ASCII.
- **STORE-033.** Each field within the RPZ zone Hash MUST uniquely identify a single trigger entry by its trigger type and trigger value. The trigger types covered are QNAME, Response-IP, NSIP, NSDNAME, and Client-IP, as fixed by `RPZ-010` through `RPZ-015` in [`011-rpz.md`](011-rpz.md). The exact encoding of the trigger type and trigger value as a Hash field name is tracked as open in section 7.
- **STORE-034.** The value associated with each RPZ Hash field MUST encode the action type and action parameters for the trigger, covering the mandatory action types fixed by `RPZ-004` through `RPZ-011` in [`011-rpz.md`](011-rpz.md). The exact serialisation of action data is tracked as open in section 7.
- **STORE-035.** At query time, RPZ trigger matching MUST NOT be performed by issuing Redis commands on the hot path. All RPZ trigger matching MUST be performed against in-process data structures — including radix trees or tries for QNAME wildcard matching and IP prefix matching — that are populated from Redis at load time and updated on dynamic RPZ modifications applied through the admin-RPC surface. Redis serves as the durable store for RPZ data; in-process structures serve as the query-time matching engine. This division is required to satisfy the sub-linear matching obligation fixed by `RPZ-024` in [`011-rpz.md`](011-rpz.md).
- **STORE-036.** On every full reload triggered by `OPS-001` in [`012-runtime-operations.md`](012-runtime-operations.md), Heimdall MUST rebuild the in-process RPZ matching structures from the current Redis state. On dynamic RPZ updates applied through the admin-RPC surface under `OPS-015` in [`012-runtime-operations.md`](012-runtime-operations.md), Heimdall MUST update both the Redis Hash entry for the affected RPZ zone and the corresponding in-process matching structures atomically from the perspective of in-flight queries, in accordance with the atomic-swap requirement of `OPS-016` through `OPS-018` in [`012-runtime-operations.md`](012-runtime-operations.md).

## 5. Key namespace

- **STORE-037.** All Redis keys used by Heimdall MUST be prefixed with `heimdall:`. This prefix is the global Heimdall namespace within the Redis keyspace and allows Heimdall to coexist with other applications on the same Redis instance without key collisions.
- **STORE-038.** The second-level namespace segment (immediately after `heimdall:`) MUST encode the component type as follows: `zone:auth:` for authoritative zone data, `cache:recursive:` for recursive resolver cache entries, `cache:forwarder:` for forwarder cache entries, and `zone:rpz:` for RPZ zone data. No other second-level namespace segments MUST be introduced without a corresponding normative requirement in this document.
- **STORE-039.** When Heimdall is deployed in a Redis Cluster, all keys that participate in multi-key atomic operations — specifically the live key and the staging key of `STORE-023` — MUST hash to the same Redis Cluster slot. This MUST be achieved by enclosing the discriminating segment of the key in curly braces as a hash tag, so that the `RENAME` atomicity guarantee of `STORE-023` is preserved under Redis Cluster. The exact hash-tag placement in the key pattern is tracked as open in section 7.

## 6. Rationale

Redis is selected as the persistence backend for three converging reasons. First, the target environment of Heimdall — extremely high load and concurrency — demands that persistent data be held in a structure whose read latency is sub-millisecond on the critical path. Redis, as an in-memory data store with optional disk-persistence, satisfies this requirement. Filesystem-backed zone stores or relational databases introduce I/O latency that would conflict with the performance targets fixed in [`008-performance-targets.md`](008-performance-targets.md). Second, a Redis backend allows multiple Heimdall instances to share the same zone data and cache state, which is a prerequisite for horizontal scaling in the high-load deployments Heimdall explicitly targets. Without a shared external store, each instance would load all zone data independently and maintain completely independent caches. Third, Redis provides atomic operations — `RENAME`, Lua scripts, transactions — that allow zone replacement and cache updates to be performed without partial-state exposure to concurrent queries, satisfying `OPS-002` in [`012-runtime-operations.md`](012-runtime-operations.md).

Unix domain sockets are the primary connection mode because they eliminate the TCP stack entirely for local connections, removing one round-trip through the kernel network layer per Redis command and reducing latency by approximately 15–30% compared to a loopback TCP connection in benchmarked Redis workloads. For a server handling hundreds of thousands of queries per second, each of which may involve one or more Redis round-trips, this margin is material. The TCP mode is provided for deployments where Redis runs on a separate host — a valid topology for large-scale deployments with a dedicated Redis tier — but carries the mandatory TLS requirement on non-loopback connections to preserve the security posture fixed in [`007-threat-model.md`](007-threat-model.md).

The choice of HSET per zone for authoritative and RPZ data, and String keys with `SET ... EX` for cache data, follows from the distinct access patterns of each domain.

For authoritative zone data, the dominant operations are a zone-scoped point lookup (`HGET` for a single RRset given owner, qtype, qclass — O(1) average), zone deletion (`DEL` on one key — O(1)), and atomic zone replacement (`RENAME` on a staging key — O(1)). HSET maps exactly to this structure: one key per zone groups all RRsets under a single Redis key, enabling zone-level operations without secondary indexes. The alternative of one String key per RRset cannot perform atomic zone-level operations without a secondary index and incurs significantly higher per-key memory overhead at zone scale (approximately 64 bytes per top-level key in Redis's global hash table, versus one such entry for the entire zone under HSET). A Sorted Set per zone is inferior in both lookup complexity (O(log N) versus O(1)) and memory usage, with no compensating advantage for the authoritative use case.

For cache data, each entry requires an independent TTL aligned to its DNS TTL, and entries are accessed by individual `(owner, qtype, qclass)` tuples with no zone-level grouping requirement. String keys with `SET ... EX` provide O(1) per-entry access and native per-entry TTL management through Redis's own active and passive expiry machinery, with no dependence on Redis 7.4's `HEXPIRE` field-level expiry, which is less operationally mature and whose active-expiry behaviour for hash fields is less aggressive than key-level expiry under high write rates.

For RPZ data, the durable store in Redis uses HSET per RPZ zone for the same reasons as authoritative zone data. Trigger matching at query time is performed by in-process data structures — radix trees for QNAME wildcard matching, Patricia tries or interval trees for IP prefix matching — populated from Redis on load. This separation is the only architecture that satisfies the sub-linear matching obligation of `RPZ-024` in [`011-rpz.md`](011-rpz.md): Redis does not provide native CIDR prefix matching or DNS wildcard evaluation, and implementing first-match RPZ evaluation purely through Redis round-trips would require multiple commands per query and cannot achieve sub-linear complexity for wildcard or CIDR triggers.

## 7. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will be specified incrementally.

- **Redis Cluster support.** Whether Heimdall supports Redis Cluster — in addition to standalone Redis and Redis Sentinel — as a deployment topology, the key sharding strategy (hash-tag placement for multi-key atomicity), and the behaviour of zone-level atomic operations in a Cluster environment, are **to be specified**.
- **Redis Sentinel support.** Whether Heimdall supports Redis Sentinel for high-availability failover — and the associated reconnection behaviour, leader-discovery mechanism, and failover detection latency budget — is **to be specified**.
- **Field name encoding for zone and cache keys.** The exact encoding of FQDN owner names, qtype values, qclass values, and RPZ trigger types and values as Hash field names and String key segments — including normalisation rules, the choice of separator characters within a field name, case normalisation, and handling of special characters in DNS labels that could appear in Redis key strings — is **to be specified**.
- **RRset serialisation format.** The binary or text format used to serialise RRsets as Redis Hash values — candidates include raw DNS wire format (RFC 1035 §3.2), a length-prefixed binary format, and a structured binary format such as MessagePack or CBOR — and the layout of associated metadata (TTL, insertion timestamp) within the serialised value, are **to be specified**.
- **Cache entry serialisation format.** The serialisation format for cache entries, extending the RRset format with the DNSSEC validation outcome per `DNSSEC-010` in [`005-dnssec-policy.md`](005-dnssec-policy.md), serve-stale metadata per RFC 8767, and negative-cache indicators per RFC 2308, is **to be specified**.
- **IXFR journal in Redis.** Whether Heimdall maintains an incremental zone-transfer journal in Redis — as a supplementary Sorted Set per zone with zone serial as the score, enabling efficient retrieval of changed RRsets since a given serial for `PROTO-039` in [`006-protocol-conformance.md`](006-protocol-conformance.md) — the journal retention window, and the interaction between journal pruning and zone replacement atomicity, are **to be specified**.
- **Connection pool parameters.** The default minimum and maximum pool size values, the connection acquisition timeout, the maximum idle connection lifetime, and the exponential backoff floor, ceiling, and jitter parameters for reconnection under `STORE-016`, are **to be specified**.
- **HSCAN COUNT hint.** The default value of the `COUNT` hint passed to `HSCAN` during zone enumeration for AXFR output under `STORE-025`, and whether it is operator-configurable, are **to be specified**.
- **Redis Cluster hash-tag placement.** For Redis Cluster deployments, the exact position of the hash tag `{...}` within zone keys and staging keys — to guarantee co-location on the same Cluster slot for the `RENAME` of `STORE-023` — is **to be specified** once the field name encoding question above is resolved.
- **Cache warm-up behaviour.** The exact behaviour of Heimdall at startup with respect to warming the in-process cache from Redis — whether warm-up is eager (all entries loaded), lazy (entries loaded on first in-process miss and then cached locally), or absent (in-process cache starts cold and Redis is the primary store on every miss until the in-process cache fills) — is **to be specified**. This item cross-references the open question on cache data structure per role in section 4 of [`004-cache-policy.md`](004-cache-policy.md).
- **TLS configuration for the TCP Redis connection.** The concrete TLS version, cipher suite, and certificate-validation requirements for the optional TCP connection mode under `STORE-009`, beyond the reference to [`003-crypto-policy.md`](003-crypto-policy.md), are **to be specified** in a revision of this document or of [`003-crypto-policy.md`](003-crypto-policy.md).

No implementation activity may proceed on the basis of assumptions about any of the items above.
