## 1. Normative requirements

### 1.1 Rust toolchain: Minimum Supported Rust Version

- **ENG-001.** Heimdall's Minimum Supported Rust Version (MSRV) MUST be the latest stable `rustc` at the time of each release. The MSRV is not pinned to a conservative stable-minus-N baseline; it tracks the current stable toolchain.
- **ENG-002.** The `rust-version` field in `Cargo.toml` MUST be kept in lock-step with `ENG-001`. Every release that occurs after a new stable release of Rust MUST update `rust-version` to the new stable version as part of that release.
- **ENG-003.** Contributors and anyone performing a source build of Heimdall MUST use the current stable `rustc`. The expected path to satisfy this requirement is `rustup update stable`; no further toolchain management is imposed on source-build audiences.
- **ENG-004.** Continuous integration MUST enforce `ENG-001` by running the full build, test, and benchmark matrix against the declared MSRV exactly. Accidental use of unstable Rust features, and accidental use of features that require a `rustc` newer than the declared MSRV, MUST cause a CI failure.
- **ENG-005.** The changelog or release notes of every official Heimdall release MUST state the MSRV in effect for that release.
- **ENG-006.** An MSRV bump on a minor version of Heimdall MUST NOT be treated as a breaking change for semantic-versioning purposes. The bump MUST nevertheless be documented explicitly in the release notes for the release in which it occurs.
- **ENG-007.** The MSRV policy fixed by `ENG-001` through `ENG-006` applies to the source-build audience only, that is, contributors and package maintainers who rebuild Heimdall from source. The pre-built distribution artefacts mandated by `ENV-024` through `ENV-026` in [`009-target-environment.md`](009-target-environment.md) remove the MSRV constraint from end users, who consume Heimdall through signed static binaries, native packages, or container images rather than through a source build.

### 1.2 Dependency acceptance: Architecture Decision Record per dependency

- **ENG-008.** Every addition of a new external crate dependency to any `Cargo.toml` in the Heimdall workspace MUST be accompanied by a dedicated Architecture Decision Record (ADR). The ADR MUST be reviewed and approved before the dependency is committed. A pull request that introduces a new dependency without an accompanying ADR MUST NOT be merged.
- **ENG-009.** Each dependency ADR MUST document all of the following items, and MUST NOT omit any of them:
  1. The problem the dependency solves, accompanied by an explicit alternatives analysis that includes "do not use a dependency" as a considered option.
  2. The audit trail of the dependency, comprising the `cargo-vet` review status, the known Common Vulnerabilities and Exposures (CVE) history, any open RustSec advisories, the dependency's maintenance activity (recent releases and active contributors), and a summary of the dependency's own transitive tree.
  3. The license compatibility of the dependency with Heimdall's own license. Heimdall's own licence is the MIT License, fixed by `ENG-090`; the ADR MUST therefore record the dependency's licence verbatim, confirm that the licence is in the whitelist fixed by `ENG-094`, confirm that it is not in the blocklist fixed by `ENG-095`, and explicitly acknowledge the file-level reciprocity trade-off if the licence is MPL-2.0.
  4. The dependency's `unsafe` code footprint, namely the number of `unsafe` blocks, whether those blocks are audited upstream, and whether Heimdall's usage of the crate exercises the `unsafe` paths.
  5. The supply-chain trust chain of the dependency, comprising the author identity and reputation, the registry ownership, and any signing or provenance metadata available.
- **ENG-010.** Each dependency ADR MUST classify the dependency as either **core-critical** (its loss would block the project) or **convenience** (it is easily replaceable). The categorisation MUST be used to prioritise the periodic review of dependencies.
- **ENG-011.** Each dependency ADR MUST record the commit and the pull request that introduce the dependency, so that the ADR, the code change, and the review discussion are linked end-to-end.
- **ENG-012.** Every removal of an external crate dependency from any `Cargo.toml` in the Heimdall workspace MUST be recorded either as a short update to the ADR that introduced the dependency or as a linked superseding ADR. The project's dependency history MUST therefore be traceable end-to-end, and not just in its current state.
- **ENG-013.** The core dependencies that are already implicit in prior specification decisions (notably `tokio`, `rustls`, `quinn`, `hyper`, `rustix`, `serde`, and their immediate kin as required by the transport, cryptographic, protocol-conformance, and platform decisions already in force) MUST be grandfathered by a one-time retrospective ADR for each crate. These retrospective ADRs MAY be batched into a single coordinated change, but every grandfathered crate MUST have its own distinct ADR entry that satisfies `ENG-009` through `ENG-011`.
- **ENG-014.** The ADR gate fixed by `ENG-008` through `ENG-013` MUST apply uniformly to runtime dependencies and to development dependencies (dependencies declared under `[dev-dependencies]` and used only for testing and tooling). Development dependencies present a real supply-chain surface at build time and MUST NOT be exempt from the gate.
- **ENG-015.** The ADR gate fixed by `ENG-008` through `ENG-014` MUST apply to every Cargo workspace and every crate contained within the Heimdall repository, without exception.
- **ENG-016.** The ADR gate fixed by `ENG-008` through `ENG-015` is the **human gate** on dependency acceptance: its purpose is to assess whether a dependency is needed and proportionate to the problem it solves. The ADR gate MUST operate in addition to, and MUST NOT replace, the automated enforcement already mandated by `THREAT-010` through `THREAT-014` in [`007-threat-model.md`](007-threat-model.md) (dependency minimisation, `cargo-audit` and `cargo-vet` as continuous-integration gates, reproducible release builds, signed release artefacts, and Software Bill of Materials per release), which together form the **mechanical gate** on whether a dependency is intact and currently un-exploited. Both gates are required; neither gate is sufficient on its own.

### 1.3 Unsafe code policy: Heimdall source

- **ENG-017.** Every crate in the Heimdall workspace MUST declare `#![deny(unsafe_code)]` at the crate root by default. The attribute MUST be present on every crate unless the crate has been explicitly and narrowly exempted under `ENG-018`.
- **ENG-018.** Introducing any `unsafe` block or `unsafe fn` within a crate MUST require an explicit `#[allow(unsafe_code)]` override at the narrowest possible scope, that is, per-module or per-item. The override MUST NOT be declared at the crate root.
- **ENG-019.** Every `unsafe` block MUST be accompanied by a `// SAFETY:` comment immediately preceding the block, and every `unsafe fn` signature MUST be accompanied by a `# Safety` section in its doc comment. Each such comment MUST state all of the following items, and MUST NOT omit any of them:
  1. The invariants that the surrounding code establishes and that the caller is expected to uphold.
  2. Why the safe alternative does not suffice, with an explicit reason (for example, performance, foreign function interface, atomic ordering not available in the safe API, SIMD intrinsic).
  3. The scope of the unsafety, comprising the assumptions made about the data, lifetimes, aliasing, and thread-safety.
- **ENG-020.** Continuous integration MUST enforce `ENG-019` by configuring `clippy::undocumented_unsafe_blocks` at the `deny` level. Missing SAFETY documentation MUST fail the build.
- **ENG-021.** The introduction of a new pattern of `unsafe` usage — that is, `unsafe` for a reason not already covered by an existing Architecture Decision Record — MUST require a dedicated ADR. The ADR MUST be reviewed and approved before the change that introduces the unsafe pattern is merged.
- **ENG-022.** Reusing a pattern of `unsafe` usage that has already been approved by an existing ADR (for example, adding a new SIMD variant to an already-approved SIMD module) MUST NOT require a fresh ADR. The mandatory SAFETY documentation fixed by `ENG-019` remains required in every case.
- **ENG-023.** All existing `unsafe` blocks and `unsafe fn`s in the Heimdall source MUST be re-reviewed at least once per year. The purpose of the review is to detect `unsafe` that has become unnecessary due to standard-library evolution and to detect `unsafe` that has become unsound due to changes in surrounding code.
- **ENG-024.** Each periodic review mandated by `ENG-023` MUST produce a written outcome for every `unsafe` item reviewed. The outcome MUST be either an explicit confirmation of continued justification or a recorded task to remove or refactor the item.
- **ENG-025.** The unsafe-code policy fixed by `ENG-017` through `ENG-024` applies to Heimdall's own source only. The `unsafe` footprint of external dependencies is out of scope for this policy and is governed instead by `ENG-009` item 4 through the dependency ADR process.

### 1.4 Testing stack: comprehensive multi-layer

- **ENG-026.** Heimdall MUST use a comprehensive, multi-layer testing stack covering all of the following nine layers: unit tests, integration tests, property-based tests, coverage-guided fuzzing, `miri`, concurrency model checking with `loom`, DNS conformance testing, a controlled integration harness, and benchmarking. All nine layers MUST be present in the project; none of them MAY be omitted. The layers are complementary, not redundant, and each catches a class of bugs the others do not.
- **ENG-027.** The testing policy fixed by `ENG-026` through `ENG-040` MUST apply uniformly to every Cargo workspace and every crate contained within the Heimdall repository, without exception.
- **ENG-028.** Unit tests MUST be written as `#[test]` functions in-module in every crate. Every public function MUST have at least basic unit-test coverage, including its edge cases.
- **ENG-029.** Integration tests MUST be written under each crate's `tests/*.rs` directory. Integration tests MUST exercise cross-module and end-to-end paths within a crate, distinct from the in-module unit tests mandated by `ENG-028`.
- **ENG-030.** Property-based tests MUST use `proptest`, selected over `quickcheck` for its shrinker. Every crate that exposes a parser, a cache, or a state machine MUST have at least one property-based test covering its core invariants. The set of invariants covered MUST include, where applicable to the crate: parser encode/decode roundtrip; cache coherence under concurrent admission and eviction; DNSSEC validation properties; and zone-file parse/serialise roundtrip.
- **ENG-031.** Coverage-guided fuzzing MUST use `cargo-fuzz` (built on `libfuzzer-sys`) and MUST be set up for every parsing surface exposed by Heimdall. The parsing surfaces to be fuzzed MUST include, at a minimum: the DNS message parser, the zone file parser, the EDNS option parser, the HPACK and QPACK decoders, and the QUIC packet demultiplexer.
- **ENG-032.** OSS-Fuzz integration MUST be pursued once Heimdall is in a state ready for submission. Continuous fuzzing against OSS-Fuzz infrastructure is part of Heimdall's long-term testing posture and MUST NOT be treated as optional once the onboarding criteria are met. The concrete timing and criteria for OSS-Fuzz onboarding are tracked as an open question in section 3.
- **ENG-033.** Heimdall's test suites MUST be runnable under `miri`, at least for every module that contains `unsafe` code. This obligation is the dynamic undefined-behaviour detector that keeps the SAFETY documentation mandated by `ENG-019` honest, and is a direct counterpart to the restricted-`unsafe` posture fixed by `ENG-017` through `ENG-025`.
- **ENG-034.** Concurrency-sensitive components MUST have model-checking tests written with `loom`. The scope of this obligation MUST include, at a minimum: segregated caches consistent with [`004-cache-policy.md`](004-cache-policy.md), connection pools, per-worker state, and atomic-ordering-sensitive code paths.
- **ENG-035.** DNS conformance tests MUST exercise RFC test vectors where they exist, including the IETF DNSSEC test vectors and the DoT, DoH, and DoQ interoperability suites. Golden-output comparison against reference implementations (Unbound, NSD, Knot) for representative queries MUST be part of the suite.
- **ENG-036.** A controlled integration-test harness MUST exist that runs Heimdall against simulated authoritatives and simulated clients, end-to-end, in a reproducible environment. The specific scaffolding (for example, Testcontainers, Docker Compose, a custom orchestrator, or an in-process mock harness) is tracked as an open question in section 3.
- **ENG-037.** Micro-benchmarks MUST use `criterion`. End-to-end benchmarks MUST use the standard tools specified in `PERF-014` in [`008-performance-targets.md`](008-performance-targets.md), namely `dnsperf`, `queryperf`, and `resperf`. Both the `criterion` micro-benchmarks and the end-to-end benchmarks MUST feed the continuous-integration regression detection mandated by `PERF-014` through `PERF-018` in [`008-performance-targets.md`](008-performance-targets.md), and the periodic external comparison against reference implementations mandated by `PERF-019` through `PERF-028` in the same file.
- **ENG-038.** The coverage targets applied to the Heimdall source (line, branch, and decision coverage), together with the tool used to measure them, are tracked as an open question in section 3. The absence of a fixed coverage target MUST NOT be interpreted as permission to merge code without the layer-specific coverage obligations fixed by `ENG-028` through `ENG-037`.
- **ENG-039.** The exact frequencies at which `miri`, `loom`, and fuzzing runs are executed in continuous integration (for example, per pull request, nightly, weekly, or continuous via OSS-Fuzz) are tracked as an open question in section 3. The absence of a fixed frequency MUST NOT be interpreted as permission to omit any of these layers from the CI pipeline entirely.
- **ENG-040.** The property-based shrinking budgets, per-test case counts, and golden-test maintenance policy (including the source of reference, the storage format, and the update cadence of golden outputs) are tracked as open questions in section 3.

### 1.5 Continuous-integration pipeline: tiered gates

- **ENG-041.** Heimdall's continuous-integration (CI) pipeline MUST be organised into four tiers, each with a distinct purpose, time budget, and blocking semantics. The four tiers are defined by `ENG-042` through `ENG-045`: Tier 1 per-pull-request fast gates, Tier 2 per-pull-request slow gates, Tier 3 nightly or periodic gates, and Tier 4 release gates. The tier structure fixed by `ENG-041` through `ENG-075` MUST apply uniformly to every Cargo workspace and every crate contained within the Heimdall repository, without exception.
- **ENG-042.** Tier 1 comprises the per-pull-request fast gates. Every check in Tier 1 MUST pass before a pull request is eligible for merge. The end-to-end wall-clock target for Tier 1 is approximately ten minutes. The concrete time budget is tracked as an open question in section 3.
- **ENG-043.** Tier 2 comprises the per-pull-request slow gates. Every check in Tier 2 MUST pass before a pull request is eligible for merge. Tier 2 is time-budgeted above Tier 1 and MUST NOT be treated as optional for merge. The concrete time budget is tracked as an open question in section 3.
- **ENG-044.** Tier 3 comprises the nightly and periodic gates. Tier 3 checks MUST NOT block individual pull requests. A failure in any Tier 3 check MUST generate a tracked issue so that the failure is visible and actionable.
- **ENG-045.** Tier 4 comprises the release gates. Every check in Tier 4 MUST pass before a Heimdall release is tagged.

#### Tier 1 — per-pull-request fast gates

- **ENG-046.** Tier 1 MUST run `cargo build --locked` on the Minimum Supported Rust Version declared by `ENG-001` through `ENG-007`, across every supported build target. The supported build targets for this purpose MUST be `x86_64`, `aarch64`, and `riscv64` on Linux (production targets per `ENV-015` and `ENV-020` in [`009-target-environment.md`](009-target-environment.md)), `x86_64` and `aarch64` on BSD (best-effort production targets), and `x86_64` and `aarch64` on macOS (development-only).
- **ENG-047.** Tier 1 MUST run `cargo test`, covering both unit tests (per `ENG-028`) and integration tests (per `ENG-029`), on the same target matrix required by `ENG-046`.
- **ENG-048.** Tier 1 MUST run `cargo fmt -- --check` against the repository's canonical `rustfmt.toml` fixed by `ENG-063` and `ENG-064`. A formatting divergence MUST fail the build.
- **ENG-049.** Tier 1 MUST run `cargo clippy -- -D warnings` with the Clippy configuration fixed by `ENG-058` through `ENG-062`.
- **ENG-050.** Tier 1 MUST run `cargo deny check` for license acceptance, banned crates, and version bans, consistent with the `cargo-deny` configuration whose content is tracked as an open question in section 3.
- **ENG-051.** Tier 1 MUST run `cargo audit` against the current Common Vulnerabilities and Exposures database, consistent with the audited-dependency gate mandated by `THREAT-011` in [`007-threat-model.md`](007-threat-model.md).
- **ENG-052.** Tier 1 MUST run `cargo vet check`, consistent with the audited-dependency gate mandated by `THREAT-011` in [`007-threat-model.md`](007-threat-model.md).
- **ENG-053.** Tier 1 MUST run `cargo doc --no-deps` to verify that the documentation builds successfully and to detect broken intra-doc links.

#### Tier 2 — per-pull-request slow gates

- **ENG-054.** Tier 2 MUST run the `proptest`-based property tests mandated by `ENG-030`, executed with bounded case counts appropriate for the per-pull-request budget.
- **ENG-055.** Tier 2 MUST run `cargo-fuzz` smoke harnesses, that is, short targeted fuzzing runs whose purpose is to catch trivial regressions against the parsing surfaces enumerated in `ENG-031`.
- **ENG-056.** Tier 2 MUST run the `loom` tests mandated by `ENG-034` on crates marked as concurrency-sensitive.
- **ENG-057.** Tier 2 MUST run the benchmark-regression detection mandated by `PERF-014` through `PERF-018` in [`008-performance-targets.md`](008-performance-targets.md), against the stored baselines. A measured regression that exceeds the configured threshold MUST fail the build.

#### Clippy configuration

- **ENG-058.** Clippy MUST be invoked with `-D warnings` as its baseline posture. Any Clippy warning MUST fail the build when it fires.
- **ENG-059.** The `clippy::pedantic` lint group MUST be enabled. An explicit allowlist MAY be maintained for individual lints within `clippy::pedantic` that produce false positives in this codebase; each allow entry MUST carry an inline comment explaining why the lint is tolerated at that site. The exact set of allowed lints is tracked as an open question in section 3.
- **ENG-060.** The `clippy::undocumented_unsafe_blocks` lint MUST be configured at the `deny` level, consistent with `ENG-020`.
- **ENG-061.** The `clippy::unwrap_used` and `clippy::expect_used` lints MUST be configured at the `warn` level in library code paths and in production-binary code paths. They MAY be allowed in test code and in `main` binary entry points.
- **ENG-062.** The `clippy::panic_in_result_fn` lint MUST be configured at the `deny` level on non-test code paths.

#### rustfmt configuration

- **ENG-063.** A single canonical `rustfmt.toml` MUST be checked into the Heimdall repository. That canonical configuration MUST be the one enforced by `ENG-048` in CI.
- **ENG-064.** The canonical `rustfmt.toml` MUST pin the Rust edition to the latest stable edition in effect at the time of each release. The concrete values of individual `rustfmt` parameters (for example, `max_width`, `reorder_imports`, and `trailing_comma`) are operational parameters and are tracked as an open question in section 3.

#### Tier 3 — nightly and periodic gates

- **ENG-065.** Tier 3 MUST run `cargo test` under `miri`, across the full supported matrix, consistent with the `miri` runnability obligation fixed by `ENG-033`.
- **ENG-066.** Tier 3 MUST run extended `cargo-fuzz` sessions (for example, overnight, multi-target runs) beyond the smoke runs mandated by `ENG-055`.
- **ENG-067.** Tier 3 MUST integrate continuous fuzzing via OSS-Fuzz, consistent with `ENG-032`.
- **ENG-068.** Tier 3 MUST run the `proptest`-based property tests mandated by `ENG-030` with large case budgets, beyond the bounded counts run in Tier 2 under `ENG-054`.
- **ENG-069.** Tier 3 MUST run the periodic external comparison benchmarks against the reference implementations identified in `PERF-020` through `PERF-028` in [`008-performance-targets.md`](008-performance-targets.md).
- **ENG-070.** Tier 3 MUST run reproducible-build verification, consistent with `THREAT-012` in [`007-threat-model.md`](007-threat-model.md).
- **ENG-071.** Tier 3 MUST regenerate the Software Bill of Materials mandated by `THREAT-014` in [`007-threat-model.md`](007-threat-model.md) and MUST produce a delta report against the previous baseline.

#### Tier 4 — release gates

- **ENG-072.** Tier 4 MUST require that every check defined in Tiers 1, 2, and 3, in its most recent completed run, passes before a release is tagged.
- **ENG-073.** Tier 4 MUST perform signing of the official release artefacts in accordance with `THREAT-013` in [`007-threat-model.md`](007-threat-model.md) (sigstore / `cosign` or an equivalent cryptographic signing process).
- **ENG-074.** Tier 4 MUST publish the Software Bill of Materials alongside the signed release artefacts, consistent with `THREAT-014` in [`007-threat-model.md`](007-threat-model.md).
- **ENG-075.** The release notes produced at Tier 4 MUST state the Minimum Supported Rust Version in effect for that release, consistent with `ENG-005` and `ENG-006`.

### 1.6 Code review policy: two-reviewer baseline with CODEOWNERS

- **ENG-076.** Every pull request proposed for merge into the primary Heimdall repository MUST be reviewed and approved before it is eligible for merge. The approval regime applied to a given pull request MUST be determined by the content of the pull request, under `ENG-077` through `ENG-080`.
- **ENG-077.** A pull request that modifies any source code MUST be reviewed and approved by at least two maintainers before it is merged. This two-reviewer baseline is the default regime for any change that is not exclusively documentation or specification content.
- **ENG-078.** A pull request whose changes are confined to documentation, to specification files under `/specification`, or to any combination of the two, and that does not modify source code, MUST be reviewed and approved by at least one maintainer before it is merged.
- **ENG-079.** A pull request that modifies any source path declared as critical under `ENG-081`, or any security-sensitive infrastructure file enumerated in `ENG-082`, or the `CODEOWNERS` file itself per `ENG-083`, MUST be reviewed and approved by at least two code owners of the affected domain, and not merely by one code owner plus one other maintainer. The two-code-owner obligation supersedes the general two-reviewer baseline fixed by `ENG-077` on those paths.
- **ENG-080.** For every pull request that modifies source code and that is not covered by `ENG-079`, at least one of the two required approvals MUST come from a code owner of each domain touched by the pull request. A pull request that spans multiple domains MUST therefore carry at least one approval per domain from a code owner of that domain.
- **ENG-081.** The Heimdall repository MUST maintain a `CODEOWNERS` file that identifies the code owners of every domain in the project, and MUST use that file to declare the critical paths for which `ENG-079` applies. The critical-path declaration MUST include, at a minimum, every path that contains `unsafe` code, every path that implements cryptographic routines, every path that implements DNSSEC validation, and every parser path, including the DNS message parser, the zone-file parser, the EDNS option parser, the HPACK decoder, the QPACK decoder, the QUIC packet demultiplexer, and the TLS record interaction surface mediated by the `rustls` configuration.
- **ENG-082.** The `CODEOWNERS` file MUST additionally declare as critical the following security-sensitive infrastructure files and MUST require two-code-owner approval under `ENG-079` for any change to them: the continuous-integration configuration, the release signing pipeline per `THREAT-013` in [`007-threat-model.md`](007-threat-model.md), the Software Bill of Materials pipeline per `THREAT-014` in [`007-threat-model.md`](007-threat-model.md), and the `cargo-deny`, `cargo-vet`, and `cargo-audit` configuration files.
- **ENG-083.** The `CODEOWNERS` file itself MUST be declared as a critical path. Any modification of the `CODEOWNERS` file MUST therefore be reviewed and approved under the two-code-owner regime fixed by `ENG-079`.
- **ENG-084.** The author of a pull request MUST NOT count as one of the reviewers required by `ENG-077`, `ENG-078`, or `ENG-079`, under any circumstance. The author's own approval, explicit or implicit, MUST NOT satisfy any part of the numerical reviewer requirement.
- **ENG-085.** A review comment that requests changes on a pull request MUST block merge until the comment is either resolved to the satisfaction of its author or explicitly converted by its author to a non-blocking suggestion. No reviewer other than the author of the comment MAY dismiss a requested-changes comment for the purpose of merge eligibility.
- **ENG-086.** An explicit approval on a pull request MUST come from an account recognised as a maintainer by the project's governance document. Approvals from accounts that are not maintainers are advisory only and MUST NOT satisfy the numerical reviewer requirements fixed by `ENG-077`, `ENG-078`, or `ENG-079`, regardless of the substance of the approving review.
- **ENG-087.** The `CODEOWNERS` file MUST be under version control in a repository location recognised by the hosting platform such that the platform enforces code-owner review on the paths declared in the file. On a GitHub-hosted repository, that location is `/.github/CODEOWNERS` or `/CODEOWNERS` at the repository root; the equivalent hosting-platform-recognised location MUST be used on any other platform.
- **ENG-088.** The list of maintainers of Heimdall and the governance for becoming a maintainer MUST be documented in a governance document under version control in the repository. The exact location and content of that governance document are tracked as open questions in section 3.
- **ENG-089.** The code-review policy fixed by `ENG-076` through `ENG-088` applies to all pull requests proposed for merge into the primary Heimdall repository. Personal branches and forks are exempt from the policy until they are proposed for merge; once proposed for merge, they are subject to the policy in full.

### 1.8 Cargo workspace layout: moderate multi-crate (four to six crates)

- **ENG-098.** Heimdall MUST be organised as a single Cargo workspace containing between four and six top-level crates, structured along natural architectural seams. The workspace MUST NOT collapse into a single crate and MUST NOT fragment into ten or more crates at the initial design stage, consistent with `ENG-105` and `ENG-106`.
- **ENG-099.** The workspace MUST contain the library crate `heimdall-core`. The scope of `heimdall-core` MUST include, and MUST be limited to, the following areas: wire-format encoding and decoding of DNS messages; the EDNS(0) extension framework and the EDNS(0) extensions required by [`006-protocol-conformance.md`](006-protocol-conformance.md), namely DNS Cookies, Padding, Extended DNS Errors, NSID, and edns-tcp-keepalive; DNSSEC primitives consistent with [`005-dnssec-policy.md`](005-dnssec-policy.md), comprising validation, signature verification, and `NSEC` / `NSEC3` handling; core error types; and project-wide domain types shared across the rest of the workspace.
- **ENG-100.** The workspace MUST contain the library crate `heimdall-runtime`. The scope of `heimdall-runtime` MUST include, and MUST be limited to, the following areas: transport implementations for UDP, TCP, DoT, DoH over HTTP/2 and HTTP/3, and DoQ, consistent with [`002-transports.md`](002-transports.md); the segregated caches for the recursive resolver and forwarder roles, consistent with [`004-cache-policy.md`](004-cache-policy.md); access-control-list (ACL) evaluation and matching, consistent with the ACL matrix fixed in [`007-threat-model.md`](007-threat-model.md); admission control; rate limiting, comprising Response Rate Limiting and per-client query rate limiting, consistent with [`007-threat-model.md`](007-threat-model.md); and observability hooks in the form of structured-event emission and metrics counters, consistent with [`007-threat-model.md`](007-threat-model.md).
- **ENG-101.** The workspace MUST contain the library crate `heimdall-roles`. The scope of `heimdall-roles` MUST include, and MUST be limited to, the following areas: authoritative zone management; the recursive iterative resolver; the forwarder; and the query-resolution precedence logic fixed by `ROLE-008` through `ROLE-012` in [`001-server-roles.md`](001-server-roles.md).
- **ENG-102.** The workspace MUST contain the binary crate `heimdall`. The binary crate MUST be thin and MUST be limited to the following responsibilities: configuration loading, parsing, and validation; the `main` entry point; supervisor logic; the dependency wiring of `heimdall-core`, `heimdall-runtime`, and `heimdall-roles`; and the command-line interface. Business logic MUST NOT live in the binary crate; it MUST live in one of the three library crates identified by `ENG-099` through `ENG-101`.
- **ENG-103.** The library crate `heimdall-config` MAY be introduced as an additional top-level workspace crate, but only once the configuration parser and validator have grown beyond what the binary crate identified by `ENG-102` can cleanly host. Until that complexity threshold is reached, the configuration parser and validator MUST remain within the binary crate.
- **ENG-104.** The library crate `heimdall-observability` MAY be introduced as an additional top-level workspace crate, but only once the surface for structured-event emission and metrics justifies its own crate boundary and its own public API surface, distinct from the observability hooks hosted by `heimdall-runtime` under `ENG-100`. Until that complexity threshold is reached, observability responsibilities MUST remain within `heimdall-runtime`.
- **ENG-105.** The workspace MUST NOT be split into ten or more top-level crates at the initial design stage. A fine-grained split of that kind crystallises module boundaries before the design has been validated and makes subsequent refactoring structurally expensive. The four-to-six-crate envelope fixed by `ENG-098` is the maximum permitted granularity for the initial workspace layout.
- **ENG-106.** The workspace MUST NOT collapse into a single crate. A single-crate layout loses the build-time enforcement of inter-module boundaries that the multi-crate layout provides, and MUST NOT be adopted.
- **ENG-107.** Every library crate in the workspace MUST carry the crate-name prefix `heimdall-`. The binary crate identified by `ENG-102` is the sole exception and MUST be named `heimdall` without the prefix.
- **ENG-108.** The workspace MUST share a single workspace-level `Cargo.lock` file, located at the root of the workspace. Per-crate lockfiles MUST NOT be committed to the repository.
- **ENG-109.** Every crate in the workspace MUST share the same Minimum Supported Rust Version, fixed by `ENG-001` through `ENG-007`. A per-crate MSRV divergence MUST NOT be introduced.
- **ENG-110.** Every crate in the workspace MUST be licensed under the MIT License, consistent with the project-wide licensing policy fixed by `ENG-090` through `ENG-097`. A per-crate licence divergence MUST NOT be introduced.
- **ENG-111.** The workspace-layout policy fixed by `ENG-098` through `ENG-110` applies to the primary Heimdall repository only. Out-of-tree crates that consume Heimdall's libraries through published artefacts are out of scope for this policy.

### 1.9 Architecture Decision Records: MADR format, location, numbering, and lifecycle

- **ENG-112.** Heimdall MUST use Architecture Decision Records (ADRs) as the project's canonical vehicle for recording architectural and engineering rationale. The ADR set and the specification set are complementary and MUST NOT duplicate each other: the specification records normative requirements (what the implementation MUST do), while ADRs record the rationale, the alternatives considered, and the consequences (why a decision was made).
- **ENG-113.** Every ADR MUST follow the MADR (Markdown Any Decision Records) format, version 3.0.0 or later. An ADR that uses a different template MUST NOT be merged.
- **ENG-114.** Every ADR MUST contain, at minimum, all of the following sections, and MUST NOT omit any of them: Title; Context and Problem Statement; Decision Drivers; Considered Options; Decision Outcome; Consequences. A Validation section MAY be included when the decision can be verified through testing, benchmarking, or another concrete mechanism.
- **ENG-115.** Every ADR MUST carry a YAML frontmatter block at the top of the file. The frontmatter MUST include, at minimum, the fields `status`, `date`, and `deciders`. The frontmatter MAY additionally include the fields `consulted` and `informed`.
- **ENG-116.** ADR files MUST live under the directory `docs/adr/` at the repository root. That directory is separate from `/specification/` and MUST NOT be collapsed into it; specification and ADR content are governed by distinct processes and serve distinct purposes.
- **ENG-117.** Each ADR file MUST be named `NNNN-kebab-case-slug.md`, where `NNNN` is a zero-padded four-digit sequence number and the slug is a short, descriptive, lowercase, hyphen-separated identifier for the decision.
- **ENG-118.** ADR sequence numbers MUST be assigned monotonically, starting from `0001`. A number that has been assigned to an ADR MUST NEVER be reused, even if the ADR to which it was assigned is later superseded, deprecated, or removed from the index of current decisions.
- **ENG-119.** The ADR at `docs/adr/0001-record-adrs.md` MUST be the meta-ADR that documents this ADR process. The meta-ADR MUST record the MADR template, the directory location fixed by `ENG-116`, the naming convention fixed by `ENG-117`, the numbering rule fixed by `ENG-118`, the lifecycle states fixed by `ENG-120`, the append-only semantics fixed by `ENG-121`, and the retention rule fixed by `ENG-122`.
- **ENG-120.** The `status` frontmatter field of every ADR MUST take exactly one of the following values: `proposed` (the ADR is a draft under review); `accepted` (the ADR has been agreed and merged); `deprecated` (the ADR is no longer applicable and has no replacement); or `superseded-by ADR-NNNN` (the ADR has been replaced by a specific later ADR, whose number `NNNN` is cited explicitly). No other value MAY appear in the `status` field.
- **ENG-121.** Status transitions on an ADR MUST be append-only in semantics. An ADR that has reached `accepted` status MUST NOT be silently edited to change its decision. A change of decision MUST be expressed by writing a new ADR and updating the superseded ADR's `status` field to `superseded-by ADR-NNNN`, where `NNNN` is the number of the new ADR.
- **ENG-122.** Deprecated and superseded ADRs MUST be retained in the repository as historical record. Deletion of an ADR, whether deprecated, superseded, or otherwise, MUST NOT occur.
- **ENG-123.** The decisions already implicit in prior specification requirements MUST receive retrospective ADRs in an initial grandfather batch, so that the project has no undocumented load-bearing decisions. The grandfather batch MUST cover, at minimum, the choice of Tokio as the asynchronous runtime, the choice of `rustls` for TLS, the choice of `quinn` for QUIC, the choice of `hyper` for HTTP, the choice of `rustix` for system calls, the Cargo workspace layout fixed by `ENG-098` through `ENG-111`, the MIT licensing fixed by `ENG-090` through `ENG-097`, and the Minimum Supported Rust Version policy fixed by `ENG-001` through `ENG-007`. The grandfather batch is expected to occupy the low sequence numbers starting at `0002`, after the meta-ADR at `0001`.
- **ENG-124.** An ADR MAY reference specification requirement identifiers of the form `<CATEGORY>-NNN` by relative link, and a specification requirement MAY reference an ADR by relative link. Cross-references in both directions MUST use relative Markdown links; hard-coded absolute URLs MUST NOT be used.
- **ENG-125.** The specification is the sole source of truth for normative content. The ADR set is the sole source of truth for rationale. Neither form MAY duplicate the other: a normative statement MUST NOT be recorded only in an ADR, and the rationale of a decision MUST NOT be recorded only as inline prose within a specification file.
- **ENG-126.** The ADR process fixed by `ENG-112` through `ENG-125` applies to all architectural and engineering decisions across the project, and in particular to the dependency-acceptance gate fixed by `ENG-008` through `ENG-016` and to the new-pattern `unsafe` gate fixed by `ENG-021`.
- **ENG-127.** The exact ADR tooling used to author, index, filter, and render the ADR set — for example, `adr-tools` as a command-line authoring aid, `log4brains` as an indexing and viewing tool, or manual authoring with no dedicated tooling — is an operational detail tracked as an open question in section 3. The specification requires only that the format and lifecycle rules fixed by `ENG-112` through `ENG-126` be enforceable by whichever toolchain the project chooses.

### 1.10 Issue tracking: GitHub Issues, with GitHub Security Advisories for security issues

- **ENG-128.** Heimdall MUST use GitHub Issues on the primary project repository as its canonical issue tracker. No external issue tracker (for example, Linear, JIRA, YouTrack) MAY be adopted as an alternative or parallel canonical tracker. The invariant MUST apply uniformly to every repository under the Heimdall project on GitHub.
- **ENG-129.** The following categories of work MUST be tracked as GitHub Issues on the primary repository: bug reports; feature requests; documentation issues; roadmap items; triage activity; and discussion of individual specification or ADR proposals. No category in this enumeration MAY be tracked elsewhere as its canonical record.
- **ENG-130.** GitHub Issues on the Heimdall repository MUST be public by default. Private issues MUST NOT be created through the standard GitHub Issues flow. Confidential security reports MUST instead use the GitHub Security Advisories (GHSA) private reporting flow fixed by `ENG-131` through `ENG-135`.
- **ENG-131.** Security-sensitive reports MUST be routed through the GHSA private reporting flow. The primary Heimdall repository MUST have GitHub's private vulnerability reporting feature enabled, so that external reporters can submit confidential reports without opening a public issue.
- **ENG-132.** A `SECURITY.md` file MUST exist at the root of the primary Heimdall repository. That file MUST document the security reporting path required by `ENG-131`, including the GHSA private-reporting entry point, the expected first-response timing, and the coordinated-disclosure policy fixed by `ENG-133`. The concrete content outline of `SECURITY.md` is tracked as an open question in section 3.
- **ENG-133.** The Heimdall project MUST operate a documented coordinated-disclosure policy. The policy MUST define the embargo window between report acceptance and public disclosure, the communication expectations with the reporter during the embargo, and the conditions under which the embargo may be shortened or extended. The concrete default value of the embargo window, and the flexibility around it, are tracked as open questions in section 3.
- **ENG-134.** The Heimdall project MUST arrange CVE assignment for every confirmed security vulnerability that is fixed under the coordinated-disclosure policy fixed by `ENG-133`. CVE assignment MUST proceed through GHSA's CVE Numbering Authority integration where applicable, or directly through MITRE where the GHSA path is not applicable to the vulnerability in question.
- **ENG-135.** Once a security vulnerability has been fixed and disclosed under `ENG-133`, the project MUST publish a GHSA advisory on the primary repository. The published advisory MUST be linked to the CVE identifier assigned under `ENG-134`, so that the public record of the vulnerability is complete and traceable end-to-end.
- **ENG-136.** Every pull request that addresses one or more GitHub Issues MUST reference those issues using the standard GitHub linkage syntax. The accepted verbs are `Fixes #NNN`, `Closes #NNN`, and `Refs #NNN`. `Fixes` and `Closes` MUST be used when the pull request fully addresses the referenced issue; `Refs` MUST be used when the pull request only partially addresses or otherwise refers to the referenced issue without resolving it.
- **ENG-137.** A pull request whose body references an issue with `Fixes #NNN` or `Closes #NNN` MUST auto-close that issue on merge. Pull-request authors MUST therefore use `Fixes` or `Closes` only for issues the pull request actually resolves in full, and MUST use `Refs` for any issue that remains open after the pull request is merged.
- **ENG-138.** The issue-tracking and coordinated-disclosure policy fixed by `ENG-128` through `ENG-137` applies to all Heimdall repositories under the project on GitHub and to all project communications about issues. Informal channels (for example, chat, email, forums) MAY be used for discussion, but MUST NOT substitute for the canonical record held by GitHub Issues and, for security matters, by GHSA.

### 1.11 Commit conventions: Conventional Commits v1.0.0

- **ENG-139.** Every commit landing on the primary branch of the Heimdall repository, and every commit landing on any release maintenance branch, MUST conform to the Conventional Commits specification, version 1.0.0 or later. A commit that does not conform MUST NOT be merged onto any of those branches.
- **ENG-140.** The subject line of every commit subject to `ENG-139` MUST take the form `type(scope): description`. The `type` component MUST be present; the `(scope)` component is OPTIONAL and, when present, MUST be enclosed in parentheses immediately after the type; the `: ` separator between the type-or-scope and the description MUST be a single colon followed by a single space; the `description` component MUST be a concise imperative summary of the substantive change.
- **ENG-141.** The `type` component of the subject line fixed by `ENG-140` MUST be exactly one of the following values: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`. No other value MAY appear in the `type` component.
- **ENG-142.** The `scope` component of the subject line fixed by `ENG-140`, when present, identifies the workspace crate or top-level project area to which the commit applies. The recommended scope vocabulary at this stage is `core`, `runtime`, `roles`, `binary`, `config`, `observability`, `ci`, `docs`, `adr`, `spec`, `deps`, aligned with the Cargo workspace layout fixed by `ENG-098` through `ENG-111` and with the top-level project areas referenced elsewhere in this file. The project MAY extend this vocabulary as new workspace crates or top-level areas appear. Whether the vocabulary is enforced as a closed enumeration in tooling or retained as free-string guidance is tracked as an open question in section 3.
- **ENG-143.** A commit that introduces a breaking change MUST signal that fact either by placing a single `!` immediately after the type-or-scope and before the `:` separator (for example, `feat!: ...` or `feat(core)!: ...`), or by including a `BREAKING CHANGE:` footer, or by both. The `BREAKING CHANGE:` footer, when present, MUST appear in its own paragraph, separated from the preceding content by a blank line.
- **ENG-144.** A commit that reverts a prior commit MUST use the type `revert` with a subject line of the form `revert: <original subject>`, where `<original subject>` is the subject line of the commit being reverted. The body of a `revert` commit MUST identify the reverted commit unambiguously, for example by its commit hash.
- **ENG-145.** A commit that carries a body in addition to its subject line MUST separate the subject line from the body by a single blank line. The body SHOULD wrap at 72 characters per line. Footers, when present, MUST appear after the body separated by a blank line.
- **ENG-146.** A commit that references one or more GitHub Issues MUST use the standard GitHub linkage syntax fixed by `ENG-136`, namely `Fixes #NNN`, `Closes #NNN`, or `Refs #NNN`, with the distinction between `Fixes`/`Closes` and `Refs` governed by `ENG-136` and `ENG-137`. Issue references SHOULD appear as footers of the commit message.
- **ENG-147.** Continuous integration MUST enforce `ENG-139` through `ENG-146` by running a Conventional-Commits validator at Tier 1 of the continuous-integration pipeline fixed by `ENG-041` through `ENG-053`. The validator MUST check every commit introduced by the pull request under review. A non-conforming commit MUST fail the Tier 1 gate, and the pull request MUST NOT be eligible for merge until every commit under review conforms. The specific validator tool (for example, `commitlint`, `committed`, `cocogitto`, or another) is tracked as an open question in section 3.
- **ENG-148.** A pre-commit hook that validates Conventional Commits locally SHOULD be offered to contributors as an opt-in convenience. The local hook MUST NOT be the authoritative enforcement point; the authoritative enforcement point is the continuous-integration gate fixed by `ENG-147`. The distribution mechanism for the pre-commit hook (for example, a script checked into `.git/hooks`, integration with the `pre-commit` framework, or a project-native command-line interface) is tracked as an open question in section 3.
- **ENG-149.** The default merge strategy for pull requests into the primary Heimdall repository MUST be squash-merge. Under squash-merge, each pull request becomes exactly one commit on the target branch at merge time, and that single squashed commit MUST itself conform to the Conventional Commits format fixed by `ENG-139` through `ENG-146`. The subject line of the squashed commit MUST reflect the substantive change introduced by the pull request.
- **ENG-150.** Merge-commit and rebase-then-merge strategies MUST NOT be the default merge strategy. They MAY be available in the repository configuration for narrow exceptions (for example, back-merges from a release maintenance branch into the primary branch), and every such exception MUST be documented in the project's contribution guide. An exception invoked without prior documentation in the contribution guide MUST NOT be applied.
- **ENG-151.** A `CHANGELOG.md` file MUST be maintained at the root of the primary Heimdall repository. The changelog SHOULD be generated from the commit history by a Conventional-Commits-aware tool (for example, `git-cliff`). Any manual curation applied to the generated changelog MUST be recorded in the pull request that updates the changelog, so that the manual edits are reviewable alongside the automated output.
- **ENG-152.** The commit-conventions policy fixed by `ENG-139` through `ENG-151` applies uniformly to the primary branch of the Heimdall repository and to every release maintenance branch. Personal branches and forks are exempt from the policy until they are proposed for merge into a branch covered by this policy; once so proposed, the commits that would land on the target branch after squash-merge are subject to the policy in full, and the continuous-integration gate fixed by `ENG-147` applies.

### 1.12 Branching strategy: GitHub Flow with protected `main`

- **ENG-153.** Heimdall MUST adopt GitHub Flow as its branching strategy. Under this model, a single long-lived branch named `main` serves as the project's trunk, all feature and fix work is carried out on short-lived branches that originate from `main` and merge back into `main` through pull requests, and releases are produced by tagging commits on `main`. Long-lived `develop`, `release`, or `hotfix` branches MUST NOT exist at this stage.
- **ENG-154.** The branch `main` MUST be the only long-lived branch of the primary Heimdall repository at this stage. Every other branch in the primary repository MUST be short-lived, MUST originate from `main`, and MUST merge back into `main` through a pull request subject to the code-review policy fixed by `ENG-076` through `ENG-089` and the merge-strategy policy fixed by `ENG-149` and `ENG-150`.
- **ENG-155.** The branch `main` MUST be configured as a protected branch on the repository host, with the enforcement mechanisms fixed by `ENG-156` through `ENG-160`. The protection MUST be enforced by the hosting platform rather than by contributor convention alone, so that the guarantees hold uniformly against every contributor, including maintainers.
- **ENG-156.** Protected-branch configuration on `main` MUST require, as blocking status checks, every Tier 1 check from `ENG-046` through `ENG-053` and every Tier 2 check from `ENG-054` through `ENG-057` of the continuous-integration pipeline fixed by `ENG-041` through `ENG-075`. No pull request MAY be merged into `main` until every such check has passed on the pull request's head commit. The concrete list of required status checks keyed to the Tier 1 and Tier 2 enumerations above is tracked as an open question in section 3.
- **ENG-157.** Protected-branch configuration on `main` MUST require, as blocking review gates, the reviewer counts fixed by the code-review policy in `ENG-076` through `ENG-089`. In particular, every pull request that modifies source code MUST carry at least two maintainer approvals under `ENG-077`, at least one code-owner approval per domain touched under `ENG-080`, and two code-owner approvals on critical paths under `ENG-079`. Pull requests confined to documentation or specification content MUST carry at least one maintainer approval under `ENG-078`. No pull request MAY be merged into `main` until the applicable review gate has been satisfied.
- **ENG-158.** Direct push to `main` MUST be disabled for all contributors, including maintainers and repository administrators. Every change that lands on `main` MUST enter through a reviewed pull request subject to `ENG-156` and `ENG-157`. An exception path that would bypass the pull-request flow for `main` MUST NOT be configured at this stage.
- **ENG-159.** Force-push to `main` MUST be disabled at the hosting-platform level. The history of `main` MUST be append-only from the perspective of the repository host, so that rewrites of the primary branch are structurally impossible rather than merely discouraged.
- **ENG-160.** Linear history MUST be enforced on `main`, consistent with the squash-merge default fixed by `ENG-149` and the prohibition of merge-commit and rebase-then-merge as defaults fixed by `ENG-150`. The protected-branch configuration MUST reject any merge that would introduce a non-linear history on `main`, other than the narrow documented exceptions admitted under `ENG-150`.
- **ENG-161.** Work-branch naming in the primary Heimdall repository SHOULD follow a prefix convention aligned with the type of change. The recommended prefixes are `feature/<short-slug>` for feature work, `fix/<issue-id-or-slug>` for bug fixes, `docs/<area>` for documentation-only changes, `chore/<task>` for maintenance work, and `refactor/<area>` for refactoring work. The convention is a recommendation rather than a strict enforcement, and external contributors working in forks MAY use any naming they prefer.
- **ENG-162.** Short-lived work branches in the primary Heimdall repository SHOULD be deleted automatically after the pull request that introduced them is merged. The repository host's default post-merge branch cleanup (for example, GitHub's "Automatically delete head branches" setting) is an acceptable implementation of this requirement.
- **ENG-163.** Dedicated long-lived release-maintenance branches (for example, `release-1.x`) MUST NOT be required at this stage. Pre-1.0.0, Heimdall releases are tags on `main` and no parallel maintenance branch is maintained. Post-1.0.0, a long-term-support (LTS) maintenance-branch model MAY be introduced under a separate decision; the current decision does NOT preclude that future introduction but does NOT require it. The concrete LTS policy, its activation threshold, and the backport process are tracked as open questions in section 3.
- **ENG-164.** Hotfixes and security fixes MUST land on `main` through the normal pull-request flow, prioritised and merged rapidly under the emergency-fix policy whose concrete content is tracked as an open question under `ENG-076` through `ENG-089`. A patch release MUST then be produced by tagging the resulting commit on `main`, consistent with `ENG-153`. Once LTS maintenance branches exist post-1.0.0 under the separate decision anticipated by `ENG-163`, security fixes MAY additionally be cherry-picked to those branches under a backport policy to be defined at that point; the backport mechanism itself is tracked as an open question in section 3.
- **ENG-165.** The concrete use of repository-host auto-merge (for example, GitHub's auto-merge on pull requests) and of a merge queue or bors-like serialisation mechanism to coordinate concurrent pull requests against the tiered continuous-integration pipeline fixed by `ENG-041` through `ENG-075` is an operational follow-up. Auto-merge and merge-queue configuration is tracked as open questions in section 3. The branching-strategy decision fixed by `ENG-153` through `ENG-164` neither mandates nor forbids either mechanism; both remain available and may be adopted under a separate decision.
- **ENG-166.** The branching-strategy policy fixed by `ENG-153` through `ENG-165` applies to the primary Heimdall repository's `main` branch and to every work branch that is proposed for merge into `main`. Personal branches and forks are exempt from the branch-naming recommendation under `ENG-161` and from the post-merge cleanup recommendation under `ENG-162` until they are proposed for merge; once so proposed, every protected-branch requirement that applies to `main` applies to the merge operation in full.

### 1.13 Release process and versioning: SemVer 2.0.0 with Tier-4 gated automation

- **ENG-167.** Heimdall MUST use Semantic Versioning, version 2.0.0, as its sole versioning scheme. Every version string produced by the project MUST take the form `MAJOR.MINOR.PATCH`, optionally followed by a pre-release identifier of the form `-alpha.N`, `-beta.N`, or `-rc.N` as permitted by section 9 of the Semantic Versioning 2.0.0 specification, where `N` is a non-negative integer. Build metadata in the form `+<identifier>` MAY be appended only when strictly necessary to distinguish otherwise-identical releases, and MUST NOT alter the precedence of the version for any dependency-resolution purpose.
- **ENG-168.** Pre-1.0.0 version strings of the form `0.x.y` MUST be treated as "API unstable" under standard Semantic Versioning practice for `0.x` series. Under this posture, `MINOR` bumps on a `0.x.y` version MAY include backwards-incompatible changes to any user-facing surface of Heimdall, without a `MAJOR` bump. Contributors and operators MUST NOT rely on the pre-1.0.0 surface for stability guarantees.
- **ENG-169.** Post-1.0.0, a `MAJOR` version bump MUST be reserved for backwards-incompatible changes to any user-facing surface of Heimdall. A `MAJOR` bump MUST be preceded by at least one `MINOR` release on the current `MAJOR` line that emits deprecation warnings for every surface affected by the upcoming breaking change. A `MAJOR` bump that is not preceded by such a deprecation-carrying `MINOR` release MUST NOT be shipped.
- **ENG-170.** Post-1.0.0, a `MINOR` version bump MUST be reserved for backwards-compatible feature additions to the user-facing surface of Heimdall. A `MINOR` bump MUST NOT introduce a breaking change to any user-facing surface.
- **ENG-171.** Post-1.0.0, a `PATCH` version bump MUST be reserved for bug fixes that are backwards-compatible on the user-facing surface of Heimdall. A `PATCH` bump MUST NOT introduce a breaking change and MUST NOT add a new feature.
- **ENG-172.** Security-release patch versions MAY be issued out-of-band at any time, without waiting for the next scheduled release cycle defined by the cadence fixed by `ENG-173` and `ENG-174`. A security-release patch MUST continue to conform to the version-form rules fixed by `ENG-167` and to the `PATCH`-bump rule fixed by `ENG-171` post-1.0.0.
- **ENG-173.** Releases MUST follow a "release-when-ready" policy. A release MUST be tagged when the content of `main` is judged ready for release under the Tier 4 gates fixed by `ENG-176`, and MUST NOT be tagged on a calendar deadline that overrides those gates.
- **ENG-174.** Post-1.0.0, `MINOR` releases SHOULD target a cadence of approximately three to six months as an orientation, without being a strict deadline. The concrete point within that range, and any adjustment to it, are tracked as an open question in section 3. Pre-1.0.0, no cadence is imposed, consistent with the release-when-ready posture fixed by `ENG-173`.
- **ENG-175.** Release tags MUST take the form `vMAJOR.MINOR.PATCH` for stable releases and `vMAJOR.MINOR.PATCH-prerelease` for pre-releases, where `prerelease` is a pre-release identifier permitted by `ENG-167`. The tag MUST match the version string of the released artefacts exactly, modulo the leading `v` prefix.
- **ENG-176.** Every Heimdall release MUST pass every Tier 4 release gate fixed by `ENG-072` through `ENG-075` before the release tag is produced. This obligation requires, at minimum and without exception: every Tier 1, Tier 2, and Tier 3 check in its most recent completed run is green; the release artefacts are signed in accordance with `THREAT-013` in [`007-threat-model.md`](007-threat-model.md); the Software Bill of Materials is published alongside the signed artefacts in accordance with `THREAT-014` in [`007-threat-model.md`](007-threat-model.md); and the release notes state the Minimum Supported Rust Version in effect for the release, consistent with `ENG-005` and `ENG-006`.
- **ENG-177.** The `CHANGELOG.md` file mandated by `ENG-151` MUST be updated at every release, such that every released version has a corresponding changelog entry. The changelog entry MAY be generated automatically from the Conventional Commits history mandated by `ENG-139` through `ENG-146` and assembled by a Conventional-Commits-aware tool, consistent with the generation path sanctioned by `ENG-151`. Any manual curation applied to the generated changelog MUST be recorded in the pull request that updates the changelog, consistent with `ENG-151`.
- **ENG-178.** Release execution SHOULD be automated end-to-end, from the production of the release tag through the publication of the signed artefacts and the Software Bill of Materials required by `ENG-176`. The concrete release-automation tooling (for example, `cargo-release`, `release-plz`, `git-cliff`, or an equivalent) is an operational follow-up and is tracked as an open question in section 3. The absence of a fixed tooling choice MUST NOT be interpreted as permission to bypass the Tier 4 gates mandated by `ENG-176`.
- **ENG-179.** A security-release patch MUST coordinate with the coordinated-disclosure and GHSA advisory timeline fixed by `ENG-131` through `ENG-135`. The published GHSA advisory MUST be made public when the fixed version is released, or immediately after, such that operators have the fix available at the moment the vulnerability is publicly described.
- **ENG-180.** A security-release patch MUST reference a single CVE identifier, assigned under `ENG-134`, consistently across every artefact associated with the release. The release tag produced under `ENG-175`, the artefact signatures produced under `ENG-176`, the Software Bill of Materials published under `ENG-176`, and the GHSA advisory published under `ENG-179` MUST all reference the same CVE identifier.
- **ENG-181.** A dedicated long-term-support (LTS) maintenance model is NOT introduced by this decision. An LTS maintenance model MAY be adopted after v1.0.0 has shipped, under a separate decision; the concrete LTS policy, its activation threshold, and its backport process remain tracked as open questions in section 3, consistent with `ENG-163` and with the open question on the LTS maintenance-branch policy already recorded in section 3.
- **ENG-182.** The authoritative distribution channel for Heimdall releases MUST be GitHub Releases on the primary Heimdall repository, hosting the signed artefacts mandated by `ENV-023` through `ENV-033` in [`009-target-environment.md`](009-target-environment.md) and by `THREAT-013` in [`007-threat-model.md`](007-threat-model.md). Alternative channels MAY coexist under `ENG-183` and `ENG-184`, but MUST NOT be treated as authoritative in place of GitHub Releases.
- **ENG-183.** If any library crate of the Heimdall workspace is published to `crates.io`, the publication MUST pass the Tier 4 release gates fixed by `ENG-176` before the crate version is pushed to the registry. A crate version pushed to `crates.io` outside the Tier 4 gate MUST NOT be produced.
- **ENG-184.** If container images of Heimdall are published to a container registry (for example, GitHub Container Registry, Docker Hub, or any other OCI-compliant registry), each image tag MUST match the corresponding Semantic Versioning release under `ENG-167` and `ENG-175`, the image MUST be signed in accordance with `THREAT-013` in [`007-threat-model.md`](007-threat-model.md), and the image MUST reference the same Software Bill of Materials published for the corresponding release under `THREAT-014` in [`007-threat-model.md`](007-threat-model.md). Container-image tagging conventions beyond the SemVer match (for example, `latest`, `stable`, `MAJOR`, `MAJOR.MINOR` aliases) are an operational follow-up and are tracked as an open question under `009-target-environment.md`.
- **ENG-185.** The release process and versioning policy fixed by `ENG-167` through `ENG-184` applies uniformly to every Heimdall release of every artefact format defined by the distribution model fixed in [`009-target-environment.md`](009-target-environment.md). No artefact format is exempt from the Tier 4 gate, the SemVer form, the tag convention, the signing obligation, or the Software Bill of Materials obligation.

### 1.14 MSRV tooling and bump cadence

- **ENG-186.** The MSRV under `ENG-001` and `ENG-002` MUST be verified on every pull request using `cargo msrv verify` (the `cargo-msrv` crate). The Tier 1 gate under `ENG-046` MUST include `cargo msrv verify` to confirm that the declared `rust-version` in every `Cargo.toml` matches the build. MSRV bumps MUST be proposed as automated pull requests opened by Renovate (or an equivalent automated dependency tool) on every new stable Rust release, and MUST NOT be applied manually without a corresponding pull request that satisfies the code-review requirements of `ENG-076` through `ENG-089`.
- **ENG-187.** The MSRV MUST advance on every new stable `rustc` release. The automated pull request created by `ENG-186` MUST update both the `rust-version` field in every `Cargo.toml` in the workspace and the changelog or release-notes entry mandated by `ENG-005`. The Tier 1 gate MUST enforce consistency between the `rust-version` field and the `rustc` version used in the build matrix; a mismatch MUST fail the build.
- **ENG-188.** Dependency-update pull requests opened by automated bots (Renovate or Dependabot) that would themselves raise the effective MSRV of a Heimdall dependency MUST be gated on explicit human review under `ENG-077`. An automated bot MUST NOT advance the MSRV of a dependency without a maintainer-reviewed pull request that confirms the new MSRV is acceptable.

### 1.15 Dependency update cadence and vendoring policy

- **ENG-189.** Dependency updates MUST be driven by an automated dependency-update bot (Renovate or Dependabot), configured to open pull requests on a weekly cadence for patch-level and minor-level dependency bumps. Major-version dependency bumps MUST NOT be automated; they MUST require a new dependency ADR under `ENG-008` through `ENG-015` or an explicit update to the existing ADR for that dependency.
- **ENG-190.** A bot-proposed version bump of an existing dependency that does not add or remove a crate from the dependency graph MUST NOT require a new ADR. The ADR gate under `ENG-008` through `ENG-012` is satisfied by the existing ADR for that dependency. The bot-proposed pull request MUST nevertheless pass the full Tier 1 gate including `cargo vet check` under `ENG-052`; a `cargo vet` failure on the new version MUST block auto-merge until a `cargo vet add-exemption` or equivalent `cargo vet` update is committed.
- **ENG-191.** Core security-critical dependencies — specifically `rustls`, `quinn`, `tokio`, and `hyper` — MUST NOT be vendored into the Heimdall repository. These dependencies MUST be pinned via `Cargo.lock` and MUST be covered by `cargo vet` certifications for every version in use. The decision not to vendor MUST be re-evaluated after the first confirmed CVE incident in any of these crates; the re-evaluation MUST produce a written outcome recorded in the relevant dependency ADR.
- **ENG-192.** The `cargo-deny` configuration MUST contain, at a minimum: the licence whitelist and blocklist fixed by `ENG-094` and `ENG-095`; a multi-version limit of one active version per crate (with explicit enumerated exceptions justified per crate); a default advisory-handling policy of `deny` for all advisories not explicitly exempted; and an explicit source allowlist restricting dependencies to `crates.io` plus a named list of approved git-sourced dependencies. No unenumerated dependency source MAY pass the `cargo deny check` gate.

### 1.16 Unsafe-code tolerance for dependencies, new-pattern threshold, and SAFETY template

- **ENG-193.** The `unsafe` footprint of an external dependency (recorded in the dependency ADR under `ENG-009` item 4) is assessed informally using the `cargo-geiger` report produced under `ENG-196`. There is no hard numerical gate on the `unsafe` block count of a dependency pre-1.0.0. A dependency whose `cargo-geiger` report shows a materially elevated `unsafe` count compared to its alternatives MUST document the difference in the dependency ADR and MUST explain why the elevated count is acceptable given the security posture fixed by [`../CLAUDE.md`](../CLAUDE.md).
- **ENG-194.** A new "pattern" of `unsafe` usage for the purposes of `ENG-021` is defined as any occurrence of `unsafe` that exercises a primitive family not already covered by an existing accepted ADR. The primitive families are: (a) foreign-function interface (FFI); (b) raw-pointer arithmetic, including reads and writes through raw pointers; (c) aliasing, `transmute`, or union field access; (d) blanket `Send` or `Sync` implementations; (e) inline assembly (`asm!`); (f) memory-mapped I/O or external memory regions (`mmap` and equivalents); and (g) atomic orderings not expressible through the `std::sync::atomic` safe API. An additional `unsafe` block within the same primitive family as an existing approved ADR is NOT a new pattern and MUST NOT require a new ADR, but MUST carry the SAFETY documentation mandated by `ENG-019` and `ENG-195`.
- **ENG-195.** The SAFETY documentation mandated by `ENG-019` MUST be supplied entirely inline, immediately preceding the `unsafe` block or as a `# Safety` doc-comment section on the `unsafe fn`. No separate crate-root `SAFETY.md` or equivalent out-of-line file MAY substitute for the per-site inline comment. Each per-site SAFETY comment MUST be one to three sentences covering all three items required by `ENG-019`: the invariants the caller must uphold, why the safe alternative does not suffice, and the scope of the unsafety (data, lifetimes, aliasing, and thread-safety assumptions).

### 1.17 `cargo-geiger` reporting and annual unsafe review cadence

- **ENG-196.** `cargo-geiger` MUST run as part of Tier 3 (nightly) of the continuous-integration pipeline. Its output report MUST be attached to the Software Bill of Materials produced under `ENG-071` and `THREAT-014` in [`007-threat-model.md`](007-threat-model.md). On every pull request that introduces or removes a dependency, `cargo-geiger` MUST be run and its diff posted as an informational comment on the pull request. The `cargo-geiger` report MUST NOT constitute a hard CI gate; its purpose is informational and decision-support.
- **ENG-197.** The annual review of `unsafe` blocks and `unsafe fn`s mandated by `ENG-023` and `ENG-024` MUST be scheduled in January of each calendar year. The release manager MUST open one GitHub Issue per `unsafe` item under review, labelled `unsafe-review`. The outcome for each item MUST be recorded as a comment on the corresponding issue and MUST be exactly one of: `still-required` (with a brief explanation); `refactor-to-safe planned` (with a linked follow-up task); or `refactored` (with a link to the pull request that removed the item). Issues opened for the review MUST be closed before the next annual review cycle begins.

### 1.18 Coverage, mutation testing, proptest, golden tests, OSS-Fuzz, and integration harness

- **ENG-198.** The coverage tool for the Heimdall source MUST be `cargo-llvm-cov`, which supports stable `rustc` via `llvm-tools-preview`. The coverage target pre-1.0.0 is 80% line coverage across the workspace. This target is informational and MUST NOT constitute a merge-blocking CI gate. Coverage reports MUST be generated as part of Tier 3 (nightly) and MUST be published as CI artefacts.
- **ENG-199.** Mutation testing using `cargo-mutants` MUST run as part of Tier 3 (nightly) on a subset of crates identified by the maintainer group. The `cargo-mutants` run is informational; a surviving mutant MUST generate a tracked GitHub Issue labelled `test:mutation-survivor` and MUST NOT block merge.
- **ENG-200.** The `proptest` case budget is `PROPTEST_CASES=256` for Tier 2 and `PROPTEST_CASES=1_000_000` for Tier 3 (nightly). The per-failure shrinking time limit is 60 seconds. Failing seeds MUST be committed to the `proptest-regressions/` directory at the root of the relevant crate and MUST remain under version control, so that previously-found failing inputs are always re-run automatically.
- **ENG-201.** The golden-output comparison tests mandated by `ENG-035` MUST be refreshable by running `cargo test -- --ignored refresh-golden`. A pull request that changes any golden output MUST include a reviewer-approved diff of the changed golden files. Reference implementations used to generate golden outputs MUST be pinned by digest in a `tests/reference/docker-compose.yml` file under version control. A quarterly refresh of golden outputs MUST be performed; if the refresh produces a non-trivial diff, the diff MUST be reviewed and, if the change is intentional, recorded in a dedicated ADR.
- **ENG-202.** OSS-Fuzz onboarding MUST satisfy all of the following criteria before submission: (a) stable fuzz targets exist for the DNS message parser, zone-file parser, EDNS option parser, HPACK decoder, QPACK decoder, and QUIC packet demultiplexer; (b) each fuzz target has a seed corpus of at least 100 valid inputs; (c) a 24-hour local fuzzing run on a CI runner produces no memory leaks and no panics; (d) the Tier 3 `cargo-fuzz` extended sessions under `ENG-066` are green. The expected submission milestone is after Heimdall v0.8.0.
- **ENG-203.** The controlled integration harness mandated by `ENG-036` MUST be implemented as a dedicated Cargo workspace crate named `heimdall-integration-tests`. The harness MUST orchestrate simulated authoritative DNS servers using `heimdall-core` primitives in-process, simulated DNS clients using `dnsperf` and `queryperf`, a Redis container, and an observability stack (Prometheus and Loki) via Docker Compose. The `heimdall-integration-tests` crate MUST NOT be published to `crates.io`.
- **ENG-204.** Fuzz targets MUST be located in a `fuzz/` subdirectory within the crate that exposes the fuzzed parsing surface, structured as a `cargo fuzz init`-compatible layout. A single shared fuzz workspace spanning multiple crates MUST NOT be created; each crate carries its own fuzz directory.

### 1.19 CI tier budgets, frequencies, platform, and benchmark runner policy

- **ENG-205.** The wall-clock budget for Tier 1 is a target maximum of 10 minutes on GitHub-hosted runners. The wall-clock budget for Tier 2 is a target maximum of 30 minutes on GitHub-hosted or self-hosted runners. Exceeding either budget by more than 50% in three consecutive runs MUST trigger a maintainer-owned optimisation task.
- **ENG-206.** The Tier 3 (nightly) frequency and budget per layer is: `miri` runs nightly on all crates that contain `unsafe` code, with no explicit wall-clock ceiling; `loom` model-checking tests run nightly with a 4-hour wall-clock ceiling per crate; `cargo-fuzz` extended sessions run nightly with a 4-hour ceiling per fuzz target; `proptest` extended sessions run nightly with `PROPTEST_CASES=1_000_000`.
- **ENG-207.** The concrete CI platform for the tiered pipeline is GitHub Actions. The runner matrix is: x86_64 Linux, macOS, and BSD jobs MUST run on GitHub-hosted runners; `aarch64` Linux jobs MUST run on the self-hosted dedicated `aarch64` runner designated by `PERF-053` in [`008-performance-targets.md`](008-performance-targets.md); `riscv64` Linux jobs MUST run on the self-hosted dedicated `riscv64` runner designated by `PERF-053`. CPU emulation (QEMU or equivalent) MUST NOT substitute for the self-hosted runners in any gate that blocks merge.
- **ENG-208.** Tier 3 benchmark regression tests and external comparison benchmarks MUST run exclusively on the dedicated self-hosted runners designated by `PERF-053`. Shared GitHub-hosted runners MUST NOT be used for Tier 3 performance measurements. Tier 2 micro-benchmark smoke runs under `ENG-057` MAY run on shared hosted runners; their purpose is relative delta detection, not absolute performance measurement.
- **ENG-209.** The `cargo-fuzz` smoke harnesses mandated by `ENG-055` for Tier 2 MUST run each fuzz target for a minimum of 60 seconds per target on the Tier 2 runner. The purpose of the smoke run is to catch trivial parsing regressions; it is not a substitute for the extended nightly session under `ENG-066`.

### 1.20 Clippy allowlist and `rustfmt` configuration

- **ENG-210.** The initial `clippy::pedantic` allowlist contains the following entries, each with its inline justification: `allow(clippy::module_name_repetitions)` (DNS record-type names unavoidably repeat module-name prefixes); `allow(clippy::similar_names)` (short identifier names are idiomatic in DNS wire-format code); `allow(clippy::too_many_lines)` (state-machine handlers and dispatch functions legitimately exceed the line threshold). Additional allow entries MAY be added as the codebase grows; each new entry MUST carry an inline justification comment. Allow entries MUST be re-reviewed at each annual MSRV bump to confirm they remain necessary.
- **ENG-211.** The canonical `rustfmt.toml` MUST contain at a minimum the following settings: `edition = "2024"`, `max_width = 100`, `tab_spaces = 4`, `imports_granularity = "Crate"`, `reorder_imports = true`, `reorder_modules = true`, `use_field_init_shorthand = true`, `use_try_shorthand = true`, `group_imports = "StdExternalCrate"`. Additional settings MAY be introduced through the normal pull-request flow reviewed under `ENG-078`.

### 1.21 Protected-branch configuration, merge queue, and auto-merge

- **ENG-212.** The `main` branch MUST be configured as a protected branch on GitHub with the following settings: require a linear history; require all Tier 1 checks from `ENG-046` through `ENG-053` and all Tier 2 checks from `ENG-054` through `ENG-057` as passing status checks; require a minimum of two approving reviews for pull requests that modify source code (one for documentation-only or specification-only pull requests per `ENG-078`); require code-owner approval on every path pattern declared in `CODEOWNERS`; disable direct push for all contributors including administrators; disable force-push for all contributors including administrators.
- **ENG-213.** GitHub's native merge queue MUST be configured for `main`. The merge queue MUST speculate up to four concurrent pull requests. The merge queue MUST run all Tier 1 and Tier 2 required status checks on each speculative merge commit before admitting the head commit to `main`. A pull request for which a CODEOWNERS-required approval is pending MUST NOT be eligible for merge-queue entry.
- **ENG-214.** Auto-merge MUST be enabled only for pull requests that simultaneously satisfy: all required status checks have passed; all required reviewer approvals under `ENG-077` through `ENG-079` have been granted; and no review with `request-changes` status remains open. Dependabot and Renovate pull requests for patch-level version bumps of crates on the `cargo-deny` allowlist MAY auto-merge when the above conditions are met. Minor-level or major-level bump pull requests from automated bots MUST NOT auto-merge and MUST receive explicit maintainer review under `ENG-077`.

### 1.22 LTS maintenance-branch policy and backport process

- **ENG-215.** Dedicated long-lived release-maintenance branches (for example, `release-1.0.x`) MUST NOT be created before the 1.0.0 release. At or after the 1.0.0 release, a single LTS maintenance branch for the `v1.0.x` patch series MUST be created. The `v1.0.x` LTS branch MUST receive security fixes and critical bug fixes for a minimum of 12 months from the date the 1.0.0 tag is produced. This LTS policy MUST be revisited no later than the v1.0.0 release milestone and recorded in a dedicated ADR at that time.
- **ENG-216.** The backport process for fixes onto any LTS maintenance branch is: (a) the fix lands on `main` first through the normal pull-request flow; (b) the relevant commits are cherry-picked to the maintenance branch; (c) the cherry-pick is submitted as a separate pull request against the maintenance branch, subject to the full code-review requirements of `ENG-076` through `ENG-089`; (d) a patch release tag is produced from the maintenance-branch head after the cherry-pick pull request is merged. Pre-1.0.0, no maintenance branches exist and no backport process applies.

### 1.23 Flaky-test policy

- **ENG-217.** A test that fails in CI but passes on a subsequent re-run MUST be treated as flaky and MUST NOT be retried automatically to pass the merge gate. Automatic retry budgets for flaky tests are prohibited. A confirmed flaky test MUST be quarantined by marking it `#[ignore]` within 24 hours of its first confirmed flaky failure; a GitHub Issue MUST be opened simultaneously with the label `test:flaky`. The quarantined test MUST be fixed or removed within 7 calendar days; tests remaining quarantined beyond 7 days MUST be removed from the codebase until a fix is ready. The release manager MUST produce a monthly flaky-test report summarising all open quarantine issues.

### 1.24 Initial CODEOWNERS domain breakdown

- **ENG-218.** The `.github/CODEOWNERS` file MUST define the following initial domain breakdown, with each domain mapped to a glob pattern: (a) `crates/heimdall-core/**` — DNS-protocol-and-DNSSEC domain; (b) `crates/heimdall-runtime/**` — transports-and-cache domain; (c) `crates/heimdall-roles/**` — DNS-role-logic domain; (d) `crates/heimdall/**` — binary-and-wiring domain; (e) `.github/**`, `deny.toml`, `Cargo.toml`, and `Cargo.lock` — CI-and-build domain (two code-owner approvals required per `ENG-082`); (f) `specification/**` — specification domain (one-reviewer approval sufficient per `ENG-078`). Critical-path patterns requiring two-code-owner approval under `ENG-079` and `ENG-081` MUST be added explicitly as they are introduced, covering at minimum: any path containing `unsafe` code, any cryptographic routine, the DNSSEC validator, every parser module, and the release-signing and SBOM pipelines.

### 1.25 Governance, review operations, and contributor policy

- **ENG-219.** A `GOVERNANCE.md` file MUST exist at the root of the primary Heimdall repository. The file MUST document: the current list of maintainers by GitHub handle; the maintainer nomination, review, and onboarding process (nomination by an existing maintainer, lazy-consensus approval among active maintainers over two weeks, then onboarding); the inactive-maintainer removal process (18 months of no substantive contribution constitutes inactivity; removal by simple majority of active maintainers); the decision-making model (lazy consensus for minor decisions, supermajority of ≥ two-thirds of active maintainers for major decisions); and the conflict-resolution path (synchronous call of active maintainers when lazy consensus fails).
- **ENG-220.** A pull request that has received no review activity from any maintainer within 14 calendar days of reaching the `ready-for-review` state MUST be flagged at the next maintainer standup. The standup MUST assign a named reviewer and record the assignment as a comment on the pull request. No automatic timeout or auto-close is applied to a pull request solely on the basis of reviewer inactivity.
- **ENG-221.** A security-fix pull request for a GHSA-identified vulnerability MAY bypass the two-reviewer baseline under `ENG-077` if and only if: (a) at least one code owner of each affected domain has approved; (b) at least one additional maintainer has approved; and (c) a dedicated post-merge full-review pull request is opened within 48 hours. The post-merge pull request MUST introduce no functional changes; its purpose is to complete the normal review cycle.
- **ENG-222.** Pull requests opened by automated dependency-update bots (Dependabot, Renovate) MUST pass the full Tier 1 and Tier 2 CI gate before any merge. Patch-level bump pull requests for crates already on the `cargo-deny` allowlist MAY auto-merge under `ENG-214` once CI passes and `cargo vet` coverage for the new version is confirmed. Minor-level or major-level bump pull requests MUST receive explicit maintainer review under `ENG-077`.
- **ENG-223.** First-time contributors MUST have a maintainer explicitly apply the label `ok-to-test` to their pull request before CI workflows run on it. The `CONTRIBUTING.md` file MUST document this requirement and the process for obtaining the label. After the first merged contribution from a given contributor, subsequent pull requests from the same contributor do not require the `ok-to-test` gate. Contributions MUST be accepted under the Developer Certificate of Origin (DCO) model; a `Signed-off-by:` trailer on every commit is required and MUST be enforced at Tier 1 by the commit-validation gate under `ENG-147`.
- **ENG-224.** A work-in-progress pull request MUST be marked as a GitHub Draft pull request. The `[WIP]` title prefix MUST NOT be used. Tier 1 fast checks (format, Clippy, basic build) MUST run on draft pull requests; the full Tier 1 and Tier 2 gates MUST run only after the pull request is marked ready for review. A draft pull request MUST NOT be merged.
- **ENG-225.** Review comments MUST use the following prefix convention: `nit:` or `suggestion:` for non-blocking cosmetic suggestions; `question:` for clarifying questions that do not block merge; `blocker:` for comments that block merge. A review submitted with the `request-changes` verdict MUST contain at least one `blocker:`-prefixed comment; a review containing only `nit:` or `suggestion:` comments MUST be submitted with the `comment` verdict, not `request-changes`. These conventions MUST be documented in `CONTRIBUTING.md`.

## 2. Rationale

The decision to track the latest stable `rustc` as the MSRV (Decision AS) is grounded in the audience to which the MSRV actually applies. Under the distribution model fixed by `ENV-023` through `ENV-033` in [`009-target-environment.md`](009-target-environment.md), production operators consume Heimdall through signed pre-built artefacts — static musl binaries, `.deb` and `.rpm` packages, and hardened OCI container images — and never observe the toolchain used to produce those artefacts. The MSRV therefore constrains only contributors and package maintainers who rebuild Heimdall from source. For that audience, `rustup` is ubiquitous and the cost of moving to the current stable is trivial: `rustup update stable` completes in seconds and does not disrupt any other Rust project on the same machine.

Tracking stable gives Heimdall direct access to Rust language features that are already material to a DNS server whose design point is extreme performance under high concurrency: `async fn in traits` (stable since 1.75), generic associated types, async closures (stable since 1.85), `LazyLock`, and the ongoing expansion of const-generic expressivity. A conservative stable-minus-N MSRV would either forbid these features outright or force idiomatic workarounds (boxed trait objects, manual `async_trait` desugaring, `once_cell` shims) that erode the ergonomic and performance advantages that motivated the choice of Rust in the first place.

The dependency ecosystem on which Heimdall relies — `tokio`, `rustls`, `quinn`, `hyper`, `serde`, `rustix`, and their transitive graph — itself tracks stable or remains within a narrow window of stable. A conservative MSRV on Heimdall would force pinning older major versions of those crates, with the twin consequences of missing security fixes that land on current lines and, in the worst case, maintaining project-local forks to backport critical changes. Both outcomes are anti-patterns for a server whose security posture is non-negotiable under the core principles fixed by [`../CLAUDE.md`](../CLAUDE.md).

Stable-minus-N MSRV policies are widely adopted by library crates that form the foundation of large downstream dependency trees, where forcing consumers to update their own toolchain has a non-trivial coordination cost. Heimdall is an application server, not a library: no external crate depends on Heimdall's `Cargo.toml`, and the stable-minus-N trade-off does not apply. The audience cost of the latest-stable policy is bounded to the project's own contributors and package maintainers, for whom the cost is trivial.

CI enforcement of the MSRV exactly under `ENG-004` prevents drift in both directions. Running the matrix against the declared MSRV catches accidental use of unstable features, which would make source builds fail on a plain stable toolchain; it also catches accidental adoption of features that require a newer-than-declared `rustc`, which would silently raise the MSRV without the corresponding `Cargo.toml` and release-notes updates required by `ENG-002` and `ENG-005`. The enforcement discipline makes the `rust-version` field a load-bearing claim rather than a loosely-maintained hint.

Documenting MSRV bumps in release notes under `ENG-005` and `ENG-006`, while not treating them as semver-breaking, follows standard Rust ecosystem practice. A contributor or package maintainer who is about to rebuild Heimdall has an unambiguous pointer to the toolchain requirement of the release they are targeting; downstream consumers of the pre-built artefacts are unaffected and see no semver churn.

The decision to require a dedicated ADR for every external crate dependency (Decision AT, `ENG-008` through `ENG-016`) rests on four considerations, each of which is not addressable by automated tooling alone.

First, the "strict minimum" obligation on dependencies inherited from [`../CLAUDE.md`](../CLAUDE.md) and reiterated by `THREAT-010` in [`007-threat-model.md`](007-threat-model.md) cannot be enforced mechanically. Tools such as `cargo-audit` and `cargo-vet` judge the integrity of the dependency graph and flag known vulnerabilities; they do not judge whether a given dependency is actually needed, or whether it is proportionate to the problem it solves. Only a human gate, informed by the specific engineering context of the change, can make that judgement.

Second, adding a dependency is structurally asymmetric with removing one. Once a dependency is introduced, its types, traits, and idioms tend to propagate quickly across modules, at which point the cost of removal is far higher than the cost of the initial refusal. A higher barrier to the introduction of a dependency is proportionate to that asymmetry and avoids the cumulative drift that is otherwise typical of large Rust projects.

Third, the ADR provides a durable historical record that survives contributor turnover. Years after the initial decision, future reviewers, operators, and auditors can open the ADR and understand why the dependency was adopted, what alternatives were considered, and what its supply-chain posture was at the time of adoption. This transparency is a first-class asset for a project whose security posture is non-negotiable, and it cannot be reconstructed after the fact from code alone.

Fourth, batching the retrospective ADRs for the core dependencies already implicit in prior decisions (`ENG-013`) is a one-time upfront cost that pays for itself twice: it establishes the gate's precedent within the project, and it produces authoritative documentation for the most load-bearing dependencies in the stack, which are precisely those whose compromise or abandonment would have the largest impact on Heimdall.

Scoping the gate uniformly to development dependencies under `ENG-014` reflects the fact that an attack on a dev-dependency is still an attack on the project: malicious code in a test-time crate can exfiltrate secrets from the build environment, tamper with test outcomes, and, in the worst case, inject artefacts into the release pipeline. The supply-chain surface of a dev-dependency is real at build time even if it is absent at runtime, and the gate's logic applies to it identically.

Scoping the gate uniformly across all Cargo workspaces within the repository under `ENG-015` prevents the trivially-defeated pattern in which a dependency is introduced through a secondary workspace member that is then pulled in transitively. A single scope boundary applied to the whole repository is the only scope boundary that holds.

Finally, the explicit complementarity with the automated enforcement in [`007-threat-model.md`](007-threat-model.md) under `ENG-016` prevents the ADR gate from being seen as a substitute for mechanical verification, and prevents the mechanical verification from being seen as a substitute for human judgement. The two gates answer different questions and are both required.

The decision to restrict, document, and gate `unsafe` code within Heimdall's own source (Decision AU, `ENG-017` through `ENG-025`) operationalises the directive in [`../CLAUDE.md`](../CLAUDE.md) that "every `unsafe` block requires an explicit justification and documented invariants". The decision translates that directive into an automated enforcement path — `#![deny(unsafe_code)]` by default plus Clippy's `undocumented_unsafe_blocks` at the `deny` level — and a human gate — a dedicated ADR for every new pattern of `unsafe` usage. The two layers together ensure that `unsafe` cannot be introduced silently and cannot be introduced without a reviewed rationale.

A zero-`unsafe` policy would be infeasible under the performance ambition fixed by Decision AO (`008-performance-targets.md`). SIMD intrinsics, zero-copy parsing on the hot path, foreign function interface into `io_uring`-style primitives on Linux, and atomic operations outside what the standard library abstracts in safe form all require `unsafe`. Forbidding `unsafe` outright would force workarounds incompatible with the targeted exceedance margin on encrypted transports, and would displace the performance risk into less auditable code. The rational response is therefore not to ban `unsafe` but to make every occurrence of it rare, narrow, documented, and deliberate.

The narrow-scope `#[allow(unsafe_code)]` override under `ENG-018` ensures that the default posture of every crate remains `unsafe`-free and that any divergence from that posture is localised and conspicuous. A crate-root override would obscure the locations where `unsafe` actually lives and would defeat the purpose of the default. The per-module or per-item scope forces contributors and reviewers to look at the precise site where the unsafety occurs.

The three mandatory items of the SAFETY comment under `ENG-019` (invariants, why-not-safe, scope of unsafety) reflect the fact that incomplete SAFETY commentary is almost as dangerous as missing commentary. An invariants-only comment tells a reviewer what is expected but not why the safe alternative was rejected, leaving the door open to a later refactor into safe code that is mistakenly assumed to be impossible. A why-not-safe-only comment tells the reviewer the motivation but not the contract, leaving the door open to callers who violate the invariants because they are nowhere stated. A scope-only comment tells the reviewer the blast radius but neither the contract nor the motivation. All three items are required because each of them addresses a distinct failure mode.

The ADR gate under `ENG-021` creates appropriate friction at the point where a new category of `unsafe` enters the codebase. The gate is scoped to new patterns, not to every new block, precisely because a per-block ADR would produce unreadable bureaucratic overhead for no additional safety benefit. A pattern is the unit at which the reasoning about unsafety is actually shared across occurrences (for example, all SIMD intrinsics in the hot-path parser share one invariants story; adding a new variant does not change the story). The combination of `ENG-021` and `ENG-022` captures this distinction precisely.

The annual review mandated by `ENG-023` and `ENG-024` addresses two distinct decay modes. First, `unsafe` written against the standard library of a given `rustc` version may become unnecessary as the standard library stabilises new safe APIs: `MaybeUninit`, stabilised atomic operations, `core::simd`, and the ongoing stabilisation of formerly-nightly intrinsics have all removed entire categories of previously justified `unsafe` over the last several Rust editions. Second, `unsafe` that was sound at the time of introduction can become unsound through a later change to surrounding code that silently violates an invariant the `unsafe` block relied on. A written outcome per item ensures that the review is not a box-ticking exercise; either continued justification is recorded or a concrete follow-up task is filed.

Finally, the explicit scope boundary under `ENG-025` prevents the Heimdall-source policy from being confused with the dependency-unsafe policy. The two are governed by different processes because they answer different questions: the Heimdall-source policy governs code that the project itself writes and can modify, whereas the dependency-unsafe footprint is an input into the acceptance decision for a third-party crate, recorded in the dependency ADR under `ENG-009` item 4 and subject to the tolerance policy listed as an open question below.

The decision to adopt a comprehensive, multi-layer testing stack (Decision AV, `ENG-026` through `ENG-040`) follows directly from the project-wide principles that Heimdall's security is non-negotiable and that its performance is the primary guide of every decision. Both principles demand testable guarantees, and a gap in any one testing layer produces a class of bugs that the other layers cannot detect.

Without coverage-guided fuzzing, parser bugs are catastrophic and have been historically frequent in DNS implementations: BIND, Unbound, and Knot have each accumulated multiple CVEs over the years in their DNS message, zone-file, and extension parsing paths. Rust mitigates the memory-safety subset of that bug class, but it does not mitigate logic bugs, state-machine confusion, or panic-on-input. Requiring `cargo-fuzz` on every parsing surface under `ENG-031`, and pursuing OSS-Fuzz integration under `ENG-032`, closes that gap structurally. OSS-Fuzz amortises continuous fuzzing by running Google infrastructure at no cost to open-source projects, which makes continuous fuzzing feasible at a scale the project could not otherwise sustain.

Without `miri`, undefined behaviour in `unsafe` code compromises soundness silently. Decision AU (`ENG-017` through `ENG-025`) permits restricted, narrow, documented `unsafe`, but that permission is only verifiable if a dynamic detector keeps the SAFETY comments honest. `miri` is the Rust ecosystem's answer to that problem, and the obligation in `ENG-033` that test suites be runnable under `miri` for every module containing `unsafe` code is the direct enforcement counterpart of `ENG-019`.

Without `loom`, race conditions in concurrent hot paths pass normal test runs but corrupt state in production under high load — precisely the workload Heimdall targets. The segregated caches (`004-cache-policy.md`), connection pools, per-worker state, and atomic-ordering-sensitive code paths that dominate Heimdall's hot path are the exact class of component that `loom` was designed to model-check. Requiring `loom` coverage on those components under `ENG-034` aligns the testing posture with the performance and concurrency ambition fixed elsewhere in the specification.

Without conformance testing, subtle divergences from the RFC text appear only in real-world interop. That is late detection and hard to fix within a release window. Requiring exercise of IETF DNSSEC test vectors and DoT, DoH, and DoQ interoperability suites under `ENG-035`, plus golden-output comparison against Unbound, NSD, and Knot for representative queries, surfaces those divergences early.

The nine layers are complementary, not redundant. Unit tests verify in-isolation correctness at the function level; integration tests verify cross-module composition; property-based tests verify invariants over input space rather than specific examples; fuzzing drives parsers to their failure modes; `miri` detects UB in `unsafe` code dynamically; `loom` model-checks concurrent interleavings exhaustively; conformance tests verify wire-protocol agreement with the rest of the DNS ecosystem; the controlled integration harness verifies end-to-end behaviour against simulated authoritatives and clients; and benchmarking verifies the performance envelope. Removing any layer creates an unmitigated class of bugs.

Requiring `proptest` over `quickcheck` under `ENG-030` reflects a concrete technical distinction: `proptest`'s shrinker produces materially better minimised failing cases, which accelerates debugging when an invariant is falsified. For a server whose correctness envelope spans DNS wire format, DNSSEC validation, cache coherence, and zone-file serialisation, the shrinker quality is load-bearing rather than cosmetic.

Requiring `criterion` for micro-benchmarks and the `dnsperf` / `queryperf` / `resperf` tools for end-to-end benchmarks under `ENG-037` reuses the benchmarking stack already fixed in [`008-performance-targets.md`](008-performance-targets.md) and feeds it into both the internal regression gate (`PERF-014` through `PERF-018`) and the external comparative gate (`PERF-019` through `PERF-028`). The two-tier benchmarking posture — `criterion` at micro scale, industry-standard tools at end-to-end scale — is the same two-tier posture used across the DNS operational ecosystem, which preserves comparability with published reference-implementation performance.

The Rust testing ecosystem is mature enough to support this layered stack without custom tooling: `proptest`, `cargo-fuzz`, `miri`, `loom`, and `criterion` are all production-grade crates with long track records. The cost of setting the stack up is bounded and one-time; the cost of not setting it up compounds over every release.

The decision to organise the CI pipeline into four tiers (Decision AW, `ENG-041` through `ENG-075`) reconciles two forces that would otherwise pull in opposite directions: the rigour required by the security and performance principles fixed by [`../CLAUDE.md`](../CLAUDE.md), and the practicality required by a per-pull-request feedback loop that contributors can actually sustain. Tier 1 and Tier 2 together are scoped to keep the end-to-end per-pull-request wall-clock within a realistic envelope (approximately thirty minutes in aggregate), which preserves a fast merge cadence. Tier 3 and Tier 4 host the heavier checks — `miri` across the full matrix, OSS-Fuzz continuous fuzzing, extended `proptest` budgets, external comparison benchmarks, reproducible-build verification, and Software Bill of Materials delta reporting — out of band, in the place where their real cost belongs.

Every gate in the four tiers traces back to an already-decided requirement. The tier decision operationalises those prior decisions rather than introducing new substantive obligations: `cargo audit` and `cargo vet` at Tier 1 enforce `THREAT-011`; benchmark regression detection at Tier 2 enforces `PERF-014` through `PERF-018`; extended fuzzing and OSS-Fuzz at Tier 3 enforce `ENG-031` and `ENG-032`; `miri` at Tier 3 enforces `ENG-033`; `loom` at Tier 2 enforces `ENG-034`; reproducible-build verification at Tier 3 enforces `THREAT-012`; Software Bill of Materials generation at Tier 3 and publication at Tier 4 enforce `THREAT-014`; artefact signing at Tier 4 enforces `THREAT-013`; the Minimum Supported Rust Version statement in release notes at Tier 4 enforces `ENG-005` and `ENG-006`. The tier structure is therefore a wiring diagram, not a new policy layer.

The tier separation honours the real cost differences between the checks. `miri` is typically an order to two orders of magnitude slower than native execution, which makes full-matrix `miri` impractical on every pull request but realistic on a nightly cadence. OSS-Fuzz, by design, runs on Google-operated infrastructure and continuously; it is not a pull-request gate and must not be framed as one. External comparison benchmarks against reference implementations require dedicated, noise-isolated hardware for reproducibility, because shared CI runners exhibit enough performance variance to swamp the signal the comparison is meant to produce. Placing those checks at Tier 3 reflects their operational reality, not a relaxation of the requirements they enforce.

Placing artefact signing and Software Bill of Materials publication at Tier 4 makes them structural obligations of the release process rather than release-time chores that depend on an individual remembering them. A release tag cannot be produced if Tier 4 fails; the signing and Software Bill of Materials obligations therefore carry the same binding force as the build-and-test obligations of Tier 1. This is the same structural-gating approach already applied to role activation in [`001-server-roles.md`](001-server-roles.md), to transport-listener instantiation in [`002-transports.md`](002-transports.md), and to performance regression detection in [`008-performance-targets.md`](008-performance-targets.md).

The Clippy posture fixed by `ENG-058` through `ENG-062` reflects the observation that disabling a whole lint group wholesale — the most common way `clippy::pedantic` is treated in Rust projects — hides the individual decisions that were made about each lint. An inline-justified allowlist, by contrast, forces every exception to be explicit and reviewable, and prevents a later drift in which new pedantic warnings are silently ignored by a pre-existing blanket disable. The `clippy::unwrap_used` and `clippy::expect_used` at `warn` in library and production paths, and the `clippy::panic_in_result_fn` at `deny` on non-test paths, together encode the fail-safe posture required by [`../CLAUDE.md`](../CLAUDE.md) directly into the lint configuration: code that would panic on a recoverable error path is either reviewed explicitly or prevented from landing.

The `rustfmt` posture fixed by `ENG-063` and `ENG-064` is deliberately minimal at the specification level: the load-bearing claim is that a single canonical `rustfmt.toml` exists, is checked in, and is enforced. The concrete values of `max_width`, import-grouping, trailing-comma handling, and analogous parameters are operational tuning that belongs in a follow-up narrow decision rather than in this decision. Fixing the obligation of a single canonical configuration is sufficient to prevent per-branch formatting drift; fixing the values of every parameter in this decision would conflate two different kinds of question.

The decision to require two-reviewer approval by default, one-reviewer approval for documentation-only or specification-only pull requests, and two-code-owner approval on critical paths (Decision AX, `ENG-076` through `ENG-089`) rests on the observation that the "security non-negotiable" principle fixed by [`../CLAUDE.md`](../CLAUDE.md) requires a robust human gate in addition to the automated gates layered throughout Decision AW. The automated gates catch mechanical regressions: formatting drift, Clippy warnings, failing tests, known CVEs, license violations, broken documentation links, measurable benchmark regressions. They do not catch design drift, architectural regressions, or the subtle security bugs that recur in DNS implementations — miscomputed DNSSEC validation order, cookie-secret lifecycle errors, cache coherence anomalies that manifest only under concurrent load, parser state-machine confusion on adversarial inputs. Those classes of defect require a human reviewer with context. A single reviewer is a single point of failure for that gate: reviewer fatigue, domain blind spots, and the author-reviewer-as-same-person social dynamic that arises when a team is small all reduce the effective value of a one-reviewer regime. Two independent reviewers materially reduce the probability that a subtle defect passes undetected.

The CODEOWNERS obligation fixed by `ENG-081` forces expertise alignment between the change under review and the reviewer. A DNSSEC validation change must be reviewed by someone with DNSSEC fluency; a QUIC transport change must be reviewed by someone with QUIC fluency; a cache-coherence change must be reviewed by someone who understands the memory-ordering model of the segregated-cache design fixed in [`004-cache-policy.md`](004-cache-policy.md). The CODEOWNERS mechanism scales this expertise requirement naturally with the contributor base and protects against the domain drift that is typical of projects in which any reviewer can approve any change. By requiring at least one code-owner approval per domain touched (`ENG-080`), the policy ensures that multi-domain pull requests are reviewed by domain-literate reviewers on every domain they modify, rather than by a single generalist reviewer who happens to hold write access.

The two-code-owner regime on critical paths (`ENG-079`, `ENG-081`, `ENG-082`, `ENG-083`) reflects the fact that those paths are the areas of maximum blast radius for a defect. `unsafe` blocks bypass Rust's compile-time memory-safety guarantees, which makes any bug in them potentially catastrophic; cryptographic routines are silently exploitable when incorrect, with consequences that include protocol downgrade, forged responses, and data exfiltration; DNSSEC validation is the chain-of-trust mechanism on which the validating roles' entire correctness envelope rests; parsers are triggerable by adversarial input directly at the network boundary. The four security-sensitive infrastructure file classes enumerated in `ENG-082` — the continuous-integration configuration, the signing pipeline, the Software Bill of Materials pipeline, and the `cargo-deny` / `cargo-vet` / `cargo-audit` configuration files — share a common property: a compromise of any of them is a compromise of the automated gates of Decision AW, the supply-chain gates of `THREAT-010` through `THREAT-014` in [`007-threat-model.md`](007-threat-model.md), and the release integrity envelope as a whole. Requiring two code-owner approvals on those paths treats them with the same gravity as the cryptographic and parsing code they protect. Requiring two code-owner approvals on the `CODEOWNERS` file itself under `ENG-083` closes the self-reference hole: without that rule, a single code owner could unilaterally rewrite the critical-path declaration and thereby disable the protection mechanism itself.

The single-reviewer carve-out for documentation-only or specification-only pull requests under `ENG-078` proportions the gate to the risk. Documentation and specification content does not execute; a defect in it cannot be directly exploited at runtime. The risk profile of a one-reviewer regime on such changes is materially lower than on code changes, and a blanket two-reviewer obligation on docs-only or spec-only pull requests would produce review overhead without a proportionate security benefit. The one-reviewer regime nevertheless preserves a human gate: nothing merges unreviewed, and the reviewer still catches factual errors, terminological drift, and inconsistencies that an automated check cannot detect.

The invariants fixed by `ENG-084` through `ENG-086` close the three common evasion paths in review policies of this shape. The author-is-not-a-reviewer invariant (`ENG-084`) prevents a contributor from self-approving, whether explicitly or by counting a co-authored contribution as an independent approval; without it, the numerical reviewer requirement degenerates to a one-person gate on any pull request where the author holds maintainer credentials. The requested-changes-blocks-merge invariant (`ENG-085`) prevents the "ignore or dismiss the objection" evasion in which a requested-changes review is overridden by a subsequent approval from a different reviewer; without it, a thoughtful objection can be silently bypassed, and the review gate collapses to "find any reviewer willing to approve". The approval-authority invariant (`ENG-086`) prevents the "approval inflation" evasion in which approvals from accounts without maintainer authority are counted toward the numerical requirement; without it, the gate's value depends on which accounts happen to drop a "looks good to me" comment, rather than on the governance-backed maintainer list.

The `CODEOWNERS` location obligation under `ENG-087` ensures that the file is not merely present but actively enforced by the hosting platform. GitHub recognises `/.github/CODEOWNERS` and `/CODEOWNERS` at the repository root as the canonical locations where the platform enforces code-owner review on pull requests; filing the same content at a non-recognised path yields a documentation artefact with no enforcement semantics. The obligation is phrased in platform-neutral terms so that the policy remains stable if Heimdall migrates to a different hosting platform: the load-bearing claim is that the file must live where the platform enforces it, whatever the platform is.

The governance document obligation under `ENG-088` addresses the prerequisite that `ENG-086` relies on: the approval-authority invariant is only enforceable if there exists an authoritative list of maintainers and a defined process for joining or leaving that list. The exact location and content of the governance document are downstream operational decisions and are recorded as open questions in section 3. What is fixed in this decision is the obligation that such a document exists and is under version control.

The scope boundary under `ENG-089` clarifies that the code-review policy applies at the point a change is proposed for merge into the primary repository, not to private work on personal branches or forks. Without this clarification, the policy could be read as requiring two-reviewer approval on every commit to any branch in any fork, which is both impractical and outside the gate's intended scope. Once a change is proposed for merge, however, it is subject to the policy in full, regardless of whether it originated in a fork or a protected branch.

The overall structure of this decision aligns with the review posture in mature open-source projects of comparable criticality: BIND, PowerDNS, Unbound, and Knot in the DNS ecosystem; tokio, rustls, and hyper in the Rust ecosystem. Each of those projects operates a multi-reviewer regime with domain ownership declarations and explicit critical-path discipline. That convergence of practice is not a proof of correctness, but it is a signal that the structure proposed here is workable at sustained scale and is understood by the population of contributors Heimdall expects to attract.

The decision to license Heimdall under the MIT License, with an enumerated permissive-licence whitelist and an enumerated copyleft-and-proprietary blocklist for dependencies (Decision AY, `ENG-090` through `ENG-097`), rests on five considerations that jointly select a permissive posture, narrow that posture to MIT specifically, and extend it by enforcement to the dependency graph.

A permissive licence enables broad adoption, including commercial and closed-source deployments. Heimdall is an infrastructure server whose utility scales with the range of environments in which it can be deployed: enterprise networks, regulated industries, commercial Software-as-a-Service providers, appliance vendors, and self-hosted operators all represent legitimate adoption surfaces. A copyleft licence on Heimdall itself would exclude a substantial share of those operators from the outset, in direct tension with the objective of a DNS server engineered for extreme load.

MIT is the simplest, shortest, and most universally understood permissive licence in the Rust ecosystem and beyond. Its text is compact enough that operators, auditors, and distributors can read and understand it in full, which lowers the friction of compliance. Within the Rust ecosystem specifically, MIT (often in combination with Apache-2.0) is the dominant licence for foundational crates, which minimises the cognitive surface a new contributor or packager must navigate.

Patent-grant protection is deliberately traded away in favour of simplicity. The MIT License contains no explicit patent grant, in contrast to Apache-2.0. That asymmetry is accepted with full awareness of the trade-off: contributors and downstream operators alike are expected to understand that the Heimdall licence chooses maximum textual simplicity over formal patent immunity, and the rationale is recorded here so that the choice is neither silent nor accidental. The contributor documentation MUST surface this trade-off explicitly to each contributor.

Copyleft licences (GPL, LGPL, AGPL) are rejected for dependencies under `ENG-095` because a copyleft dependency, even a transitive one, would force Heimdall to re-license under the copyleft terms — the well-known GPL-propagation effect. That outcome would directly contradict the Heimdall-own-licence choice fixed by `ENG-090`, and would re-impose on Heimdall's operators the very deployment restrictions that the MIT choice is intended to remove. The blocklist is therefore not a philosophical preference but a structural consequence of the licence choice: the two are interlocked, and loosening either one would compromise the other. Proprietary and non-commercial-restricted licences are rejected for the analogous reason — they would impose obligations on Heimdall's own redistributors that MIT expressly does not impose, which would break the MIT promise end-to-end.

MPL-2.0 is accepted under `ENG-094` item 8 with an explicit acknowledgement of the file-level reciprocity trade-off. Under MPL-2.0, modifications to the MPL-covered files themselves must remain under MPL, but the MPL does not propagate to the aggregate work, which preserves MIT coverage of Heimdall as a whole. The dependency ADR under `ENG-009` item 3 is therefore the site at which the project acknowledges, per MPL-2.0 dependency, that any modifications Heimdall makes to those files will remain MPL-licensed. This explicit acknowledgement is the difference between an unreviewed adoption of MPL code and a deliberate one, and it is what makes MPL-2.0 admissible without compromising the MIT posture of the rest of the project.

Encoding the whitelist and blocklist in the `cargo-deny` configuration under `ENG-096` makes the licensing policy load-bearing rather than aspirational. `cargo-deny` is already in the Tier 1 CI gate under `ENG-050`; attaching the licence enforcement to the same invocation ensures that every pull request is checked automatically, that licence drift cannot occur silently through a transitive-dependency bump, and that the human ADR gate under `ENG-008` and the mechanical licence gate under `ENG-096` converge on the same answer. This is the same two-gate pattern — human plus mechanical — applied across the engineering policies of this file.

The decision to organise Heimdall as a Cargo workspace with four to six top-level crates structured along natural architectural seams (Decision AZ, `ENG-098` through `ENG-111`) reconciles the opposing failure modes of an under-split and an over-split Rust project. The four mandatory crates identified by `ENG-099` through `ENG-102` — `heimdall-core` for protocol primitives, `heimdall-runtime` for runtime machinery, `heimdall-roles` for DNS role logic, and the thin `heimdall` binary for wiring — correspond to the architectural seams that have already emerged from the prior specification decisions. Each crate has a coherent responsibility and a defensible public surface, which makes the boundaries enforceable at build time rather than by convention alone.

The binary crate is deliberately thin under `ENG-102`. Keeping business logic out of the binary preserves testability at the library level, where unit tests, integration tests, and property-based tests under `ENG-028` through `ENG-030` can reach every code path without a process boundary. It also enables future embedded-use scenarios in which Heimdall's libraries are linked into another service as a library dependency rather than run as a standalone binary. Neither property is available if business logic is allowed to drift into the binary crate.

Rust's incremental compilation model operates at the crate level: a touched file in a single-crate layout triggers recompilation of the whole crate, whereas the same file in a multi-crate layout triggers recompilation only of the crates downstream of the change. Four to six crates along natural seams gives Heimdall the compile-time parallelism benefit of a multi-crate layout without the coordination overhead that a finer-grained split would introduce. A ten-or-more-crate layout saturates the parallelism benefit and imposes a different cost — cross-crate version bumps, coordinated renames, cascading visibility changes — that dominates its marginal return.

The moderate split aligns with the code-owner domain decision fixed by `ENG-081` and `ENG-082` under Decision AX. Crates map cleanly to ownership domains: `heimdall-core` corresponds to the parser and DNSSEC-primitive owners, `heimdall-runtime` corresponds to the transport and cache owners, and `heimdall-roles` corresponds to the DNS-behaviour owners. The `CODEOWNERS` mechanism can therefore attach domain-literate reviewers to crate-level path patterns without the boundary ambiguity that a monolithic or hyper-fragmented layout would produce.

A fine-grained split of ten or more crates, rejected by `ENG-105`, crystallises module boundaries before the design has been validated by real implementation. Subsequent refactoring of such a split is high-friction: every rename, every move of a type across crates, and every collapse of two crates into one cascades through `Cargo.toml` dependencies, public-API exports, crate versions, and the dependency-ADR record under `ENG-008` through `ENG-016`. The moderate split leaves structural room for growth — through the optional split-out crates sanctioned by `ENG-103` and `ENG-104` once their complexity thresholds are reached — without committing to premature boundary choices.

A single-crate layout, rejected by `ENG-106`, forgoes the build-time enforcement of boundaries that the compiler provides across crate boundaries. Within a single crate, module visibility can be widened ad hoc, internal types can be exposed to adjacent modules as a convenience, and the cumulative effect is a monolithic coupling that is difficult to reverse. A moderate multi-crate split makes the natural architectural discipline load-bearing rather than aspirational.

The two optional split-out crates under `ENG-103` and `ENG-104` — `heimdall-config` and `heimdall-observability` — are recorded here precisely so that they are not introduced prematurely. Both are plausible future crates, and both are pre-named so that if and when they are introduced the naming convention fixed by `ENG-107` is observed from day one. Introducing them before their complexity threshold is reached would fragment the workspace without a proportionate benefit; deferring them until the threshold is reached preserves the four-crate baseline for as long as the baseline remains workable.

The workspace-wide conventions fixed by `ENG-107` through `ENG-110` — the `heimdall-*` prefix on library crate names, the single workspace-level `Cargo.lock`, the shared MSRV, and the shared licence — together prevent the most common drift modes in multi-crate Rust projects. A stray per-crate lockfile masks reproducibility failures; a per-crate MSRV divergence silently raises the effective MSRV of consumers who build the workspace as a whole; a per-crate licence divergence produces an aggregate work whose licensing is ambiguous. The conventions collapse each of those failure modes at the level at which they originate.

The decision to adopt Architecture Decision Records in the MADR format, to locate them under `docs/adr/`, to number them as a monotonic zero-padded four-digit sequence, and to govern their lifecycle through an explicit set of statuses with append-only semantics (Decision BA, `ENG-112` through `ENG-127`) rests on six considerations that jointly select the format, the location, the numbering scheme, and the retention rule.

MADR is the de facto standard for ADRs in the Rust ecosystem and beyond. Kubernetes, Spring, and a large body of production Rust projects use MADR directly or as a close derivative. Its template covers precisely the shape of decision the Heimdall clarification process already produces under [`../CLAUDE.md`](../CLAUDE.md): a context and problem statement, the decision drivers, the options considered, the recommended option with its justification, and the consequences of the choice. Reusing a recognised format minimises tooling friction, minimises reader ramp-up for new contributors and external auditors, and guarantees that any ADR tool in the ecosystem — authoring aids, index generators, viewers — will interoperate with the Heimdall ADR set without adaptation.

Separating `docs/adr/` from `/specification/` under `ENG-116` makes the distinction between what (normative requirements) and why (architectural rationale) structurally explicit. Operators, auditors, and future maintainers know where to look for each kind of content without cross-referencing a convention document. The two folders are governed by distinct update processes: specification content is tied to functional, security, performance, and deployment decisions and is referenced from the source by requirement identifier; ADR content is tied to architectural decisions and is referenced from the specification by link. Collapsing the two folders into one would re-introduce the duplication that `ENG-125` explicitly forbids.

Sequential zero-padded four-digit numbering under `ENG-117` and `ENG-118` gives stable, unambiguous reference identifiers of the form `ADR-0042`. The four-digit width carries the project well beyond any realistic ADR count; the zero-padding guarantees lexicographic sort equivalent to numeric sort on filesystem listings, index pages, and tooling output. Date-based numbering, considered as an alternative, was rejected because it introduces ambiguity when multiple ADRs are drafted on the same day, does not sort naturally without additional suffixes, and obscures the sequential relationship between decisions. The non-reuse rule is the same rule already applied to specification file numbers under [`README.md`](README.md) and therefore aligns the two numbering systems behaviourally.

The YAML frontmatter under `ENG-115` and the explicit lifecycle states under `ENG-120` together make the ADR set machine-parseable. Tooling can filter ADRs by status, verify that every `superseded-by` reference resolves to an ADR that actually exists, and prevent silent rewrites of accepted ADRs by diffing frontmatter across revisions. Without the frontmatter, the same semantics would have to be reconstructed by prose parsing, which is brittle and tool-specific.

The append-only semantics under `ENG-121` and the retention rule under `ENG-122` together preserve the decision history of the project. An accepted ADR that is later overturned is evidence of how the project's thinking evolved, which is useful both to auditors and to future maintainers who would otherwise rediscover the reasoning the hard way. Deleting a superseded ADR would amount to rewriting history, which is incompatible with the audit posture the project adopts elsewhere — signed releases under `THREAT-013`, Software Bill of Materials publication under `THREAT-014`, and the dependency-history preservation under `ENG-012`. Retention of deprecated and superseded ADRs aligns the ADR process with those broader transparency obligations.

The retrospective grandfathering of core decisions under `ENG-123` is a one-time upfront cost that eliminates the "history ghost" of undocumented load-bearing choices. Tokio, `rustls`, `quinn`, `hyper`, and `rustix` are already foundational in prior specification requirements; the workspace layout, the MIT licence, and the MSRV policy are already fixed; yet none of these carries an ADR that documents the alternatives considered and the trade-offs accepted at the time of choice. Without the grandfather batch, the rationale behind those foundational decisions would be recoverable only from conversation history and inference, which is precisely what the ADR process is designed to avoid. Batching the retrospective ADRs into an initial run at the low sequence numbers starting at `0002` establishes the ADR gate's precedent, produces authoritative documentation for the most load-bearing decisions in the stack, and ensures that the meta-ADR at `0001` is immediately followed by concrete worked examples of the format.

The cross-reference protocol fixed by `ENG-124` preserves bidirectional traceability without duplication. A specification requirement that is downstream of an architectural decision can point at the ADR that records the rationale; the ADR can point back at the specification requirements that encode the decision's normative content. Relative links under `ENG-124` survive repository moves, mirror deployments, and offline audit copies, whereas absolute URLs would break silently in any of those scenarios.

The decision to adopt GitHub Issues as the canonical issue tracker and GitHub Security Advisories (GHSA) as the canonical private reporting flow for security-sensitive reports (Decision BB, `ENG-128` through `ENG-138`) rests on five considerations that jointly select the tracker, exclude external alternatives, and extend the posture by an explicit coordinated-disclosure path.

The Heimdall repository is hosted on GitHub, a fact that is already implicit in prior decisions referencing GitHub Actions for continuous-integration tooling, `/.github/CODEOWNERS` for hosting-platform-enforced code-owner review under `ENG-087`, and signed GitHub Releases for artefact distribution under `THREAT-013` in [`007-threat-model.md`](007-threat-model.md) and `ENV-023` through `ENV-033` in [`009-target-environment.md`](009-target-environment.md). Adopting GitHub Issues as the canonical tracker aligns the issue surface with the repository host, which minimises the friction any external contributor faces when reporting a bug, requesting a feature, or joining the project. A contributor who already has a GitHub account to open pull requests does not need a second account, a second notification channel, or a second identity to participate in issue triage.

Native issue-to-pull-request linkage — the `#NNN` cross-link and the `Fixes #NNN` / `Closes #NNN` auto-close semantics — is free, reliable, and already understood by every contributor familiar with GitHub. Splitting the tracker out into an external system (Linear, JIRA, YouTrack) would duplicate project-management state across two systems, require bridging infrastructure to keep the two in sync, and introduce drift whenever the bridge lags behind. The consistent use of `Fixes`, `Closes`, and `Refs` fixed by `ENG-136` and the auto-close behaviour fixed by `ENG-137` preserve the end-to-end link between the issue, the pull request that resolves it, and the merged commit, without any additional tooling.

GHSA is the de facto standard for coordinated vulnerability disclosure on open-source projects hosted on GitHub. It provides a private reporting flow in which external researchers can submit confidential reports without opening a public issue, CVE assignment through its integration as a CVE Numbering Authority, a private embargo workspace in which the maintainers and the reporter can coordinate the fix, and automated publication of the advisory with a stable URL once the fix is disclosed. Reconstructing an equivalent workflow outside GHSA would require bespoke infrastructure (a private mailing list, a private issue tracker, a manual CVE request path to MITRE, a separate advisory-publication process) for no proportionate gain. The one-line path — private report through GHSA, CVE assigned through the same system, advisory published at disclosure — is the industry-standard coordinated-disclosure template for this class of project, and `ENG-131` through `ENG-135` encode it directly.

A public-by-default issue policy under `ENG-130` aligns with open-source norms and supports contributor trust. Visibility of bug reports, triage discussion, and roadmap items allows prospective contributors to gauge project activity, identify issues they can help resolve, and verify that the project's maintenance posture is active. Private issues, by contrast, would be invisible to precisely the contributor population the project depends on for sustained maintenance. Security-sensitive reports are the narrow and well-defined exception, and they are routed through GHSA precisely because GHSA provides a purpose-built private channel that preserves confidentiality without conflating "private" with "absent from public activity signals". The separation — public by default on GitHub Issues, private where warranted on GHSA — gives both properties without compromising either.

External tools such as Linear, JIRA, and YouTrack are rejected because their added features are not proportionate to the needs of an open-source DNS server project at this stage. Those tools are designed for in-house product teams with workflow automation, story-point estimation, release-train orchestration, and stakeholder-facing dashboards. Heimdall's issue surface at this stage is bug reports, feature requests, documentation issues, roadmap items, triage, and individual specification or ADR proposals. GitHub Issues handles that surface without friction; an external tool would introduce a secondary account requirement, a secondary notification channel, and a secondary workflow for external contributors to learn, for no benefit Heimdall can currently justify. The `ENG-128` exclusion is therefore not a rejection of those tools in general but an acknowledgement that they are not proportionate to the project's current needs.

The `SECURITY.md` obligation under `ENG-132` ensures that the reporting path is discoverable and machine-readable. GitHub surfaces `SECURITY.md` prominently from the repository root and links to it automatically from the repository's security tab, so a researcher who lands on the repository through any path — a clone, a web browse, a package-manager-sourced link — can find the reporting instructions without any project-specific knowledge. Without `SECURITY.md`, the reporting path would depend on the reporter knowing to look for the GHSA "Report a vulnerability" button, which is itself not discoverable without a specific GitHub UI path.

The coordinated-disclosure policy under `ENG-133` codifies the expectations that both sides — the project and the reporter — operate under. Without a documented policy, the embargo window is implicit and variable, the first-response timing is undefined, and the conditions for extending or shortening the embargo are ambiguous. A documented policy closes those gaps and aligns Heimdall with the coordinated-disclosure norms practised by the wider security-research community. The concrete embargo default and its flexibility are operational parameters and are recorded as open questions in section 3.

The decision to adopt Conventional Commits version 1.0.0 or later as the mandatory commit format on the primary branch and on release maintenance branches, to enforce the format at Tier 1 of the continuous-integration pipeline, to adopt squash-merge as the default merge strategy, and to maintain a `CHANGELOG.md` generated from the commit history (Decision BC, `ENG-139` through `ENG-152`) rests on five considerations that jointly select the format, the enforcement location, the merge strategy, and the changelog path.

Conventional Commits is the de facto standard commit format across the Rust ecosystem. Foundational crates on which Heimdall already depends — `tokio`, `rustls`, `quinn`, `hyper`, and `rustix` — either follow Conventional Commits directly or adhere to close variants. Adopting the same format reduces ramp-up friction for external contributors who already work with those ecosystems and for reviewers who move across projects, and aligns Heimdall's commit log with the ecosystem norm against which it will be read.

Automated changelog generation is only realistic when the commit log is machine-readable. Tools in the ecosystem designed to produce changelogs from commit history — `git-cliff` is the canonical example, and several alternatives exist — all require Conventional-Commits-shaped input. Without a structured format, changelog generation collapses into either bespoke tooling on top of ad hoc parsing heuristics or manual curation per release; both paths scale poorly and drift silently against the actual history. The `CHANGELOG.md` obligation under `ENG-151` is therefore coupled to the format obligation under `ENG-139` through `ENG-146`: the two must be adopted together to produce durable value.

Semantic-versioning impact signalling is load-bearing for the upcoming release and versioning decisions Heimdall will face as it approaches first public release. The `feat!` subject-line marker and the `BREAKING CHANGE:` footer mandated by `ENG-143` make the breaking-change signal unambiguous at the commit level, machine-extractable at the changelog level, and auditable after the fact. A format without explicit breaking-change markers — freeform commit messages, for example — forces the project to reconstruct semver impact from code review discussion or from post hoc code diffing, both of which are fragile and known to produce incorrect version bumps.

Scopes aligned with the Cargo workspace layout fixed by Decision AZ (`ENG-098` through `ENG-111`) make `git log` filtering and search per crate trivial. A reviewer who needs to see every change to `heimdall-core` can filter the log with a single predicate; an auditor who wants to understand the full history of the `rustls` integration inside `heimdall-runtime` has a stable scope string to search on. This scope-to-crate alignment also ties commit-level classification to the `CODEOWNERS` domain structure fixed by Decision AX (`ENG-081` and `ENG-082`), so that the same taxonomy governs review routing and commit classification. The recommended scope vocabulary under `ENG-142` names the workspace crates and the top-level project areas explicitly; extending the vocabulary as the workspace evolves is anticipated and is part of the decision.

Squash-merge as the default merge strategy under `ENG-149` and `ENG-150` ensures a linear, readable history on the primary branch, one Conventional Commit per pull request, and an unambiguous one-to-one relationship between pull requests and landed commits. That one-to-one relationship is what makes changelog generation, `git bisect`, and targeted reverts tractable. Merge-commit as the default would produce a non-linear history with scattered feature-branch commits whose individual messages may or may not conform to the format; rebase-then-merge would produce a linear history but would land every intermediate pull-request commit on the primary branch, requiring every intermediate commit to conform to Conventional Commits independently. Both defaults would undermine the clean-history property that drives the adoption of Conventional Commits in the first place. The narrow-exception carve-out under `ENG-150` for documented cases (back-merges from release maintenance branches are the canonical example) preserves flexibility where it is operationally necessary without eroding the default.

Placing the enforcement point at Tier 1 of continuous integration under `ENG-147` — as opposed to a pre-commit hook alone — reflects the same human-gate-plus-mechanical-gate pattern applied across the engineering policies of this file. A pre-commit hook that runs on a contributor's local machine is opt-in and bypassable; a Tier 1 continuous-integration check is authoritative, unbypassable, and applies uniformly to every pull request regardless of the contributor's local tooling. Offering the pre-commit hook under `ENG-148` as an opt-in local convenience shortens the feedback loop for contributors who prefer to catch format errors before they push, but the authoritative gate remains in continuous integration, consistent with the structural-gating approach already applied to license enforcement (`ENG-050`, `ENG-096`), audited-dependency enforcement (`ENG-051`, `ENG-052`), and formatting enforcement (`ENG-048`).

The decision to adopt GitHub Flow as the branching strategy, with a single long-lived `main` branch protected by required status checks, required reviewer counts, disabled direct push, disabled force-push, and enforced linear history (Decision BD, `ENG-153` through `ENG-166`), rests on five considerations that jointly select the branching model, the enforcement mechanism, the release path, and the deferral of long-lived maintenance branches.

GitHub Flow is the de facto standard branching model for modern Rust projects of Heimdall's class. The foundational crates on which Heimdall depends — `tokio`, `rustls`, `quinn`, `hyper`, `tonic`, `serde` — all operate single-trunk workflows with short-lived feature branches merged back to the trunk through pull requests. External contributors approaching Heimdall therefore encounter an immediately familiar flow, and the repository's tooling, status-check wiring, and review routing align with the hosting platform's default behaviours without adaptation. Adopting a more complex model such as Git Flow, with its parallel `develop` and release branches, would impose an additional mental model on every contributor, would require bespoke tooling to keep the parallel branches in sync, and would diverge from the ecosystem norm against which Heimdall's workflow will be read.

The simpler structure of GitHub Flow — one long-lived branch, short-lived work branches, no parallel release or hotfix branches — requires fewer rules and less social discipline than Git Flow to maintain. The quality guarantees that Git Flow used to enforce socially through branch-specific review policies are supplied in Heimdall by the tier gates of Decision AW (`ENG-041` through `ENG-075`) and by the code-review regime of Decision AX (`ENG-076` through `ENG-089`). Those two decisions are independently justified and independently load-bearing; layering Git Flow on top of them would not add quality but would add procedural overhead. GitHub Flow allows the existing gates to do the work they are designed to do, without duplication.

Protected-branch configuration on `main` under `ENG-155` through `ENG-160` is the authoritative enforcement point for three previously decided policies: the two-reviewer and two-code-owner regime under Decision AX, the squash-merge default and the linear-history property under Decision BC (`ENG-149` and `ENG-150`), and the Tier 1 and Tier 2 gating under Decision AW. All three decisions converge on a single technical mechanism at the hosting-platform level — required reviewers, required status checks, disabled direct push, disabled force-push, enforced linear history — and the branching-strategy decision is the place where that convergence is made explicit. Without protected-branch enforcement, each of those three decisions would depend on maintainer discipline alone, which is exactly the social-enforcement posture the specification rejects elsewhere (for example, through the Tier 1 license gate under `ENG-096` and the formatter gate under `ENG-048`).

Deferring long-lived release-maintenance branches until after the 1.0.0 release under `ENG-163` avoids a premature LTS commitment. Maintaining multiple long-lived branches in parallel — a current-development trunk plus one or more maintenance lines — imposes an operational tax that is only justified once a stable 1.0 public API exists and once downstream operators have begun to depend on specific release lines. Pre-1.0.0, every released tag is expected to be superseded by a newer tag on a relatively short cycle, which makes a second long-lived branch redundant. Committing to a maintenance-branch model now would either produce unused infrastructure or force a premature 1.0.0 declaration to justify the model, neither of which is desirable. Post-1.0.0 introduction of LTS branches under a separate decision preserves full flexibility without locking the project into a heavy model today, and `ENG-164` defines a hotfix path — fast-tracked pull request against `main`, then a patch-tag — that works without any maintenance-branch scaffolding in place.

Strict trunk-based development, in which branches live for less than a day and merge to `main` multiple times a day, is a reasonable long-term target once the continuous-integration end-to-end wall-clock is very low and the contributor base is large enough that the merge rate justifies the discipline. Neither of those conditions holds for Heimdall at this stage: the Tier 1 budget targeted under `ENG-042` is approximately ten minutes, the full Tier 1-plus-Tier 2 budget is higher, and the contributor base is bootstrapping. GitHub Flow is the pragmatic starting point that delivers the essential quality guarantees today and leaves a migration path to strict trunk-based development open as the project matures. The branching-strategy decision does not preclude that migration; it positions the project to take it when the prerequisites are met.

The decision to adopt Semantic Versioning 2.0.0 as the sole versioning scheme, a release-when-ready cadence with an orientation of approximately three to six months for `MINOR` releases post-1.0.0, Tier 4 gates as the structural precondition for every release, out-of-band security-release patches coordinated with the GHSA advisory timeline, and deferral of an LTS model until after v1.0.0 (Decision BE, `ENG-167` through `ENG-185`), rests on six considerations that jointly select the scheme, the cadence, the enforcement surface, the security-release path, and the LTS deferral.

Semantic Versioning is the ecosystem norm against which Heimdall's release metadata will be read. Cargo's dependency-range notation (`^`, `~`, and caret-style defaults), `crates.io` metadata, the `rust-version` field consumed under `ENG-001` through `ENG-007`, `cargo-audit`'s vulnerability-matching logic, and the broader Rust-ecosystem discussion of Minimum Supported Rust Version all assume SemVer. A version scheme that did not convey compatibility impact — CalVer, for example, whose `YYYY.MM.DD` form signals release time but not whether an upgrade is safe — would force downstream consumers, tooling, and auditors to reconstruct compatibility impact from release notes or from code diffs, both of which are fragile and known to produce incorrect upgrade decisions. Rejecting CalVer on that basis is a deliberate trade-off: the value of unambiguous compatibility signalling outweighs the calendar-legibility advantage that CalVer would have provided.

Release-when-ready is the pragmatic cadence posture pre-1.0.0. Before the public API has been stabilised under Decision BE's own `ENG-169` preconditions, any attempt to release on a fixed calendar would either force arbitrary breaking bumps (to ship a `MINOR` that has not stabilised) or defer obviously-finished work indefinitely (to wait out the calendar). Neither outcome serves operators or contributors. The three-to-six-month orientation fixed by `ENG-174` post-1.0.0 reflects the point at which a cadence hint becomes useful: downstream operators planning upgrade windows benefit from a directional signal about how often feature releases are expected, but a strict deadline would recreate the pre-1.0.0 failure mode in a different form. The orientation is therefore a `SHOULD`, not a `MUST`, and the concrete value within the range remains an open question for later refinement when actual release history provides empirical grounding.

Tier 4 gates under `ENG-176` make release a structurally-auditable event. A release tag that is produced outside the Tier 4 gate is indistinguishable from a compromised or careless release, and the whole supply-chain posture fixed by `THREAT-010` through `THREAT-014` in [`007-threat-model.md`](007-threat-model.md) depends on that distinction being mechanical rather than social. The gate composition — every Tier 1, Tier 2, and Tier 3 check green; signed artefacts per `THREAT-013`; Software Bill of Materials per `THREAT-014`; release notes with Minimum Supported Rust Version per `ENG-005` and `ENG-006` — is the same composition already decided under Decision AW (`ENG-041` through `ENG-075`), applied at release boundary without carve-outs. Reiterating that composition in the release decision rather than merely cross-referencing it ensures that the release-time obligations are visible in the release specification itself, and that a reader of Decision BE does not need to reconstruct them from three other decisions.

Security-release urgency is structurally different from feature-release cadence, and decoupling them under `ENG-172`, `ENG-179`, and `ENG-180` is deliberate. A fixed release calendar aligned with `MINOR` cadence would force security fixes to wait for the next scheduled window, during which the vulnerability would either remain under embargo with known-to-be-fixed code accumulating in `main`, or would be disclosed without a released fix available. Both outcomes fail operators. Permitting an out-of-band `PATCH` release at any time, coordinating its publication with the GHSA advisory timeline fixed by `ENG-131` through `ENG-135`, and requiring the same CVE identifier across the release tag, the artefact signatures, the Software Bill of Materials, and the published advisory, produces a release whose relationship to the vulnerability is machine-verifiable after the fact. The audit trail from disclosure to fix is unambiguous, which is the property operators need to decide whether to upgrade and auditors need to verify that the fix was delivered.

Deferring a dedicated LTS maintenance model until after v1.0.0 under `ENG-181` avoids a premature operational tax and is consistent with the LTS deferral already fixed under `ENG-163` for branching. Pre-1.0.0, every released tag is expected to be superseded by a newer tag on a relatively short cycle, which makes an LTS commitment redundant. Committing to an LTS model now would either produce unused maintenance infrastructure or force a premature 1.0.0 declaration to justify it, neither of which is desirable. Post-1.0.0 introduction of an LTS model under a separate decision preserves full flexibility without locking the project into a heavy maintenance posture today, and the security-release path fixed by `ENG-172` and `ENG-179` works without any LTS scaffolding in place.

The distribution-channel obligations under `ENG-182` through `ENG-184` tie the release decision to the artefact formats already fixed under Decision AR in [`009-target-environment.md`](009-target-environment.md). GitHub Releases as the authoritative channel is the location where the signed artefacts mandated by that decision and by `THREAT-013` land; `crates.io` and container registries, where used, are additional channels that must inherit the same Tier 4 gate so that the release integrity envelope is preserved end-to-end. Without that inheritance, a library crate pushed to `crates.io` or a container image pushed to a registry outside the Tier 4 gate would be a supply-chain blind spot for consumers that install Heimdall through those channels. Applying Tier 4 uniformly across every publication surface closes that gap structurally, and the uniform scope clause under `ENG-185` prevents new artefact formats added in the future from silently escaping the gate.
