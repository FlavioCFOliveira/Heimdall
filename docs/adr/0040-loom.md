# ADR-0040: loom for concurrency model-checking

## Context

ENG-056 mandates exhaustive concurrency model-checking on all concurrency-sensitive code
in Heimdall. Sprint 17 task #233 requires loom tests covering:

- The drain counter (`AtomicUsize` + `AtomicBool`) — verifying that the in-flight count
  is always consistent across concurrent acquire/release and drain initiation.
- The ArcSwap-like atomic pointer access pattern — verifying that concurrent readers and
  a swapper never observe an inconsistent state.

Stress-testing (running concurrent code in loops) finds bugs probabilistically but cannot
prove absence of data races on all memory-ordering interleavings. Loom performs exhaustive
state-space exploration: it replaces `std::sync::atomic`, `std::thread`, and
`std::sync::Mutex` with controlled implementations that enumerate all valid memory
orderings, detecting data races and invariant violations that stress tests would miss.

## Decision

Use **`loom` v0.7.2** as a `[dev-dependencies]` entry in `crates/heimdall-runtime`.

Loom tests are activated only when `RUSTFLAGS='--cfg loom'` is passed, which substitutes
loom's controlled implementations for the standard library's concurrency primitives. This
flag is set only in the dedicated Tier 2/3 CI job (`ci-tier2.yml`, `loom` step); normal
`cargo test` and Tier 1 CI do not run loom tests.

## Considered options

### Option A — `loom` v0.7.2 (selected)

- Pros: developed and maintained by the Tokio team; the state-of-the-art tool for
  exhaustive Rust concurrency testing; MIT licence; no runtime impact (dev-only);
  supports futures via the `futures` feature (required for async drain tests).
- Cons: significant `unsafe` internally (it replaces atomics and threads with controlled
  proxies, which is inherently `unsafe`); slow on models with large state spaces (must be
  scoped to small, focused models); MIT only (not dual-licensed).

### Option B — ThreadSanitizer (TSan)

- Pros: catches real races at runtime; integrates with `cargo test`.
- Cons: requires nightly (`-Zsanitizer=thread`); probabilistic (does not enumerate all
  interleavings); higher false-negative rate than loom for subtle orderings. Not a
  replacement for exhaustive checking.

### Option C — stress-test loops

- Pros: zero dependencies; simple to write.
- Cons: probabilistic; provides no coverage guarantee; would not satisfy ENG-056.
  Rejected as the sole concurrency validation strategy.

### Option D — `trybuild`

- Cons: compile-error testing; unrelated to concurrency correctness. Not applicable.

## Classification

**Convenience dev-dependency** (test infrastructure only). Has no effect on the compiled
runtime binary. Cannot be removed without violating ENG-056.

## Unsafe code posture

Loom's `unsafe` is intentional: to replace standard library atomics and thread
primitives with controlled proxies, loom must use `unsafe` to intercept and schedule
operations. This unsafe lives entirely within the `loom` crate and is active only when
`--cfg loom` is set (i.e., never in production builds).

Heimdall's loom test code is written against loom's `loom::sync::atomic` and
`loom::thread` APIs (which mirror `std::sync::atomic` and `std::thread`), not against
any custom unsafe. The `#![deny(unsafe_code)]` attribute applies to Heimdall source; the
loom test module is subject to the same constraint.

## Consequences

- `loom` with `features = ["futures"]` is added under `[dev-dependencies]` in
  `crates/heimdall-runtime/Cargo.toml`.
- Loom tests are placed in `crates/heimdall-runtime/tests/loom_tests.rs` under
  `#[cfg(loom)]` guards.
- The Tier 2 CI job runs: `RUSTFLAGS='--cfg loom' cargo test -p heimdall-runtime
  --test loom_tests`.
- Normal `cargo test -p heimdall-runtime` does NOT execute loom tests.
