# Runtime operations

**Purpose.** This document defines Heimdall's runtime-operations policy. It fixes the decisions that govern how running state is mutated without restarting the process, including the `SIGHUP`-triggered full reload of the configuration file and of all zone files, the admin-RPC surface for granular runtime operations that do not map naturally to a configuration-file edit, the atomic-swap implementation requirement that governs both mechanisms, and the scope decisions that keep alternative triggers out of the primary mechanism set at this stage. It does not redefine which roles Heimdall supports, which transports are instantiated per role, or how DNSSEC validation and policy-zone evaluation work; those questions are settled in [`001-server-roles.md`](001-server-roles.md), [`002-transports.md`](002-transports.md), [`005-dnssec-policy.md`](005-dnssec-policy.md), and [`011-rpz.md`](011-rpz.md) respectively.

**Status.** Stable.

**Requirement category.** `OPS`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md). For the cryptographic policy whose TLS-ticket-encryption-key and QUIC `NEW_TOKEN` key rotation obligations are mutation targets of the admin-RPC surface, see [`003-crypto-policy.md`](003-crypto-policy.md). For the DNSSEC policy whose Negative Trust Anchor lifecycle is a mutation target of the admin-RPC surface, see [`005-dnssec-policy.md`](005-dnssec-policy.md). For the protocol-conformance policy whose hot-reload open question under Decision AA is partially resolved by this document, see [`006-protocol-conformance.md`](006-protocol-conformance.md). For the threat-model policy whose ACL, rate-limiting, and structured-event taxonomy are referenced by the admin-RPC surface, see [`007-threat-model.md`](007-threat-model.md). For the RPZ policy whose policy-zone entries are a mutation target of the admin-RPC surface, see [`011-rpz.md`](011-rpz.md).

## 1. Scope

This document applies to all three DNS server roles supported by Heimdall — the authoritative server role, the recursive resolver role, and the forwarder role, as defined in [`001-server-roles.md`](001-server-roles.md) — and to every running configuration of a Heimdall instance, regardless of which combination of roles is active. The requirements govern the mutation of running state performed through the two complementary mechanisms fixed by this document: the `SIGHUP`-triggered full reload in section 2 and the admin-RPC surface in section 3. They do not govern the initial process start, which is outside the scope of this document.

## 2. SIGHUP full reload

### 2.1 Normative requirements

- **OPS-001.** Heimdall MUST reload its entire configuration file and all zone files on receipt of the `SIGHUP` signal. The reload MUST encompass every input file that contributes to the running state, without distinction between the main configuration file and the zone files it references.
- **OPS-002.** The reload required by `OPS-001` MUST be atomic from the perspective of in-flight queries. Every new query arriving at Heimdall after the swap MUST see the new running state. Every query already dispatched against the old running state at the moment of the swap MUST complete against the old running state. Partial-state exposure, in which a single query observes a mixture of old and new state, MUST NOT be permitted.
- **OPS-003.** When the new configuration file or any zone file referenced by it fails to parse or fails to validate, the reload required by `OPS-001` MUST be rejected in its entirety. The previously running state MUST be preserved unchanged, and no partial update MUST be applied. Heimdall MUST log the failure with sufficient context — including the offending file, the parse or validation error, and the line or location where available — for the operator to diagnose the cause.
- **OPS-004.** The reload required by `OPS-001` MUST NOT tear down running listeners. Running TCP connections, DoT sessions, DoH sessions on HTTP/2 and HTTP/3, and DoQ QUIC sessions MUST remain bound to the same sockets across the reload and MUST NOT be closed as a side effect of the reload. The sockets themselves MUST remain bound to the same addresses and ports across the reload.
- **OPS-005.** The outcome of every reload triggered by `SIGHUP` — whether accepted and applied, or rejected under `OPS-003` — MUST be emitted as a structured audit event consistent with `THREAT-080` in [`007-threat-model.md`](007-threat-model.md).
- **OPS-006.** `SIGHUP` MUST be the primary signal-based trigger for the full reload defined by `OPS-001` through `OPS-005`. Heimdall MUST NOT expose any DNS-based signalling — including a special query name that triggers a reload — as a reload trigger.

## 3. Admin RPC

### 3.1 Normative requirements

- **OPS-007.** Heimdall MUST expose an admin-RPC surface for granular runtime operations on the running state that do not map naturally to a configuration-file edit. The admin-RPC surface MUST be distinct from the `SIGHUP` full-reload mechanism fixed by section 2 and MUST be usable independently of it.
- **OPS-008.** The admin-RPC surface MUST bind by default to a local Unix domain socket. Authentication on the default binding MUST be provided by the filesystem permissions of the Unix domain socket; no additional authentication layer is required on the default binding.
- **OPS-009.** Heimdall MAY expose the admin-RPC surface over a TCP binding in addition to, or in place of, the default Unix domain socket, when explicitly enabled by configuration. When a TCP binding is enabled, Heimdall MUST require mutual TLS authentication in accordance with `SEC-012` through `SEC-016` in [`003-crypto-policy.md`](003-crypto-policy.md) on TLS-over-TCP, or the equivalent mutual TLS authentication at the TLS-1.3-inside-QUIC layer in accordance with `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md) on QUIC, and MUST subject every admitted request to ACL evaluation in accordance with `THREAT-033` through `THREAT-047` in [`007-threat-model.md`](007-threat-model.md).
- **OPS-010.** The admin-RPC surface MUST expose, at minimum, the capability to add a zone to the running state at runtime, the capability to remove a zone from the running state at runtime, and the capability to reload a single zone without touching any other zone or the configuration as a whole. The exact protocol encoding of these capabilities is tracked as open in section 8.
- **OPS-011.** The admin-RPC surface MUST expose, at minimum, the capability to register a Negative Trust Anchor, the capability to revoke a Negative Trust Anchor, and the capability to list the currently registered Negative Trust Anchors, consistent with `DNSSEC-017` in [`005-dnssec-policy.md`](005-dnssec-policy.md). The exact protocol encoding of these capabilities is tracked as open in section 8.
- **OPS-012.** The admin-RPC surface MUST expose, at minimum, the capability to rotate the TLS ticket-encryption key as a runtime operation, consistent with the open question on TLS session-ticket operational defaults tracked in section 10 of [`003-crypto-policy.md`](003-crypto-policy.md), and the capability to rotate the QUIC `NEW_TOKEN` key as a runtime operation, consistent with the open question on QUIC `NEW_TOKEN` operational defaults tracked in section 10 of [`003-crypto-policy.md`](003-crypto-policy.md). The exact protocol encoding of these capabilities is tracked as open in section 8.
- **OPS-013.** The admin-RPC surface MUST expose, at minimum, the capability to tune rate-limit parameters at runtime, within the bounds permitted by the operator's configuration, consistent with `THREAT-048` through `THREAT-060` in [`007-threat-model.md`](007-threat-model.md). The exact protocol encoding and the expression of the permitted bounds are tracked as open in section 8.
- **OPS-014.** The admin-RPC surface MUST expose, at minimum, the capability to trigger a controlled drain of the running instance. A controlled drain MUST stop accepting new connections, MUST allow in-flight work to complete, and MUST then exit cleanly. The exact protocol encoding of this capability, together with the semantics of its bounded-wait behaviour if in-flight work does not complete within a configured window, is tracked as open in section 8.
- **OPS-015.** The admin-RPC surface MUST expose, at minimum, the capability to query diagnostic state, including cache statistics, the set of active connections per transport, and validation traces for a specific query. The admin-RPC surface MUST also expose, at minimum, the capability to add a Response Policy Zone entry at runtime and the capability to remove a Response Policy Zone entry at runtime, consistent with `RPZ-016` through `RPZ-020` in [`011-rpz.md`](011-rpz.md). Every admin-RPC operation MUST emit a structured audit event consistent with `THREAT-080` in [`007-threat-model.md`](007-threat-model.md). The exact protocol encoding, the depth of the diagnostic-state exposure, and the audit-event format are tracked as open in section 8.

## 4. Atomic-swap implementation

### 4.1 Normative requirements

- **OPS-016.** The running state that the `SIGHUP` full reload under section 2 swaps MUST be held behind a single atomic pointer. The standard Rust pattern for this property — `ArcSwap` or equivalent — satisfies the requirement; the concrete primitive is an implementation detail. Queries on the hot path MUST follow the atomic pointer at entry and MUST retain the snapshot they observed across the completion of the query.
- **OPS-017.** Partial updates of the running state during the swap required by `OPS-016` MUST NOT be permitted. The swap MUST expose exactly two observable states to any in-flight query: the pre-swap state and the post-swap state. No observer MUST be able to observe a mixed state composed of fields from both.
- **OPS-018.** The atomic-swap implementation required by `OPS-016` and `OPS-017` MUST satisfy the atomicity obligation fixed by `OPS-002` for the `SIGHUP` full reload. Admin-RPC operations under section 3 that mutate running state MUST either reuse the same atomic-swap mechanism or MUST provide an equivalent guarantee on the subset of state they mutate.

## 5. HTTP observability endpoint

### 5.1 Normative requirements

- **OPS-021.** Heimdall MUST expose an HTTP observability endpoint, distinct from both the admin-RPC surface fixed by section 3 and from every DoH listener instantiated under [`002-transports.md`](002-transports.md). The observability endpoint MUST NOT share a TCP port with any DoH transport.
- **OPS-022.** The HTTP observability endpoint required by `OPS-021` MUST expose, at minimum, the following four endpoints over HTTP `GET`: `/healthz` as fixed by `OPS-023`, `/readyz` as fixed by `OPS-024`, `/metrics` as fixed by `OPS-025`, and `/version` as fixed by `OPS-026`.
- **OPS-023.** The `GET /healthz` endpoint required by `OPS-022` MUST return HTTP status 200 OK whenever the Heimdall process is running and capable of serving queries in principle. This endpoint is the liveness probe intended for integration with Kubernetes `livenessProbe`, `systemd` watchdog, Docker `HEALTHCHECK`, and equivalent mechanisms. The endpoint MUST NOT evaluate per-role readiness conditions; role-specific readiness is the responsibility of `OPS-024`.
- **OPS-024.** The `GET /readyz` endpoint required by `OPS-022` MUST return HTTP status 200 OK only when the instance is ready to serve. Readiness MUST require, at minimum, that every configured listener is bound; that the DNSSEC trust anchor is initialised when a validating role is active; that zones are loaded when the authoritative role is active; and that configured upstreams are reachable when the forwarder role is active, subject to a grace window for transient upstream failures. The exact set of readiness predicates per role combination, including the duration of the upstream-reachability grace window, is tracked as open in section 8.
- **OPS-025.** The `GET /metrics` endpoint required by `OPS-022` MUST expose metrics in the OpenMetrics format compatible with Prometheus scraping, in accordance with `THREAT-083` in [`007-threat-model.md`](007-threat-model.md).
- **OPS-026.** The `GET /version` endpoint required by `OPS-022` MUST return build and runtime metadata as a JSON object, including at minimum the project version, the git commit SHA of the build, the build timestamp, the Minimum Supported Rust Version in effect for the build, the set of transports compiled in, the set of roles compiled in, and a reference to the hash of the Software Bill of Materials associated with the build. The concrete output granularity, including whether the exact git commit SHA is returned on an unauthenticated response given the fingerprinting trade-off, is tracked as open in section 8.
- **OPS-027.** The HTTP observability endpoint required by `OPS-021` MUST bind by default to `127.0.0.1` (loopback only). Binding the endpoint to a broader interface MUST require an explicit operator configuration choice. The concrete default TCP port for the endpoint is an operational parameter; its concrete value is tracked as open in section 8.
- **OPS-028.** The HTTP observability endpoint required by `OPS-021` MAY be served unauthenticated when bound to a loopback-only address. When the endpoint is bound to a non-loopback interface, Heimdall MUST require mutual TLS authentication in accordance with `SEC-012` through `SEC-016` in [`003-crypto-policy.md`](003-crypto-policy.md), and MUST subject every admitted request to ACL evaluation in accordance with `THREAT-033` through `THREAT-047` in [`007-threat-model.md`](007-threat-model.md).
- **OPS-029.** The HTTP observability endpoint required by `OPS-021` MUST be read-only. It MUST NOT expose any mutation or control operation. All mutation and control operations MUST be routed through the admin-RPC surface fixed by `OPS-007` through `OPS-015`.
- **OPS-030.** Debug and profiling endpoints — including `/debug/pprof`-style runtime profiling endpoints and flamegraph endpoints — are OPTIONAL, MUST be opt-in by explicit configuration, and are outside the scope of this document. When not explicitly enabled, such endpoints MUST NOT be exposed by Heimdall. Any future introduction of such endpoints is tracked as open in section 8.
- **OPS-031.** DNS-based health probing — for example, a special query name that returns status as a `TXT` record — is OPTIONAL and MAY be added later as a complement to the HTTP observability endpoint fixed by `OPS-021` through `OPS-029`. A DNS-based health probe MUST NOT replace the HTTP observability endpoint as the primary health interface. Any such complement is tracked as open in section 8 and is consistent with the exclusion of DNS-based signalling as a reload trigger under `OPS-020`.
- **OPS-032.** When Heimdall runs under `systemd`, Heimdall SHOULD call `sd_notify(READY=1)` upon reaching the readiness condition evaluated by `OPS-024`, and SHOULD participate in `systemd` `WatchdogSec`-based liveness when the service unit configures one. The exact `sd_notify` payload set, including the `READY`, `STATUS`, and watchdog-keepalive intervals, is tracked as open in section 8.

## 6. Scope and exclusions

### 6.1 Normative requirements

- **OPS-019.** Filesystem-watch notification — including `inotify` on Linux and `kqueue` on BSD and macOS — MUST NOT be treated as a primary trigger for the full reload fixed by section 2 at this stage. A filesystem-watch complement MAY be added later as an opt-in trigger, and when added, it MUST NOT replace the `SIGHUP` mechanism fixed by `OPS-001` through `OPS-006`.
- **OPS-020.** DNS-based signalling — including a special query name that triggers a reload — MUST NOT be implemented as a reload trigger. This exclusion is consistent with `OPS-006` and MUST NOT be re-opened without a joint revision of this document and of [`006-protocol-conformance.md`](006-protocol-conformance.md).

## 7. Rationale

Supporting `SIGHUP` full reload together with a dedicated admin-RPC surface is the operational pattern that every mature DNS server in 2026 exposes. BIND exposes `rndc`, Unbound exposes `unbound-control`, Knot exposes `knotc`, and PowerDNS exposes `pdns_control` and `pdnsutil`; in each case, a signal-triggered full reload sits alongside a dedicated control channel for granular operations. Operators arriving at Heimdall from any of these ecosystems expect the pair. A runtime-operations surface that offered only one of the two mechanisms would restrict Heimdall's applicability in deployments whose operational tooling is already wired to the deployed baseline, and would force operators to either rewrite their tooling or run Heimdall outside the conventions the rest of the ecosystem has converged on.

`SIGHUP` is universally scriptable from `systemd`, from `init.d` scripts, from container orchestrators, and from process supervisors. Every Unix-adjacent platform Heimdall targets under [`009-target-environment.md`](009-target-environment.md) delivers `SIGHUP` through the standard signal-handling interface, and every process-management tool that operators reach for already emits `SIGHUP` as a first-class operation. Fixing `SIGHUP` as the primary signal-based trigger therefore incurs no additional integration cost on the operator side. The alternative of fixing a bespoke reload mechanism as the primary trigger would require operators to adapt their existing tooling to Heimdall, which is the inverse of the ecosystem-alignment posture already adopted for other operational surfaces.

The admin-RPC surface covers the state mutations that do not map naturally to a file-based configuration. Negative Trust Anchors under `DNSSEC-016` through `DNSSEC-018` in [`005-dnssec-policy.md`](005-dnssec-policy.md) are inherently runtime state: each one carries a mandatory expiry and a lifecycle that is intentionally time-bounded, and representing their registration and revocation as configuration-file edits would force the operator to choose between a file that drifts away from the running state or a continuous reload loop that mutates the file at every registration and expiry event. Incident-response zone injections — for example, adding a sinkhole zone in response to an ongoing attack, or adding a policy-zone entry under `RPZ-016` in [`011-rpz.md`](011-rpz.md) — must happen on the timescale of the incident, not on the timescale of a configuration-deployment pipeline that may include human review, change-management checkpoints, and staged rollout. Introspection for debugging — cache statistics, active connections, validation traces for a specific query — cannot live in a configuration file at all, because it is observation rather than mutation. Covering each of these categories through a single admin-RPC surface keeps the mutation-and-introspection interface unified and removes a class of latent drift between the running state and whatever file-based view might otherwise be asked to represent it.

The authentication model on the admin-RPC surface is structural rather than bolt-on. The Unix-domain-socket default under `OPS-008` confines access to local operators by construction, because filesystem permissions on the socket are enforced by the kernel, with no cryptographic negotiation on the hot path and no additional authentication code to audit. The optional TCP binding under `OPS-009` extends access to controlled remote scenarios — a centralised operations endpoint, a Kubernetes operator, a managed-DNS control plane — and does so by reusing the mutual TLS infrastructure already fixed for the DNS transports by `SEC-012` through `SEC-016` and `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md) and the ACL matrix already fixed by `THREAT-033` through `THREAT-047` in [`007-threat-model.md`](007-threat-model.md). No new cryptographic primitive and no new authorisation vocabulary is introduced; the admin-RPC surface participates in the same trust and admission model as the DNS transports.

Atomic swap via a single shared pointer is the standard Rust pattern for zero-downtime reconfiguration. Crates such as `arc-swap` provide the pointer-swap primitive with the memory-ordering guarantees required by the atomicity obligation fixed by `OPS-002` and `OPS-016`. Queries on the hot path read the pointer once at entry, retain the snapshot across the completion of the query, and therefore never observe a mixed state composed of pre-swap and post-swap fields; queries arriving after the swap read the new pointer and see the new state. This property is obtained without locking the hot path: the pointer read is a single atomic load, which is the dominant performance characteristic required by the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md). A lock-based alternative — where a reader-writer lock would be held across each query — would reintroduce contention between the hot path and the reload path, and would degrade throughput measurably under reload even when reloads are rare.

Rejecting the entire reload on any parse or validation failure under `OPS-003` applies the fail-safe principle fixed by [`../CLAUDE.md`](../CLAUDE.md) to the reload surface itself. A partial reload — in which some files are applied and others are skipped because they failed to parse — would expose the operator to a silent divergence between the intended state and the served state, which is the exact class of misconfiguration that a specification-level discipline exists to prevent. Preserving the previously running state on any failure keeps the served state known and recoverable, and shifts the cost of a reload failure onto the reload attempt rather than onto the clients being served.

Preserving running listeners across the reload under `OPS-004` is required by the zero-disruption obligation. Tearing down running TCP, DoT, DoH on HTTP/2 and HTTP/3, or DoQ connections as a side effect of a reload would force every connected client to re-establish its transport state, which means a fresh TLS or QUIC handshake on encrypted transports, renegotiation of session tickets, and loss of the edns-tcp-keepalive state fixed by `PROTO-014` in [`006-protocol-conformance.md`](006-protocol-conformance.md). None of that is necessary: the sockets are bound to the same addresses and ports, and the running transport state is orthogonal to the running configuration and zone state. Reload therefore touches what it must and nothing else.

Filesystem-watch notifications as a primary trigger for reload are fragile across the combinations of editor behaviour and atomic-rename semantics that a zone-file deployment pipeline typically uses. A file written through a temporary-file-plus-rename sequence emits different events on Linux `inotify` than on BSD `kqueue`; a file written through in-place truncation emits yet another set of events; network filesystems add further divergence. Treating filesystem-watch as the primary trigger would therefore couple the correctness of the reload mechanism to the specific write pattern used by each operator, which is exactly the class of deployment-specific fragility that a specification-level discipline exists to prevent. Leaving `SIGHUP` as the primary trigger, and allowing filesystem-watch as an opt-in complement later under `OPS-019`, keeps the primary mechanism robust while not foreclosing the opt-in improvement.

DNS-based signalling as a reload trigger, excluded by `OPS-006` and `OPS-020`, is rejected because it would couple a control-plane operation to the data-plane protocol that the server is serving. A special query name that triggered a reload would be subject to the same ACL matrix, rate-limiting, and cache pipeline as every other DNS query, which would either weaken the admission posture on the reload operation or would introduce a carved-out exemption on the hot path that every other part of the specification has been careful to avoid. The admin-RPC surface under section 3 is the correct home for control-plane operations, and it already covers the remote-trigger use case through its TCP binding under `OPS-009`.

The HTTP observability endpoint fixed by `OPS-021` through `OPS-032` in section 5 is required at the specification level for six reinforcing reasons. First, ecosystem alignment: Kubernetes `livenessProbe` and `readinessProbe`, Docker `HEALTHCHECK`, AWS Elastic Load Balancing target groups, Google Cloud Run health checks, and Prometheus scraping are all HTTP-native. The ecosystem tooling that operators deploy around Heimdall expects HTTP `GET` on well-known endpoints; anything else requires bridges and introduces operational friction. Second, format alignment with the observability decisions already taken: the OpenMetrics format required by `THREAT-083` in [`007-threat-model.md`](007-threat-model.md) is an HTTP-scraped format, and exposing it on a dedicated HTTP listener is the natural home rather than an adapter over some other transport. Third, clean separation of concerns: the observability surface is read-only and in the common Kubernetes pattern runs unauthenticated on a trusted internal network, whereas the admin-RPC surface fixed by `OPS-007` through `OPS-015` is mutating and always authenticated. Mixing the two on a single HTTP surface would create a risk of an accidentally-exposed control endpoint under a permissive authentication posture; keeping them distinct makes the authentication model auditable by construction. Fourth, safe default binding: the loopback-only default under `OPS-027` reflects the Kubernetes pod-local scraping pattern and prevents accidental exposure of internal state to untrusted networks; external exposure is made an explicit operator choice with mandatory mTLS and ACL evaluation under `OPS-028`. Fifth, port-namespace separation from DoH: fixing the observability endpoint on a listener distinct from every DoH listener under `OPS-021` prevents path-namespace collisions between operator-facing URIs such as `/metrics` and `/readyz` and DoH endpoints such as `/dns-query`, and simplifies host-based firewall rules that discriminate data-plane from control-plane traffic on different TCP ports. Sixth, complementarity with DNS-based health probing: DNS-based health probing is not standard in the surrounding ecosystem, and the HTTP endpoint is what operators actually reach for; allowing a DNS-based complement under `OPS-031` while refusing to let it replace the HTTP endpoint matches how operators actually integrate health signals with orchestrators and monitoring systems. The `systemd` integration fixed by `OPS-032` extends the readiness semantics already fixed by `OPS-024` to platforms that consume `sd_notify` rather than HTTP polling, without introducing a new readiness model: the same readiness condition drives both surfaces.

## 8. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will be specified incrementally in subsequent revisions of this document.

- **Admin RPC protocol encoding.** The concrete wire-format encoding of the admin-RPC surface fixed by `OPS-007` through `OPS-015` — gRPC over Protocol Buffers on the Unix domain socket, HTTP plus JSON, a custom text protocol in the style of BIND's `rndc`, or another encoding — is **to be specified**.
- **Exact command set and argument shape per capability.** The exact command names, argument shapes, response shapes, and error vocabulary for each capability fixed by `OPS-010` through `OPS-015` are **to be specified**.
- **Zone-file input-path atomicity expectations.** The atomicity expectations that Heimdall places on the input side of a zone-file reload — whether a temporary-file-plus-rename sequence is required for operators providing zone-file content, and whether `fsync` on the directory is required before a reload is triggered — are **to be specified**.
- **Config-validation strictness.** The strictness model applied to the configuration-file reload under `OPS-003` — specifically, whether the entire reload is rejected on any error across all files, or whether a per-zone apply-and-log-rejected model is permitted for the zone-file subset of the reload while the main configuration is still rejected atomically — is **to be specified**.
- **In-flight query handling policy during swap.** The precise in-flight handling policy under `OPS-002` — whether in-flight queries always complete against the old state, or whether certain classes of in-flight queries are instead forced to retry against the new state — is **to be specified**.
- **Introspection surface depth.** The depth of the introspection surface exposed under `OPS-015` — how much internal state is exposed via the admin RPC, including whether full cache contents are exposed, whether active validations are exposed, and whether connection-pool internals are exposed — is **to be specified**.
- **Reload idempotency guarantees.** The idempotency guarantees that Heimdall offers across repeated reload attempts under `OPS-001` and across repeated admin-RPC mutations under section 3 — specifically, whether a second reload triggered before the first has committed is coalesced, queued, or rejected — are **to be specified**.
- **Audit-log format for admin-RPC operations.** The concrete audit-log format for admin-RPC operations under `OPS-015`, keyed to the structured-event taxonomy fixed by `THREAT-080` in [`007-threat-model.md`](007-threat-model.md), including per-operation field sets and the identification of the operator responsible for each mutation, is **to be specified**.
- **Default TCP port for the HTTP observability endpoint.** The concrete default TCP port assigned to the HTTP observability endpoint fixed by `OPS-021` through `OPS-027` is **to be specified**.
- **Readiness predicates per role combination.** The exact set of readiness predicates applied by `GET /readyz` under `OPS-024` for each supported role combination across the authoritative, recursive, and forwarder roles defined in [`001-server-roles.md`](001-server-roles.md), together with the duration of the upstream-reachability grace window for the forwarder role, is **to be specified**.
- **`/version` output granularity.** The concrete output granularity of the `GET /version` endpoint under `OPS-026`, and in particular whether the exact git commit SHA is returned on an unauthenticated response in light of the fingerprinting trade-off, is **to be specified**.
- **Rate limiting of the HTTP observability endpoint.** The rate-limiting posture applied to the HTTP observability endpoint fixed by `OPS-021` through `OPS-029` to prevent scrape-based denial-of-service amplification, including whether the rate-limiting family fixed by `THREAT-048` through `THREAT-060` in [`007-threat-model.md`](007-threat-model.md) applies uniformly to this endpoint or whether a dedicated scrape-oriented mechanism is required, is **to be specified**.
- **`sd_notify` payload set.** The exact `sd_notify` payload set emitted by Heimdall under `OPS-032`, including the `READY`, `STATUS`, and `WATCHDOG` keepalive intervals and the mapping between internal readiness transitions and payload emissions, is **to be specified**.
- **Debug and profiling endpoints.** The concrete future introduction of debug and profiling endpoints — `/debug/pprof`-style runtime profiling endpoints, flamegraph endpoints, and equivalents — whose exclusion from the default scope is fixed by `OPS-030`, is **to be specified** as a separate narrower decision if pursued.
- **OpenAPI specification for the HTTP endpoints.** The OpenAPI specification document describing the HTTP observability endpoints fixed by `OPS-021` through `OPS-029`, including whether it is auto-generated from the implementation or hand-maintained and its location in the repository, is **to be specified**.
- **DNS-based health probe as an optional complement.** The concrete design of the optional DNS-based health probe allowed by `OPS-031` as a complement to the HTTP observability endpoint, including the reserved query name, the record type of the status response, the applicable ACL posture, and the interaction with the exclusion of DNS-based signalling under `OPS-006` and `OPS-020`, is **to be specified** if pursued.

No implementation activity may proceed on the basis of assumptions about any of the items above.
