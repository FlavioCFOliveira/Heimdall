# Performance targets

**Purpose.** This document defines the framework within which Heimdall's quantitative performance targets are expressed, measured, and enforced. It fixes the per-cell matrix that organises the targets, the dimensions quantified for each cell, the reference hardware baselines against which the targets are evaluated, the measurement methodology and continuous-integration enforcement mechanism, and the comparative ambition level that Heimdall adopts against state-of-the-art reference implementations (parity on plain DNS, exceed on encrypted transports). The concrete numerical values of the targets are deferred to the first implementation baseline and are tracked as open questions in section 7.

**Status.** Stable.

**Requirement category.** `PERF`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model these requirements depend on, see [`001-server-roles.md`](001-server-roles.md). For the transport model these requirements depend on, see [`002-transports.md`](002-transports.md). For the resource-limits and admission-control family whose caps interact with the performance targets, see [`007-threat-model.md`](007-threat-model.md), in particular the requirements `THREAT-061` through `THREAT-078`.

## 1. Framework and per-cell matrix

- **PERF-001.** Heimdall MUST define its quantitative performance targets on a per-cell basis, where a cell is the pair `(role, transport)` formed by one of the three roles defined in [`001-server-roles.md`](001-server-roles.md) and one of the transport profiles enumerated in `PERF-002`. Each cell in the matrix MUST have its own, independent set of numerical targets.
- **PERF-002.** For the purposes of the per-cell matrix, the transport axis MUST enumerate the following six transport profiles: DNS classic over UDP on port 53; DNS classic over TCP on port 53; DNS-over-TLS on port 853; DNS-over-HTTPS served over HTTP/2; DNS-over-HTTPS served over HTTP/3; DNS-over-QUIC on port 853. DNS-over-HTTPS over HTTP/2 and DNS-over-HTTPS over HTTP/3 MUST be treated as two distinct cells and MUST NOT be collapsed into a single DoH cell, because their hot-path profiles differ materially.
- **PERF-003.** The cells of the matrix MUST span every combination of role and transport profile defined in `PERF-001` and `PERF-002`, consistent with the `(role, transport)` matrix made supported at the software level by `NET-011` in [`002-transports.md`](002-transports.md). The set of cells exists at the software level: every cell has targets that the code MUST achieve when that cell is activated, independent of whether any particular deployment configuration activates that cell.
- **PERF-004.** The numerical values of the per-cell targets are deferred to the first implementation baseline and MUST NOT be assumed before they are specified. They are tracked as open questions in section 7.

## 2. Dimensions quantified per cell

- **PERF-005.** Each cell defined under `PERF-001` MUST have numerical targets in at least the following dimensions: sustainable queries per second (QPS); response latency at the p50, p99, and p99.9 percentiles; memory footprint, quantified both per cache and per process; and concurrent connection count. Additional dimensions MAY be added for specific cells where justified, but the dimensions listed above MUST be present for every cell.
- **PERF-006.** Sustainable QPS under `PERF-005` MUST be interpreted as the query rate the cell can sustain indefinitely under steady-state conditions without degradation of the latency, memory, or connection-count targets of the same cell. Peak burst rates MUST NOT be substituted for sustainable QPS.
- **PERF-007.** Latency percentiles under `PERF-005` MUST be measured end-to-end from the point at which the query is observable on the listening socket to the point at which the response is handed back to the operating system for transmission. Percentiles MUST be computed over the measurement window and MUST NOT be approximated by averages.
- **PERF-008.** Memory-footprint targets under `PERF-005` MUST be expressed separately for each cache that belongs to the cell's role, consistent with the segregated-cache model established in [`004-cache-policy.md`](004-cache-policy.md), and for the overall process resident set. Per-cache targets and per-process targets MUST be enforced independently.
- **PERF-009.** The concurrent-connection-count targets under `PERF-005` MUST be compatible with the per-listener and per-source admission caps established by `THREAT-061` through `THREAT-078` in [`007-threat-model.md`](007-threat-model.md). In the event of conflict between a performance target and an enforced admission cap, the admission cap prevails, and the performance target for the affected cell MUST be reconciled accordingly.

## 3. Reference hardware

- **PERF-010.** Performance targets under `PERF-005` MUST be expressed for three parallel reference hardware baselines: an `x86_64` modern baseline, an `aarch64` modern baseline, and a `riscv64` modern baseline, consistent with the tri-architecture production scope fixed by `ENV-015` and `ENV-020` in [`009-target-environment.md`](009-target-environment.md). Heimdall MUST be a first-class citizen on all three architectures; no target MUST be defined on one baseline without corresponding targets on the other two.
- **PERF-011.** The `x86_64` modern reference baseline is defined as follows: CPU class AMD Epyc 9004 or 9005 series, or Intel Xeon Sapphire Rapids or Emerald Rapids; 32 physical cores and 64 hardware threads; 128 GB of DDR5 memory; a 25 Gb/s network interface card of the Mellanox ConnectX-6 or ConnectX-7 family, or the Intel E810 family; NVMe storage; Linux kernel 6.x. The exact kernel minor version, sysctl values, and network-interface offload configuration applied to this baseline are tracked as open questions in section 7.
- **PERF-012.** The `aarch64` modern reference baseline is defined as follows: CPU class Ampere Altra, AmpereOne, AWS Graviton3, AWS Graviton4, or equivalent; core count, memory capacity, networking bandwidth, and storage comparable to the `x86_64` baseline defined by `PERF-011`; Linux kernel 6.x. The exact kernel minor version, sysctl values, and network-interface offload configuration applied to this baseline are tracked as open questions in section 7.
- **PERF-013.** When a performance target is expressed against any reference baseline, the measurement MUST be performed on hardware that matches the baseline definition in `PERF-011`, `PERF-012`, or `PERF-029` as applicable. A measurement performed on hardware that does not match the applicable baseline MUST NOT be treated as evidence for or against the target.

- **PERF-029.** The `riscv64` modern reference baseline is defined as a server-class RISC-V system running the `riscv64gc` ABI (as established for the Linux target by `ENV-015` in [`009-target-environment.md`](009-target-environment.md)) on Linux kernel 6.x, with a documented minimum physical core count and, when generally available, hardware support for the RISC-V Vector Extension (RVV). The specific RISC-V platform model, the minimum core count, the availability and generation of RVV, the memory capacity, and the network-interface card applied to this baseline are tracked as open questions in section 7.
- **PERF-030.** Absolute per-cell performance targets on the `riscv64` reference baseline under `PERF-029` MAY be more modest in absolute QPS and latency terms than the targets on the `x86_64` and `aarch64` baselines under `PERF-011` and `PERF-012`, reflecting the current state of RISC-V server-class hardware. This variation MUST NOT be treated as a defect of the architecture column; it is an expected property of the reference set at this stage of the RISC-V server ecosystem.
- **PERF-031.** The external comparison fixed by `PERF-019` through `PERF-028` MUST be applied per architecture column, including the `riscv64` column introduced by `PERF-029`. On a given cell, Heimdall MUST be compared only against reference implementations that are themselves running on the same architecture as Heimdall; a cross-architecture comparison (for example, Heimdall on `riscv64` against a reference implementation on `x86_64`) MUST NOT be treated as evidence for or against `PERF-021` through `PERF-024`. The "parity on plain DNS / exceed on encrypted transports" ambition applies per architecture column.
- **PERF-032.** The pinned reference-implementation set recorded under `PERF-020` MUST record, per comparison cycle and per architecture column, the reference implementations that are available and viable on that architecture. When a reference implementation is not available on a given architecture at a given comparison cycle, the absence MUST be documented alongside the measurement results of that cycle, and the cell MUST be compared against the best-in-class reference implementation that is available on that architecture.

## 4. Measurement methodology and CI enforcement

- **PERF-014.** Performance measurements for the per-cell targets under `PERF-005` MUST be produced using the following standard tools: `dnsperf`, `kdig`, and `queryperf` for QPS and latency measurements; `resperf` for resolver stress measurements; and purpose-built measurement harnesses, maintained in the Heimdall repository, for the memory-footprint and concurrent-connection-count dimensions. Additional tools MAY be used to complement these, but the tools enumerated in this requirement MUST be the baseline from which the official per-cell measurements are derived.
- **PERF-015.** Performance benchmarks under `PERF-014` MUST be executed as part of the continuous-integration (CI) pipeline, and their results MUST be recorded on a per-cell basis consistent with the matrix defined under `PERF-001`.
- **PERF-016.** The CI pipeline MUST enforce regression detection on every cell. A performance regression exceeding the configured threshold on any dimension listed under `PERF-005` MUST fail the CI build for the affected cell. The specific percentage thresholds applied to each dimension, and whether the threshold is applied per cell or at an aggregate level, are tracked as open questions in section 7.
- **PERF-017.** Any substantive change submitted as a pull request MUST report the measured performance deltas, relative to the prior baseline, for every cell affected by the change. A change that does not affect a given cell MUST NOT be required to report deltas for that cell.
- **PERF-018.** The tooling, the benchmark harnesses, and the stored baselines that implement `PERF-014` through `PERF-017` MUST be maintained under version control in the Heimdall repository so that they evolve together with the code under measurement.

## 5. Comparative performance ambition against reference implementations

- **PERF-019.** Heimdall's performance targets MUST be expressed not only in terms of internal regression detection under `PERF-015` through `PERF-017`, but also as an external comparison against state-of-the-art reference implementations, evaluated on a per-cell basis consistent with `PERF-001`. Internal regression detection alone detects degradation but does not detect stagnation at a mediocre level; the external comparison provides the forward pressure required by the performance principle stated in [`../CLAUDE.md`](../CLAUDE.md).
- **PERF-020.** The reference implementation set used for the external comparison under `PERF-019` MUST include, as the starting set, the following implementations per role: for the authoritative role, NSD (NLnet Labs), Knot Authoritative (CZ.NIC), and PowerDNS Authoritative; for the recursive resolver role, Unbound (NLnet Labs), Knot Resolver (CZ.NIC), and PowerDNS Recursor; for the forwarder role, dnsdist (PowerDNS), CoreDNS with its `forward` plugin, and Unbound operated in forward mode; for the encrypted-transport cells across every applicable role, dnsdist, cloudflared (DoH), Flamethrower, and any other mature encrypted-DNS implementations available at the time of the comparison cycle. The exact version of each reference implementation used for a given comparison cycle MUST be pinned and documented together with the measurement results.
- **PERF-021.** For every cell of the matrix defined under `PERF-001` whose transport is DNS classic over UDP on port 53 or DNS classic over TCP on port 53 (hereafter "plain-DNS cells"), Heimdall MUST achieve parity with the best-in-class reference implementation identified under `PERF-020` for that cell, on the reference hardware defined under `PERF-010` through `PERF-012`. Parity is the minimum ambition level for plain-DNS cells.
- **PERF-022.** Parity under `PERF-021` is defined jointly by the following bounds, each evaluated against the best-in-class reference implementation for the same cell on the same reference hardware: the sustainable QPS measured under `PERF-006` MUST be within approximately 5 percent of the reference; the response latency at the p99 percentile measured under `PERF-007` MUST NOT exceed the reference by more than approximately 20 percent; the per-process memory footprint and the CPU efficiency measured under `PERF-005` MUST NOT be worse than the reference. All four bounds MUST be satisfied simultaneously for parity to hold on a given cell.
- **PERF-023.** For every cell of the matrix defined under `PERF-001` whose transport is DoT, DoH served over HTTP/2, DoH served over HTTP/3, or DoQ (hereafter "encrypted-transport cells"), Heimdall MUST exceed the best-in-class reference implementation identified under `PERF-020` for that cell, on the reference hardware defined under `PERF-010` through `PERF-012`, by a measurable margin. Exceeding the reference is the target ambition level for encrypted-transport cells; parity is not sufficient.
- **PERF-024.** The measurable margin required by `PERF-023` targets, as an order of magnitude pending first-benchmark calibration, approximately 20 percent additional sustainable QPS or approximately 80 percent of the reference p99 latency, evaluated jointly with no regression on memory footprint or CPU efficiency. The per-process memory footprint and the CPU efficiency measured under `PERF-005` for encrypted-transport cells MUST be at most equal to the best-in-class reference. The exact margin values applied to each encrypted-transport cell are tracked as an open question in section 7.
- **PERF-025.** The external comparison under `PERF-019` MUST be performed on the same reference hardware defined under `PERF-010` through `PERF-012` that applies to the cell under evaluation. A comparison measurement produced on hardware that does not match the applicable baseline MUST NOT be treated as evidence for or against `PERF-021` through `PERF-024`, consistent with `PERF-013`.
- **PERF-026.** The external comparison under `PERF-019` MUST be revisited at least once per calendar year. Each revisit MUST re-identify the best-in-class reference implementation per cell, update the pinned versions recorded under `PERF-020`, and confirm or adjust the target margins applied under `PERF-021` through `PERF-024`. The revisit MUST produce a dated record of the reference set, versions, and margins that applied for the corresponding comparison cycle.
- **PERF-027.** The scope of the external comparison under `PERF-019` through `PERF-026` MUST span every cell of the `(role, transport, architecture)` matrix established by `PERF-001`, `PERF-002`, and `PERF-010`. A cell of that matrix MUST NOT be exempted from comparative evaluation except through an explicit exclusion recorded under the normalisation policy tracked as an open question in section 7.
- **PERF-028.** When the external comparison under `PERF-021` through `PERF-024` fails on a given cell, the failure MUST be surfaced as a reviewable signal alongside the internal regression signal produced under `PERF-016`. The specific mechanism by which comparative failures are surfaced in the CI pipeline, and whether such a failure blocks release independently of the internal regression gate, are tracked as open questions in section 7.

## 6. Rationale

`../CLAUDE.md` states that performance is the primary guide for every decision across the entire stack. Without quantified, enforced targets, the principle remains aspirational and is easily eroded by incremental changes whose individual cost is small but whose cumulative effect is material. Fixing targets per cell, rather than as a single aggregate number for the whole server, follows from the observation that the performance profiles of the cells in the matrix differ by orders of magnitude: authoritative UDP on port 53 is a stateless, low-overhead path dominated by packet processing, while DoH over HTTP/3 on the recursive resolver role is a stateful, encrypted, multi-layered path dominated by TLS, QUIC, and HTTP processing. A single aggregate target would hide regressions in any individual cell behind improvements in another and would provide no actionable signal. Treating DoH over HTTP/2 and DoH over HTTP/3 as separate cells under `PERF-002` is a direct consequence of the same reasoning: their hot-path profiles differ materially, with QUIC processing on HTTP/3 imposing a qualitatively different cost structure from the TLS-over-TCP and HPACK processing on HTTP/2, so collapsing them would obscure regressions that affect only one of the two.

Expressing targets across two parallel reference baselines (`PERF-010`, `PERF-011`, `PERF-012`) reflects the deployment reality of 2026. Server `aarch64` hardware, including AWS Graviton3 and Graviton4 in the cloud and Ampere Altra and AmpereOne on bare-metal and co-located deployments, is mainstream and is no longer an afterthought. Expressing targets only on `x86_64` would deliver second-class behaviour on a significant share of modern deployments, would accumulate architecture-specific regressions silently between releases, and would contradict the principle that Heimdall is a first-class citizen on both architectures. Expressing the targets in parallel on both baselines imposes the discipline of architecture-aware design from the outset.

Deferring the absolute numerical targets under `PERF-004` is a direct application of the "Assume Nothing" rule. The project has not yet produced an implementation, so fixing values such as "5 million QPS on UDP/53" now would be speculation dressed up as requirement, and would anchor future measurements to numbers that have no empirical basis. Methodology and regression thresholds, by contrast, can be fixed now without empirical input, because they are policy decisions rather than performance claims; doing so lets the first implementation baseline produce numerical targets that are already subject to regression enforcement as soon as they are recorded, and the absolute values will stabilise across the initial releases.

Requiring the use of standard benchmarking tools under `PERF-014` (`dnsperf`, `queryperf`, `resperf`, `kdig`) avoids a "not invented here" posture and produces measurements that are directly comparable with the published performance of reference implementations such as BIND, Knot Authoritative, NSD, PowerDNS Authoritative, Unbound, Knot Resolver, PowerDNS Recursor, and dnsdist. The literature and the operational ecosystem are aligned around these tools; adopting them preserves the ability to cross-check Heimdall's published performance against the state of the art and reduces the burden of independent tool audits. Purpose-built harnesses are explicitly carved out for the memory and connection-count dimensions because no standard tool covers those dimensions in a way that is directly aligned with the per-cell matrix.

Enforcing regression detection in CI under `PERF-016` places the performance principle on the structural side of the system rather than on the reviewer-discipline side. A target that is not enforced by an automated gate degrades silently; a target that fails the build when it is violated cannot degrade without a visible, reviewable event. This is the same structural-gating approach already applied to role activation in [`001-server-roles.md`](001-server-roles.md) and to transport-listener instantiation in [`002-transports.md`](002-transports.md): the safe behaviour is built in rather than selected. Requiring pull requests to report performance deltas for affected cells under `PERF-017` completes the loop by making the performance impact of each substantive change visible at the point at which the change is reviewed, rather than post-hoc when a regression has already landed on the main branch.

Plain-DNS reference implementations (NSD, Knot Authoritative, PowerDNS Authoritative, Unbound, Knot Resolver, PowerDNS Recursor, dnsdist) have accumulated decades of C-level optimisation on top of transport paths (UDP and TCP on port 53) whose cost structure is comparatively stable and well understood. Targeting a significant margin of improvement against them, on plain-DNS cells, in the first release is not realistic; parity is itself a state-of-the-art outcome in practice, and it is the correct ambition level for `PERF-021` and `PERF-022`. The 5 percent QPS tolerance and the 20 percent p99 tolerance under `PERF-022` are deliberately calibrated to accept the residual constant-factor cost of safer abstractions without permitting any structural slippage; the memory and CPU bounds enforce that parity is not obtained by trading resource usage for throughput.

Encrypted-transport cells are, by contrast, the clearest differentiator for Heimdall. A Rust codebase built from the start around Tokio asynchronous I/O, rustls for TLS 1.3, quinn for QUIC, zero-copy parsing on the hot path, and encrypted-transport acceptors treated as first-class citizens rather than as bolt-on layers on top of a legacy plain-DNS engine can outperform existing implementations, many of which expose encrypted DNS as an add-on path whose optimisation has lagged the plain-DNS path. Targeting "exceed" under `PERF-023` and `PERF-024` articulates the project's positioning and ties the performance story to the architectural choices that justify the investment in Rust. The order of magnitude of approximately 20 percent additional QPS or approximately 80 percent of reference p99 latency under `PERF-024` is stated as an operational target pending first-benchmark calibration, rather than as a fixed contractual value, because no empirical baseline yet exists; `PERF-024` therefore explicitly defers the exact margin to the open question in section 7, while fixing the direction of travel.

Memory and CPU efficiency improvements come structurally from the Rust type system (zero-copy parsing, per-thread arenas, `#[repr(C)]` layouts for cache-friendly record representations, safe SIMD) rather than from point optimisation, so requiring them on both plain-DNS cells (`PERF-022`) and encrypted-transport cells (`PERF-024`) is appropriate at the specification level. A design that matches QPS but regresses memory or CPU would have obtained its throughput by consuming more of the envelope; the requirement prevents that outcome.

The stricter alternative — requiring Heimdall to exceed every reference implementation on every cell, including plain DNS — was rejected because it is unrealistic against decades of C-level optimisation on the plain-DNS path and would force the specification into a claim the project cannot honestly defend. The softer alternative — "competitive within 20 percent on every cell" — was rejected because it contradicts the "extreme performance" posture in [`../CLAUDE.md`](../CLAUDE.md) and, by treating every cell uniformly, gives up the positioning that encrypted transports afford. The softest alternative — removing external comparison entirely and relying solely on internal regression detection — was rejected because it leaves the project vulnerable to unnoticed stagnation at a mediocre absolute level: internal regressions would fail, but a project that stabilises at half the throughput of the state of the art would never trip any gate. Annual revisiting of the comparison under `PERF-026` reflects that the reference set itself evolves, that a fixed reference set will drift out of date, and that what counts as best-in-class is not static.

Extending the reference-hardware subsection under `PERF-029` through `PERF-032` to a third architecture column for `riscv64` is a natural extension of the per-architecture framework already in place. The two-axis matrix fixed by `PERF-010` through `PERF-013` was designed from the outset to carry per-architecture targets independently, so adding a third column does not alter the structure of the framework: it populates it with an additional baseline. Allowing the absolute targets on `riscv64` to be more modest under `PERF-030` honours the current state of RISC-V server-class hardware without weakening the comparative ambition under `PERF-019` through `PERF-024`: the external comparison is applied per architecture column (`PERF-031`) against reference implementations running on the same architecture, so the "parity on plain DNS / exceed on encrypted transports" posture remains meaningful on `riscv64` even when the absolute numbers are lower than on `x86_64` or `aarch64`. The pinned reference-implementation bookkeeping under `PERF-032` recognises that not every reference implementation is available on every architecture at every comparison cycle, and fixes the documentation obligation accordingly.

## 7. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will need to be specified before the performance targets defined by this document can be evaluated or enforced.

- **Concrete per-cell numerical targets.** The concrete numerical values of the per-cell targets under `PERF-005`, expressed per `(role, transport, architecture)` cell, are **to be specified**. This includes the sustainable QPS target, the response-latency targets at p50, p99, and p99.9, the per-cache and per-process memory-footprint targets, and the concurrent-connection-count target for every cell of the matrix.
- **Regression thresholds.** The specific percentage regression thresholds under `PERF-016` are **to be specified**, including whether a single threshold applies to every dimension or whether the threshold is dimension-specific, and whether the threshold is enforced per cell, at an aggregate level across cells, or both.
- **CI infrastructure for performance benchmarks.** The CI infrastructure that executes the performance benchmarks under `PERF-015`, including the choice between self-hosted runners backed by dedicated reference hardware and cloud CI runners, and the reproducibility constraints applied to the execution environment (noise isolation, pinned kernel version, pinned firmware, pinned NIC driver, CPU frequency governor configuration, hardware interrupt affinity, NUMA topology), are **to be specified**.
- **Benchmark corpora.** The benchmark corpora used to drive the tools under `PERF-014`, including the split between synthetic query sets and replays of real traffic captures (for example CAIDA DNS data sets and B-Root archives), the licensing constraints applicable to each corpus, and the handling of privacy-sensitive content within replayed captures, are **to be specified**.
- **Warm-up policy for steady-state measurement.** The warm-up policy applied before a measurement is recorded for the sustainable-QPS and latency dimensions under `PERF-006` and `PERF-007`, including the duration of the warm-up phase, the criteria that mark the system as having entered steady state, and the handling of cold-start effects (caches empty, connection pools empty, JIT and branch predictors unwarmed), is **to be specified**.
- **Stability criterion for a measurement to count.** The stability criterion that a measurement must satisfy before it counts as evidence for or against a target, including the maximum coefficient of variation across repeated runs, the minimum number of runs required, and the handling of outliers, is **to be specified**.
- **Kernel version and tuning per reference baseline.** The exact Linux kernel minor version, sysctl values, and network-interface offload configuration applied to the `x86_64`, `aarch64`, and `riscv64` reference baselines under `PERF-011`, `PERF-012`, and `PERF-029` are **to be specified** and MUST be documented per baseline.
- **Concrete RISC-V reference hardware for benchmarking.** The concrete RISC-V reference-hardware model introduced by `PERF-029` — including the specific server-class RISC-V platform, the minimum physical core count, the availability and generation of the RISC-V Vector Extension (RVV), the memory capacity, and the network-interface card — is **to be specified** once representative hardware is selected, and MUST be fixed before any official measurement on the `riscv64` baseline is recorded.
- **Pinned reference-implementation versions per comparison cycle.** The exact versions of each reference implementation listed under `PERF-020` that apply to each comparison cycle under `PERF-019` and `PERF-026` are **to be specified**, and MUST be recorded alongside the measurement results of that cycle.
- **Exact margin for encrypted-transport cells.** The exact comparative margin applied to the encrypted-transport cells under `PERF-023` and `PERF-024`, for which this document fixes the order of magnitude (approximately 20 percent additional QPS or approximately 80 percent of reference p99 latency) but defers the precise value to first-benchmark calibration, is **to be specified**, per encrypted-transport cell and per reference-hardware baseline.
- **Feature-parity normalisation policy for the external comparison.** The normalisation policy that applies when Heimdall and a reference implementation differ in feature parity on a given cell — for example, when the reference supports Response Policy Zones or any other feature that Heimdall does not implement, or conversely — is **to be specified**. The policy MUST determine which cells are comparable, which cells are excluded from the external comparison under `PERF-019` through `PERF-027`, and how exclusions are recorded.
- **Reproducibility requirements for the comparison bench.** The reproducibility requirements that apply to the external comparison bench under `PERF-025`, including the obligation that Heimdall and every reference implementation under comparison run on identical operating-system images, identical Linux kernel versions, identical sysctl and tuning values, identical network-interface offload configuration, and identical client-side load-generation setup, are **to be specified**.
- **Schedule and governance for the annual comparison-cycle revisit.** The schedule and governance for the annual revisit required by `PERF-026`, including the fixed calendar anchor of the revisit, the party responsible for driving it, the review and sign-off procedure for updated reference versions and margins, and the retention policy for past comparison-cycle records, are **to be specified**.
- **Mechanism for surfacing comparative failures.** The specific mechanism by which failures of the external comparison under `PERF-021` through `PERF-024` are surfaced alongside the internal regression signal under `PERF-028`, including whether such a failure blocks release independently of the internal regression gate, is **to be specified**.

No implementation activity may proceed on the basis of assumptions about any of the items above.
