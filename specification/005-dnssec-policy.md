# DNSSEC policy

**Purpose.** This document defines Heimdall's DNSSEC policy. It fixes the decisions that govern how DNSSEC data is produced, stored, served, validated, and otherwise handled across the roles supported by Heimdall, independently of which roles or transports are active in a given deployment. It does not restate which roles Heimdall supports or when their state is allocated; those questions are settled in [`001-server-roles.md`](001-server-roles.md).

**Status.** Stable.

**Requirement category.** `DNSSEC`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md).

## 1. Scope

This document fixes the DNSSEC scope for all three roles supported by Heimdall: the authoritative server role (section 2), the recursive resolver role (section 2A), the forwarder role (section 2B), aggressive use of the DNSSEC-validated cache on the validating roles (section 2C), the signature-algorithm acceptance policy applied by the validating roles (section 2D), validation hardening against denial-of-service vectors on the validating roles (section 2E), and the DS digest algorithm acceptance policy applied by the validating roles (section 2F). The remaining DNSSEC items — Extended DNS Error codes and Negative Trust Anchor operational defaults — apply uniformly across the validating roles (recursive resolver and forwarder) and are tracked as open in section 4. QNAME minimisation is a privacy-adjacent behaviour orthogonal to DNSSEC validation; it is fixed by `PROTO-020` through `PROTO-024` in [`006-protocol-conformance.md`](006-protocol-conformance.md) and is not duplicated here. Glue-record handling on the recursive resolver role, in accordance with [RFC 9471](https://www.rfc-editor.org/rfc/rfc9471), is fixed by `PROTO-050` through `PROTO-054` in [`006-protocol-conformance.md`](006-protocol-conformance.md) and is not duplicated here; the child-side validation step required by `PROTO-053` reuses the validation pipeline fixed by `DNSSEC-008` through `DNSSEC-014` of this document.

## 2. Authoritative server: pre-signed zones only

### 2.1 Normative requirements

- **DNSSEC-001.** Heimdall MUST support serving pre-signed DNSSEC zones on the authoritative server role. Zones are signed by an external signer outside the Heimdall process, and Heimdall serves the resulting signed zone data.
- **DNSSEC-002.** Heimdall MUST accept zone data that already contains DNSSEC records produced by an external signer. The record types that Heimdall MUST accept as part of a pre-signed zone are `RRSIG`, `DNSKEY`, `NSEC`, `NSEC3`, `NSEC3PARAM`, `DS`, `CDNSKEY`, `CDS`, and `CSYNC`.
- **DNSSEC-003.** Heimdall MUST NOT generate, derive, store, import, or otherwise possess DNSSEC private signing keys inside the authoritative server process. Private-key material for DNSSEC signing is out of scope for the authoritative server process and MUST reside exclusively in the external signer.
- **DNSSEC-004.** Heimdall MUST NOT sign DNS responses on the fly on the authoritative server role. Every DNSSEC record returned on the authoritative path MUST originate from the pre-signed zone data loaded into the server, not from a signing operation performed at query time.
- **DNSSEC-005.** On the authoritative server role, when a DNSSEC-aware query arrives — that is, a query whose `OPT` record has the `DO` flag set, as specified by [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035) — Heimdall MUST serve the DNSSEC records present in the zone that are required to prove the response, including the corresponding `RRSIG`, `NSEC`, and `NSEC3` records.
- **DNSSEC-006.** On the authoritative server role, Heimdall MUST handle the `DO` flag and the `AD` flag in accordance with [RFC 4035, section 3](https://www.rfc-editor.org/rfc/rfc4035#section-3).
- **DNSSEC-007.** Online signing, DNSSEC private-key management inside Heimdall, and automated DNSSEC key-rollover orchestration inside Heimdall are out of scope at this stage and MUST NOT be implemented. This decision MAY be reconsidered in a future revision of this document if a concrete use case emerges.

## 2A. Recursive resolver: validation by default

### 2A.1 Normative requirements

- **DNSSEC-008.** Heimdall MUST implement DNSSEC validation on the recursive resolver role in accordance with [RFC 4033](https://www.rfc-editor.org/rfc/rfc4033), [RFC 4034](https://www.rfc-editor.org/rfc/rfc4034), [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035), [RFC 6840](https://www.rfc-editor.org/rfc/rfc6840), and [RFC 9364](https://www.rfc-editor.org/rfc/rfc9364).
- **DNSSEC-009.** DNSSEC validation on the recursive resolver role MUST be enabled by default. Heimdall MUST NOT expose any configuration switch that disables DNSSEC validation globally for the running instance.
- **DNSSEC-010.** The outcome of DNSSEC validation for a response produced by the recursive resolver role MUST be exactly one of the four states defined by [RFC 4034](https://www.rfc-editor.org/rfc/rfc4034): `secure`, `insecure`, `bogus`, or `indeterminate`.
- **DNSSEC-011.** When validation yields `secure`, Heimdall MUST return the response with the `AD` flag set, and MUST include the corresponding `RRSIG` records in the response when the originating query had the `DO` flag set, as specified by [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035).
- **DNSSEC-012.** When validation yields `insecure`, Heimdall MUST return the response without the `AD` flag set. `insecure` is a valid outcome for data that is not covered by a chain of trust reaching a configured trust anchor.
- **DNSSEC-013.** When validation yields `bogus`, Heimdall MUST respond with `SERVFAIL`, and SHOULD include an Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) that identifies the class of validation failure.
- **DNSSEC-014.** When validation yields `indeterminate`, Heimdall MUST return the response without the `AD` flag set.
- **DNSSEC-015.** The default trust anchor used by the recursive resolver role MUST be the IANA root key, distributed in accordance with [RFC 7958](https://www.rfc-editor.org/rfc/rfc7958). Heimdall MUST support automatic trust-anchor key rollover in accordance with [RFC 5011](https://www.rfc-editor.org/rfc/rfc5011).
- **DNSSEC-016.** Heimdall MAY accept operator registration of Negative Trust Anchors as specified by [RFC 7646](https://www.rfc-editor.org/rfc/rfc7646) for individual zones. Each Negative Trust Anchor MUST carry an explicit expiry time. Heimdall MUST reject the registration of a Negative Trust Anchor that does not specify an expiry time.
- **DNSSEC-017.** Heimdall MUST automatically remove each registered Negative Trust Anchor at or before its expiry time. After removal, the zone previously covered by the Negative Trust Anchor MUST again be subject to the validation outcomes defined in `DNSSEC-010` through `DNSSEC-014`.
- **DNSSEC-018.** Negative Trust Anchor registration, expiry, and manual revocation MUST be auditable. Heimdall MUST emit a log entry for each registration event, each expiry-driven removal event, and each manual revocation event.

## 2B. Forwarder: independent validation of upstream responses

### 2B.1 Normative requirements

- **DNSSEC-019.** Heimdall MUST implement DNSSEC validation on the forwarder role in accordance with [RFC 4033](https://www.rfc-editor.org/rfc/rfc4033), [RFC 4034](https://www.rfc-editor.org/rfc/rfc4034), [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035), [RFC 6840](https://www.rfc-editor.org/rfc/rfc6840), and [RFC 9364](https://www.rfc-editor.org/rfc/rfc9364). Validation on the forwarder role MUST be performed locally, on every response received from every configured upstream, and MUST apply the same validation policy as the recursive resolver role defined in section 2A.
- **DNSSEC-020.** DNSSEC validation on the forwarder role MUST be enabled by default. Heimdall MUST NOT expose any configuration switch that disables DNSSEC validation for the forwarder role.
- **DNSSEC-021.** The outcome of DNSSEC validation for a response received from a configured upstream MUST be exactly one of the four states defined by [RFC 4034](https://www.rfc-editor.org/rfc/rfc4034): `secure`, `insecure`, `bogus`, or `indeterminate`. Heimdall MUST handle each outcome on the forwarder role identically to the recursive resolver role, as fixed by `DNSSEC-010` through `DNSSEC-014`.
- **DNSSEC-022.** Heimdall MUST NOT rely on the `AD` flag set by any configured upstream when determining the validation outcome on the forwarder role. The `AD` flag is set in the response returned to the client only after local validation on Heimdall itself yields `secure`, in accordance with `DNSSEC-011`.
- **DNSSEC-023.** The trust anchor applied on the forwarder role MUST be the same trust anchor applied on the recursive resolver role, as fixed by `DNSSEC-015`. A single trust-anchor state, including the IANA root key distributed under [RFC 7958](https://www.rfc-editor.org/rfc/rfc7958) and the automatic key-rollover state maintained under [RFC 5011](https://www.rfc-editor.org/rfc/rfc5011), applies to the running instance and is shared between the recursive resolver role and the forwarder role. Heimdall MUST NOT maintain a separate trust-anchor state per role.
- **DNSSEC-024.** Negative Trust Anchors registered under `DNSSEC-016` MUST apply uniformly to whichever validating role triggers the validation on the running instance. Heimdall MUST NOT partition Negative Trust Anchor state per role, and MUST NOT expose a configuration option that binds a Negative Trust Anchor to a specific role.

## 2C. Aggressive use of the DNSSEC-validated cache

### 2C.1 Normative requirements

- **DNSSEC-025.** Heimdall MUST implement aggressive use of `NSEC` and `NSEC3` records from the DNSSEC-validated cache in accordance with [RFC 8198](https://www.rfc-editor.org/rfc/rfc8198). This requirement applies to both validating roles defined in sections 2A and 2B: the recursive resolver role and the forwarder role.
- **DNSSEC-026.** When a cached `NSEC` or `NSEC3` record proves the non-existence of a name or of a type that is the subject of a new incoming query, Heimdall MUST synthesise the corresponding `NXDOMAIN` or `NODATA` response locally from the cached proof, and MUST NOT issue an upstream query for that resolution step. For the recursive resolver role, "upstream query" refers to a query emitted toward an authoritative server during iterative resolution. For the forwarder role, "upstream query" refers to a query emitted toward a configured upstream resolver.
- **DNSSEC-027.** Aggressive synthesis under `DNSSEC-025` and `DNSSEC-026` MUST use only `NSEC` and `NSEC3` records that have been validated to the `secure` state as defined in `DNSSEC-010` for the recursive resolver role and as defined in `DNSSEC-021` for the forwarder role. Heimdall MUST NOT synthesise responses from cached `NSEC` or `NSEC3` records whose validation outcome is `insecure`, `bogus`, or `indeterminate`.
- **DNSSEC-028.** Aggressive synthesis under `DNSSEC-025` and `DNSSEC-026` MUST respect the `NSEC3` opt-out semantics defined in [RFC 5155, section 6](https://www.rfc-editor.org/rfc/rfc5155#section-6). An opt-out `NSEC3` record proves the non-existence only of signed delegations within the covered interval and does not prove the non-existence of names that could be covered by an unsigned delegation spanned by the opt-out. Heimdall MUST NOT synthesise a non-existence response for any name that could be covered by an opt-out-spanned unsigned delegation, and MUST instead resolve such a query through the normal resolution path.
- **DNSSEC-029.** Wildcard interactions during aggressive synthesis under `DNSSEC-025` and `DNSSEC-026` MUST follow the proof rules specified in [RFC 7129](https://www.rfc-editor.org/rfc/rfc7129) and in [RFC 8198, section 4](https://www.rfc-editor.org/rfc/rfc8198#section-4). Heimdall MUST NOT synthesise a non-existence response when the cached `NSEC` or `NSEC3` proof does not exclude the possibility of a wildcard expansion that would produce a positive answer for the queried name.
- **DNSSEC-030.** Heimdall MUST NOT expose any configuration switch that disables aggressive use of `NSEC` and `NSEC3` records under `DNSSEC-025` through `DNSSEC-029`, either globally on the running instance or per zone. There is no off-switch for aggressive synthesis on either validating role.

## 2D. Algorithm acceptance policy for the validating roles

### 2D.1 Normative requirements

- **DNSSEC-031.** The algorithm acceptance policy fixed by `DNSSEC-032` through `DNSSEC-040` applies to the validating roles defined in sections 2A and 2B: the recursive resolver role and the forwarder role. The authoritative server role serves pre-signed zones produced by an external signer, as fixed by `DNSSEC-001` through `DNSSEC-004`; algorithm selection on the authoritative path is made by the external signer and is outside the scope of this section. The signature-algorithm acceptance policy applied by the authoritative server role when serving pre-signed zones remains open as tracked in section 4.
- **DNSSEC-032.** Heimdall MUST implement DNSSEC signature validation for the algorithms marked as MUST in [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). A correctly signed response whose chain of trust reaches a configured trust anchor under one of these algorithms MUST yield the `secure` outcome defined by `DNSSEC-010` and `DNSSEC-021`. The algorithms covered by this requirement are algorithm 8 (RSASHA256) and algorithm 13 (ECDSAP256SHA256).
- **DNSSEC-033.** Heimdall SHOULD implement DNSSEC signature validation for the algorithms marked as RECOMMENDED in [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). Where Heimdall implements validation for these algorithms, a correctly signed response whose chain of trust reaches a configured trust anchor under one of them MUST yield the `secure` outcome defined by `DNSSEC-010` and `DNSSEC-021`. The algorithms covered by this requirement are algorithm 14 (ECDSAP384SHA384), algorithm 15 (Ed25519), and algorithm 16 (Ed448).
- **DNSSEC-034.** Heimdall MAY implement DNSSEC signature validation for the algorithms marked as MAY in [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). Where Heimdall implements validation for these algorithms, a correctly signed response whose chain of trust reaches a configured trust anchor under one of them MUST yield the `secure` outcome defined by `DNSSEC-010` and `DNSSEC-021`. The algorithms covered by this requirement are algorithm 5 (RSASHA1), algorithm 7 (RSASHA1-NSEC3-SHA1), and algorithm 10 (RSASHA512).
- **DNSSEC-035.** Heimdall MUST NOT implement DNSSEC signature validation for the algorithms marked as MUST NOT in [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). Any `RRSIG` produced under one of these algorithms MUST NOT contribute to a `secure` outcome. The algorithms covered by this requirement are algorithm 1 (RSAMD5), algorithm 3 (DSA-SHA1), algorithm 6 (DSA-NSEC3-SHA1), and algorithm 12 (ECC-GOST).
- **DNSSEC-036.** When processing an RRset that carries an `RRSIG` produced under an algorithm covered by `DNSSEC-035`, Heimdall MUST treat that signature as if it were not present and MUST NOT attempt cryptographic verification against it. The validation pipeline MUST then proceed to consider any remaining `RRSIG` records covering the same RRset under algorithms supported by `DNSSEC-032`, `DNSSEC-033`, or `DNSSEC-034` for which Heimdall implements validation.
- **DNSSEC-037.** When, after applying `DNSSEC-036`, no `RRSIG` remains for which Heimdall implements validation and which verifies against a chain of trust reaching a configured trust anchor, the validation outcome for the RRset MUST be determined in accordance with [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035): `insecure` when the RRset lies in a zone for which no chain of trust to a configured trust anchor exists, and `bogus` when the RRset lies in a zone whose chain of trust to a configured trust anchor requires a signature that is absent, unsupported, or invalid. The corresponding response handling is governed by `DNSSEC-012` and `DNSSEC-013`.
- **DNSSEC-038.** When Heimdall determines that an RRset, or the zone containing it, is signed exclusively under algorithms covered by `DNSSEC-034` (algorithms 5, 7, and 10), the response returned to the client MUST include an Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) that signals the use of a deprecated signing algorithm. This requirement applies whether the validation outcome is `secure`, `insecure`, or `bogus`, and is additional to any Extended DNS Error already required by `DNSSEC-013`.
- **DNSSEC-039.** When the condition described in `DNSSEC-038` is detected, Heimdall MUST emit a structured log entry that identifies the affected zone, the algorithm number, and the validation outcome produced by the pipeline. The log entry MUST be suitable for operator metrics and alerting on deprecated-algorithm use observed in the resolution path.
- **DNSSEC-040.** The algorithm acceptance policy fixed by `DNSSEC-032` through `DNSSEC-039` tracks [RFC 8624](https://www.rfc-editor.org/rfc/rfc8624). When RFC 8624 is updated or superseded, the policy MUST be revised accordingly in a future revision of this document. Heimdall MUST NOT anticipate such revisions by tightening or relaxing the policy unilaterally.

## 2E. Validation hardening against denial-of-service vectors

### 2E.1 Normative requirements

- **DNSSEC-041.** When validating a single RRset on either validating role defined in sections 2A and 2B, Heimdall MUST bound the number of `DNSKEY` × `RRSIG` combinations that it attempts to verify, in order to contain the combinatorial cost exploited by the KeyTrap class of attacks identified as [CVE-2023-50387](https://nvd.nist.gov/vuln/detail/CVE-2023-50387). The precise numeric values of this cap are fixed by `DNSSEC-086`.
- **DNSSEC-042.** When the bound fixed by `DNSSEC-041` is reached during the validation of an RRset, Heimdall MUST NOT continue the combinatorial expansion for that RRset. The validation outcome for that RRset MUST be treated as `bogus` in the sense of `DNSSEC-010` and `DNSSEC-021`, unless the validation pipeline has already reached `secure` for the same RRset through a simpler path that does not require further combinatorial expansion. When the outcome is `bogus` under this requirement, the response returned to the client is governed by `DNSSEC-013`, and Heimdall SHOULD include an Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) indicating that the validation complexity bound was reached.
- **DNSSEC-043.** The KeyTrap mitigation fixed by `DNSSEC-041` and `DNSSEC-042` MUST apply uniformly across the recursive resolver role and the forwarder role. Heimdall MUST NOT expose a configuration switch that disables this mitigation on either role, whether globally on the running instance or per zone.
- **DNSSEC-044.** Heimdall MUST NOT perform more than 150 `NSEC3` iterations for any single query on either validating role defined in sections 2A and 2B, in accordance with [RFC 9276, section 3.2](https://www.rfc-editor.org/rfc/rfc9276#section-3.2).
- **DNSSEC-045.** When a response arrives from an upstream — an authoritative server on the recursive resolver role, or a configured upstream resolver on the forwarder role — signed with `NSEC3` parameters whose iteration count exceeds 150, Heimdall MUST treat the answer as `insecure` in the sense of `DNSSEC-010` and `DNSSEC-021`, and MUST NOT treat it as `bogus`. The response MUST be returned to the client without the `AD` flag set, in accordance with `DNSSEC-012`, and MAY include an Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) indicating that the `NSEC3` iteration count was excessive.
- **DNSSEC-046.** `NSEC3` iteration counts at or below 150 MUST be honoured normally by the validation pipeline on either validating role. [RFC 9276, section 3.1](https://www.rfc-editor.org/rfc/rfc9276#section-3.1) expresses a preference for an iteration count of 0, but any value in the inclusive range 0 to 150 is accepted and does not trigger the handling defined in `DNSSEC-045`.
- **DNSSEC-047.** The `NSEC3` iteration mitigation fixed by `DNSSEC-044` through `DNSSEC-046` MUST apply uniformly across the recursive resolver role and the forwarder role. Heimdall MUST NOT expose a configuration switch that disables this mitigation on either role, whether globally on the running instance or per zone.

## 2F. DS digest algorithm acceptance policy for the validating roles

### 2F.1 Normative requirements

- **DNSSEC-048.** The DS digest algorithm acceptance policy fixed by `DNSSEC-049` through `DNSSEC-054` applies to the validating roles defined in sections 2A and 2B: the recursive resolver role and the forwarder role. The authoritative server role serves pre-signed zones produced by an external signer, as fixed by `DNSSEC-001` through `DNSSEC-004`; DS digest selection on the authoritative path is made by the external signer and is outside the scope of this section.
- **DNSSEC-049.** Heimdall MUST implement DS digest validation for digest type 2 (SHA-256) in accordance with [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2). A DS record carrying digest type 2 that matches the hash of the delegated `DNSKEY` MUST be accepted as a link in the chain of trust reaching a configured trust anchor.
- **DNSSEC-050.** Heimdall MAY implement DS digest validation for digest type 4 (SHA-384) in accordance with [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2). Where Heimdall implements validation for digest type 4, a DS record carrying digest type 4 that matches the hash of the delegated `DNSKEY` MUST be accepted as a link in the chain of trust reaching a configured trust anchor.
- **DNSSEC-051.** Heimdall MAY implement DS digest validation for digest type 1 (SHA-1) in accordance with [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2). When a delegation has no DS record of a non-deprecated digest type that Heimdall implements — that is, no DS record of digest type 2 and, where Heimdall implements digest type 4, no DS record of digest type 4 — and digest type 1 is the only digest present, Heimdall MUST continue the validation path using digest type 1. When the chain of trust for the response is closed exclusively through a DS record of digest type 1, the response returned to the client MUST include an Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) signalling the use of a deprecated DS digest algorithm, and Heimdall MUST emit a structured log entry that identifies the affected delegation point, the digest type, and the validation outcome produced by the pipeline. The log entry MUST be suitable for operator metrics and alerting on deprecated-DS-digest use observed in the resolution path.
- **DNSSEC-052.** Heimdall MUST NOT implement DS digest validation for digest type 3 (GOST) in accordance with [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2). Any DS record carrying digest type 3 MUST NOT contribute to a chain of trust reaching a configured trust anchor.
- **DNSSEC-053.** Any DS digest type not defined in [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2) — including any digest type assigned by IANA but not yet covered by that section — MUST be treated as unsupported by the validating roles. A DS record carrying an unsupported digest type MUST NOT contribute to a chain of trust reaching a configured trust anchor, and the validation pipeline MUST proceed by considering any remaining DS records at the same delegation point whose digest type is covered by `DNSSEC-049`, `DNSSEC-050`, or `DNSSEC-051`.
- **DNSSEC-054.** The DS digest algorithm acceptance policy fixed by `DNSSEC-049` through `DNSSEC-053` tracks [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2). When RFC 8624 is updated or superseded, the policy MUST be revised accordingly in a future revision of this document. Heimdall MUST NOT anticipate such revisions by tightening or relaxing the policy unilaterally.

## 3. Rationale

Serving pre-signed zones rather than signing on the fly follows directly from the security posture stated in [`../CLAUDE.md`](../CLAUDE.md). Private signing keys never reside in the authoritative-server process, so a compromise of the Heimdall binary does not imply a compromise of the zones it serves. Signing is performed once, offline, by an external signer under its own operational controls, and the result is loaded into Heimdall as data. This is the strongest available trust boundary between the zone's long-term signing material and the network-facing process.

On the performance axis, signing is pre-computed and cached in the zone data itself. The authoritative-serve path has no per-query cryptographic cost beyond what is already materialised in the loaded zone, which aligns with the hot-path objectives stated in [`../CLAUDE.md`](../CLAUDE.md). Online signing, by contrast, would introduce a per-query asymmetric-cryptography cost that grows with query volume and scales poorly under the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md).

The decision aligns with mature industry practice. Authoritative servers that serve pre-signed zones produced by an external signer include NSD, Knot Authoritative, BIND operating in secondary mode over zones signed elsewhere, and PowerDNS Authoritative paired with an external signer such as OpenDNSSEC. The separation between a signer component and a serving component is the established operational model for DNSSEC at scale.

Keeping online signing, private-key management, and automated key-rollover orchestration out of scope contains the initial surface of the authoritative role. Each of these capabilities would require a substantial additional subsystem: key generation, KSK and ZSK separation, secure key storage, rollover timing per [RFC 7583](https://www.rfc-editor.org/rfc/rfc7583), NSEC3 salt rotation, and the operational tooling required to run them safely. Adding that surface to the authoritative server at this stage would expand the attack surface inside the network-facing process without a corresponding gain, and would delay the delivery of the authoritative role itself. The decision is reversible; reopening it later remains possible if a concrete use case emerges that cannot be satisfied by the external-signer model.

Validating DNSSEC by default on the recursive resolver role follows directly from the non-negotiable security posture stated in [`../CLAUDE.md`](../CLAUDE.md). A recursive resolver that does not validate cannot detect cache-poisoning, on-path answer substitution, or upstream misdirection, and reduces the integrity guarantees of the entire resolution path to those of the weakest delegation it traverses. Making validation the default, and refusing to expose a global off-switch under `DNSSEC-009`, makes the strong property part of the system's structure rather than a configuration-dependent property that can be weakened by accident or by template inheritance.

The decision is aligned with mature industry practice. In 2026, all widely-deployed modern recursive resolvers validate by default, including Unbound, Knot Resolver, PowerDNS Recursor, Google Public DNS, Cloudflare 1.1.1.1, and Quad9. A recursive resolver that does not validate by default would be regressive relative to this baseline and would force operators who deploy Heimdall into a weaker integrity posture than they obtain from the resolvers they are replacing.

Negative Trust Anchors under [RFC 7646](https://www.rfc-editor.org/rfc/rfc7646) with mandatory expiry, as fixed by `DNSSEC-016` and `DNSSEC-017`, provide a responsible escape valve for the well-understood case in which an upstream zone is documented as broken at the DNSSEC level and cannot be repaired on the operator's timescale. Requiring an expiry on every registration prevents a Negative Trust Anchor from becoming a permanent validation hole, and automatic removal at or before expiry keeps the escape valve time-bounded by construction. Requiring every registration, expiry, and manual revocation event to be auditable under `DNSSEC-018` ensures that the use of the escape valve is observable by operators and does not silently erode the integrity posture of the resolver.

Validation cost is amortised by the segregated recursive cache established by `CACHE-001` through `CACHE-006` in [`004-cache-policy.md`](004-cache-policy.md). Under realistic traffic the majority of queries are served from cache, so the per-query cost of the validating path is concentrated on the minority of queries that are cache misses, and the hot path for cache-served responses remains aligned with the performance objectives stated in [`../CLAUDE.md`](../CLAUDE.md).

Returning `SERVFAIL` on a `bogus` outcome, with an optional Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) to identify the failure class as required by `DNSSEC-013`, is RFC-conformant, is standard practice across the validating recursive resolvers listed above, and is the only response code that correctly signals that Heimdall has detected a cryptographic integrity failure and refuses to serve the data.

Requiring the forwarder role to validate DNSSEC locally on every response received from every configured upstream, as fixed by `DNSSEC-019` through `DNSSEC-024`, is a defence-in-depth measure. Local validation closes the trust chain regardless of upstream integrity. An upstream may be honest and still be buggy, slowly compromised, or reachable only through network infrastructure that the operator of Heimdall does not control; none of those failure modes can be detected by accepting the upstream's own assertion of validity. The `AD` flag, in particular, is set by the upstream on its own authority and can be forged, stripped, or silently preserved across a downgrade by a compromised upstream that returns plausible but non-validating data. Refusing to rely on the upstream `AD` flag under `DNSSEC-022`, and instead recomputing the validation outcome locally, removes that downgrade vector by construction.

Applying exactly the same validation policy to the recursive resolver role and the forwarder role, as fixed by `DNSSEC-019` and `DNSSEC-021`, keeps a single code path, a single trust anchor under `DNSSEC-023`, a single Negative Trust Anchor mechanism under `DNSSEC-024`, and a single four-state outcome across both validating roles. One mental model applies to both roles, which reduces implementation complexity, shrinks the audit surface, and eliminates a class of cross-role inconsistencies in which a response would be considered `secure` by one role and not by the other on the same instance.

Refusing to expose a configuration switch that disables DNSSEC validation on the forwarder role under `DNSSEC-020` applies to the forwarder role the same structural-gating rationale already stated for the recursive resolver role under `DNSSEC-009`. Validation is part of the system's structure, not a configuration-dependent property that can be weakened by accident or by template inheritance.

Validation cost on the forwarder role is amortised by the segregated forwarder cache fixed by `CACHE-001` through `CACHE-006` in [`004-cache-policy.md`](004-cache-policy.md). Under realistic traffic the majority of queries are served from cache, so the per-query cost of the validating path is concentrated on the minority of queries that are cache misses, and the hot path for cache-served responses remains aligned with the performance objectives stated in [`../CLAUDE.md`](../CLAUDE.md).

Mandating aggressive use of the DNSSEC-validated cache on both validating roles, fixed by `DNSSEC-025` through `DNSSEC-030`, follows directly from the decisions already taken on DNSSEC validation in sections 2A and 2B. Validation is mandatory on both validating roles under `DNSSEC-008`, `DNSSEC-009`, `DNSSEC-019`, and `DNSSEC-020`, so the cryptographic cost of obtaining a `secure`-validated `NSEC` or `NSEC3` proof is already paid on the cache-miss path. Reusing that proof to synthesise `NXDOMAIN` or `NODATA` responses locally for subsequent queries that fall within the proven interval is therefore a pure optimisation: no additional trust assumption is introduced, because the synthesised response derives entirely from data that has already crossed the `secure` bar fixed by `DNSSEC-010` and `DNSSEC-021`.

Aggressive synthesis also provides a structural mitigation against "random subdomain" attacks, in which an attacker floods the resolver with queries for non-existent names beneath a target zone in order to overwhelm the authoritative servers for that zone. A single cached `NSEC` interval can cover a large portion of the namespace; once the interval is cached, queries that fall inside it are absorbed locally by Heimdall rather than propagated to the authoritative server. The absorption is automatic, requires no additional rate-limiting surface, and strengthens Heimdall's posture on both validating roles relative to the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md).

On the privacy and performance axes, aggressive synthesis reduces the number of upstream queries emitted by either validating role. Fewer queries leave the resolver, which lowers the hot-path latency observed by clients and is coherent with the privacy posture already fixed by QNAME minimisation under `PROTO-020` through `PROTO-024` in [`006-protocol-conformance.md`](006-protocol-conformance.md): names that fall inside an already-cached proven interval are never disclosed to upstream parties at all.

The decision aligns with mature industry practice. In 2026, Unbound, Knot Resolver, and BIND all implement aggressive use of `NSEC` and `NSEC3` from the validated cache by default. A new validating resolver and forwarder that did not implement it would be regressive relative to the deployed baseline, and the synthesised responses derive from `secure`-validated data anyway, so there is no realistic operational reason to disable the behaviour. Refusing to expose an off-switch under `DNSSEC-030`, whether global or per zone, applies to aggressive synthesis the same structural-gating rationale already stated for DNSSEC validation itself under `DNSSEC-009` and `DNSSEC-020`, for QNAME minimisation under `PROTO-023`, and for 0x20 case randomisation under `PROTO-030`: a property that derives from already-validated data and improves both security and performance is made part of the system's structure, not a configuration-dependent property that can be weakened by accident or by template inheritance.

Respecting `NSEC3` opt-out under `DNSSEC-028` and the wildcard proof rules under `DNSSEC-029` is not an optional refinement; it is a correctness requirement. An opt-out `NSEC3` record proves only the non-existence of signed delegations within the covered interval, so synthesising a non-existence response for a name that could be covered by an opt-out-spanned unsigned delegation would be a spurious `NXDOMAIN` that contradicts the actual state of the DNS. Similarly, synthesising a non-existence response without excluding a possible wildcard expansion would contradict the answer the authoritative server would have produced. Both cases are standard aggressive-use pitfalls documented in [RFC 8198, section 4](https://www.rfc-editor.org/rfc/rfc8198#section-4) and [RFC 7129](https://www.rfc-editor.org/rfc/rfc7129); fixing them as normative requirements makes the correctness boundary explicit rather than implicit.

Tracking [RFC 8624](https://www.rfc-editor.org/rfc/rfc8624) for the algorithm acceptance policy applied by the validating roles, as fixed by `DNSSEC-031` through `DNSSEC-040`, delivers interoperability without concealing degradation. Zones signed with the RSASHA1 variants (algorithms 5 and 7) and with RSASHA512 (algorithm 10) still exist in production in 2026; rejecting them outright would break resolution of those zones without a local defect on Heimdall's side, and RFC 8624 marks these algorithms as MAY validate rather than MUST NOT. Supporting them under `DNSSEC-034` preserves resolution while exposing their continued use to the operator.

Exposing deprecated-algorithm use is the purpose of `DNSSEC-038` and `DNSSEC-039`. The Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) signals the condition downstream to the stub resolver and ultimately to the application or user, creating pressure on the zone operator to migrate. The structured log entry exposes the same signal to the Heimdall operator, enabling metrics and alerting on the proportion of resolution traffic that still depends on deprecated signing material. Both channels turn a silent interoperability compromise into a visible, measurable signal.

Hard rejection of the MUST NOT algorithms under `DNSSEC-035` — RSAMD5 (algorithm 1), DSA-SHA1 (algorithm 3), DSA-NSEC3-SHA1 (algorithm 6), and ECC-GOST (algorithm 12) — is non-negotiable. Each of these algorithms is either cryptographically broken or has been superseded by algorithms of materially higher assurance, and [RFC 8624](https://www.rfc-editor.org/rfc/rfc8624) classifies them as MUST NOT for validation. Compliance with that classification is a security requirement in line with the non-negotiable security posture stated in [`../CLAUDE.md`](../CLAUDE.md).

Treating a signature produced under a MUST-NOT algorithm as if it did not exist, as fixed by `DNSSEC-036`, rather than as an outright validation failure, is the correct behaviour specified by [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1) and by [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035): the validation pipeline proceeds to consider any remaining supported signatures on the same RRset. The final outcome under `DNSSEC-037` is then determined by the chain-of-trust requirements of RFC 4035: `insecure` when no chain of trust to a configured trust anchor exists, and `bogus` when a chain of trust does exist but requires a supported signature that is not present. This fallback preserves resolution for zones that are multiply signed with both supported and unsupported algorithms, and fails closed for zones whose chain of trust depends entirely on an unsupported signature.

Fixing the policy as a mirror of [RFC 8624](https://www.rfc-editor.org/rfc/rfc8624) under `DNSSEC-040`, and revising it only when the RFC is itself revised, selects incremental evolution over speculative hardening. Tracking the current standard is auditable, interoperable, and aligned with the rest of the DNSSEC-capable ecosystem. Anticipating contentious future deprecations inside Heimdall would expose operators to isolated resolution failures that do not reproduce on other validating resolvers, and would shift the cost of a policy change onto Heimdall's users rather than onto the standards process where it belongs.

Bounding the combinatorial cost of validating a single RRset under `DNSSEC-041` through `DNSSEC-043`, and capping the `NSEC3` iteration count under `DNSSEC-044` through `DNSSEC-047`, is the point at which the project's non-negotiable security posture and performance-as-primary-guide posture, both stated in [`../CLAUDE.md`](../CLAUDE.md), converge on the same answer. Both attack classes exploit the same structural weakness: they turn DNSSEC validation, which is nominally a bounded-work operation, into DNSSEC denial of service by driving the validator into unbounded work on inputs that a single adversarial zone can produce. Without structural caps on these two dimensions, a new validating resolver is trivially DoS-able by a zone that the operator of Heimdall does not control.

The 150-iteration cap fixed by `DNSSEC-044` is not a project choice — it is an obligation imposed by [RFC 9276, section 3.2](https://www.rfc-editor.org/rfc/rfc9276#section-3.2), and applies equally to every conforming validator. Accepting any iteration count in the inclusive range 0 to 150 under `DNSSEC-046`, while preferring the lower end as stated in [RFC 9276, section 3.1](https://www.rfc-editor.org/rfc/rfc9276#section-3.1), matches the guidance in the RFC: 0 is preferred for new zones, but existing zones published with non-zero counts within the cap must still resolve.

The KeyTrap cap fixed by `DNSSEC-041` and `DNSSEC-042` became baseline across mature validating resolvers — Unbound, BIND, and Knot Resolver — in 2024, following the public disclosure of [CVE-2023-50387](https://nvd.nist.gov/vuln/detail/CVE-2023-50387). A new validating resolver delivered in 2026 without an equivalent cap would be regressive relative to that deployed baseline, and would expose every operator of Heimdall to a DoS vector that the rest of the ecosystem has already closed. The precise numeric value of the cap is deferred as an open question in section 4 because it is an empirical parameter that must be fixed by benchmarking on the Heimdall validation path; the obligation to have a cap, and the handling when the cap is exceeded, are fixed here.

The asymmetry between the two mitigations on the outcome side is deliberate. Treating an `NSEC3` response whose iteration count exceeds 150 as `insecure` rather than `bogus` under `DNSSEC-045` preserves client-visible resolution where possible: the zone is still returning data, and the client receives it without the `AD` flag so that any downstream policy based on validation status can still react. Treating a KeyTrap exhaustion event as `bogus` under `DNSSEC-042` is stricter because the combinatorial blow-up is adversarial by construction: there is no honest operational reason for an RRset to exceed the validation complexity bound, and returning `SERVFAIL` refuses to serve data whose validation has been deliberately sabotaged.

Extended DNS Error signalling under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) is attached to both mitigations — as a SHOULD for the KeyTrap `bogus` outcome under `DNSSEC-042`, and as a MAY for the excessive-iteration `insecure` outcome under `DNSSEC-045` — so that the stub resolver, application, or monitoring pipeline downstream of Heimdall can observe that a mitigation has fired. The signal creates operational visibility of otherwise opaque validation trimming, and creates pressure on the operators of the zones that trigger it.

Refusing to expose a configuration switch that disables either mitigation on either validating role under `DNSSEC-043` and `DNSSEC-047` applies the same structural-gating rationale already stated for DNSSEC validation itself under `DNSSEC-009` and `DNSSEC-020`, for aggressive synthesis under `DNSSEC-030`, for QNAME minimisation under `PROTO-023`, and for 0x20 case randomisation under `PROTO-030`: a property that closes a structural DoS vector is made part of the system's structure, not a configuration-dependent property that can be weakened by accident or by template inheritance.

Tracking [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2) for the DS digest algorithm acceptance policy applied by the validating roles, as fixed by `DNSSEC-048` through `DNSSEC-054`, is the mirror decision for DS records of the signature-algorithm decision already taken for `RRSIG` records in section 2D. The two policies share a common shape: accept the digest types the RFC marks as MAY validate rather than rejecting them outright, reject only the digest types the RFC marks as MUST NOT validate, and surface deprecated usage through Extended DNS Error and structured logs so that the compromise remains visible rather than silent.

Unconditional rejection of DS records carrying digest type 1 (SHA-1) would produce `SERVFAIL` for any delegation whose parent zone still publishes only SHA-1 DS records, with no local defect on Heimdall's side. Such delegations continue to exist in production in 2026, and [RFC 8624](https://www.rfc-editor.org/rfc/rfc8624) marks digest type 1 as MAY validate rather than MUST NOT. Supporting it under `DNSSEC-051`, but only as a fallback when no DS record of a non-deprecated digest type is present at the same delegation point, preserves resolution for those zones while keeping modern delegations on modern digest types.

The security balance underlying `DNSSEC-051` is materially different from the corresponding balance for RSASHA1 signatures under `DNSSEC-034`. A DS record authenticates the child `DNSKEY` RRset at a single delegation point; exploiting a SHA-1 collision against a DS record requires the adversary to construct a colliding `DNSKEY` and to have it published at the child, which is a substantially harder capability than forging a single signature over arbitrary RRset contents. The residual risk of continuing to honour SHA-1 DS records as a fallback is therefore lower than the risk of breaking resolution for deployed zones, and the visibility mechanisms required by `DNSSEC-051` — Extended DNS Error under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) and a structured log entry — keep the operational cost of the fallback observable.

Hard rejection of DS digest type 3 (GOST) under `DNSSEC-052` follows directly from the MUST NOT classification in [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2). GOST as a DNSSEC digest is obsolete; compliance with the MUST NOT classification is a security requirement in line with the non-negotiable security posture stated in [`../CLAUDE.md`](../CLAUDE.md). Treating any digest type not defined in RFC 8624 as unsupported under `DNSSEC-053`, rather than as an outright validation failure, mirrors the fallback behaviour fixed by `DNSSEC-036` and `DNSSEC-037` for signature algorithms: the validation pipeline proceeds to consider any remaining DS records at the same delegation point under supported digest types, and fails closed only when no supported DS record remains for which a chain of trust can be closed.

Fixing the policy as a mirror of [RFC 8624, section 3.2](https://www.rfc-editor.org/rfc/rfc8624#section-3.2) under `DNSSEC-054`, and revising it only when the RFC is itself revised, selects incremental evolution over speculative hardening, for the same reasons already stated for the signature-algorithm policy under `DNSSEC-040`.

## 2G. Zone input formats for the authoritative server role

### 2G.1 Normative requirements

- **DNSSEC-055.** The canonical primary input format for pre-signed zones on the authoritative server role is the RFC 1035 master-file text format as specified by [RFC 1035, section 5](https://www.rfc-editor.org/rfc/rfc1035#section-5), extended by the `$INCLUDE`, `$ORIGIN`, `$TTL`, and `$GENERATE` directives in the BIND-compatible dialect, and further extended to accept the DNSSEC record types listed in `DNSSEC-002` inline as zone-file records. The master-file parser MUST accept all DNSSEC-record types listed in `DNSSEC-002` without stripping or ignoring them.
- **DNSSEC-056.** Zone transfer via AXFR ([RFC 5936](https://www.rfc-editor.org/rfc/rfc5936)) and IXFR ([RFC 1995](https://www.rfc-editor.org/rfc/rfc1995)) is the secondary input channel for pre-signed zones on the authoritative server role, used exclusively when Heimdall is operating as a secondary authoritative for a zone whose primary is an external server. The zone-transfer path MUST accept all DNSSEC-record types listed in `DNSSEC-002`. The zone-transfer implementation requirements are fixed by `PROTO-036` through `PROTO-049` in [`006-protocol-conformance.md`](006-protocol-conformance.md) and are not duplicated here.
- **DNSSEC-057.** JSON, YAML, and any other structured-data format are NOT supported as zone-input formats on the authoritative server role and MUST NOT be implemented. This restriction is a deliberate scope boundary, not a temporary omission.
- **DNSSEC-058.** A REST API or any HTTP-based mechanism is NOT a zone-load channel on the authoritative server role. Point edits to individual resource records, if ever needed, are performed exclusively through the admin-RPC interface defined by `OPS-007` through `OPS-015` in [`012-runtime-operations.md`](012-runtime-operations.md). The authoritative server role does not expose a REST API for zone-data ingestion.

### 2G.2 Rationale

RFC 1035 master-file text is the universal lowest common denominator for zone data: every DNS signing tool, DNSSEC signer, and authoritative server can produce and consume it. Selecting it as the canonical primary format maximises compatibility with the existing ecosystem of external signers — the only parties that supply zone data to the authoritative server role, as fixed by `DNSSEC-001` through `DNSSEC-004`. AXFR and IXFR are the standard zone-distribution mechanisms for secondary authoritative deployments and are already within scope for the authoritative server role; supporting them as a zone-input channel follows directly from the secondary-server use case rather than representing an additional format. JSON and YAML have no established ecosystem of DNS signing tools and add format-translation surface without a corresponding gain, so they are explicitly excluded. A REST API as a zone-load channel would expose zone data ingestion over the network on the authoritative path, widening the attack surface inside the network-facing process without a corresponding operational gain; the admin-RPC interface is the correct mechanism for any runtime zone-data modification.

## 2H. Signature-algorithm acceptance policy for the authoritative server role

### 2H.1 Normative requirements

- **DNSSEC-059.** The signature-algorithm acceptance policy fixed by `DNSSEC-060` through `DNSSEC-064` applies to the authoritative server role when loading and serving pre-signed zones. The policy is distinct from the validation-path algorithm acceptance policy fixed by `DNSSEC-031` through `DNSSEC-040` for the validating roles. On the authoritative path, Heimdall does not perform cryptographic verification of signatures at query time (see `DNSSEC-004`); this policy instead governs which signed zones Heimdall is willing to load and which RRSIG records it is willing to serve.
- **DNSSEC-060.** Heimdall MUST accept pre-signed zones containing `RRSIG` records produced under algorithms 8 (RSASHA256), 13 (ECDSAP256SHA256), 14 (ECDSAP384SHA384), 15 (Ed25519), and 16 (Ed448). Zones signed with any of these algorithms MUST be loaded and served without restriction.
- **DNSSEC-061.** Heimdall MUST accept pre-signed zones containing `RRSIG` records produced under algorithms 5 (RSASHA1), 7 (RSASHA1-NSEC3-SHA1), and 10 (RSASHA512), which are marked as deprecated in [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). Such zones MUST be loaded and served without refusing the zone. However, when Heimdall serves a response carrying an `RRSIG` produced under one of these deprecated algorithms, it MUST include an Extended DNS Error option under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) — specifically EDE info-code 7 (`Signature Expired`) is not appropriate; instead the EDE MUST use the PRIVATE USE range or the closest applicable code that signals deprecated algorithm use — and MUST emit a structured log entry identifying the zone, the algorithm number, and the fact that a deprecated algorithm was used in a served response.
- **DNSSEC-062.** Heimdall MUST NOT accept pre-signed zones containing `RRSIG` records produced exclusively under algorithms 1 (RSAMD5), 3 (DSA-SHA1), 6 (DSA-NSEC3-SHA1), or 12 (ECC-GOST), which are marked as MUST NOT by [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). When zone loading encounters a zone signed only under one or more of these algorithms — that is, no `RRSIG` record under a supported algorithm is present for any covered RRset — loading MUST be refused with a structured error that identifies the algorithm(s) involved. When a zone carries signatures under both MUST-NOT and supported algorithms, loading MUST succeed, and only the RRSIG records under supported algorithms MUST be served; RRSIG records under MUST-NOT algorithms MUST be dropped and MUST NOT be placed in the served zone.
- **DNSSEC-063.** The EDE signalling required by `DNSSEC-061` for deprecated-algorithm responses MUST be implemented as a response-time annotation: the authoritative server role MUST check whether the RRSIGs being attached to a response are under a deprecated algorithm and, if so, MUST append the EDE option before returning the response. The check MUST NOT add measurable latency to the hot path for responses signed exclusively under non-deprecated algorithms (algorithms 8, 13, 14, 15, 16).
- **DNSSEC-064.** The signature-algorithm acceptance policy for the authoritative server role fixed by `DNSSEC-059` through `DNSSEC-063` tracks [RFC 8624, section 3.1](https://www.rfc-editor.org/rfc/rfc8624#section-3.1). When RFC 8624 is updated or superseded, the policy MUST be revised accordingly in a future revision of this document. Heimdall MUST NOT anticipate such revisions by tightening or relaxing the policy unilaterally.

### 2H.2 Rationale

Accepting zones signed under the non-deprecated RFC 8624 algorithms — 8, 13, 14, 15, and 16 — unconditionally is straightforward: these are the algorithms the standard marks as MUST or RECOMMENDED for implementation, so refusing them would make Heimdall incompatible with all modern external signers. Accepting zones signed under the deprecated algorithms — 5, 7, and 10 — but signalling the deprecation at serve time mirrors the policy already fixed for the validating roles under `DNSSEC-034` and `DNSSEC-038`: the algorithm is not rejected outright because deployed zones still use it, but its use is surfaced to the operator and downstream client through an EDE signal so that the zone operator has a visible incentive to migrate. Refusing to load zones signed exclusively under the MUST-NOT algorithms — 1, 3, 6, and 12 — is non-negotiable: those algorithms are cryptographically broken or obsolete, and serving data signed only under them would be misleading because downstream validators will reject the signatures anyway. Dropping RRSIG records under MUST-NOT algorithms when a zone is multiply signed preserves resolution for zones that include those records alongside valid signatures, without propagating cryptographically invalid data to clients.

## 2I. NSEC and NSEC3 support on the authoritative server role

### 2I.1 Normative requirements

- **DNSSEC-065.** Heimdall MUST serve pre-signed zones authenticated with `NSEC` records in accordance with [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035) and [RFC 4034](https://www.rfc-editor.org/rfc/rfc4034). There is no restriction on the use of `NSEC` in a pre-signed zone loaded into Heimdall.
- **DNSSEC-066.** Heimdall MUST serve pre-signed zones authenticated with `NSEC3` records in accordance with [RFC 5155](https://www.rfc-editor.org/rfc/rfc5155). There is no restriction on the use of `NSEC3` in a pre-signed zone loaded into Heimdall, subject to the constraint in `DNSSEC-067`.
- **DNSSEC-067.** When a pre-signed zone uses `NSEC3`, the `NSEC3PARAM` record MUST be present in the apex RRset of the loaded zone. Heimdall MUST refuse to load a zone that contains `NSEC3` records but lacks a valid `NSEC3PARAM` record at the zone apex, and MUST emit a structured error identifying the zone and the missing record.
- **DNSSEC-068.** Zones authenticated with both `NSEC` and `NSEC3` records in the same zone are not a valid configuration under the DNSSEC standards; Heimdall MUST refuse to load a zone that contains records of both types, and MUST emit a structured error identifying the zone and the conflict.
- **DNSSEC-069.** When a pre-signed zone uses `NSEC3` with the opt-out flag set as specified in [RFC 5155, section 6](https://www.rfc-editor.org/rfc/rfc5155#section-6), Heimdall MUST serve the zone including the opt-out `NSEC3` records exactly as present in the zone data. Heimdall MUST NOT modify, strip, or alter the opt-out flag when serving `NSEC3` records.
- **DNSSEC-070.** On the authoritative server role, Heimdall does not perform aggressive NSEC/NSEC3 synthesis (that mechanism is specific to the validating roles under `DNSSEC-025` through `DNSSEC-030`). The authoritative role serves the NSEC or NSEC3 records that are present in the loaded zone in accordance with the response rules fixed by the normative requirements for the authoritative role.

### 2I.2 Rationale

Because Heimdall serves pre-signed zones on the authoritative path — and does not generate NSEC or NSEC3 records itself — the only policy question on the authoritative side is which denial-of-existence schemes it will load and serve. Accepting both NSEC and NSEC3 unconditionally covers the full range of external signers: older signers and many current signers produce NSEC; newer deployments may prefer NSEC3 for name enumeration resistance. Rejecting either scheme would arbitrarily narrow the set of zones Heimdall can serve without a corresponding security gain. The opt-out flag is a standard NSEC3 feature used in large unsigned-delegation zones; stripping it would produce incorrect denial-of-existence proofs. Refusing to load zones that mix NSEC and NSEC3, or that use NSEC3 without NSEC3PARAM, enforces correctness requirements that are already obligations of the external signer: any well-formed signed zone must satisfy these constraints, so refusing to load malformed zones fails safely at zone-load time rather than silently propagating incorrect data.

## 2J. CDS and CDNSKEY handling on the authoritative server role

### 2J.1 Normative requirements

- **DNSSEC-071.** On the authoritative server role, Heimdall MUST serve `CDS` and `CDNSKEY` records that are present in the pre-signed zone data loaded into the server. These records MUST be served in response to queries for the `CDS` and `CDNSKEY` types in accordance with the response rules for the authoritative role, exactly as any other record type present in the zone.
- **DNSSEC-072.** Heimdall MUST NOT generate, derive, or synthesise `CDS` or `CDNSKEY` records from any source other than the pre-signed zone data. In particular, Heimdall MUST NOT construct `CDS` or `CDNSKEY` records from the `DNSKEY` or `DS` records present in the zone, from trust-anchor state, or from any other local state. The production of `CDS` and `CDNSKEY` records is the exclusive responsibility of the external signer.
- **DNSSEC-073.** The handling of `CDS` and `CDNSKEY` queries fixed by `DNSSEC-071` and `DNSSEC-072` applies uniformly regardless of whether the `CDS` DELETE record specified by [RFC 8078, section 4](https://www.rfc-editor.org/rfc/rfc8078#section-4) is present in the zone: Heimdall serves whatever `CDS` and `CDNSKEY` records the external signer has placed in the zone and does not interpret their content.

### 2J.2 Rationale

`CDS` and `CDNSKEY` are signalling records whose content is produced by the external signer in coordination with the parent zone's DS management process. Generating them from local state would place Heimdall in a signalling role that it does not occupy: the authoritative server role is a serving role, not a key-management role. Generating these records from local DNSKEY data could produce signals that diverge from the external signer's intended delegation-management state, corrupting the DS update protocol for the zone. The constraint mirrors the pre-signed-zone model of `DNSSEC-001` through `DNSSEC-004`: all DNSSEC material originates outside Heimdall, and Heimdall's function is to serve it faithfully.

## 2K. Load-time integrity validation of pre-signed zones

### 2K.1 Normative requirements

- **DNSSEC-074.** Heimdall MUST NOT perform cryptographic `RRSIG` verification against the `DNSKEY` RRset at zone-load time. Load-time signature verification is an offline operation that belongs to the external signer's workflow, not to the serving process. Repeating it inside Heimdall at zone-load time would introduce a significant CPU cost on every zone load or SIGHUP reload without providing a corresponding security gain, because Heimdall never generates or holds the private keys required to regenerate or re-verify signatures from a position of trust.
- **DNSSEC-075.** Heimdall MUST perform structural and syntactic validation of every DNSSEC record in a zone at load time. This validation MUST include: (a) confirming that each `RRSIG` record has a valid wire-format structure as specified by [RFC 4034](https://www.rfc-editor.org/rfc/rfc4034), including that the `Signature Inception` and `Signature Expiration` fields are parseable timestamps and that the `Algorithm` field contains a value consistent with the record's structure; (b) confirming that each `DNSKEY` record has valid wire-format structure; (c) confirming that the `NSEC3PARAM` record at the apex, if present, contains a syntactically valid iteration count, salt, and hash algorithm. Structural validation failures MUST cause the load to be refused.
- **DNSSEC-076.** Heimdall MUST check, at zone-load time, that every `RRSIG` record in the zone covers an RRset that is present in the zone. An `RRSIG` record that covers a type-name pair absent from the zone is a dangling signature; Heimdall MUST emit a structured warning log entry for each dangling signature encountered and MUST drop the dangling `RRSIG` from the loaded zone data without refusing the entire load.
- **DNSSEC-077.** Heimdall MUST check, at zone-load time, that `RRSIG` records in the zone are not expired at the time of loading, using the server's local clock. A zone in which all `RRSIG` records covering a given RRset have already expired MUST NOT be loaded; Heimdall MUST refuse the load and MUST emit a structured error identifying the zone and the first expired RRset encountered. A zone in which some but not all `RRSIG` records are expired MUST be loaded, but Heimdall MUST emit a structured warning log entry for each expired `RRSIG` encountered.
- **DNSSEC-078.** The load-time checks defined in `DNSSEC-075` through `DNSSEC-077` MUST be applied consistently on both the initial load and every subsequent SIGHUP-triggered reload. A reload that would produce a zone that fails any of these checks MUST be refused; the previously loaded zone data MUST remain active until a valid replacement is presented.

### 2K.2 Rationale

Refusing to perform full cryptographic RRSIG verification at load time is a deliberate performance decision: zone loads happen on startup and on every SIGHUP reload, and verifying every signature in a large zone would introduce a CPU-intensive blocking operation on the reload path, conflicting with the atomic-swap, no-listener-tear-down reload requirement fixed by `OPS-016` through `OPS-018` in [`012-runtime-operations.md`](012-runtime-operations.md). The security justification for that omission is that Heimdall is not in a position of trust with respect to the zone's signing keys — it cannot distinguish a correct signature from a substituted one without the corresponding private key — so the verification would only catch accidental corruption, not adversarial substitution. Structural validation under `DNSSEC-075`, expiry checking under `DNSSEC-077`, and dangling-signature detection under `DNSSEC-076` together provide the defences that are both achievable and useful on the load path: they catch malformed input, expired zones that would serve invalid data, and minor inconsistencies introduced by a signer that deleted a record without regenerating the covering `RRSIG`. Refusing a reload that would replace a valid zone with an invalid one under `DNSSEC-078` applies the fail-safe posture stated in [`../CLAUDE.md`](../CLAUDE.md): the server remains in a known-good state rather than transitioning to a partially invalid one.

## 2M. KeyTrap cap: DNSKEY × RRSIG combination limit

### 2M.1 Normative requirements

- **DNSSEC-086.** The KeyTrap cap applied by `DNSSEC-041` MUST enforce two separate sub-limits and one product limit per RRset validation attempt on either validating role:
  - Maximum **4** distinct `DNSKEY` records considered during the validation of any single RRset (the key limit). When more than 4 `DNSKEY` records are present in the zone's `DNSKEY` RRset, the 4 records considered MUST be selected by preferring, in order: records whose algorithm matches the algorithm of a covering `RRSIG` on the RRset being validated, records flagged as Zone Signing Key (ZSK) over records flagged as Key Signing Key (KSK) for non-`DNSKEY` RRsets, and finally records in wire-format order. This selection rule is deterministic and repeatable.
  - Maximum **8** distinct `RRSIG` records considered during the validation of any single RRset (the signature limit). When more than 8 `RRSIG` records cover the same RRset, the 8 records considered MUST be selected by preferring records whose `Algorithm` field corresponds to an algorithm supported by `DNSSEC-032` and `DNSSEC-033` (MUST and RECOMMENDED), then records under algorithms supported by `DNSSEC-034` (MAY), in wire-format order within each preference tier.
  - Maximum **32** total cryptographic verification attempts across all `DNSKEY` × `RRSIG` pairs considered for a single RRset (the product cap). If, after applying the key limit and the signature limit, the number of pairs would exceed 32, Heimdall MUST stop after 32 attempts, regardless of whether `secure` has been reached, and proceed to the outcome determination step under `DNSSEC-042`.
- **DNSSEC-087.** The numeric values fixed by `DNSSEC-086` — key limit 4, signature limit 8, product cap 32 — are the initial defaults established at project inception. These values MUST be verified by benchmarking on the Heimdall validation path during the implementation sprint for the DNSSEC primitives crate (Sprint 16). If benchmarking shows that the product cap of 32 causes a measurable performance regression on realistic inputs, or that the defaults are insufficient to contain the KeyTrap vector, the values MUST be revised in a subsequent revision of this document before that sprint is closed. The structural obligation (`DNSSEC-041`), the outcome semantics (`DNSSEC-042`), and the no-off-switch rule (`DNSSEC-043`) remain fixed regardless of the numeric revision.

### 2M.2 Rationale

The values 4 (key limit), 8 (signature limit), and 32 (product cap) match the post-CVE-2023-50387 defaults shipped by Unbound 1.19.0 — the first widely deployed recursive resolver to publish a fix for the KeyTrap class of attacks. Aligning with Unbound's post-patch defaults provides a conservative starting point that has been validated in production, while the benchmarking requirement under `DNSSEC-087` ensures that the values are calibrated to the Heimdall validation path before the implementation is shipped. The DNSKEY selection rule (prefer ZSK for non-DNSKEY RRsets, prefer algorithms matching a covering RRSIG) maximises the probability of reaching `secure` within the cap on honest inputs, while the signature selection rule (prefer MUST/RECOMMENDED algorithms) ensures that the most secure signatures are attempted first. Together, these selection rules make the cap transparent to honest zones that use a single signing algorithm and a single key rollover window, which is the overwhelming majority of DNSSEC-signed zones in production.

## 2L. Handling of queries for DNSSEC meta-types on the authoritative server role

### 2L.1 Normative requirements

- **DNSSEC-079.** On the authoritative server role, queries for the following DNSSEC meta-type record types MUST be handled as regular type queries from the loaded zone data: `DNSKEY`, `NSEC`, `NSEC3`, `NSEC3PARAM`, `RRSIG`, `DS`, `CDS`, `CDNSKEY`, and `CSYNC`. Heimdall MUST return the RRset for the queried type if one is present in the zone at the queried owner name, following the standard authoritative response rules of [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035) and [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035).
- **DNSSEC-080.** When a `DNSKEY` query arrives at the authoritative server role, Heimdall MUST serve the `DNSKEY` RRset present in the zone at the queried owner name. Heimdall MUST include the corresponding `RRSIG` records covering the `DNSKEY` RRset when the originating query had the `DO` flag set. If the queried name is not the zone apex, Heimdall MUST respond with `NOERROR` and an empty answer section if no `DNSKEY` RRset is present at that name, with the appropriate `NSEC` or `NSEC3` denial-of-existence proof when the query had the `DO` flag set.
- **DNSSEC-081.** When a `DS` query arrives at the authoritative server role for a name within the served zone, Heimdall MUST check whether the queried name is a zone cut (a delegation boundary). If the name is a zone cut, Heimdall MUST return the `DS` RRset present in the zone at that name if one exists, or an authoritative `NOERROR` with a denial-of-existence proof if no `DS` RRset is present. If the name is not a zone cut, Heimdall MUST respond with `NOERROR` and an empty answer section or denial-of-existence proof as appropriate. Heimdall MUST NOT return `REFUSED` or `NOTIMP` for `DS` queries.
- **DNSSEC-082.** When a `NSEC` or `NSEC3` query arrives at the authoritative server role, Heimdall MUST serve the `NSEC` or `NSEC3` record present in the zone at the queried owner name, if one exists. These records are regular zone data on the authoritative path and MUST be served on direct query.
- **DNSSEC-083.** When a `CDS` or `CDNSKEY` query arrives at the authoritative server role, Heimdall MUST serve the `CDS` or `CDNSKEY` RRset present in the zone at the queried owner name, if one exists, in accordance with `DNSSEC-071`. These are regular query types and MUST NOT be suppressed or intercepted.
- **DNSSEC-084.** When a `CSYNC` query arrives at the authoritative server role, Heimdall MUST serve the `CSYNC` RRset present in the zone at the queried owner name, if one exists. `CSYNC` records are regular zone data and MUST be served on direct query.
- **DNSSEC-085.** None of the DNSSEC meta-types listed in `DNSSEC-079` through `DNSSEC-084` require special treatment beyond what is already required for any other query type on the authoritative server role: correct `RCODE`, correct `AA` flag, correct denial-of-existence proof when the `DO` flag is set, and correct `RRSIG` inclusion when the `DO` flag is set and an RRset is returned. Heimdall MUST NOT respond with `NOTIMP` or `REFUSED` for any of these types unless the queried name is not within a zone served by this instance.

### 2L.2 Rationale

DNSSEC meta-type records are first-class zone data on the authoritative path: they originate from the external signer, live in the zone file, and are loaded alongside all other records. There is no operational reason to intercept or suppress queries for these types, and doing so would break DNSSEC clients and parent-side automation — DS queries, CDS/CDNSKEY automation under [RFC 8078](https://www.rfc-editor.org/rfc/rfc8078), and CSYNC automation under [RFC 7477](https://www.rfc-editor.org/rfc/rfc7477) — that rely on being able to query these types directly from the authoritative. The DS record requires a zone-cut check because its ownership semantics differ from most other types: it is owned by the parent of the delegation, so the authoritative server for the parent zone serves it as a regular record, while the authoritative server for the child zone is not authoritative for the DS at the child's apex. This is standard RFC 4035 behaviour and requires no special-casing beyond the normal zone-cut detection already present in the authoritative query-processing path.

## 4. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will be specified incrementally in subsequent revisions of this document.

- **Per-query CPU budget for DNSSEC validation.** An upper bound on the wall-clock CPU time spent validating a single query on the validating roles (recursive resolver, forwarder), together with the abort-and-Extended-DNS-Error semantics when the bound is exceeded, is **to be specified**. This is a supplementary protection beyond the two structural caps fixed by `DNSSEC-041` through `DNSSEC-047`. The policy applies uniformly across both validating roles.
- **Extended DNS Error codes accompanying a bogus SERVFAIL.** The precise set of Extended DNS Error codes under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) that MUST accompany a `SERVFAIL` response produced by `DNSSEC-013` on either validating role (recursive resolver, forwarder) is **to be specified**. The policy applies uniformly across both validating roles.
- **Negative Trust Anchor operational defaults.** The operational defaults applicable to Negative Trust Anchors under `DNSSEC-016` through `DNSSEC-018` — including the default expiry time, the maximum admissible expiry time, the listing interface exposed to operators, and the audit-log format — are **to be specified**. The policy applies uniformly to the validating roles (recursive resolver, forwarder) in accordance with `DNSSEC-024`.

No implementation activity may proceed on the basis of assumptions about any of the items above.
