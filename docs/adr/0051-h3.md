# ADR-0051: h3 for HTTP/3 server implementation

## Context

Sprint 25 requires a DoH over HTTP/3 listener (RFC 9114) to satisfy `NET-006`,
`NET-007`, and `SEC-036..046` in `specification/002-transports.md` and
`specification/003-crypto-policy.md`.

HTTP/3 is substantially more complex than HTTP/2:

- It runs over QUIC rather than TCP; framing, stream multiplexing, and flow
  control are delegated to the QUIC layer.
- Header compression uses QPACK (RFC 9204) instead of HPACK; QPACK requires
  separate encoder and decoder streams in addition to the request/response
  streams, and dynamic table management must be bounded to prevent decompression
  bombs (`SEC-040`).
- The HTTP/3 control stream carries SETTINGS, GOAWAY, and MAX_PUSH_ID frames
  with their own rate-limiting concerns (`SEC-043`).
- Request and response framing uses DATA and HEADERS frames over QUIC
  unidirectional and bidirectional streams rather than the TCP framing of HTTP/2.

Implementing HTTP/3 framing, QPACK, and control-stream state management from
scratch on top of quinn's raw QUIC streams would constitute a re-implementation
of the h3 crate without the benefit of that crate's development and test history.
QPACK in particular is non-trivial: the static and dynamic table lookups, the
encoder/decoder stream protocol, and the blocking avoidance semantics are
specified across RFC 9204 and its errata.

A purpose-built HTTP/3 library is therefore required.

## Decision

Use **`h3` v0.0.6** as the HTTP/3 server (and client, for tests) implementation
over QUIC.

`h3` is added under `[dependencies]` in `crates/heimdall-runtime/Cargo.toml`.

## Considered options

### Option A — `h3` v0.0.6 (selected)

Pure-Rust HTTP/3 implementation providing:
- `h3::server::Connection` for the server accept loop.
- `h3::server::Builder` with `max_header_list_size` and QPACK table cap configuration.
- HEADERS frame parsing and emission with QPACK encoder/decoder.
- Control-stream handling (SETTINGS, GOAWAY, MAX_PUSH_ID).
- MIT OR Apache-2.0 licence.
- Zero `unsafe` blocks in the Rust layer (QPACK and framing are pure Rust).
- The only production-quality pure-Rust HTTP/3 implementation available in 2026.
- Maintained as part of the hyperium organisation (same stewardship as hyper and h2).

Cons: pre-1.0 status (`0.0.x`); semver guarantees are not stable — breaking API
changes may occur between minor versions. This risk is accepted because:
(a) there is no viable alternative for pure-Rust HTTP/3; (b) the h3 API surface
used by Heimdall is limited to the server accept loop, SETTINGS configuration,
and request/response framing — a stable and narrow subset; (c) h3's version pin
in `Cargo.toml` bounds the risk to explicit upgrade decisions.

### Option B — Manual HTTP/3 framing on top of quinn raw streams

Build HTTP/3 framing, QPACK, control-stream management, and SETTINGS handling
directly on top of `quinn::Connection` and its bidirectional/unidirectional
streams.

- Pros: no additional dependency.
- Cons: QPACK is the dominant complexity; RFC 9204 requires a static table of
  61 predefined entries, a dynamic table with insertion, eviction, and
  reference-counting, and the encoder-stream/decoder-stream protocol. Implementing
  this correctly, securely, and with the QPACK decompression-bomb mitigations
  required by `SEC-040` would take significantly more effort than the integration
  work for the `h3` crate, and the result would be less tested. Rejected.

### Option C — `salvo` or other web-framework HTTP/3 layers

High-level web frameworks (salvo, poem, axum) expose HTTP/3 listeners but
abstract away the SETTINGS/QPACK/control-stream configuration that `SEC-036`
through `SEC-045` require Heimdall to apply explicitly. Heimdall's hardening
requirements prohibit relying on default library behaviour; a framework that hides
those configuration knobs is therefore not usable. Rejected.

## Classification

**Core-critical** (runtime dependency). `h3` is required to implement the DoH
over HTTP/3 listener mandated by `NET-006`. Without it, HTTP/3 framing and QPACK
must be reimplemented from scratch. It cannot be removed without replacing the
entire DoH/H3 transport.

## Pre-1.0 semver implications

Because `h3` is at `0.0.x`, the crate's own semver policy allows breaking changes
at any minor version bump. The `Cargo.toml` pin uses an exact minor-version
constraint (`"0.0.6"`) rather than a caret range, so cargo will not automatically
pull in a breaking `0.0.7` without an explicit version bump and review. Every
future upgrade of `h3` must be accompanied by a review of the API changes and an
update to this ADR.

## Consequences

- `h3 = "0.0.6"` is added under `[dependencies]` in
  `crates/heimdall-runtime/Cargo.toml`.
- The `Doh3Listener` in `transport/doh3.rs` uses `h3::server::builder()` with
  explicit `max_header_list_size` and QPACK table cap settings (SEC-037, SEC-040).
- HTTP/3 features excluded by `NET-028` (server push, WebTransport, CONNECT,
  HTTP/3 CONNECT, DATAGRAMS) are not enabled; `h3`'s API does not expose them by
  default on the server accept path, and push is explicitly disabled via builder
  configuration.
- The `h3` crate version is pinned; any upgrade requires this ADR to be updated.
