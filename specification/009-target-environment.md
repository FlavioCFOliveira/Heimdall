# Target environment: operating systems and CPU architectures

**Purpose.** This document defines the operating systems and CPU architectures that Heimdall targets, the minimum kernel or platform baselines that apply to each target tier, and the mandatory artefact formats through which Heimdall is distributed. It fixes Linux as the first-class production platform, FreeBSD and OpenBSD as best-effort production platforms, macOS as a development-only platform, and Windows as explicitly out of scope; it fixes `x86_64`, `aarch64`, and `riscv64` as the three production CPU architectures on Linux, with a narrower architecture set on BSD and macOS; and it fixes a multi-format primary distribution model covering static Linux binaries, native Linux packages, and a hardened OCI container image, with `cargo install` retained as an additional source path for developers. Other dimensions of the target environment (runtime-privilege model beyond what is already fixed by the threat model) are not fixed by this document and are deferred to subsequent environment decisions.

**Status.** Stable.

**Requirement category.** `ENV`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the operational-hardening profile whose platform-specific mechanisms are cross-referenced by this document (`pledge`, `unveil`, the macOS sandbox profile), see [`007-threat-model.md`](007-threat-model.md). For the reference hardware against which quantitative performance targets are evaluated on Linux, see [`008-performance-targets.md`](008-performance-targets.md).

## 1. Target tiers

Heimdall classifies every supported platform into one of four tiers. The tier of a platform determines the obligations that apply to Heimdall on that platform.

- **First-class production target.** The platform on which Heimdall is optimised, on which production deployment is supported, and against which quantitative performance targets are evaluated.
- **Best-effort production target.** A platform on which Heimdall MUST remain buildable and functional, on which production deployment is permitted, and on which no optimisation work disproportionate to the first-class target is obligated.
- **Development-only target.** A platform on which Heimdall MUST build and run its tests to support developer machines. Production deployment on this platform is not supported.
- **Out of scope.** A platform that Heimdall MUST NOT attempt to support as a build, test, or deployment target.

## 2. Normative requirements

### 2.1 Linux

- **ENV-001.** Linux MUST be treated as Heimdall's first-class production target. The optimisation effort invested in the Linux path MAY exceed the effort invested in any other platform, and the quantitative performance targets defined in [`008-performance-targets.md`](008-performance-targets.md) MUST be evaluated on Linux reference baselines.
- **ENV-002.** For production deployments on Linux, the Linux kernel version running on the host MUST be 6.1 LTS or later. A Linux host whose kernel version is older than 6.1 MUST NOT be treated as a supported production environment.
- **ENV-003.** Heimdall's hot path on Linux MUST target `io_uring` as the primary I/O mechanism. The `io_uring` hot path MAY rely on the following features that are mature in the 6.1 LTS baseline fixed by `ENV-002`: multishot operations, buffer rings, and zerocopy send.
- **ENV-004.** Heimdall on Linux MAY leverage the following additional kernel features when they are available on the host and when their use is justified by a measured performance or security benefit: `AF_XDP` as an optional optimisation, the BPF Linux Security Module (BPF LSM), `cgroups v2`, and the MGLRU memory-reclaim subsystem. Use of these features MUST NOT be made mandatory on all Linux deployments; a deployment on the baseline fixed by `ENV-002` MUST remain functional without them.
- **ENV-005.** Heimdall MAY provide a fallback I/O path on Linux that relies on older or widely-used kernel primitives (`epoll`, classical socket APIs) in place of `io_uring`. When such a fallback path is provided, it MUST be controlled by a feature flag, MUST NOT be the primary path on kernels that satisfy `ENV-002`, and MUST NOT compromise the security invariants established elsewhere in this specification.

### 2.2 BSD

- **ENV-006.** FreeBSD and OpenBSD MUST be treated as best-effort production targets. Heimdall MUST remain buildable and functional on both FreeBSD and OpenBSD, and MAY be deployed in production on either platform at the operator's discretion.
- **ENV-007.** On OpenBSD, the Heimdall process MUST use `pledge(2)` and `unveil(2)` as required by `THREAT-029` in [`007-threat-model.md`](007-threat-model.md). This document does not relax that requirement; it reaffirms it as part of the BSD target tier.
- **ENV-008.** On FreeBSD and OpenBSD, Heimdall's I/O path MAY rely on `kqueue` as the primary event-notification mechanism. The throughput ceiling achievable on a `kqueue`-based path MAY be below the ceiling achievable on the Linux `io_uring` path established by `ENV-003`, and this difference MUST NOT be treated as a defect on the BSD target tier.
- **ENV-009.** No optimisation work on the BSD target tier is obligated to match, in scope or depth, the optimisation work invested in the Linux target tier under `ENV-001` through `ENV-005`. Platform-specific optimisations on BSD MAY be undertaken, but they MUST NOT compromise the Linux first-class posture fixed by `ENV-001`.

### 2.3 macOS

- **ENV-010.** macOS MUST be treated as a development-only target. Heimdall MUST build on macOS and MUST run its test suite on macOS so that macOS developer machines remain a usable development environment.
- **ENV-011.** macOS MUST NOT be treated as a production target. Production deployment of Heimdall on macOS is not supported, even when the process is capable of running on that platform. Heimdall MUST NOT be described, documented, or configured as if macOS were a production target.
- **ENV-012.** A reference sandbox profile for macOS developer use MUST be provided as required by `THREAT-030` in [`007-threat-model.md`](007-threat-model.md). This document does not relax that requirement; it reaffirms it as part of the macOS target tier.

### 2.4 Windows

- **ENV-013.** Windows is explicitly OUT OF SCOPE at this stage. Heimdall MUST NOT attempt to support Windows as a build target, as a test target, or as a deployment target. No code path, configuration option, or reference artefact of Heimdall MAY be added for the purpose of supporting Windows.
- **ENV-014.** Windows MUST NOT be described, documented, or configured as a platform on which Heimdall runs. The decision to exclude Windows MAY be reconsidered in a later iteration of this document if the operational landscape changes; until such a revision is explicitly made, Windows remains out of scope.

### 2.5 CPU architectures

- **ENV-015.** Heimdall MUST support three production CPU architectures on Linux: `x86_64` (Intel and AMD), `aarch64` (ARMv8-A and later), and `riscv64` (the `riscv64gc` ABI per the rustc default, which includes the IMAFDC extensions). All three architectures MUST be treated as first-class on the Linux target tier fixed by `ENV-001` and `ENV-002`, with the caveat that absolute performance targets on current RISC-V server-class hardware MAY be more modest than on the `x86_64` or `aarch64` baselines while the RISC-V ecosystem matures.
- **ENV-016.** On the BSD best-effort production target tier fixed by `ENV-006` through `ENV-009`, Heimdall MUST support `x86_64` and `aarch64`. `riscv64` is OUT OF SCOPE on FreeBSD and OpenBSD at this stage and MUST NOT be treated as a supported build, test, or deployment target on those platforms.
- **ENV-017.** On the macOS development-only target tier fixed by `ENV-010` through `ENV-012`, Heimdall MUST support `x86_64` (Intel Macs) and `aarch64` (Apple Silicon). `riscv64` is OUT OF SCOPE on macOS and MUST NOT be treated as a supported build or test target on that platform.
- **ENV-018.** 32-bit CPU architectures, including `i386`, `armv7`, and `riscv32`, are explicitly OUT OF SCOPE on every target tier. Heimdall MUST NOT attempt to support any 32-bit architecture as a build, test, or deployment target. No code path, configuration option, or reference artefact MAY be added for the purpose of supporting a 32-bit architecture.
- **ENV-019.** Big-endian variants of the supported architectures are explicitly OUT OF SCOPE on every target tier. Heimdall MUST NOT attempt to support big-endian targets. No code path, configuration option, or reference artefact MAY be added for the purpose of supporting a big-endian architecture.
- **ENV-020.** The per-cell performance matrix defined in [`008-performance-targets.md`](008-performance-targets.md), in particular the reference-hardware subsection under `PERF-010` through `PERF-013`, MUST be extended to a third architecture column for `riscv64` on the Linux target tier. Targets on `riscv64` MUST be expressed on a separate RISC-V reference-hardware baseline and MAY be more modest in absolute QPS and latency terms than on the `x86_64` or `aarch64` baselines, reflecting the current state of RISC-V server-class hardware. The "parity on plain DNS / exceed on encrypted transports" ambition fixed by `PERF-019` through `PERF-024` MUST be applied per architecture column, comparing Heimdall against best-in-class reference implementations that are themselves running on the same architecture.
- **ENV-021.** The Heimdall release process MUST build, test, and distribute artefacts for all three production CPU architectures on Linux (`x86_64`, `aarch64`, `riscv64`). The release process MUST NOT omit artefacts for any of the three architectures except through an explicit, dated exception recorded alongside the release.
- **ENV-022.** The `riscv64gc-unknown-linux-gnu` Rust target is Tier 2 at the time of writing. The project MUST track the tier status of this target across rustc releases and MUST address any Tier 2 specific issues (including but not limited to reduced host-test coverage and the absence of automated pre-release testing on this target) as they arise. A promotion of `riscv64gc-unknown-linux-gnu` to Tier 1 MUST NOT be assumed before it is officially announced.

### 2.6 Deployment model and artefact formats

- **ENV-023.** Heimdall MUST be distributed as first-class artefacts in multiple formats, so that the full range of realistic deployment patterns on the target tiers fixed by `ENV-001` through `ENV-022` is supported without forcing operators through any particular ecosystem. The mandatory formats are fixed by `ENV-024` through `ENV-026`. No one of these formats MAY be treated as the exclusive primary format; all three MUST be produced for every official release.
- **ENV-024.** Heimdall MUST be distributed, for every supported Linux production architecture fixed by `ENV-015` (`x86_64`, `aarch64`, `riscv64`), as a statically linked Linux binary produced against musl libc. Each architecture-specific static binary MUST be released as a `.tar.gz` archive. The static binary MUST have no runtime dependency on any dynamically loaded shared library on the host.
- **ENV-025.** Heimdall MUST be distributed, for every supported Linux production architecture fixed by `ENV-015`, as native Linux packages in both `.deb` format (for Debian-family distributions) and `.rpm` format (for Red Hat, Fedora, and SUSE families). Each native package MUST install the Heimdall binary, the configuration templates, and the reference systemd unit file required by `THREAT-025` in [`007-threat-model.md`](007-threat-model.md). A native package that does not statically link the Heimdall binary MUST bundle a pinned runtime; native packages that rely on an unpinned host-provided dynamic runtime MUST NOT be produced.
- **ENV-026.** Heimdall MUST be distributed as a hardened OCI container image, published for every supported Linux production architecture fixed by `ENV-015`. The container image MUST be built on a minimal base (for example, a distroless, Alpine, or equivalent minimal base image; the exact base image choice is an operational follow-up tracked in section 4) and MUST contain only the Heimdall binary, the trust-anchor material required by the validating roles, the root certificate material required by the encrypted transports, and the irreducible minimum of runtime files necessary for the process to start and serve traffic. The production container image MUST NOT contain a shell, a package manager, or any debugging tool.
- **ENV-027.** The reference integration files that Heimdall ships under version control per `THREAT-031` in [`007-threat-model.md`](007-threat-model.md) — the reference systemd unit file required by `THREAT-025`, the reference OpenBSD `rc.d` script required by `THREAT-029`, and the reference macOS sandbox profile for developer use required by `THREAT-030` — MUST be included in the release artefact set. The `.deb` and `.rpm` packages mandated by `ENV-025` MUST install the reference systemd unit file; the OpenBSD `rc.d` script and the macOS sandbox profile MUST be published alongside the release as source-distribution artefacts.
- **ENV-028.** `cargo install` MUST remain a viable installation path for developers and contributors building Heimdall from source. This path is additional to, not a substitute for, the artefact formats mandated by `ENV-024` through `ENV-026`, and its availability MUST NOT be used to reduce the scope of those formats.
- **ENV-029.** Every released artefact in the formats mandated by `ENV-024`, `ENV-025`, and `ENV-026` — the static `.tar.gz` archive, each `.deb` package, each `.rpm` package, and each container image — MUST be signed in accordance with the supply-chain signing requirements fixed by `THREAT-013` in [`007-threat-model.md`](007-threat-model.md). The signing metadata for each artefact MUST be published alongside that artefact.
- **ENV-030.** A Software Bill of Materials (SBOM) MUST accompany each official release in accordance with `THREAT-014` in [`007-threat-model.md`](007-threat-model.md). The SBOM MUST cover every artefact produced for the release across every format mandated by `ENV-024` through `ENV-026` and every supported architecture fixed by `ENV-015`.
- **ENV-031.** Dynamically linked Linux binaries MUST NOT be a first-class primary distribution format for Heimdall. A dynamically linked build against glibc MAY be produced for developer convenience under `ENV-028`, but it MUST NOT be the primary distribution path and MUST NOT replace any of the formats mandated by `ENV-024` through `ENV-026`.
- **ENV-032.** Per-operating-system curl-pipe installers (installation scripts fetched and executed directly from a URL) MUST NOT be provided by the project. Installation MUST proceed through one of the formats mandated by `ENV-024` through `ENV-026`, or through the source path permitted by `ENV-028`.
- **ENV-033.** The deployment-model obligations fixed by `ENV-023` through `ENV-032` MUST apply uniformly across every supported Linux production architecture fixed by `ENV-015`. The release process MUST NOT omit any mandated format for any supported architecture except through an explicit, dated exception recorded alongside the release, on the same basis already established by `ENV-021` for architecture coverage.

## 3. Rationale

Classifying Linux as the first-class production target under `ENV-001` and `ENV-002` reflects the operational reality of the DNS deployment ecosystem in 2026. Server operators, cloud virtual machines, Kubernetes clusters, and container platforms on which Heimdall is realistically deployed are virtually entirely Linux-based. Concentrating the optimisation investment on Linux therefore yields the largest return; spreading the investment uniformly across every Unix-like platform would dilute focus without serving a corresponding share of operators.

Fixing Linux 6.1 LTS as the minimum kernel version under `ENV-002` is driven by the maturity profile of the kernel features on which Heimdall's hot path depends. `io_uring` in particular became materially more capable between 5.x and 6.1, with multishot operations, buffer rings, and zerocopy send reaching a level of stability and breadth suitable for a production DNS hot path. Targeting an older baseline would either force Heimdall to avoid these primitives (forgoing the performance differentiator they provide) or force it to paper over kernel-version differences at runtime (adding complexity and attack surface). 6.1 is a long-term-support release, so operators following standard distribution practices already have access to it without opting into bleeding-edge kernels.

Targeting `io_uring` as the primary hot path under `ENV-003` is a direct consequence of the "exceed on encrypted transports" ambition fixed by `PERF-023` and `PERF-024` in [`008-performance-targets.md`](008-performance-targets.md). `kqueue`, the native BSD event-notification primitive, lacks equivalents for some hot-path patterns on which `io_uring` is relied upon (in particular, the batched submission and completion model that reduces syscall overhead in the high-connection-count encrypted-transport cells). A Linux path that forwent `io_uring` would be bound by the same ceiling that constrains BSD and would not meet the encrypted-transport ambition.

Providing an optional `epoll`-based fallback under `ENV-005` preserves the ability to run Heimdall in constrained environments where `io_uring` is either absent (older kernels) or unavailable at runtime (for example, restrictive seccomp filters in some container deployments that block the `io_uring` syscalls). Gating the fallback behind a feature flag ensures that the primary path remains the one optimised for, and that the fallback never silently replaces the hot path on a modern kernel.

Classifying FreeBSD and OpenBSD as best-effort production targets under `ENV-006` through `ENV-009` honours two legitimate niches. OpenBSD provides the process-isolation primitives (`pledge`, `unveil`) already required by `THREAT-029`, and its user base explicitly values those primitives as a structural part of the deployment security posture. FreeBSD provides jails, a mature networking stack, and a substantial share of the general-purpose server ecosystem. Both platforms can be supported with minimal additional engineering effort via `cfg`-gated code, without forcing the Linux path to regress. Treating them as best-effort, rather than as first-class, makes explicit that no obligation exists to match the Linux throughput ceiling on these platforms.

Classifying macOS as development-only under `ENV-010` through `ENV-012` is an honest statement of the operational reality: no DNS server whose design point is extreme performance under high concurrency is deployed in production on macOS. Developer machines on Mac remain a significant share of the contributor base, however, so the build-and-test obligation is retained as a first-class concern. Documenting macOS as non-production avoids the misleading impression that the process running on a developer's machine constitutes a supported deployment.

Excluding Windows outright under `ENV-013` and `ENV-014` reflects three convergent factors: Windows' network stack, its capability model, and its deployment ecosystem differ from the Unix-like platforms in ways that would require substantial parallel engineering effort; there is no established demand for a high-performance DNS server on Windows that would justify that effort; and admitting Windows to any scope — even best-effort — would force the project to carry a parallel set of reference artefacts (equivalent to the systemd unit, the OpenBSD `rc.d` script, and the macOS sandbox profile required by `THREAT-031`) that would dilute focus without serving operators. The decision is open to revision if the operational landscape changes, but until a revision is made explicitly, the exclusion is total.

Extending the supported CPU-architecture set under `ENV-015` through `ENV-022` to a tri-architecture model (`x86_64`, `aarch64`, `riscv64`) on Linux positions Heimdall ahead of the deployment curve rather than behind it. `x86_64` and `aarch64` are both mainstream in 2026, with `aarch64` no longer a niche in the server ecosystem: AWS Graviton3 and Graviton4, Ampere Altra and AmpereOne, and comparable Arm-based servers are widely deployed. `riscv64` is emerging fast: cloud providers have begun offering RISC-V instances, the major Linux distributions (Debian, Ubuntu, Fedora) treat `riscv64` as a first-class port, and rustc supports the `riscv64gc-unknown-linux-gnu` target at Tier 2 with an upward trajectory. Committing to `riscv64` as a first-class architecture on Linux at the specification stage avoids the alternative posture of a late, retrofitted port and aligns the project's toolchain, release, and benchmarking processes with RISC-V from the outset.

Absolute performance on current RISC-V server-class hardware is behind `x86_64` and `aarch64`; `ENV-015` and `ENV-020` acknowledge this directly rather than forcing unrealistic uniform absolute targets across architectures. The per-architecture framework already built into `PERF-010` through `PERF-013` accommodates this naturally: the comparative ambition fixed by `PERF-019` through `PERF-024` is evaluated per architecture against best-in-class reference implementations running on the same architecture, so the "parity on plain DNS / exceed on encrypted transports" posture applies cleanly to `riscv64` without requiring it to match `x86_64` absolute numbers. The engineering effort to support three architectures, rather than two, is not proportional to the number of architectures: parsing, DNSSEC signature verification, and hashing all have vectorisable kernels whose SIMD abstractions are written once and specialised per architecture intrinsic (AVX2 / AVX-512 on `x86_64`, NEON and SVE / SVE2 on `aarch64`, RVV on `riscv64` where hardware and toolchain support permit), so the incremental delta of adding a third architecture is on the order of 1.5x rather than 3x.

Excluding 32-bit architectures under `ENV-018` and big-endian variants under `ENV-019` reflects the operational reality that a high-performance DNS server targeting extreme load has no use case on those platforms. 32-bit address spaces are incompatible with the memory footprints that Heimdall's cache policy and connection caps under `THREAT-061` through `THREAT-078` imply under load; big-endian hardware is vanishingly rare in production DNS deployments in 2026. Admitting either to any tier would force the project to carry conditional code paths and test matrix obligations without serving operators. Excluding `riscv64` on BSD and macOS under `ENV-016` and `ENV-017` reflects the corresponding ecosystem reality: FreeBSD and OpenBSD `riscv64` ports exist but are not production-relevant at this stage, and macOS does not run on RISC-V hardware at all. Restricting `riscv64` to the Linux target tier, where distributions, toolchains, and cloud instances align, keeps the commitment concrete.

Adopting a multi-format primary distribution model under `ENV-023` through `ENV-033` acknowledges the deployment diversity that characterises DNS operations in 2026. Bare-metal operators predominantly consume native packages and drive the process through systemd; cloud and Kubernetes operators consume container images; developers consume binaries directly for local testing and experimentation. A single primary format would alienate part of the realistic user base; committing to three mandatory first-class formats serves each of these audiences on its own terms without forcing it through another ecosystem.

Statically linking against musl under `ENV-024` reduces runtime dependencies on the host to zero. This simplifies emergency deployment (a single file, no installer, no runtime assumption about the host libc), makes the `.tar.gz` artefact trivially portable across Linux distributions within a supported architecture, and is also the construction that cleanly feeds the distroless-style container path under `ENV-026` without dragging in a shared-library runtime.

The hardened minimal container base required by `ENV-026` (distroless or equivalent) is aligned with modern Kubernetes operation and minimises the container attack surface directly: the absence of a shell, of a package manager, and of debugging tools inside the production image removes the post-exploitation footholds that full-distribution base images expose. This construction is coherent with the process-hardening posture already fixed by `THREAT-022` through `THREAT-028` and with the supply-chain posture fixed by `THREAT-010` through `THREAT-014`.

The reference integration files already mandated by `THREAT-025` (systemd unit), `THREAT-029` (OpenBSD `rc.d`), and `THREAT-030` (macOS sandbox profile), and collectively required under version control by `THREAT-031`, are operationalised by `ENV-025` and `ENV-027`: native packages install the systemd unit where it applies, and the remaining integration files travel with the release as source-distribution artefacts. No new hardening obligation is introduced by the deployment model; the deployment model delivers the hardening obligations already fixed by the threat model through the channels through which operators actually install software.

The multi-format obligation is a one-time engineering investment in the release pipeline, not a per-deployment cost. Modern continuous-integration tooling (for example, GitHub Actions orchestration, `cargo-dist`, `cargo-deb`, `cargo-rpm`, and Docker buildx multi-architecture workflows) collapses the effort of producing static binaries, native packages, and multi-architecture container images into a single pipeline. The marginal cost of adding a format to a pipeline that already produces another format is modest; the marginal cost to operators of not having their preferred format is high.

Applying the supply-chain signing and SBOM obligations under `ENV-029` and `ENV-030` uniformly across every format preserves the supply-chain coherence fixed by `THREAT-010` through `THREAT-014`. A format whose integrity cannot be verified through the documented signing procedure, or whose dependency composition is not covered by the SBOM, would be a gap in the supply chain that the project does not tolerate. By fixing the obligations at the deployment-model layer, the specification avoids the risk that a convenience format is introduced without the corresponding supply-chain coverage.

Excluding dynamically linked Linux binaries as a primary format under `ENV-031` avoids the class of deployment problems that arise when the binary and the host libc version disagree. A dynamically linked build remains available for developer convenience through the source path under `ENV-028`, but releasing such binaries as primary artefacts would conflict with the portability rationale behind the static-musl choice and would fragment the release matrix with no commensurate benefit. Excluding per-OS curl-pipe installers under `ENV-032` follows the same principle: an installation channel that bypasses the signed-artefact discipline, executes arbitrary code fetched over the network, and does not integrate with the operator's native package manager would undermine the supply-chain posture that the rest of the deployment model enforces.

## 4. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will need to be specified before the target-environment policy can be fully operationalised.

- **Minimum versions for FreeBSD and OpenBSD.** The minimum supported versions for FreeBSD (within the 14.x series) and for OpenBSD (within the 7.x series) are **to be specified**.
- **Minimum version for the macOS developer platform.** The minimum supported macOS version for the development-only target tier (Sonoma 14, Sequoia 15, or a later release) is **to be specified**.
- **`io_uring`-to-`epoll` fallback policy.** The exact policy governing the fallback path introduced by `ENV-005` is **to be specified**, including whether the fallback MUST be available on every Linux build or MAY be selected at compile time through a feature flag, and the behaviour that Heimdall MUST exhibit when the running kernel is older than 6.1 or when `io_uring` is unavailable at runtime (for example, in containers whose seccomp filter blocks the `io_uring` syscalls).
- **Specific distribution versions for `.deb` and `.rpm` targeting.** The concrete distribution versions against which the native Linux packages mandated by `ENV-025` are built and validated — for example, Ubuntu 24.04 LTS, Debian 13, RHEL 9, RHEL 10, Fedora (current and previous supported releases), and Alpine 3.x — are **to be specified**, together with the criterion that classifies a distribution version as in-scope for release validation.
- **Container base image choice.** The concrete container base image selected for the hardened OCI image mandated by `ENV-026` — for example, distroless static, Alpine, Debian slim, or Red Hat `ubi-micro` — is **to be specified**, together with the rationale that drives the choice (minimum surface, supply-chain transparency, tooling availability, and reproducibility implications).
- **CI pipeline for artefact generation.** The concrete continuous-integration pipeline that produces the artefacts mandated by `ENV-024` through `ENV-026` across every supported Linux production architecture — including the orchestrator (for example, GitHub Actions or an equivalent), the runner topology (hosted runners where available, self-hosted runners where required for `aarch64` and `riscv64`), and the build isolation guarantees — is **to be specified**.
- **Cross-compilation strategy for `aarch64` and `riscv64`.** The concrete strategy applied to produce `aarch64` and `riscv64` artefacts from the release pipeline — whether through a native build farm, through cross-compilation via toolchain containers, through emulation, or through a hybrid arrangement — is **to be specified**, together with the implications of each choice for build reproducibility under `THREAT-012` and for test coverage.
- **Release cadence and semantic-versioning policy.** The concrete release cadence for official Heimdall releases and the concrete semantic-versioning policy that governs version numbers — including the rules for major, minor, and patch increments, the handling of pre-release and release-candidate identifiers, and the long-term-support posture, if any — are **to be specified**.
- **Container image tagging convention.** The concrete tagging convention for the container images mandated by `ENV-026` — including the use of immutable digest-pinned tags, moving tags (for example, `latest` or major-version floating tags), semantic-versioning tags, and date-based tags, and the compatibility guarantees attached to each tag class — is **to be specified**.
- **RISC-V reference hardware for the performance baseline.** The concrete RISC-V reference-hardware model introduced by `ENV-020` — including the specific server-class RISC-V platform, the minimum core count, the availability and generation of the RISC-V Vector Extension (RVV), the memory capacity, and the network-interface card — is **to be specified** and MUST be fixed once representative hardware is selected.
- **Minimum micro-architecture baselines per architecture.** The minimum micro-architecture baseline that Heimdall targets on each of the three production architectures is **to be specified**: x86-64-v2 versus x86-64-v3 versus x86-64-v4 for `x86_64`; ARMv8.2-A versus ARMv9 (and the specific feature set, including SVE and SVE2 availability) for `aarch64`; and the specific RISC-V profile (for example, RVA22, RVA23) for `riscv64`.
- **Runtime dispatch versus single-baseline binary strategy per architecture.** The strategy applied per architecture for SIMD and architecture-feature usage is **to be specified**: whether Heimdall ships a single binary that detects SIMD and extension features at runtime and dispatches to the appropriate kernel, or whether it ships separate binaries per feature level, or a hybrid arrangement, together with the implications of each choice for binary size, build-matrix complexity, and performance.
- **Tier-promotion tracking for `riscv64gc-unknown-linux-gnu`.** The process by which the project reacts to tier changes of the `riscv64gc-unknown-linux-gnu` Rust target, as required by `ENV-022`, is **to be specified**, including the monitoring cadence, the party responsible, and the response procedure when a tier change is announced.

No implementation activity may proceed on the basis of assumptions about any of the items above.
