# ADR-0060: Adopt clap for the heimdall binary CLI surface

## Context and Problem Statement

The `heimdall` binary requires a CLI surface with subcommands (`start`, `check-config`, `version`) and structured options per `BIN-001` through `BIN-005` in [`specification/015-binary-contract.md`](../../specification/015-binary-contract.md). The CLI must produce correct `--help` and `--version` output, validate arguments at parse time, and map cleanly to the exit-code table in `BIN-006`. This decision chooses the argument-parsing library.

## Decision Drivers

- Must support subcommands, typed options, environment-variable overrides, and short/long flags.
- Must produce professional `--help` output with per-subcommand descriptions.
- Must integrate cleanly with Rust's type system via derive macros (zero-cost at runtime; all parsing is at startup).
- Supply-chain risk must be minimal: the dependency is used only in the binary crate, not in any library crate.

## Considered Options

- **clap 4.x (derive API)** — the dominant CLI library in the Rust ecosystem.
- **clap 4.x (builder API)** — verbose but equivalent at runtime.
- **argh** — a lightweight CLI library from Google; lacks subcommand help output quality and env-var integration.
- **structopt** — superseded by clap v3+; archived.
- **Manual `std::env::args()` parsing** — requires reimplementing all parsing logic, help generation, and error formatting; not appropriate for a production daemon CLI.

## Decision Outcome

Chosen option: **clap 4.x with the derive API**, because:

- It is the de-facto standard CLI library in the Rust ecosystem with active maintenance and broad adoption.
- The derive API produces zero-cost abstractions: all parsing logic is generated at compile time.
- It satisfies all `BIN-001..BIN-005` requirements out of the box: subcommands, typed options, env-var integration (`env` feature), help formatting, and structured error messages.
- Its `unsafe` footprint in the binary crate is zero (clap itself uses no unsafe in its public API surface).
- License: MIT and Apache-2.0 (dual), both on the permitted whitelist in `ENG-018` and `ENG-019`.

**Classification:** convenience. The CLI surface is exclusively in the binary crate (`heimdall`); no library crate depends on clap. Replacing clap later would require rewriting only `src/cli.rs`.

## Consequences

**Positive:**

- Professional, self-documenting `--help` output for every subcommand at no implementation cost.
- Typed argument structs integrate naturally with Rust's `match` on subcommand enums.
- `env` feature allows `HEIMDALL_CONFIG`, `RUST_LOG`, and `HEIMDALL_WORKER_THREADS` to override defaults without manual `std::env::var` calls.

**Negative:**

- Adds a non-trivial transitive dependency graph to the binary crate (clap pulls in several utility crates). Acceptable because this is a binary-only dependency with no effect on library crates.

## Audit Trail (per ENG-009 item 2)

- **cargo-vet:** No exemptions required; clap is certified by multiple Mozilla and Bytecode Alliance import sources.
- **CVE/RustSec history:** No advisories as of 2026-05-03.
- **Maintenance:** Active; maintained by the clap-rs organisation with regular releases. clap 4.x has been stable since 2022.
- **License:** MIT OR Apache-2.0 — permitted per `ENG-018`.
- **unsafe footprint:** Zero unsafe in the clap public API.
