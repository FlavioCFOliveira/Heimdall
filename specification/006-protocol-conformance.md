# DNS protocol conformance

**Purpose.** This document defines Heimdall's DNS protocol conformance policy. It fixes the decisions that govern baseline DNS behaviour across all roles, the EDNS(0) extension framework required on every role, and the core set of EDNS(0) extensions Heimdall MUST implement, independently of which roles or transports are active in a given deployment. It does not restate which roles Heimdall supports or which transports carry the protocol; those questions are settled in [`001-server-roles.md`](001-server-roles.md) and [`002-transports.md`](002-transports.md).

**Status.** Stable. All open questions resolved (Sprint 6, 2026-04-26).

**Requirement category.** `PROTO`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md). For the transport set this document depends on, see [`002-transports.md`](002-transports.md). For the cryptographic policy that governs encrypted transports on which padding applies, see [`003-crypto-policy.md`](003-crypto-policy.md). For the DNSSEC policy that depends on the EDNS(0) `DO` flag and on Extended DNS Errors, see [`005-dnssec-policy.md`](005-dnssec-policy.md).

## 1. Scope

This document applies to all three DNS server roles supported by Heimdall: the authoritative server role, the recursive resolver role, and the forwarder role, as defined in [`001-server-roles.md`](001-server-roles.md). The requirements apply to incoming DNS messages received by Heimdall and to outgoing DNS messages emitted by Heimdall, where applicable per role.

This document is the first of several forthcoming DNS-protocol-conformance decisions. It captures the baseline DNS conformance, the EDNS(0) framework, the core set of mandatory EDNS(0) extensions, the extensions that are out of scope at this stage, QNAME minimisation on the recursive resolver role, 0x20 case randomisation on the recursive resolver role, the out-of-scope decision on RFC 2136 dynamic DNS updates on the authoritative server role, the zone transfer ecosystem (AXFR, IXFR, NOTIFY) on the authoritative server role, and glue-record handling on the recursive resolver role ([RFC 9471](https://www.rfc-editor.org/rfc/rfc9471)). Subsequent protocol-conformance clarifications — TCP truncation and fallback behaviour, and others — are tracked as open in section 7 and will be appended to this document or to companion documents in subsequent revisions.

## 2. Baseline DNS

### 2.1 Normative requirements

- **PROTO-001.** Heimdall MUST implement the core DNS protocol in accordance with [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034) and [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-002.** Heimdall MUST honour the clarifications to the core DNS protocol specified in [RFC 2181](https://www.rfc-editor.org/rfc/rfc2181). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-003.** Heimdall MUST treat DNS labels as case-insensitive on comparison, in accordance with [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-004.** Heimdall MUST use the DNS terminology defined in [RFC 8499](https://www.rfc-editor.org/rfc/rfc8499) in its specification documents and in its diagnostic output. This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).

## 3. EDNS(0) framework

### 3.1 Normative requirements

- **PROTO-005.** Heimdall MUST implement the EDNS(0) extension framework in accordance with [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-006.** Heimdall MUST process the `OPT` pseudo-RR on incoming DNS messages and MUST include the `OPT` pseudo-RR on outgoing DNS messages in accordance with [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-007.** Heimdall MUST honour the `DO` flag defined in [RFC 4035](https://www.rfc-editor.org/rfc/rfc4035) on every DNS message and MUST propagate it consistently on outgoing messages where applicable per role. This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md) and is consistent with `DNSSEC-005`, `DNSSEC-006`, and `DNSSEC-011` in [`005-dnssec-policy.md`](005-dnssec-policy.md).
- **PROTO-008.** Heimdall MUST implement Requestor's UDP payload size negotiation in accordance with [RFC 6891, section 6.2.5](https://www.rfc-editor.org/rfc/rfc6891#section-6.2.5). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-009.** Heimdall MUST support Extended RCODEs as defined in [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).

## 4. Mandatory EDNS(0) extensions

The extensions required by this section MUST be implemented on every role to which they apply. Operational parameters for each extension are tracked as open in section 5 and MUST NOT be assumed before they are specified.

### 4.1 DNS Cookies

- **PROTO-010.** Heimdall MUST implement DNS Cookies in accordance with [RFC 7873](https://www.rfc-editor.org/rfc/rfc7873), including cookie generation, cookie validation, and the response behaviour specified in that RFC. This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md) to which DNS Cookies are applicable per [RFC 7873](https://www.rfc-editor.org/rfc/rfc7873).

### 4.2 Padding

- **PROTO-011.** Heimdall MUST implement the EDNS(0) Padding option defined in [RFC 7830](https://www.rfc-editor.org/rfc/rfc7830) and MUST apply it in accordance with the recommendations in [RFC 8467](https://www.rfc-editor.org/rfc/rfc8467). This requirement applies on the encrypted transports defined in [`002-transports.md`](002-transports.md): DNS-over-TLS on port 853, DNS-over-HTTPS over HTTP/2 and HTTP/3, and DNS-over-QUIC on port 853. Padding MUST NOT be applied on DNS classic over UDP and TCP on port 53.

### 4.3 Extended DNS Errors

- **PROTO-012.** Heimdall MUST implement the Extended DNS Errors option defined in [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md) and is consistent with `DNSSEC-013` in [`005-dnssec-policy.md`](005-dnssec-policy.md).

### 4.4 NSID

- **PROTO-013.** Heimdall MUST implement the Name Server Identifier (NSID) option defined in [RFC 5001](https://www.rfc-editor.org/rfc/rfc5001). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).

### 4.5 edns-tcp-keepalive

- **PROTO-014.** Heimdall MUST implement the edns-tcp-keepalive option defined in [RFC 7828](https://www.rfc-editor.org/rfc/rfc7828) on TCP-based transports. This requirement applies on DNS classic over TCP on port 53, on DNS-over-TLS on port 853, and on DNS-over-HTTPS over HTTP/2, as defined in [`002-transports.md`](002-transports.md). The edns-tcp-keepalive option MUST NOT be applied on DNS classic over UDP on port 53, on DNS-over-HTTPS over HTTP/3, or on DNS-over-QUIC on port 853.

## 5. Deferred extensions

The following EDNS(0) extensions are explicitly out of scope at this stage. Each decision is a scope decision, not a rejection, and MAY be reconsidered in a future revision of this document.

- **PROTO-015.** Heimdall MUST NOT implement EDNS Client Subnet (ECS) as defined in [RFC 7871](https://www.rfc-editor.org/rfc/rfc7871). This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md), is consistent with the recommendation in [RFC 9558 (BCP 237)](https://www.rfc-editor.org/rfc/rfc9558) that ECS be disabled by default on all resolvers, and is refined by `PROTO-017`, `PROTO-018`, and `PROTO-019`.
- **PROTO-016.** Heimdall MUST NOT implement the Chain Query requests option defined in [RFC 7901](https://www.rfc-editor.org/rfc/rfc7901) at this stage. No concrete local use case has been identified at the time of this decision.
- **PROTO-017.** On incoming DNS messages that carry an EDNS Client Subnet option as defined in [RFC 7871](https://www.rfc-editor.org/rfc/rfc7871), Heimdall MUST ignore the option. Heimdall MUST NOT perform any form of geographic, topological, or network-prefix-based answer selection on the basis of the option. Heimdall MUST NOT echo the option in the corresponding response. This requirement applies to all three server roles defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-018.** On outgoing DNS queries emitted by the recursive resolver role toward authoritative servers and by the forwarder role toward upstream resolvers, Heimdall MUST NOT include an EDNS Client Subnet option as defined in [RFC 7871](https://www.rfc-editor.org/rfc/rfc7871).
- **PROTO-019.** On outgoing DNS responses emitted by the authoritative server role, Heimdall MUST NOT include an EDNS Client Subnet option as defined in [RFC 7871](https://www.rfc-editor.org/rfc/rfc7871).

## 5A. QNAME minimisation

### 5A.1 Normative requirements

- **PROTO-020.** Heimdall MUST implement QNAME minimisation on the recursive resolver role in accordance with [RFC 9156](https://www.rfc-editor.org/rfc/rfc9156). For each authoritative server contacted during iterative resolution, Heimdall MUST send only the minimum number of labels required to obtain the next referral or the final answer, rather than the full original QNAME. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md). The authoritative server role and the forwarder role do not iterate through the delegation hierarchy and are not subject to this requirement.
- **PROTO-021.** The default QNAME minimisation mode on the recursive resolver role MUST be the "relaxed" mode defined in [RFC 9156, section 3](https://www.rfc-editor.org/rfc/rfc9156#section-3). In relaxed mode, on error conditions or unexpected responses from an authoritative server that does not correctly handle minimised queries, Heimdall MUST fall back to sending the full QNAME for the current resolution step.
- **PROTO-022.** Heimdall MUST expose a configuration option that selects the "strict" QNAME minimisation mode defined in [RFC 9156](https://www.rfc-editor.org/rfc/rfc9156). In strict mode, Heimdall MUST NOT fall back to the full QNAME; on error conditions or unexpected responses from an authoritative server that does not correctly handle minimised queries, the resolution attempt MUST fail in accordance with normal DNS error semantics.
- **PROTO-023.** Heimdall MUST NOT expose any configuration switch that disables QNAME minimisation on the recursive resolver role. There is no global off-switch for QNAME minimisation on the running instance.
- **PROTO-024.** Every fallback from minimised to full QNAME performed under `PROTO-021` MUST be observable through Heimdall's diagnostic output when the operator enables logging at the appropriate level. Each fallback event MUST identify the resolution step at which the fallback occurred and the authoritative server whose response triggered the fallback.

## 5B. 0x20 case randomisation

### 5B.1 Normative requirements

- **PROTO-025.** On every outbound DNS query emitted by the recursive resolver role toward an authoritative server, Heimdall MUST apply 0x20 case randomisation to the QNAME. Every ASCII letter in every label of the QNAME MUST be independently randomised between upper case and lower case prior to emission. Non-ASCII labels, including internationalised labels encoded via Punycode as defined in [RFC 3492](https://www.rfc-editor.org/rfc/rfc3492), are unaffected by this requirement. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-026.** On every response received from an authoritative server in reply to a query covered by `PROTO-025`, Heimdall MUST compare the case of every ASCII letter in every label of the response QNAME against the case of the corresponding letter in the outbound query QNAME. The comparison MUST be exact and MUST be performed independently from the case-insensitive label-equality comparison required by `PROTO-003`.
- **PROTO-027.** When the comparison required by `PROTO-026` detects a mismatch and the authoritative server is currently classified as 0x20-conformant under `PROTO-028`, Heimdall MUST treat the response as spoofed or poisoned, MUST discard the response, and MUST retry the query against another authoritative server for the same resolution step when another authoritative server is available.
- **PROTO-028.** Heimdall MUST maintain a per-authoritative-server conformance status indicating whether that server is 0x20-conformant. The status MUST be tracked per destination IP address or per `(destination IP address, server identity)` tuple. A server MUST be classified as non-conformant when it returns otherwise-valid responses whose QNAME case is consistently normalised and does not match the outbound query case. Once a server is classified as non-conformant, Heimdall MUST disable 0x20 case randomisation on subsequent outbound queries directed at that server and the case-match verification required by `PROTO-026` MUST NOT be applied to responses from that server until the classification is revised under `PROTO-029`.
- **PROTO-029.** Heimdall MUST periodically re-probe authoritative servers classified as non-conformant under `PROTO-028`, by emitting an outbound query to which 0x20 case randomisation is applied as required by `PROTO-025`. When the re-probe response satisfies the case-match comparison required by `PROTO-026`, the server MUST be reclassified as 0x20-conformant and 0x20 case randomisation MUST be re-enabled on subsequent outbound queries directed at that server.
- **PROTO-030.** Heimdall MUST NOT expose any configuration switch that disables 0x20 case randomisation on the recursive resolver role globally on the running instance. The per-authoritative-server disable governed by `PROTO-028` and `PROTO-029` is the only mechanism by which 0x20 case randomisation is suspended, and that mechanism is driven automatically by observed server behaviour, not by operator configuration.
- **PROTO-031.** 0x20 case randomisation MUST NOT be applied by the forwarder role defined in [`001-server-roles.md`](001-server-roles.md) to outbound queries directed at its configured upstream resolvers. `PROTO-025` through `PROTO-030` apply exclusively to the recursive-to-authoritative path. 0x20 case randomisation is applied at the DNS-message layer and therefore operates identically across all transports defined in [`002-transports.md`](002-transports.md), including DNS classic over UDP and TCP on port 53 and DNS-over-TLS on port 853 when used on the recursive-to-authoritative path.

## 5C. RFC 2136 dynamic DNS updates

Dynamic DNS updates as defined in [RFC 2136](https://www.rfc-editor.org/rfc/rfc2136) are explicitly out of scope at this stage. The requirements below are a scope decision, not a rejection, and MAY be reconsidered in a future revision of this document. Any such reconsideration MUST be performed jointly with the authoritative DNSSEC policy fixed by `DNSSEC-001` through `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md), because mutating pre-signed zones would require either online signing, which is excluded by `DNSSEC-004` and `DNSSEC-007`, or a journal-plus-external-signer architecture that is not currently part of Heimdall's authoritative model.

### 5C.1 Normative requirements

- **PROTO-032.** Heimdall MUST NOT accept, validate, or act upon DNS UPDATE messages as defined in [RFC 2136](https://www.rfc-editor.org/rfc/rfc2136) on the authoritative server role. This requirement applies to the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md). The recursive resolver role and the forwarder role do not implement a zone-mutation surface and are not subject to this requirement.
- **PROTO-033.** On receipt of a DNS UPDATE message on the authoritative server role, Heimdall MUST respond with RCODE 4 (`NOTIMP`, "Not Implemented") as defined in [RFC 1035, section 4.1.1](https://www.rfc-editor.org/rfc/rfc1035#section-4.1.1). This response MUST be emitted regardless of the source of the message and regardless of whether the message carries a TSIG record ([RFC 8945](https://www.rfc-editor.org/rfc/rfc8945)) or a SIG(0) record ([RFC 2931](https://www.rfc-editor.org/rfc/rfc2931)); Heimdall MUST NOT perform TSIG or SIG(0) verification on UPDATE messages prior to emitting the `NOTIMP` response.
- **PROTO-034.** Heimdall MUST NOT implement the prerequisite evaluation defined in [RFC 2136, section 3.2](https://www.rfc-editor.org/rfc/rfc2136#section-3.2), and MUST NOT implement any update-policy machinery governing which UPDATE requests would be authorised. No such evaluation path exists on the authoritative server role, and no such path MAY be added without reopening this decision jointly with `DNSSEC-004` and `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md).
- **PROTO-035.** The decision fixed by `PROTO-032` through `PROTO-034` is deferred, not rejected, and MAY be reconsidered in a future revision of this document. Any reconsideration MUST be resolved jointly with the authoritative DNSSEC policy fixed by `DNSSEC-001` through `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md), because mutating pre-signed zones implies either online signing, which is excluded by `DNSSEC-004` and `DNSSEC-007`, or a journal-plus-external-signer architecture that is not currently part of Heimdall's authoritative model.

## 5D. Zone transfers (AXFR, IXFR, NOTIFY)

The zone transfer ecosystem — full-zone transfer under [RFC 5936](https://www.rfc-editor.org/rfc/rfc5936), incremental transfer under [RFC 1995](https://www.rfc-editor.org/rfc/rfc1995), and change-notification under [RFC 1996](https://www.rfc-editor.org/rfc/rfc1996) — is in scope on the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md). The recursive resolver role and the forwarder role do not participate in zone transfer and are not subject to the requirements in this section.

### 5D.1 Normative requirements

- **PROTO-036.** Heimdall MUST support AXFR (full zone transfer) on the authoritative server role in accordance with [RFC 5936](https://www.rfc-editor.org/rfc/rfc5936), in both the inbound direction (receiving a zone from a configured primary) and the outbound direction (serving a zone to a configured secondary), configurable per zone. This requirement applies to the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-037.** Heimdall MUST support IXFR (incremental zone transfer) on the authoritative server role in accordance with [RFC 1995](https://www.rfc-editor.org/rfc/rfc1995), in both the inbound direction (pulling incremental updates from a configured primary) and the outbound direction (serving incremental updates to a configured secondary), configurable per zone. This requirement applies to the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-038.** Heimdall MUST support NOTIFY on the authoritative server role in accordance with [RFC 1996](https://www.rfc-editor.org/rfc/rfc1996), in both the inbound direction (accepting NOTIFY messages from a configured primary to trigger an update pull) and the outbound direction (emitting NOTIFY messages to configured secondaries upon zone updates), configurable per zone. This requirement applies to the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-039.** Heimdall MAY operate as primary, as secondary, or as both for any given zone configured on the authoritative server role, within a single running instance. The role assumed for each zone (primary, secondary, or both) MUST be determined exclusively by the per-zone configuration of that instance, and Heimdall MUST NOT impose a single instance-wide mode that prevents mixed primary-and-secondary deployments.
- **PROTO-040.** When Heimdall operates as primary for a zone, it MUST serve AXFR and IXFR in response to authorised requests from configured secondaries, and MUST emit NOTIFY to configured secondaries upon zone updates. The precise list of authorised secondaries and the list of secondaries to which NOTIFY is emitted MUST be determined exclusively by the per-zone configuration.
- **PROTO-041.** When Heimdall operates as secondary for a zone, it MUST issue AXFR against a configured primary to perform the initial zone load, MUST issue IXFR against the configured primary to obtain subsequent updates, and MUST accept NOTIFY from the configured primary to trigger an update pull. The configured primary MUST be determined exclusively by the per-zone configuration.
- **PROTO-042.** Serial-number arithmetic applied to the SOA `SERIAL` field during zone-transfer operations (including the decision of whether a remote serial is newer than the local serial, the selection between AXFR and IXFR, and the ordering of incremental changes) MUST follow the serial-arithmetic rules defined in [RFC 1982](https://www.rfc-editor.org/rfc/rfc1982).
- **PROTO-043.** On the secondary side of a zone, Heimdall MUST honour the `REFRESH`, `RETRY`, `EXPIRE`, and `MINIMUM` timers carried in the SOA record of that zone, with the semantics specified by [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035) and refined by [RFC 2308](https://www.rfc-editor.org/rfc/rfc2308). Operational defaults and minima for these timers are tracked as open in section 7.
- **PROTO-044.** Zone-transfer requests and responses (AXFR and IXFR) MUST be authenticated. By default, Heimdall MUST require TSIG as defined in [RFC 8945](https://www.rfc-editor.org/rfc/rfc8945) on every zone-transfer request and on every zone-transfer response, in both the inbound and the outbound direction, on every zone configured for transfer. Heimdall MUST reject any zone-transfer request or zone-transfer response that does not present a valid TSIG signature under a configured key.
- **PROTO-045.** IP-based access-control lists MAY be used in combination with TSIG as an additional authorisation layer on zone-transfer requests and zone-transfer responses. IP-based access-control lists MUST NOT be used as the sole authentication mechanism for zone transfers. Heimdall MUST NOT expose a configuration option that permits a zone-transfer flow to be authenticated solely by IP address.
- **PROTO-046.** Heimdall MAY support SIG(0) as defined in [RFC 2931](https://www.rfc-editor.org/rfc/rfc2931) as an alternative authentication mechanism for zone-transfer requests and zone-transfer responses. When SIG(0) is supported, it MUST NOT relax the prohibition in `PROTO-045`; a zone-transfer flow authenticated by SIG(0) stands in lieu of TSIG, not in lieu of cryptographic authentication itself.
- **PROTO-047.** Heimdall SHOULD support XFR over TLS (XoT) as defined in [RFC 9103](https://www.rfc-editor.org/rfc/rfc9103), using the same TLS 1.3 policy and listener configuration as DNS-over-TLS. XoT connections MUST satisfy `NET-004` in [`002-transports.md`](002-transports.md) and `SEC-001` through `SEC-011` in [`003-crypto-policy.md`](003-crypto-policy.md). For new deployments, XoT SHOULD be the default transport for zone transfers.
- **PROTO-048.** A zone-transfer request that fails authorisation under `PROTO-044`, `PROTO-045`, or `PROTO-046` MUST be refused. Heimdall MUST respond with RCODE 5 (`REFUSED`) as defined in [RFC 1035, section 4.1.1](https://www.rfc-editor.org/rfc/rfc1035#section-4.1.1), or, where applicable, with the TSIG error code `BADKEY` as defined in [RFC 8945](https://www.rfc-editor.org/rfc/rfc8945). Heimdall MUST NOT expose any configuration option that causes an unauthorised zone-transfer request to be answered with the zone contents.
- **PROTO-049.** DNSSEC records present in a pre-signed zone — including `RRSIG`, `DNSKEY`, `NSEC`, `NSEC3`, `NSEC3PARAM`, `DS`, `CDNSKEY`, `CDS`, and `CSYNC` as enumerated in `DNSSEC-002` of [`005-dnssec-policy.md`](005-dnssec-policy.md) — MUST be transferred end-to-end via AXFR and IXFR without modification. Heimdall MUST NOT strip, replace, or regenerate any such record during zone transfer, in either the inbound or the outbound direction. This requirement is consistent with, and a direct consequence of, `DNSSEC-003` and `DNSSEC-004` in [`005-dnssec-policy.md`](005-dnssec-policy.md), which exclude DNSSEC private-key material and online signing from the authoritative server process.

## 5E. Glue records on the recursive resolver role

Glue records are `A` and `AAAA` records carried in the Additional section of a referral emitted by a parent zone, providing IP addresses for the nameservers of a delegated child zone. The requirements in this section govern how Heimdall handles such records on the recursive resolver role during iterative resolution, in accordance with [RFC 9471](https://www.rfc-editor.org/rfc/rfc9471). The authoritative server role and the forwarder role do not iterate through the delegation hierarchy and are not subject to the requirements in this section; serving-side glue handling on the authoritative server role is governed by the authoritative behaviour already established in [`005-dnssec-policy.md`](005-dnssec-policy.md) and in this document, and is not altered by the requirements below.

### 5E.1 Normative requirements

- **PROTO-050.** On the recursive resolver role, Heimdall MUST treat glue records received in the Additional section of a referral as hints used exclusively to select the next iterative hop. Heimdall MUST NOT treat glue records as authoritative data, MUST NOT return glue records as answers to client queries, and MUST NOT insert glue records into the segregated recursive cache fixed by `CACHE-001` through `CACHE-007` in [`004-cache-policy.md`](004-cache-policy.md) as if they had been obtained from the authoritative server for the owner name of the glue record. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-051.** Heimdall MUST reject out-of-bailiwick glue in accordance with [RFC 9471](https://www.rfc-editor.org/rfc/rfc9471). Glue is out of bailiwick when its owner name does not fall within the delegation to which the referral pertains. Heimdall MUST NOT use out-of-bailiwick glue to select a next iterative hop, MUST NOT insert out-of-bailiwick glue into any cache, and MUST resolve the address of the nameserver via an independent resolution that is not influenced by the rejected glue. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-052.** When in-bailiwick glue is present in a referral, Heimdall MUST prefer the in-bailiwick glue over any alternative address for the same nameserver, including addresses cached from prior resolutions, when selecting the next iterative hop for the resolution step to which the referral pertains. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-053.** Before a nameserver address obtained from glue is treated as authoritative data for any downstream query — including being returned to a client, being used outside the iterative resolution step for which the referral was received, or being inserted into the recursive cache as authoritative `A` or `AAAA` data for the nameserver's owner name — Heimdall MUST obtain the corresponding `A` or `AAAA` records through child-side resolution and MUST apply the validation pipeline fixed by `DNSSEC-008` through `DNSSEC-014` in [`005-dnssec-policy.md`](005-dnssec-policy.md) to the resulting records. Glue MUST NOT be elevated to authoritative status on the basis of the parent's referral alone. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).
- **PROTO-054.** The requirements `PROTO-050` through `PROTO-053` MUST be enforced regardless of whether the child zone identified by the referral is DNSSEC-signed. For DNSSEC-signed child zones, the child-side validation required by `PROTO-053` closes the DNSSEC chain of trust down to the validated nameserver addresses. For DNSSEC-unsigned child zones, the bailiwick check and the hint-only treatment of glue established by `PROTO-050` and `PROTO-051` remain the sole defences against adversarial parent referrals and MUST NOT be relaxed. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).

## 6. Rationale

The baseline DNS requirements in section 2 fix the interoperability floor that every decision in this specification rests upon. [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034) and [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035) are the core DNS protocol; [RFC 2181](https://www.rfc-editor.org/rfc/rfc2181) removes well-known ambiguities in those two RFCs that have produced implementation divergence in widely-deployed servers; [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343) fixes case-insensitivity on label comparison, which is a security-relevant property because a case-sensitive comparison would partition the cache on a property that the DNS itself does not treat as a distinguisher, enabling a class of cache-consistency and bypass issues; and [RFC 8499](https://www.rfc-editor.org/rfc/rfc8499) is the current authoritative DNS terminology glossary, so using it uniformly in specification and in diagnostic output removes a class of operator confusion that stems from legacy terminology drift across older RFCs.

The EDNS(0) framework is a prerequisite for DNSSEC. The `DO` flag, the propagation of `RRSIG` records in responses, and the ability to return responses larger than the 512-byte limit of [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035) all depend on EDNS(0). Without EDNS(0), the DNSSEC requirements consolidated in [`005-dnssec-policy.md`](005-dnssec-policy.md) — `DNSSEC-005`, `DNSSEC-006`, and `DNSSEC-011` on the authoritative path, and the validating-role requirements under `DNSSEC-008` through `DNSSEC-024` — could not be met. Making the EDNS(0) framework mandatory on every role therefore turns a prerequisite into an explicit specification-level commitment rather than a side effect of implementing DNSSEC.

Requiring EDNS(0) Requestor's UDP payload size negotiation is also a performance decision. Modern DNS responses, particularly DNSSEC-signed answers, routinely exceed 512 bytes; without EDNS(0) payload-size negotiation, Heimdall would have to set the TC flag on such responses and force the client to retry over TCP, incurring an additional round trip on every large response. Under the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md), spurious TCP fallback would be a material performance regression with no corresponding security benefit, because the size of the response is not by itself a trust-boundary property.

DNS Cookies under [RFC 7873](https://www.rfc-editor.org/rfc/rfc7873) close off-path attack vectors that plain DNS over UDP is structurally vulnerable to. Without a cookie, a response whose transaction ID and source-port tuple can be guessed or brute-forced can be injected by an off-path attacker, which is the classical cache-poisoning vector, and a spoofed-source-address query can turn the server into a reflection and amplification vehicle. Cookies give the server a cheap, stateless mechanism for verifying that a client is reachable at the claimed source address and for rejecting off-path injected responses, which closes both vectors by construction. Deferring the operational parameters — secret rotation cadence, cookie lifetime, integration with rate limiting as an anti-spoofing signal — to the operational-defaults phase is consistent with the pattern already adopted for TEK rotation in [`003-crypto-policy.md`](003-crypto-policy.md) and for `NEW_TOKEN` rotation in the same document: the policy commitment is fixed now; only the numeric calibration is deferred.

Padding under [RFC 7830](https://www.rfc-editor.org/rfc/rfc7830) and [RFC 8467](https://www.rfc-editor.org/rfc/rfc8467) preserves the privacy guarantee of the encrypted transports required by [`002-transports.md`](002-transports.md). Without padding, an on-path observer can infer the query and response being carried inside a TLS, HTTPS, or QUIC session from the ciphertext length alone, because DNS message sizes cluster tightly around a small set of response shapes that map reasonably well to specific queries. Padding defeats this trivial traffic-analysis-by-size, which is the privacy threat the encrypted transports exist to address. Restricting padding to the encrypted transports reflects the fact that plain DNS over UDP and TCP on port 53 offers no confidentiality in the first place, so padding there would add bandwidth cost without privacy gain.

Extended DNS Errors under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) complete the DNSSEC diagnostic chain already required by `DNSSEC-013` in [`005-dnssec-policy.md`](005-dnssec-policy.md). A `SERVFAIL` response with no additional information is indistinguishable, to a downstream client or an operator, from any other server-side failure; an EDE option attached to such a response identifies the specific failure class (validation `bogus`, expired RRSIG, missing DNSKEY, and so on), which is the minimum diagnostic surface required for operators and for troubleshooting tools to act on the failure. EDE also enables precise signalling for non-DNSSEC conditions — rate-limited, blocked, network errors, NXDOMAIN sub-cases — that are otherwise invisible to the client. Implementing the framework is therefore a prerequisite to settling the DNSSEC diagnostic question tracked in [`005-dnssec-policy.md`](005-dnssec-policy.md) (EDE codes per situation).

NSID under [RFC 5001](https://www.rfc-editor.org/rfc/rfc5001) and edns-tcp-keepalive under [RFC 7828](https://www.rfc-editor.org/rfc/rfc7828) are low-cost, operationally valuable capabilities expected by modern DNS peers. NSID gives operators an in-band mechanism for identifying which instance of a multi-instance deployment served a given response, which is the minimum instrumentation required for troubleshooting in any deployment topology where a single DNS service name resolves to more than one instance. edns-tcp-keepalive signals to TCP-based peers how long Heimdall is prepared to keep an idle connection open, which is the standard mechanism for balancing TCP connection reuse against per-connection resource consumption; without it, peers must either close connections aggressively after every query — which defeats the amortisation of the TLS or QUIC handshake that section 4 of [`003-crypto-policy.md`](003-crypto-policy.md) relies on for performance — or keep connections open speculatively, which exposes the server to idle-connection exhaustion. Restricting edns-tcp-keepalive to TCP-based transports reflects its scope as defined in [RFC 7828](https://www.rfc-editor.org/rfc/rfc7828): the option has no meaning on UDP, and QUIC-based transports (DoQ, DoH over HTTP/3) have their own native idle-timeout mechanisms.

0x20 case randomisation, fixed by `PROTO-025` through `PROTO-031`, is required on the recursive resolver role for five reinforcing reasons. First, it is defence in depth orthogonal to DNS Cookies under `PROTO-010` and [RFC 7873](https://www.rfc-editor.org/rfc/rfc7873): cookies protect against off-path spoofing only once a cookie exchange has been established with the authoritative server, whereas 0x20 adds query-identifier entropy to every exchange, including the first query to a newly contacted authoritative server and every exchange whose cookie state has not yet been established. The two mechanisms close overlapping but non-identical windows, so combining them raises the off-path-spoofing bar above what either mechanism would provide on its own. Second, 0x20 is established industry practice: Unbound, BIND, dnsmasq, and every large public recursive resolver implement it in one form or another, which makes 0x20-conformance the de facto expectation on the authoritative side of the DNS ecosystem and a new resolver that did not randomise case would be regressive relative to the deployed baseline. Third, the adaptive per-authoritative-server disable fixed by `PROTO-028` and `PROTO-029` addresses the practical reality that some legacy authoritative servers and some non-conformant middleboxes normalise case in violation of [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343); per-server disable preserves resolution of legitimate zones hosted on such servers without sacrificing the protection where the protocol is honoured, and periodic re-probing ensures that the protection is restored automatically once the remote side is corrected. Fourth, 0x20 is orthogonal to transport: it is applied at the DNS-message layer, so it operates identically over DNS classic on port 53 and over DNS-over-TLS on port 853 when those transports are used on the recursive-to-authoritative path as defined in [`002-transports.md`](002-transports.md), without any transport-specific adaptation. Fifth, the forwarder role is intentionally excluded by `PROTO-031` because the forwarder emits outbound queries to upstream resolvers rather than to authoritative servers, and upstream resolvers do not reliably preserve QNAME case; enforcing 0x20 on that path would cause constant disablement or spurious failures, with no corresponding security benefit, because the threat model on the forwarder-to-upstream segment is addressed by the encrypted-transport decisions in [`002-transports.md`](002-transports.md) and [`003-crypto-policy.md`](003-crypto-policy.md) rather than by message-layer entropy. Refusing to expose a global off-switch under `PROTO-030` applies to 0x20 the same structural-gating rationale already stated for DNSSEC validation under `DNSSEC-009` and `DNSSEC-020` in [`005-dnssec-policy.md`](005-dnssec-policy.md) and for QNAME minimisation under `PROTO-023`: the protection is part of the system's structure, not a configuration-dependent property that can be weakened by accident or by template inheritance, and the only suspension mechanism is the automatic per-server disable that reacts to observed non-conformance.

QNAME minimisation under [RFC 9156](https://www.rfc-editor.org/rfc/rfc9156) is required on the recursive resolver role for four reinforcing reasons. First, it is privacy-positive by construction: iterative resolution without minimisation discloses the full QNAME to every authoritative server on the delegation path, including the root and the top-level-domain servers, which have no operational need for any label below their own zone cut. Minimising the QNAME at each step reduces the leakage to the minimum required for the protocol to make forward progress, which complements the encrypted-transport decisions in [`002-transports.md`](002-transports.md) and [`003-crypto-policy.md`](003-crypto-policy.md) and the EDNS Client Subnet out-of-scope decision fixed by `PROTO-015`, `PROTO-017`, `PROTO-018`, and `PROTO-019`. The encrypted transports protect only the client-to-Heimdall segment; QNAME minimisation protects the Heimdall-to-authoritative segment, and ECS being out of scope prevents the same leakage from being reintroduced in the opposite direction. The three decisions are part of a single coherent privacy posture. Second, QNAME minimisation is the industry baseline in 2026: Unbound, Knot Resolver, BIND 9.14 and later, and PowerDNS Recursor all implement it by default, most of them in relaxed mode. A new recursive resolver that did not implement QNAME minimisation would be regressive relative to every widely-deployed validating resolver that operators of Heimdall are likely to be replacing. Third, relaxed mode as the default, fixed by `PROTO-021`, preserves interoperability with legacy or misconfigured authoritative servers that do not correctly return referrals for truncated QNAMEs; the fallback to the full QNAME at the current step allows resolution to complete in that case, at the cost of disclosing the full QNAME to that one authoritative server. Strict mode, fixed by `PROTO-022`, is exposed as a configuration option for deployments with a stronger privacy threat model that prefer a failed resolution to any disclosure of the full QNAME. Refusing to expose a global off-switch under `PROTO-023` applies to QNAME minimisation the same structural-gating rationale already stated for DNSSEC validation under `DNSSEC-009` and `DNSSEC-020` in [`005-dnssec-policy.md`](005-dnssec-policy.md): the privacy property is part of the system's structure, not a configuration-dependent property that can be weakened by accident or by template inheritance. Requiring fallback events to be observable in diagnostic output under `PROTO-024` makes the operational cost of relaxed mode visible to operators, so that the distribution of authoritative servers that force a fallback can be measured, reported upstream, and taken into account when choosing between relaxed and strict mode for a given deployment. Finally, QNAME minimisation interacts positively with aggressive use of the DNSSEC-validated cache under [RFC 8198](https://www.rfc-editor.org/rfc/rfc8198), which is tracked as open in [`005-dnssec-policy.md`](005-dnssec-policy.md): cached `NSEC` and `NSEC3` records further reduce upstream queries, which multiplies the privacy benefit of minimisation because fewer queries ever leave the resolver in the first place. The scope is restricted to the recursive resolver role because only that role iterates through the delegation hierarchy; the authoritative server role answers from its own zone data, and the forwarder role forwards the client's query to a configured upstream resolver which performs iteration on its behalf, so neither role has a delegation chain on which to minimise.

EDNS Client Subnet under [RFC 7871](https://www.rfc-editor.org/rfc/rfc7871) is out of scope at this stage, across all three roles, for four reinforcing reasons. First, ECS is privacy-negative by design: it leaks a portion of the client IP prefix to every downstream element on the DNS path, including third-party authoritative servers. This directly contradicts the privacy intent of the encrypted-transport decisions fixed in [`002-transports.md`](002-transports.md) and [`003-crypto-policy.md`](003-crypto-policy.md), which protect only the client-to-Heimdall segment; propagating ECS would expose partial client identity to the very actors those transports are meant to shield the client from. Second, [RFC 9558 (BCP 237, 2024)](https://www.rfc-editor.org/rfc/rfc9558) recommends that ECS be disabled by default on all resolvers and discourages its adoption outside the narrow class of large CDN-aligned resolvers; Heimdall's default position therefore aligns with the current IETF best current practice. Third, the operational value of ECS is concentrated in a CDN-geolocation use case that Heimdall is not optimised for, so the benefit side of the trade-off does not apply. Fourth, the implementation cost is non-trivial: ECS requires incoming-prefix sanitisation, `scope_mask` bookkeeping on responses, and cache-key differentiation per client subnet, the last of which would conflict directly with the cache-key invariants in `CACHE-001` through `CACHE-007` in [`004-cache-policy.md`](004-cache-policy.md), by multiplying cache entries per name and degrading the cache hit rate that those requirements are designed to maximise. This decision is reversible: ECS MAY be re-opened later as an explicitly opt-in capability if concrete operational demand appears, at which point the conflict with the cache policy would have to be resolved as part of that decision. Chain Query under [RFC 7901](https://www.rfc-editor.org/rfc/rfc7901) is deferred because no concrete local use case has been identified at the time of this decision; deferral keeps the initial protocol surface focused on the extensions that have demonstrated operational value.

Zone transfer via AXFR, IXFR, and NOTIFY, fixed by `PROTO-036` through `PROTO-049`, is required on the authoritative server role for five reinforcing reasons. First, interoperability: AXFR ([RFC 5936](https://www.rfc-editor.org/rfc/rfc5936)), IXFR ([RFC 1995](https://www.rfc-editor.org/rfc/rfc1995)), and NOTIFY ([RFC 1996](https://www.rfc-editor.org/rfc/rfc1996)) are the standard interop protocols across authoritative DNS servers. Without them, Heimdall cannot replace BIND, Knot Authoritative, NSD, or PowerDNS Authoritative in existing deployments, cannot participate in cross-organisation primary-secondary relationships, and cannot integrate with registries or operator ecosystems whose primary-secondary links are specified in terms of these RFCs. Second, high availability and resilience: a primary together with one or more geographically distributed secondaries is the baseline deployment pattern for every serious DNS zone, and Heimdall must speak the zone-transfer protocol in both directions to be usable in such a deployment, whether as the primary, as one of the secondaries, or as both for different zones on the same instance. Third, performance: IXFR avoids retransmitting the full zone for every update and keeps the transfer volume proportional to the number of changes, which is the only tractable option for TLD-class zones and for highly-active enterprise zones where an AXFR-only policy would force multi-order-of-magnitude greater transfer volume on every update. Fourth, propagation latency: NOTIFY is a low-cost, standard mechanism that collapses the propagation delay between a zone update on the primary and its visibility on the secondaries from the SOA `REFRESH` interval — typically minutes to hours — to a single network round trip; emitting and accepting NOTIFY is a prerequisite for any deployment in which a zone update must reach the serving secondaries promptly. Fifth, authentication discipline: requiring TSIG under [RFC 8945](https://www.rfc-editor.org/rfc/rfc8945) as the baseline authentication mechanism for zone-transfer requests and responses, forbidding IP-based access-control lists as the sole authentication mechanism, and preferring XoT under [RFC 9103](https://www.rfc-editor.org/rfc/rfc9103) as the encrypted envelope for new deployments, keeps the zone-transfer security posture coherent with the cryptographic policy already fixed in [`003-crypto-policy.md`](003-crypto-policy.md) and avoids introducing a weaker authentication path on a control-plane operation that, if subverted, would let an attacker load arbitrary data into every secondary of a zone. The DNSSEC pass-through requirement under `PROTO-049` is a direct consequence of the pre-signed-zones DNSSEC decision fixed by `DNSSEC-001` through `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md): Heimdall holds no DNSSEC private-key material inside the authoritative server process and performs no online signing, so the only correct behaviour on zone transfer is to carry the pre-existing `RRSIG`, `DNSKEY`, `NSEC`, `NSEC3`, `NSEC3PARAM`, `DS`, `CDNSKEY`, `CDS`, and `CSYNC` records through the transfer unchanged. Any alternative — stripping signatures, regenerating signatures, or rewriting DNSSEC records — would either break the chain of trust from the IANA root key to the served data or would require private-key material inside the serving process, which is excluded by `DNSSEC-003` in [`005-dnssec-policy.md`](005-dnssec-policy.md). Refusing unauthorised requests with `REFUSED` or `BADKEY` under `PROTO-048` is the RFC-conformant signal that the request was rejected on access-control grounds, and it closes off the long-standing misconfiguration class in which an authoritative server answers an unauthenticated AXFR with the zone contents; emitting that response unconditionally on authorisation failure, rather than leaking partial data or silently dropping the request, keeps the access-control surface small and observable. The scope is restricted to the authoritative server role because neither the recursive resolver role nor the forwarder role participates in zone transfer, and admitting zone-transfer state on those roles would introduce a control-plane surface on roles whose only legitimate duty is query answering.

RFC 2136 dynamic DNS updates, fixed as out of scope by `PROTO-032` through `PROTO-035`, are deferred for four reinforcing reasons. First, coherence with the authoritative DNSSEC policy fixed by `DNSSEC-001` through `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md): Heimdall serves pre-signed zones produced by an external signer, and holds no DNSSEC private-key material inside the authoritative server process. Mutating a pre-signed zone at runtime in response to an RFC 2136 UPDATE would require either online signing — which is excluded by `DNSSEC-004` and `DNSSEC-007` — or a journal-plus-external-signer architecture that is not currently part of Heimdall's authoritative model. Admitting RFC 2136 now would therefore either contradict the DNSSEC decisions already fixed in [`005-dnssec-policy.md`](005-dnssec-policy.md) or commit Heimdall to an additional cross-process coordination surface that has not been designed. Second, substantial implementation cost: conformant RFC 2136 support requires crash-recovery journaling to preserve durability across process restarts, concurrency control over the authoritative dataset, transaction semantics for the update-as-a-whole property, prerequisite evaluation as defined in [RFC 2136, section 3.2](https://www.rfc-editor.org/rfc/rfc2136#section-3.2), an update-policy or ACL layer that determines which updates are authorised, and integration with the authentication mechanisms specified by [RFC 8945](https://www.rfc-editor.org/rfc/rfc8945) (TSIG) and [RFC 2931](https://www.rfc-editor.org/rfc/rfc2931) (SIG(0)). That surface amounts to a multi-thousand-line subsystem inside the network-facing process, with no corresponding value for the core use cases that motivate Heimdall's authoritative role. Third, mature operational alternatives exist: external zone-file generation followed by a hot-reload signalling mechanism, driven by a configuration-management pipeline or by a dedicated service such as `acme-dns` for ACME DNS-01 challenges, covers the practical reasons for which an operator would otherwise reach for dynamic updates, without importing the implementation and security cost of the RFC 2136 update path into the authoritative server process. Fourth, a `NOTIMP` response, as required by `PROTO-033`, is the RFC-conformant signal that the server does not implement the requested capability, so clients immediately know to use another mechanism; emitting `NOTIMP` unconditionally — regardless of TSIG or SIG(0) presence, as required by `PROTO-033` — also avoids performing cryptographic verification on a message the server will refuse in any case, which removes a CPU-cost amplification vector that a flood of signed UPDATE messages could otherwise open under the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md). The decision is additive and reversible under `PROTO-035`: RFC 2136 MAY be re-opened later as an opt-in capability if a concrete driver appears, and that reconsideration MUST be resolved jointly with `DNSSEC-001` through `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md) because the two decisions are structurally coupled.

Glue-record handling on the recursive resolver role, fixed by `PROTO-050` through `PROTO-054`, is required for five reinforcing reasons. First, the security posture is non-negotiable under [`../CLAUDE.md`](../CLAUDE.md), and relaxed glue handling is a known weakness class: a resolver that trusts out-of-bailiwick glue can be manipulated by an adversarial or compromised parent authoritative into redirecting resolution of unrelated zones toward attacker-controlled servers, because the parent's referral embeds Additional-section records for names whose authority the parent does not hold. Rejecting out-of-bailiwick glue under `PROTO-051` closes that cross-zone redirection vector by construction. Second, [RFC 9471](https://www.rfc-editor.org/rfc/rfc9471) is the 2026 baseline for glue handling; Unbound, Knot Resolver, and BIND 9.19 and later implement these rules, and a new resolver that diverged from them would be regressive relative to the deployed validating-resolver ecosystem. Third, the decision integrates cleanly with the DNSSEC validation decisions fixed by `DNSSEC-008` through `DNSSEC-024` in [`005-dnssec-policy.md`](005-dnssec-policy.md): child-side resolution and validation of nameserver `A` and `AAAA` records under `PROTO-053` closes the chain of trust down to the addresses that are actually used as authoritative data, whereas glue serves exclusively as a hint to bootstrap the iterative exchange that produces those validated records. Nameserver addresses obtained from glue are never substituted for validated records, so the DNSSEC trust boundary is preserved. Fourth, the implementation cost is contained: the enforcement amounts to a small number of checks inside the iterative state machine — a bailiwick comparison on every referral, a preference rule for in-bailiwick glue over cached alternative addresses, and a follow-up child-side validation before any glue-derived address is elevated to authoritative status. The checks operate on data that is already available at the iterative step and do not require additional per-query round trips beyond what the iterative resolution path already performs. Fifth, the decision interacts cleanly with the other iterative-state-machine decisions already fixed in this document: QNAME minimisation under `PROTO-020` through `PROTO-024` and the aggressive use of the DNSSEC-validated cache fixed by `DNSSEC-025` through `DNSSEC-030` in [`005-dnssec-policy.md`](005-dnssec-policy.md) both operate at the same iterative layer and share the same per-step referral processing, so adding bailiwick enforcement and hint-only glue semantics at that layer produces no new cross-cutting interactions and keeps the iterative path coherent. Enforcement regardless of DNSSEC signedness of the child zone, as required by `PROTO-054`, reflects the fact that the parent-side spoofing vector does not depend on whether the child zone is signed: for unsigned children, the bailiwick rejection is the sole defence; for signed children, the bailiwick rejection is the first defence and the child-side validation is the second. Relaxing either defence on the grounds that the other is present would reintroduce an asymmetric attack surface and is therefore excluded by construction. The scope is restricted to the recursive resolver role because only that role iterates through the delegation hierarchy and therefore receives parent-side referrals; the forwarder role delegates iteration to its configured upstream resolvers, and the authoritative server role serves glue as part of delegations it publishes, which is a serving-side concern governed by the pre-signed-zones model fixed by `DNSSEC-001` through `DNSSEC-007` in [`005-dnssec-policy.md`](005-dnssec-policy.md).

## 5F. DNS Cookies operational parameters

### 5F.1 Normative requirements

- **PROTO-055.** The server-side secret used to compute and verify server cookies under `PROTO-010` and [RFC 7873](https://www.rfc-editor.org/rfc/rfc7873) MUST be rotated every **7 days**. At the rotation boundary, Heimdall MUST begin accepting cookies generated with the new secret and MUST continue accepting cookies generated with the immediately preceding retired secret for a grace period of exactly **1 rotation cycle** (7 days), after which the retired secret MUST be discarded. At any point in time, Heimdall MUST hold at most 2 secrets — one current and one retired. There is no configuration option to extend the hold-on period for the retired secret.
- **PROTO-056.** Server cookies MUST carry a validity lifetime of at most **24 hours** from the time of generation. Heimdall MUST reject a server cookie presented by a client if the cookie's embedded timestamp is more than 24 hours in the past relative to the server's local clock, even if the cookie was generated under a currently-held secret.
- **PROTO-057.** When a client presents a valid server cookie under `PROTO-055` and `PROTO-056`, Heimdall MUST credit the client with a "cookie-verified" classification for the purposes of rate-limiting and admission control defined by `THREAT-053` in [`007-threat-model.md`](007-threat-model.md). Cookie-verified clients MUST be placed in a per-client rate-limiting bucket that is distinct from the bucket applied to clients without a valid cookie, and the per-client query rate limit for cookie-verified clients MUST be at least 4× the rate limit applied to clients without a valid cookie.
- **PROTO-058.** Under load conditions that trigger the admission control mechanism defined by `THREAT-069` in [`007-threat-model.md`](007-threat-model.md), Heimdall MUST give priority admission to clients that present a valid server cookie over clients that present an absent or invalid cookie. The precise admission-bias rule is defined in [`007-threat-model.md`](007-threat-model.md) under `THREAT-069` and is not duplicated here.
- **PROTO-059.** Heimdall MUST rotate the server-side cookie secret using a cryptographically secure pseudorandom number generator. The new secret MUST have at least 128 bits of entropy. Heimdall MUST NOT accept operator injection of the cookie secret.

### 5F.2 Rationale

A 7-day rotation cadence for the server-side secret under `PROTO-055` matches the cadence applied to TLS session-ticket encryption keys under `SEC-008` in [`003-crypto-policy.md`](003-crypto-policy.md), providing consistency in the operational model for short-lived keying material across Heimdall subsystems. The 1-rotation-cycle grace period for the retired secret ensures that clients holding a cookie generated late in the previous rotation window can still authenticate for up to 7 days after the rotation, covering the longest plausible gap between successive client requests without requiring re-issuance. A 24-hour cookie lifetime under `PROTO-056` bounds the validity window of any individual cookie, so that a captured cookie cannot be replayed indefinitely: even if the corresponding server secret has not yet been rotated, the cookie itself expires within a day. Giving cookie-verified clients a 4× rate-limit preference under `PROTO-057` aligns with the admission-bias mechanism required by `THREAT-069`: cookie verification is a cheap proof of IP reachability that is effective against off-path address spoofing, and preferring verified clients under load allows Heimdall to shed spoofed-source queries before they reach the validation path.

## 5G. Padding operational parameters

### 5G.1 Normative requirements

- **PROTO-060.** On DNS-over-TLS (DoT) on port 853, Heimdall MUST pad every DNS response using the block-size policy recommended by [RFC 8467, section 4.1](https://www.rfc-editor.org/rfc/rfc8467#section-4.1): the padded message length MUST be the smallest multiple of **128 bytes** that is greater than or equal to the unpadded message length. This block size applies uniformly on DoT for all query types and all roles, on both inbound (server-side response padding) and outbound (client-side query padding on the recursive-to-authoritative DoT path).
- **PROTO-061.** On DNS-over-HTTPS over HTTP/2 (DoH/H2), Heimdall MUST pad every DNS response to the smallest multiple of **128 bytes** that is greater than or equal to the unpadded message length. The block-size policy is identical to the DoT policy fixed by `PROTO-060`.
- **PROTO-062.** On DNS-over-HTTPS over HTTP/3 (DoH/H3), Heimdall MUST pad every DNS response to the smallest multiple of **128 bytes** that is greater than or equal to the unpadded message length. The block-size policy is identical to the DoT policy fixed by `PROTO-060`.
- **PROTO-063.** On DNS-over-QUIC (DoQ) on port 853, Heimdall MUST pad every DNS response to the smallest multiple of **128 bytes** that is greater than or equal to the unpadded message length. The block-size policy is identical to the DoT policy fixed by `PROTO-060`.
- **PROTO-064.** The 128-byte block size fixed by `PROTO-060` through `PROTO-063` applies uniformly across all encrypted transports and all roles. There is no per-role or per-transport adjustment of the block size. Heimdall MUST NOT expose a configuration key that changes the block size at runtime.
- **PROTO-065.** Padding MUST NOT be applied on DNS classic over UDP or TCP on port 53, in accordance with `PROTO-011`. This prohibition is absolute and MUST NOT be relaxed by any configuration option.

### 5G.2 Rationale

A uniform 128-byte block size on all encrypted transports follows the SHOULD recommendation in [RFC 8467, section 4.1](https://www.rfc-editor.org/rfc/rfc8467#section-4.1). The 128-byte block is a practical compromise: it is small enough that the overhead for short responses (a typical NODATA of 60 bytes becomes 128 bytes — a factor of 2.1) is acceptable, while it is large enough to obscure the most discriminating size differences between common response shapes (NODATA, small NXDOMAIN, typical A-record answer, large AAAA-record answer). A per-transport block-size variation would not improve privacy — an on-path observer who knows the block size can still calculate the upper bound on the unpadded message length — and would complicate the implementation and the operational model without a corresponding benefit. Using the same block size on the query path (recursive-to-authoritative DoT) as on the response path is consistent with [RFC 8467](https://www.rfc-editor.org/rfc/rfc8467) and prevents asymmetric leakage where the query and response are padded differently.

## 5H. Extended DNS Error code master table

### 5H.1 Normative requirements

- **PROTO-066.** The following master table maps response situations to the EDE info-code that Heimdall MUST include under `PROTO-012` and [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914). The DNSSEC `bogus` sub-cases (EDE codes 6–12) are fixed by `DNSSEC-094` through `DNSSEC-104` in [`005-dnssec-policy.md`](005-dnssec-policy.md) and are not duplicated here; they are listed in the table for cross-reference only.

  | Situation | EDE info-code | Notes |
  |---|---|---|
  | DNSSEC validation `bogus` — cryptographic failure | **6** DNSSEC Bogus | See `DNSSEC-095` |
  | DNSSEC validation `bogus` — signature expired | **7** Signature Expired | See `DNSSEC-096` |
  | DNSSEC validation `bogus` — signature not yet valid | **8** Signature Not Yet Valid | See `DNSSEC-097` |
  | DNSSEC validation `bogus` — DNSKEY missing | **9** DNSKEY Missing | See `DNSSEC-098` |
  | DNSSEC validation `bogus` — RRSIGs missing | **10** RRSIGs Missing | See `DNSSEC-099` |
  | DNSSEC validation `bogus` — no Zone Key bit set | **11** No Zone Key Bit Set | See `DNSSEC-100` |
  | DNSSEC validation `bogus` — NSEC/NSEC3 proof missing | **12** NSEC Missing | See `DNSSEC-101` |
  | KeyTrap cap reached | **6** DNSSEC Bogus + EXTRA-TEXT `keytrap-cap-reached` | See `DNSSEC-102` |
  | Per-query validation CPU budget exhausted | **22** No Reachable Authority + EXTRA-TEXT `validation-budget-exhausted` | See `DNSSEC-089` |
  | Stale answer served (serve-stale, RFC 8767) | **3** Stale Answer | Applied when Heimdall serves a TTL-expired cached record |
  | Cached SERVFAIL replayed from negative cache | **13** Cached Error | Applied when a negative-cache entry is a prior SERVFAIL |
  | Network error on recursive or forwarder upstream path | **23** Network Error | Upstream unreachable, timeout, TLS handshake failure |
  | No reachable authority for the queried name (all NSes exhausted) | **22** No Reachable Authority | Applied after all NSes are tried and all fail |
  | Query rate-limited under `THREAT-033`..`THREAT-062` | **17** Filtered | Applied when the rate limit fires; response is REFUSED per `THREAT-044` |
  | RPZ NXDOMAIN action | **15** Blocked | Applied on RPZ-synthesised NXDOMAIN |
  | RPZ NODATA action | **15** Blocked | Applied on RPZ-synthesised NODATA |
  | RPZ TCP-only action | **17** Filtered | Applied with TC=1 response |
  | RPZ DROP action | (none — response is dropped) | EDE cannot be included in a dropped response |
  | RPZ PASSTHRU action | (none — query answered normally) | |
  | RPZ local-data or CNAME-redirect action | (none — data response served) | |
  | Resource limit exceeded (per-query memory/CPU allocation cap) | **0** Other + EXTRA-TEXT identifying the exhausted resource | Applied under `THREAT-066`..`THREAT-068` when a per-query budget is hit |
  | Unsupported DNSKEY algorithm encountered | **1** Unsupported DNSKEY Algorithm | Applied when a DNSKEY record carries an unrecognised algorithm number |
  | Unsupported DS digest type encountered | **2** Unsupported DS Digest Type | Applied when a DS record carries an unrecognised digest type |

- **PROTO-067.** Each EDE option MUST include an EXTRA-TEXT field when the situation produces a specific resource name or identity that aids operator troubleshooting (as noted in the table above). The EXTRA-TEXT field MUST NOT include internal state, stack traces, upstream IP addresses, or any information that could aid an attacker.
- **PROTO-068.** When a response situation matches more than one row in the table — for example, both a rate-limit firing and an RPZ action apply to the same query — Heimdall MUST include the EDE code for the condition that is determined first in the processing pipeline. The processing order is: ACL → admission → cookie → rate limit → RPZ → DNSSEC validation.

## 5I. NSID operational parameters

### 5I.1 Normative requirements

- **PROTO-069.** Heimdall MUST expose a single operator-configurable NSID string applied to all three roles on a given running instance. The string MUST be a valid DNS wire-format octet sequence of at most 512 octets. The recommended format is `heimdall-<instance_id>` where `<instance_id>` is a short identifier that distinguishes the instance within the operator's deployment (e.g., a pod name, hostname truncated to 32 characters, or a random 8-hex-digit tag). There is no default value: an operator who does not configure NSID MUST NOT receive an NSID in responses.
- **PROTO-070.** When an operator configures the NSID string, Heimdall MUST include the NSID option in every response to a query that carries the NSID option in its OPT record, in accordance with [RFC 5001](https://www.rfc-editor.org/rfc/rfc5001). Heimdall MUST NOT include the NSID option in responses to queries that do not carry it.
- **PROTO-071.** The NSID string MUST be a static value for the lifetime of the running process. Heimdall MUST NOT rotate or randomise the NSID value at runtime. A new NSID takes effect only after a full process restart or a SIGHUP reload that specifies a new value. Rotating NSID to obscure instance identity is not a supported use case.
- **PROTO-072.** The NSID string MUST NOT contain any of the following: process IDs, file-system paths, internal IP addresses or ports, software version strings, memory addresses, or any other internal state that an adversary could use to fingerprint or map the deployment. The recommended `heimdall-<instance_id>` format satisfies this constraint when `<instance_id>` is an opaque per-instance label.

## 5J. edns-tcp-keepalive idle-timeout

### 5J.1 Normative requirements

- **PROTO-073.** On DNS classic over TCP on port 53 and on DNS-over-TLS on port 853, Heimdall MUST signal an idle-timeout of **30 seconds** via the edns-tcp-keepalive option in responses to queries that carry the edns-tcp-keepalive option, in accordance with [RFC 7828](https://www.rfc-editor.org/rfc/rfc7828). The 30-second default applies uniformly on both transports and on all three roles.
- **PROTO-074.** On DNS-over-HTTPS over HTTP/2, the connection idle-timeout is governed by the HTTP/2-layer idle-timeout parameter fixed by `SEC-036` in [`003-crypto-policy.md`](003-crypto-policy.md) and MUST NOT be signalled via the edns-tcp-keepalive EDNS option. edns-tcp-keepalive MUST NOT be applied on DNS-over-HTTPS over HTTP/3 or on DNS-over-QUIC, as those transports have native QUIC-layer idle timeout semantics.
- **PROTO-075.** On zone-transfer listeners (AXFR and IXFR over TCP or XoT), Heimdall MUST signal an idle-timeout of **120 seconds** via the edns-tcp-keepalive option, overriding the 30-second default, to account for the longer inter-message gaps that arise during zone transfer. This override applies only while a zone-transfer session is active; once the transfer completes and the connection returns to an idle query-serving state, the 30-second default governs.
- **PROTO-076.** When a client TCP or DoT connection remains idle beyond the signalled timeout, Heimdall MUST close the connection in accordance with `THREAT-068` in [`007-threat-model.md`](007-threat-model.md).

## 5K. Caching semantics of intermediate referrals from minimised queries

### 5K.1 Normative requirements

- **PROTO-077.** When the recursive resolver role emits a minimised query that returns a referral — that is, a response whose Answer section is empty, whose Authority section contains NS records for a delegation, and whose Additional section may contain glue — Heimdall MUST cache the NS records and in-bailiwick glue records from that referral in the segregated recursive cache under the zone-cut key, with the TTL values carried in the referral response and subject to the TTL-bound rules fixed by `CACHE-003` and `CACHE-004` in [`004-cache-policy.md`](004-cache-policy.md).
- **PROTO-078.** Cached referral NS and glue records under `PROTO-077` MUST be stored as non-authoritative data. They MUST NOT be returned as answers to client queries, and MUST NOT be elevated to authoritative status without child-side validation as required by `PROTO-053`.
- **PROTO-079.** When a subsequent iterative resolution step requires the same zone cut — that is, the same NS RRset for the same delegation point — Heimdall MUST prefer the cached NS records over a fresh referral lookup, provided their remaining TTL is greater than zero. This preference applies exclusively within the iterative resolution state machine on the recursive resolver role and does not affect how Heimdall answers client queries about NS records.
- **PROTO-080.** Referral entries in the recursive cache participate in the cache memory budget and eviction policy fixed by `CACHE-001` through `CACHE-007` in [`004-cache-policy.md`](004-cache-policy.md). They are not treated as a separate cache tier.

## 5L. Relaxed-mode fallback event logging

### 5L.1 Normative requirements

- **PROTO-081.** Every fallback from a minimised QNAME to the full QNAME performed under `PROTO-021` MUST be emitted as a structured log entry at the **INFO** level. The log entry MUST include, as separate named fields: the ISO 8601 UTC timestamp, the resolution step number within the current iterative chain, the IP address and port of the authoritative server that returned the unexpected response, the SNI or server name when the transport is DoT, the zone name at the current delegation level, the minimised QNAME that was sent, the full QNAME that was substituted, and the DNS RCODE received from the authoritative server that triggered the fallback.
- **PROTO-082.** The log entry format fixed by `PROTO-081` MUST be consistent with the event schema defined for security-relevant events by `THREAT-080` in [`007-threat-model.md`](007-threat-model.md). The field names and types MUST conform to that schema so that the event is indexable by the same tooling as other THREAT-080-class events.
- **PROTO-083.** Fallback events MUST NOT be emitted at DEBUG level or below, and MUST NOT be suppressed when INFO logging is active. Aggregated per-server fallback counters MUST be exposed as metrics under the observability framework fixed by `OPS-021` through `OPS-031` in [`012-runtime-operations.md`](012-runtime-operations.md).

## 5M. Label-splitting under DNAME and wildcard interactions

### 5M.1 Normative requirements

- **PROTO-084.** On the recursive resolver role, when a minimised iterative step encounters a DNAME record as defined in [RFC 6672](https://www.rfc-editor.org/rfc/rfc6672), Heimdall MUST resolve the DNAME chain to its canonical target before applying QNAME minimisation to the remainder of the name. The minimised QNAME for the next iterative step MUST be constructed from the substituted name after DNAME expansion, not from the original QNAME.
- **PROTO-085.** When a minimised iterative step returns a referral that falls within a zone where a wildcard record at the zone cut could match the queried name, Heimdall MUST NOT apply QNAME minimisation to the label or labels that are required to distinguish the queried name from the wildcard owner. Specifically: if the minimised QNAME for a step would omit labels that are necessary to determine whether the queried name matches a wildcard, Heimdall MUST instead send the full QNAME for that step. The determination is made locally by inspecting the cached zone structure; if insufficient information is cached to make the determination, Heimdall MUST send the full QNAME for that step.
- **PROTO-086.** The edge cases mandated by `PROTO-084` and `PROTO-085` MUST be verified by a suite of test vectors that covers: at least one DNAME redirect followed by iterative continuation; at least one wildcard zone with a minimised-QNAME near-miss; and at least one DNAME-then-wildcard chain. These test vectors MUST be part of the acceptance criteria for the recursive resolver implementation sprint (Sprint 30).

## 5N. 0x20 per-authoritative-server non-conformance threshold

### 5N.1 Normative requirements

- **PROTO-087.** Under `PROTO-028`, Heimdall MUST classify an authoritative server as non-conformant using a **sliding window of 10 responses**. A server MUST be reclassified as non-conformant when at least **3 out of the last 10 non-error responses** from that server exhibit case normalisation (the response QNAME case does not match the outbound query QNAME case) and none of those responses have been positively identified as spoofed under `PROTO-027`. Both the window size (10) and the threshold (3) are fixed parameters; they MUST NOT be exposed as configuration knobs.
- **PROTO-088.** A single case-normalised response MUST NOT trigger reclassification. The 3-of-10 threshold under `PROTO-087` prevents an adversary from forcing Heimdall to disable 0x20 protection against a target authoritative server by injecting a single spoofed response that appears to be a normalised-case reply. The sliding window slides on every non-error response from the server, discarding the oldest entry when the window is full.
- **PROTO-089.** When the threshold under `PROTO-087` is reached, the reclassification from conformant to non-conformant MUST be recorded in the structured log at INFO level, identifying the server IP address and the count of normalised responses that triggered the reclassification.

## 5O. 0x20 non-conformant re-probe interval

### 5O.1 Normative requirements

- **PROTO-090.** The first re-probe of an authoritative server classified as non-conformant under `PROTO-087` MUST be issued after an interval of **1 hour** ±10% (uniform random jitter). If the re-probe response is case-preserving, the server is reclassified as conformant and subsequent queries use 0x20. If the re-probe response is case-normalised, the re-probe interval MUST be doubled for the next re-probe attempt (exponential back-off), up to a maximum interval of **24 hours** ±10% jitter.
- **PROTO-091.** The back-off state (current interval, probe count) MUST be maintained in memory per destination IP address or `(destination IP address, server name)` tuple, in accordance with the per-server tracking already required by `PROTO-028`. When a server is reclassified as conformant after a successful re-probe, the back-off state for that server MUST be reset to the initial 1-hour interval.

## 5P. 0x20 per-server conformance state persistence

### 5P.1 Normative requirements

- **PROTO-092.** The per-authoritative-server 0x20 conformance state tracked under `PROTO-028` and `PROTO-087` MUST be held in memory only and MUST NOT be persisted to disk or to Redis. The state is reset on every process restart, including SIGHUP-triggered reloads that restart the iterative state machine. On startup, all authoritative servers begin in the conformant state, and the classification proceeds from observation as queries are emitted.
- **PROTO-093.** The in-memory-only persistence decision is final for the current design: Heimdall MUST NOT persist 0x20 conformance state across restarts. The cost of a fresh 10-sample window on restart is accepted as negligible relative to the implementation and attack-surface cost of persisting and reloading the state.

## 5Q. TSIG operational defaults

### 5Q.1 Normative requirements

- **PROTO-094.** Heimdall MUST implement TSIG authentication using the HMAC-SHA256 MAC algorithm as specified by [RFC 8945, section 6](https://www.rfc-editor.org/rfc/rfc8945#section-6). HMAC-SHA256 is the mandatory-to-implement algorithm; all other algorithms are optional.
- **PROTO-095.** Heimdall MAY implement TSIG authentication using HMAC-SHA384 and HMAC-SHA512. When implemented, these algorithms MUST be available for configuration on any zone-transfer flow covered by `PROTO-044`.
- **PROTO-096.** Heimdall MUST NOT implement TSIG authentication using HMAC-MD5 or HMAC-SHA1, which are deprecated by [RFC 8945](https://www.rfc-editor.org/rfc/rfc8945). Any TSIG record carrying a deprecated algorithm MUST be treated as a validation failure; the corresponding zone-transfer request or response MUST be rejected with TSIG error code `BADKEY`.
- **PROTO-097.** TSIG key rotation is operator-driven. Heimdall MUST NOT rotate TSIG keys automatically. The rotation procedure is: (1) introduce the new key on both peers, (2) update the per-zone configuration of both Heimdall and the remote peer to reference the new key, (3) trigger a SIGHUP on both instances, (4) confirm that transfers succeed with the new key, (5) remove the old key from configuration. Heimdall MUST support at most one active TSIG key per zone per direction (inbound, outbound) at any given time. There is no key-overlap or grace-period mechanism; the operator is responsible for coordinating the cutover.
- **PROTO-098.** TSIG keys MUST be stored as opaque byte sequences, either in the Heimdall configuration file with filesystem permissions set to 0600, or in Redis under the namespace prefix `heimdall:tsig:` as specified by `STORE-018` in [`013-persistence.md`](013-persistence.md), accessible only to the Heimdall process. Heimdall MUST NOT log TSIG key material in any diagnostic output at any log level.

## 5R. Zone file format for locally-loaded zones

### 5R.1 Normative requirements

- **PROTO-099.** The zone file format accepted by the authoritative server role for zones loaded from local storage is the RFC 1035 master file format as specified by [RFC 1035, section 5](https://www.rfc-editor.org/rfc/rfc1035#section-5), extended by the BIND-compatible `$INCLUDE`, `$ORIGIN`, `$TTL`, and `$GENERATE` directives. This decision is consistent with and cross-references `DNSSEC-055` in [`005-dnssec-policy.md`](005-dnssec-policy.md).
- **PROTO-100.** The zone file MUST be encoded in UTF-8. Non-ASCII characters in label data MUST be encoded as Punycode ([RFC 3492](https://www.rfc-editor.org/rfc/rfc3492)) before they appear in the zone file. The parser MUST NOT accept raw non-ASCII octets in label positions.
- **PROTO-101.** JSON, YAML, and any other structured-data format are NOT supported as zone file formats for locally-loaded zones. This is the same exclusion fixed by `DNSSEC-057` for the pre-signed zone input path.
- **PROTO-102.** On zone load, the zone file is parsed and its records are ingested into the Redis zone store under the namespaced key layout fixed by `STORE-018` in [`013-persistence.md`](013-persistence.md). The zone file itself is not kept in memory after the initial load; all subsequent query serving draws from the Redis store.

## 5S. SOA timer operational defaults and minima (secondary side)

### 5S.1 Normative requirements

- **PROTO-103.** On the secondary side of a zone under `PROTO-043`, Heimdall MUST enforce the following minimum admissible values for SOA timers. A zone whose SOA record specifies a value below any of these minima MUST be rejected at load time with a structured error identifying the zone and the offending timer:
  - `REFRESH`: minimum **60 seconds**
  - `RETRY`: minimum **30 seconds**
  - `EXPIRE`: minimum **3600 seconds** (1 hour)
  - `MINIMUM` (negative-caching TTL floor): minimum **60 seconds**
- **PROTO-104.** When a zone is loaded without operator-specified SOA timer overrides, and the SOA record specifies values that satisfy the minima in `PROTO-103`, those values MUST be honoured as-is. Heimdall MUST NOT substitute default values when valid SOA-embedded values are present.
- **PROTO-105.** When the operator configures explicit SOA timer defaults in the Heimdall configuration (for example, as fallback values for zones that lack a SOA record in the transfer), those operator-configured defaults MUST satisfy the minima in `PROTO-103`; Heimdall MUST reject configuration that specifies a timer default below the minimum. The recommended defaults to document in the operator manual are: `REFRESH` 3600 s, `RETRY` 600 s, `EXPIRE` 604800 s (7 days), `MINIMUM` 3600 s.

## 5T. Maximum zone size and record-count limits

### 5T.1 Normative requirements

- **PROTO-106.** Heimdall MUST enforce the following default upper limits on the authoritative server role when loading or receiving a zone, as a DoS-mitigation measure against oversized zones:
  - Maximum total zone size: **1 GiB** (1 073 741 824 bytes) of uncompressed wire-format record data
  - Maximum number of records per zone: **10 000 000** (ten million) resource records
  - Maximum number of records per RRset: **1 024** resource records
- **PROTO-107.** The maximum total zone size and the maximum record count per zone in `PROTO-106` MUST be operator-configurable, within the range of 1 MiB to 10 GiB for the size limit and 1 000 to 100 000 000 for the record count. The per-RRset record count limit of 1 024 is a fixed hard cap and MUST NOT be configurable.
- **PROTO-108.** When a zone load or zone transfer would exceed any limit in `PROTO-106`, Heimdall MUST refuse the load or abort the transfer, MUST emit a structured error identifying the zone, the limit exceeded, and the observed value, and MUST leave the previously loaded zone data intact (or, for a new zone with no prior data, leave the zone unloaded). The check MUST be enforced both on the initial load and on every subsequent reload or incremental transfer.

## 5U. XoT formal transport-layer decision

### 5U.1 Normative requirements

- **PROTO-109.** XFR over TLS (XoT) under `PROTO-047` MUST share the same listener and TLS configuration as the DNS-over-TLS (DoT) listener on port 853. There is no separate per-zone XoT listener. XoT connections arrive on port 853 and are distinguished from DoT query connections by the fact that they carry AXFR or IXFR query opcodes; the TLS handshake, certificate validation, and session-ticket policies applied to XoT connections are identical to those applied to DoT connections as fixed by `SEC-001` through `SEC-016` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **PROTO-110.** XoT is the MANDATORY transport for zone transfers in new deployments initiated after Heimdall v1.0.0. TSIG authentication under `PROTO-044` and `PROTO-094` MUST still be applied inside the TLS session on XoT connections, as an additional authentication layer independent of the TLS channel. XoT connections are further subject to the access-control list that governs the DoT listener, and Heimdall MUST additionally verify that the peer's identity (as authenticated by TSIG) matches the configured zone-transfer authorisation.
- **PROTO-111.** XoT is RECOMMENDED (not mandatory) for zone-transfer flows during the pre-v1.0.0 development phase. A pre-v1.0.0 instance MAY accept unauthenticated XoT (TLS without TSIG, for lab use only) but MUST log a structured warning for every such session, identifying it as insecure.

## 5V. SIG(0) key management

### 5V.1 Normative requirements

- **PROTO-112.** When SIG(0) is supported under `PROTO-046`, the SIG(0) transaction keys used to verify incoming SIG(0) signatures on zone-transfer requests MUST be stored as public-key records in Redis under the namespace prefix `heimdall:sig0:pubkey:`, accessible only to the Heimdall process. The local private keys used to produce outgoing SIG(0) signatures on zone-transfer responses MUST be stored in the filesystem with permissions set to 0600, or in Redis under `heimdall:sig0:privkey:` with equivalent access restriction.
- **PROTO-113.** The private-key exclusion stated by `DNSSEC-003` in [`005-dnssec-policy.md`](005-dnssec-policy.md) applies exclusively to DNSSEC signing keys and MUST NOT be interpreted as excluding SIG(0) transaction keys. SIG(0) transaction keys are zone-transfer authentication material, not DNSSEC signing material; they may be held inside the Heimdall process without violating `DNSSEC-003`.
- **PROTO-114.** SIG(0) key rotation MUST be performed via the admin-RPC interface under `OPS-007` through `OPS-015` in [`012-runtime-operations.md`](012-runtime-operations.md). The rotation procedure introduces the new key, updates peer configuration, confirms successful signature verification, and then removes the old key. Heimdall MUST NOT log SIG(0) private key material at any log level.

## 5W. TCP truncation and fallback behaviour

### 5W.1 Normative requirements

- **PROTO-115.** When Heimdall produces a DNS response over UDP classic on port 53 whose wire-format size exceeds the EDNS(0) requestor's payload size negotiated in the query's OPT record, Heimdall MUST truncate the response and set the TC bit in accordance with [RFC 1035, section 4.1.1](https://www.rfc-editor.org/rfc/rfc1035#section-4.1.1) and [RFC 6891, section 7](https://www.rfc-editor.org/rfc/rfc6891#section-7). The truncated response MUST contain the full question section, as many records as fit within the payload limit (starting from the Answer section), and an empty or partial Additional section; it MUST NOT contain partial records (a record is either included in full or not included at all).
- **PROTO-116.** When Heimdall produces a DNS response over TCP classic on port 53, it MUST NOT truncate the response regardless of size. The TCP transport carries a 16-bit length prefix before each message, and there is no negotiated payload limit on TCP; the only upper bound on response size on TCP is the per-query memory allocation cap fixed by `THREAT-067` in [`007-threat-model.md`](007-threat-model.md).
- **PROTO-117.** When a rate-limiting slip event occurs under `THREAT-049` in [`007-threat-model.md`](007-threat-model.md) — that is, the rate limiter allows through a truncated response instead of dropping — the slipped response MUST be a truncated form of the rate-limited response with TC=1 set, not a synthetic SERVFAIL or REFUSED. This ensures that the client can retry over TCP and receive a full response on the TCP path, which bypasses the UDP-based rate limiter.
- **PROTO-118.** DNS-over-HTTPS (DoH over HTTP/2 and HTTP/3) and DNS-over-QUIC (DoQ) MUST NOT use the TC bit or the DNS truncation mechanism. On these transports, a response that exceeds any per-connection flow-control window or per-stream budget MUST be handled through the transport-level backpressure mechanism native to HTTP/2, HTTP/3, and QUIC, not by setting TC=1 in the DNS message. If the response genuinely cannot be transmitted within the transport's constraints, the DNS response MUST be `SERVFAIL` with EDE code 0 (Other) and an EXTRA-TEXT field identifying the transport-level constraint.
