# Heimdall Specification

This folder is the **single source of truth** for Heimdall's functional requirements, architectural decisions, invariants, and constraints. Any statement about what Heimdall is, does, or must not do is authoritative only if it appears here.

For project-wide principles (security posture, performance as the primary guide, the "Assume Nothing" rule, and the mandatory Specify → Implement → Test → Document workflow), see the root-level [`CLAUDE.md`](../CLAUDE.md). This specification does not duplicate those principles; it applies them.

## Status

The specification is being built **incrementally**, as decisions are clarified with the user one at a time. At any given moment it captures only what has been explicitly agreed. Nothing that is not written here may be assumed about the project.

Unresolved items are either:

- Omitted entirely, or
- Listed in the "Open questions" section of the relevant specification file, clearly marked as **to be specified**.

No implementation work may rely on items that are absent or marked as open.

## Conventions

### File naming

All specification documents live directly under `/specification` as flat Markdown files and follow this pattern:

```
NNN-<kebab-case-slug>.md
```

- `NNN` is a three-digit, zero-padded, monotonically increasing sequence number, reserved at the moment the file is created. Numbers are never reused, even if a file is deleted or renamed.
- `<kebab-case-slug>` is a short, descriptive, lowercase, hyphen-separated identifier for the topic (for example, `server-roles`, `transports`, `threat-model`).
- Numbers are not semantic: they do not imply dependency, priority, or grouping. Thematic grouping is expressed through the index below and through explicit cross-references between files, not through the numbers.
- The `README.md` in this folder is the only exception to the numeric prefix rule. It is always the index and is never numbered.

### File header

Every specification file starts with:

1. A level-1 heading with the document title.
2. A short "Purpose" paragraph stating the scope of the file in one or two sentences.
3. A "Status" line indicating whether the content is **Stable** (agreed with the user) or **Draft** (work in progress pending user confirmation).

### Requirement identifiers

When a file states normative requirements (things the implementation must, must not, or may do), each requirement is given a stable identifier of the form:

```
<CATEGORY>-<NNN>
```

- `<CATEGORY>` is a short uppercase tag describing the area (for example, `ROLE` for server roles, `SEC` for security requirements, `PERF` for performance requirements, `NET` for network transports).
- `<NNN>` is a three-digit sequence number, local to the file and category, never reused within that file.
- The category tag for a file is declared at the top of that file, immediately under the "Status" line.

### Normative language

Requirements use the verbs **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY**, with the meanings defined in RFC 2119 and RFC 8174, restricted to their plain technical sense. The specification otherwise uses plain English.

### Cross-references

Links between specification files are always **relative** (for example, `[server roles](001-server-roles.md)`). Links to the root `CLAUDE.md` use `../CLAUDE.md`.
