# Response Policy Zones (RPZ)

**Purpose.** This document defines Heimdall's Response Policy Zones (RPZ) policy. It fixes the decisions that govern how Heimdall evaluates, applies, and reports policy-based answer modification on the recursive resolver role, including the mandatory action types, the mandatory trigger types, the transport and lifecycle of policy zones, the interaction with DNSSEC validation, and the performance obligations. It does not restate which roles Heimdall supports, how zone transfers are authenticated, or how DNSSEC validation is performed; those questions are settled in [`001-server-roles.md`](001-server-roles.md), [`006-protocol-conformance.md`](006-protocol-conformance.md), and [`005-dnssec-policy.md`](005-dnssec-policy.md) respectively.

**Status.** Stable.

**Requirement category.** `RPZ`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md). For the DNSSEC validation pipeline whose `AD`-flag semantics are preserved by the requirements in section 2D, see [`005-dnssec-policy.md`](005-dnssec-policy.md). For the zone-transfer and Extended DNS Error infrastructure reused by the requirements in sections 2C and 2D, see [`006-protocol-conformance.md`](006-protocol-conformance.md). For the structured-event taxonomy that the observability open question is keyed to, see [`007-threat-model.md`](007-threat-model.md).

## 1. Scope

This document applies exclusively to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md). RPZ evaluation takes place on the recursive resolver path and does not apply to the authoritative server role, which serves its own zones with no resolver-side filtering, or to the forwarder role directly. A Heimdall instance on which both the recursive resolver role and the forwarder role are active sees RPZ evaluation only on the queries that are resolved by the recursive resolver role under the precedence fixed by `ROLE-008` through `ROLE-015` in [`001-server-roles.md`](001-server-roles.md).

Subsequent narrower clarifications — the precise mapping from RPZ action to Extended DNS Error code, the ordering tie-break rules between multiple policy zones, the concrete trigger-matching data structures, extended wildcard semantics, the configuration format, dynamic reload semantics, per-hit observability, and the TTL policy for policy-modified responses — are tracked as open in section 4 and MUST NOT be assumed before they are specified.

## 2. Normative requirements

### 2A. Scope and role applicability

- **RPZ-001.** Heimdall MUST implement Response Policy Zones (RPZ) on the recursive resolver role, following the de facto standard documented by [`draft-vixie-dns-rpz`](https://datatracker.ietf.org/doc/html/draft-vixie-dns-rpz) and implemented by BIND, Unbound, PowerDNS Recursor, and Knot Resolver. This requirement applies to the recursive resolver role defined in [`001-server-roles.md`](001-server-roles.md).
- **RPZ-002.** Heimdall MUST NOT apply RPZ evaluation on the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md). The authoritative server role serves its own zones with no resolver-side filtering.
- **RPZ-003.** Heimdall MUST NOT apply RPZ evaluation on the forwarder role defined in [`001-server-roles.md`](001-server-roles.md). On an instance where the recursive resolver role and the forwarder role are both active, RPZ evaluation applies only to the queries resolved by the recursive resolver role under the precedence fixed by `ROLE-008` through `ROLE-015` in [`001-server-roles.md`](001-server-roles.md).

### 2B. Mandatory action types

- **RPZ-004.** Heimdall MUST implement the `NXDOMAIN` action. When an RPZ match triggers the `NXDOMAIN` action, Heimdall MUST respond as if the queried name does not exist, in accordance with the `NXDOMAIN` semantics of [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035) as refined by [RFC 2308](https://www.rfc-editor.org/rfc/rfc2308).
- **RPZ-005.** Heimdall MUST implement the `NODATA` action. When an RPZ match triggers the `NODATA` action, Heimdall MUST respond with an empty Answer section and a SOA record in the Authority section, in accordance with the `NODATA` semantics of [RFC 2308](https://www.rfc-editor.org/rfc/rfc2308).
- **RPZ-006.** Heimdall MUST implement the `PASSTHRU` action. When an RPZ match triggers the `PASSTHRU` action, Heimdall MUST treat the query as if no RPZ match had occurred for that resolution step and MUST NOT evaluate any subsequent policy zone against the same query. `PASSTHRU` is the explicit allow action and overrides earlier matches in other policy zones evaluated in the same evaluation order.
- **RPZ-007.** Heimdall MUST implement the `DROP` action. When an RPZ match triggers the `DROP` action, Heimdall MUST NOT emit any response to the client for the triggering query.
- **RPZ-008.** Heimdall MUST implement the `TCP-only` action. When an RPZ match triggers the `TCP-only` action on a query received over UDP, Heimdall MUST reply with a response whose TC (truncation) flag is set, in accordance with [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), in order to force the client to retry the query over TCP. When the triggering query has already arrived over a TCP-based transport, the `TCP-only` action has no further effect and MUST be equivalent to `PASSTHRU` for the purposes of response generation.
- **RPZ-009.** Heimdall MUST implement the `Local-data` action. When an RPZ match triggers the `Local-data` action, Heimdall MUST return the operator-specified local data associated with the triggered name in place of the answer that would otherwise have been produced by the recursive resolver role.
- **RPZ-010.** Heimdall MUST implement the `CNAME redirect` action. When an RPZ match triggers the `CNAME redirect` action, Heimdall MUST respond with a CNAME record pointing to the operator-specified redirection target. A `CNAME redirect` whose target is the root name `.` MUST be treated as a synonym of the `NXDOMAIN` action, in accordance with deployed RPZ conventions.

### 2C. Mandatory trigger types

- **RPZ-011.** Heimdall MUST implement the QNAME trigger. The QNAME trigger matches on the query name, and MUST support both exact-match entries and wildcarded entries in the policy zone. The exact wildcard semantics beyond the basic suffix-wildcard form are tracked as open in section 4.
- **RPZ-012.** Heimdall MUST implement the Response-IP trigger (IP trigger). The Response-IP trigger matches when the answer that the recursive resolver role would otherwise return contains an IP address, or an address falling within an operator-specified CIDR range, declared in the policy zone.
- **RPZ-013.** Heimdall MUST implement the NSIP trigger. The NSIP trigger matches when the IP address of an authoritative nameserver encountered during iterative resolution, as resolved by the recursive resolver role, falls within an operator-specified CIDR range declared in the policy zone.
- **RPZ-014.** Heimdall MUST implement the NSDNAME trigger. The NSDNAME trigger matches when the name of an authoritative nameserver encountered during iterative resolution matches an operator-specified name or suffix declared in the policy zone.
- **RPZ-015.** Heimdall MUST implement the Client-IP trigger. The Client-IP trigger matches when the source IP address of the requesting client falls within an operator-specified CIDR range declared in the policy zone.

### 2D. Policy-zone transport and lifecycle

- **RPZ-016.** Heimdall MUST be capable of loading policy zones via the zone-transfer infrastructure fixed by `PROTO-036` through `PROTO-049` in [`006-protocol-conformance.md`](006-protocol-conformance.md), so that external threat-intelligence feeds distributed as zone transfers can be consumed as policy zones. AXFR, IXFR, NOTIFY, the authentication requirements under `PROTO-044` through `PROTO-048`, and the DNSSEC-record pass-through requirement under `PROTO-049` MUST apply to policy-zone transport identically to any other zone transfer.
- **RPZ-017.** Heimdall MUST be capable of loading policy zones from local files as an alternative to zone transfer. The concrete file format and configuration surface for local policy-zone loading are tracked as open in section 4.
- **RPZ-018.** Heimdall MUST support the configuration of more than one policy zone per instance.
- **RPZ-019.** When more than one policy zone is configured on an instance, Heimdall MUST evaluate the policy zones in the operator-configured order and MUST apply first-match-wins semantics: the first policy zone whose triggers match the query produces the action for that query, and no subsequent policy zone MUST be evaluated for that query, with the single exception of the `PASSTHRU` action under `RPZ-006`, which explicitly suppresses any remaining match from any other policy zone.
- **RPZ-020.** The ordering tie-break rules applicable when more than one trigger within a single policy zone matches the same query are tracked as open in section 4 and MUST NOT be assumed before they are specified.

### 2E. DNSSEC interaction

- **RPZ-021.** When a response has been modified by an RPZ action other than `PASSTHRU`, Heimdall MUST NOT set the `AD` flag on that response. A response modified by RPZ is no longer a cryptographically validated response from the original data source within the meaning of `DNSSEC-011` in [`005-dnssec-policy.md`](005-dnssec-policy.md), and the `AD` flag would misrepresent its provenance.
- **RPZ-022.** A response that matches a policy zone with the `PASSTHRU` action is, by construction, not modified by RPZ, and is subject to the normal DNSSEC handling fixed by `DNSSEC-008` through `DNSSEC-014` in [`005-dnssec-policy.md`](005-dnssec-policy.md), including the setting of the `AD` flag on a `secure` outcome under `DNSSEC-011`.
- **RPZ-023.** When Heimdall modifies a response under an RPZ action other than `PASSTHRU`, Heimdall SHOULD include an Extended DNS Error option, as specified by [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914) and consistent with `PROTO-012` in [`006-protocol-conformance.md`](006-protocol-conformance.md), identifying that the response has been modified by policy. The precise mapping from RPZ action to Extended DNS Error code is tracked as open in section 4.

### 2F. Performance obligations

- **RPZ-024.** RPZ evaluation MUST be designed so that the per-query cost on the recursive resolver hot path is bounded and small. The following structural properties MUST be preserved: QNAME trigger matching MUST use a data structure whose per-query lookup cost is sub-linear in the number of policy-zone entries; Client-IP and Response-IP trigger matching MUST use a data structure whose per-query lookup cost is sub-linear in the number of configured CIDR ranges; NSIP and NSDNAME trigger matching MUST reuse the same classes of structures applied to the nameserver metadata produced by iterative resolution. The concrete data-structure choices are tracked as open in section 4 and are implementation details; the obligation fixed here is that the structures chosen MUST satisfy the sub-linear per-query cost property.
- **RPZ-025.** RPZ evaluation on the recursive resolver role MUST NOT impose a hot-path cost that degrades the performance targets fixed by [`008-performance-targets.md`](008-performance-targets.md). An implementation MUST measure the per-query cost of RPZ evaluation and MUST demonstrate, on each `(role, transport, architecture)` cell where the recursive resolver role is active, that the configured RPZ workload does not cause a regression against the baselines enforced under [`008-performance-targets.md`](008-performance-targets.md).

## 3. Rationale

Response Policy Zones are the industry-standard mechanism for DNS-level filtering and policy enforcement on recursive resolvers in 2026. BIND, Unbound, PowerDNS Recursor, and Knot Resolver all implement RPZ against the de facto draft-vixie-dns-rpz specification, and enterprise and ISP operators expect it as a baseline capability. A new recursive resolver delivered without RPZ support would be excluded from integration with the threat-intelligence ecosystem that distributes block-lists, sinkhole-lists, and known-indicator feeds (Spamhaus DROP, URLhaus, abuse.ch, and commercial feeds) in RPZ format; the absence of this integration would materially restrict Heimdall's applicability in the recursive role that the specification already requires.

RPZ is a natural complement to the decisions already taken across the specification: DNSSEC validation under `DNSSEC-008` through `DNSSEC-024` in [`005-dnssec-policy.md`](005-dnssec-policy.md) secures the trust path by which answers reach Heimdall; the multi-axis ACL matrix under `THREAT-033` through `THREAT-047` in [`007-threat-model.md`](007-threat-model.md) gates who may query the resolver; RPZ filters what the resolver will ultimately answer once the query has been accepted and resolved. The three layers are independent and additive. ACLs decide whether a client may interact with the resolver at all; DNSSEC decides whether a response is cryptographically trustworthy; RPZ decides whether a response is consistent with operator policy. None of these layers substitutes for another.

The transport infrastructure required by RPZ is already in place. Policy zones distributed as zone transfers reuse the AXFR, IXFR, and NOTIFY machinery fixed by `PROTO-036` through `PROTO-049` in [`006-protocol-conformance.md`](006-protocol-conformance.md), including the TSIG authentication required by `PROTO-044`, the prohibition of IP-only authentication under `PROTO-045`, and the optional XoT envelope recommended by `PROTO-047`. RPZ adds action semantics at the resolver but introduces no new transport, no new authentication surface, and no new control-plane protocol. The cost of admitting RPZ at the specification level is therefore contained to the resolver-side evaluation logic and the operator-facing configuration surface.

The DNSSEC interaction fixed by `RPZ-021` through `RPZ-023` follows the handling already adopted by deployed validating resolvers. A response that has been modified by RPZ is no longer a faithful rendition of the validated answer, so attaching the `AD` flag to it would misrepresent its provenance and would contradict the `secure`-outcome semantics fixed by `DNSSEC-011` in [`005-dnssec-policy.md`](005-dnssec-policy.md). Suppressing the `AD` flag on RPZ-modified responses is the correct and the standard behaviour. Exempting the `PASSTHRU` action from this rule under `RPZ-022` reflects the fact that `PASSTHRU` does not modify the response; by construction, `PASSTHRU` preserves whatever the resolver would otherwise have produced, including its DNSSEC validation outcome and its `AD`-flag status. Signalling policy modification through an Extended DNS Error option under `RPZ-023`, with the concrete code mapping tracked as an open question, turns the otherwise-opaque policy modification into an observable signal for downstream clients and diagnostic tooling, in the same spirit as the EDE requirements already attached to other mitigations by `DNSSEC-013`, `DNSSEC-038`, `DNSSEC-042`, `DNSSEC-045`, and `DNSSEC-051` in [`005-dnssec-policy.md`](005-dnssec-policy.md) and by `PROTO-012` in [`006-protocol-conformance.md`](006-protocol-conformance.md).

The performance obligations fixed by `RPZ-024` and `RPZ-025` are required by the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md). Without a sub-linear cost property on trigger matching, a deployment with a large policy-zone feed — threat-intelligence feeds routinely contain millions of entries — would add linear-in-feed-size cost to every cache-miss query on the recursive resolver path, which would degrade the performance targets that [`008-performance-targets.md`](008-performance-targets.md) commits to enforce per `(role, transport, architecture)` cell. Fixing the sub-linear-cost property as a structural obligation, while leaving the concrete data-structure choices open, is consistent with the pattern already adopted for ACL evaluation under `THREAT-046` in [`007-threat-model.md`](007-threat-model.md): the property is mandated at the specification level; the concrete implementation strategy is fixed at the implementation level.

## 4. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will be specified incrementally in subsequent revisions of this document.

- **RPZ action to Extended DNS Error code mapping.** The precise mapping from each RPZ action defined by `RPZ-004` through `RPZ-010` to the Extended DNS Error code signalled under `RPZ-023` — including whether the mapping uses EDE 15 ("Blocked"), EDE 16 ("Censored"), EDE 17 ("Filtered"), or other codes assigned by [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914), and any action-specific refinements — is **to be specified**. This item will be resolved jointly with the open question "Extended DNS Errors code set per situation" tracked in [`006-protocol-conformance.md`](006-protocol-conformance.md).
- **Policy-zone ordering tie-break rules.** The tie-break rules applicable under `RPZ-019` when more than one trigger within a single policy zone matches the same query — in particular the relative precedence of the five trigger types defined by `RPZ-011` through `RPZ-015` when they match on the same query, and the tie-break rule between two QNAME triggers of different specificity — are **to be specified**.
- **Concrete data structures for trigger matching.** The concrete data structures used to satisfy the sub-linear per-query cost property required by `RPZ-024` — including the choice among radix trees, PATRICIA tries, hash structures, CIDR tries, and any combination thereof, per trigger class — is **to be specified**. The obligation fixed by `RPZ-024` applies regardless of the choice.
- **Wildcard semantics for QNAME triggers beyond the basic suffix-wildcard form.** The exact semantics of wildcarded QNAME entries under `RPZ-011` beyond the basic suffix-wildcard form implemented by deployed RPZ resolvers — including any extensions, any restrictions, and the precise handling of wildcard labels in the middle of a name — are **to be specified**.
- **Configuration format for policy-zone source lists.** The concrete configuration format for declaring the list of configured policy zones, their source (zone transfer under `RPZ-016` versus local file under `RPZ-017`), their refresh intervals on the zone-transfer path, and any per-zone overrides — including per-zone action whitelists, per-zone trigger whitelists, and per-zone evaluation-order hints — is **to be specified**.
- **Dynamic reload of policy zones.** The dynamic-reload semantics applicable to policy zones — including the signalling mechanism that triggers a reload, the atomicity of a reload with respect to in-flight queries, and the interaction with the hot-reload decision tracked as open in [`006-protocol-conformance.md`](006-protocol-conformance.md) and with the ACL dynamic-reload decision tracked as open in [`007-threat-model.md`](007-threat-model.md) — is **to be specified**.
- **Observability: per-hit counters per action and per zone.** The concrete observability surface for RPZ — including per-action and per-zone hit counters, per-trigger-class hit counters, the structured-event schema for RPZ match events, and the integration with the structured-event taxonomy fixed by `THREAT-080` in [`007-threat-model.md`](007-threat-model.md) — is **to be specified**. This item will be resolved jointly with the open question "Exact schema of structured events" tracked in [`007-threat-model.md`](007-threat-model.md).
- **Response TTL for policy-modified responses.** The TTL policy applicable to responses produced under `RPZ-004`, `RPZ-005`, `RPZ-009`, and `RPZ-010` — given that the RFC 2308 negative-cache TTL semantics do not straightforwardly apply to synthesised policy responses — is **to be specified**. The open question covers both the TTL value returned to the client and the local caching behaviour on the recursive resolver role.

No implementation activity may proceed on the basis of assumptions about any of the items above.
