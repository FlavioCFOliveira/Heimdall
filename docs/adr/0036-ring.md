# ADR-0036: ring for cryptographic primitives

## Context

Several protocol features in the Heimdall core require cryptographic operations:

- **DNS Cookies** (RFC 7873, PROTO-010): server cookie derivation via HMAC-SHA256 over
  (client-cookie ∥ client-IP ∥ server-secret).
- **TSIG** (RFC 8945, PROTO-044): message authentication codes via HMAC-SHA256,
  HMAC-SHA384, or HMAC-SHA512. HMAC-MD5 and HMAC-SHA1 are rejected.
- **SIG(0)** (RFC 2931, PROTO-046): message signing and verification via RSA, ECDSA
  (P-256), or Ed25519.

A cryptographic library is required that:
- Provides HMAC (SHA-256, SHA-384, SHA-512).
- Provides RSA signing/verification, ECDSA P-256 signing/verification, and Ed25519
  signing/verification.
- Is already present in the transitive dependency tree (minimising supply-chain growth).
- Has an audited `unsafe` footprint with a documented security posture.
- Is maintained by identifiable, trusted authors.

`ring` is already a **transitive** dependency via `rustls` (ADR-0027). This ADR
promotes it to a **direct** dependency in `heimdall-core`.

## Decision

Use **`ring` v0.17** (or the latest stable version at the time of each dependency bump)
as the cryptographic primitive library for HMAC, RSA, ECDSA, and Ed25519 operations
in the Heimdall workspace.

`ring` is added as a `[dependencies]` entry in `heimdall-core`.

## Considered options

### Option A — `ring` v0.17 (selected)

BoringSSL-derived Rust library providing HMAC, SHA-2, RSA, ECDSA (P-256, P-384), and
Ed25519.

- Pros: already in the transitive tree via `rustls` (no new supply-chain surface);
  actively maintained by briansmith; used by major security-critical projects (rustls,
  webpki, Cloudflare); HMAC and signature APIs match the exact operations Heimdall needs;
  extensive fuzz corpus and test suite derived from BoringSSL; ISC-style licence.
- Cons: `unsafe`-heavy implementation (intentional, for performance and side-channel
  resistance via constant-time implementations); API is intentionally minimal and
  opinionated; does not expose low-level primitives that could be misused.

### Option B — RustCrypto (`hmac` + `sha2` + `p256` + `ed25519-dalek`)

Pure-Rust independent implementations from the RustCrypto organisation.

- Pros: zero `unsafe` in the trait layer; modular (add only what is needed).
- Cons: four separate crates = four separate ADRs; no pre-existing transitive presence
  (increases supply-chain surface); side-channel resistance is less battle-tested than
  `ring`'s BoringSSL-derived constant-time code. Rejected in favour of `ring`.

### Option C — `aws-lc-rs`

Amazon's AWS-LC-backed Rust crate; drop-in partial replacement for `ring`.

- Cons: C/C++ build dependency (AWS-LC); not yet in the transitive tree;
  significantly larger supply chain. Rejected.

### Option D — `openssl`

FFI bindings to the system OpenSSL installation.

- Cons: system-library dependency violates reproducible-build requirements (THREAT-012);
  not in the transitive tree; linking is platform-dependent. Rejected.

## Classification

**Core-critical** (runtime dependency). Loss of `ring` would require replacing all
cryptographic operations in TSIG, DNS Cookies, and SIG(0) — features that are load-bearing
for zone-transfer security and for the rate-limiting and anti-spoofing guarantees in the
spec. It cannot be removed without significant rework.

## Unsafe code posture

`ring`'s `unsafe` is intentional and security-motivated: the constant-time assembly
implementations of AES-GCM, ChaCha20-Poly1305, SHA-2, and P-256 rely on platform
intrinsics that cannot be expressed in safe Rust. Briansmith has publicly committed that
`ring`'s `unsafe` is treated with the same rigour as BoringSSL's C code.

Heimdall's own code does NOT introduce new `unsafe` to interact with `ring` — ring's
public API is entirely safe Rust. The `unsafe` footprint belongs entirely to ring's
own implementation, which is outside Heimdall's `#![deny(unsafe_code)]` boundary.

## Consequences

- `ring` is added under `[dependencies]` in `crates/heimdall-core/Cargo.toml` with the
  `features = []` default (HMAC and signature support are included by default).
- HMAC keys are treated as `ring::hmac::Key` objects; they MUST NOT be stored in plain
  memory accessible to non-cryptographic code. The `ring` API ensures keys are not
  `Clone` or `Debug`, which enforces this separation at compile time.
- DNS Cookie server secrets, TSIG key material, and SIG(0) private keys MUST be zeroed
  from memory after use. `ring` key objects implement `Drop` with zeroing where applicable.
- The `unsafe` footprint of `ring` is excluded from `cargo-geiger` Heimdall-internal
  counts; it is tracked separately in the SBOM delta report (ENG-071, ADR-0035).
