# ADR-0039: toml for TOML config parsing

## Context

ENG-225 (Sprint 9) selected TOML as the configuration file format for Heimdall. The
rationale: TOML is human-readable, unambiguous, strongly typed, and has a stable
specification (TOML v1.0). It avoids the whitespace-sensitivity of YAML and is more
expressive for structured config than INI.

A TOML parsing crate is needed that:

- Deserialises TOML into `serde::Deserialize` types (the `serde` framework is already
  selected per ADR-0038).
- Produces structured error messages that are actionable for operators.
- Is actively maintained and widely used.
- Has no `unsafe` code.

## Decision

Use **`toml` v1.1.2** as the TOML-to-serde deserialisation layer for
`heimdall-runtime`'s config loader.

## Considered options

### Option A — `toml` v1.1 (selected)

Pure-Rust, serde-based TOML v1.0 parser. The canonical TOML crate in the Rust ecosystem.

- Pros: widely used (Cargo itself uses it for `Cargo.toml` parsing); zero `unsafe`;
  serde integration is first-class; MIT/Apache-2.0; actively maintained by the Cargo/Rust
  tooling team.
- Cons: none significant.

### Option B — `figment`

Multi-source configuration library (TOML + env vars + JSON + ...).

- Pros: handles config layering natively.
- Cons: more complex API; the layering feature is not required at this stage; adds surface
  area. May be reconsidered if multi-source config is needed post-1.0.

### Option C — `config-rs`

Another multi-source config crate.

- Pros: similar to figment.
- Cons: same trade-off as figment; no advantage over plain `toml` for the current use
  case. Rejected.

### Option D — `serde_yaml`

YAML deserialisation via serde.

- Pros: YAML is familiar.
- Cons: TOML was already selected (ENG-225); YAML has well-documented footguns (implicit
  type coercions, indentation-sensitivity). Rejected.

### Option E — `serde_json` for config

- Cons: JSON is not ergonomic for hand-authored config files (no comments, mandatory
  quoting of all keys). Rejected.

## Classification

**Convenience** (config-path only; not on any query hot path). Can be removed or replaced
without affecting DNS protocol handling.

## Unsafe code posture

Zero `unsafe` in `toml` or its transitive dependencies (`toml_edit`, `toml_datetime`).
No `unsafe` is introduced in Heimdall source.

## Consequences

- `toml` is added under `[dependencies]` in `crates/heimdall-runtime/Cargo.toml`.
- Config is loaded by reading the file to a `String`, then calling
  `toml::from_str::<Config>(&contents)`.
- Parse errors are surfaced as `ConfigError::Parse(toml::de::Error)` with full positional
  information.
