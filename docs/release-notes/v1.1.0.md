# Heimdall v1.1.0 — Release Notes

**Release date:** 2026-05-05
**Type:** General Availability (first functional release)
**Tag:** `v1.1.0`

---

## Important: v1.0.0 was a non-functional placeholder

v1.0.0 shipped with a stub `fn main() {}` binary entry-point.  All library
crates (`heimdall-core`, `heimdall-runtime`, `heimdall-roles`) were real and
passing their test suites at that tag, but the server could not be started.

**v1.1.0 is the first release you can actually run.**

---

## Minimum Supported Rust Version (MSRV)

**Rust 1.94.0** — required for Rust Edition 2024 features and `let`-chain
stabilisation (RFC 2497).  The workspace is pinned to the nightly channel via
`rust-toolchain.toml` until Rust 1.94 is released as stable.

---

## Quickstart

```text
heimdall start --config /etc/heimdall/heimdall.toml
```

Validate configuration without starting the server:

```text
heimdall check-config --config /etc/heimdall/heimdall.toml
```

Print version and build metadata:

```text
heimdall version
```

Run a diagnostic DNS query:

```text
heimdall probe query example.com A @127.0.0.1:53
```

## What is new in v1.1.0

### Binary entry-point (Sprints 43–45)

The `heimdall` binary now fully implements the contract specified in
`specification/015-binary-contract.md`:

- `start` — full boot sequence: config parse → logging init → privilege drop →
  Tokio runtime → role initialisation → listener bind → `sd_notify(READY=1)`.
- `check-config` — validates TOML without starting the server; exits 0 on
  success, 2 on parse/validation failure.
- `version` — emits JSON with 10 fields: `name`, `version`, `git_hash`,
  `build_date`, `rustc_version`, `target_triple`, `features`, `tier`, `msrv`,
  and `runtime` (uid/gid).
- `probe` — diagnostic CLI for DNS query, DNSSEC chain check, server health
  poll, and latency benchmarking.

### Five-stage admission pipeline (Sprints 43–46)

Each inbound query passes through:

1. **ACL** — CIDR/transport/role/qname-pattern matching (THREAT-033..047).
2. **Connection limits** — per-listener in-flight cap (THREAT-062..068).
3. **Cookie/load gate** — DNS Cookie validation under load (THREAT-069).
4. **RRL** — Response Rate Limiting per RFC 8906 (THREAT-048..050).
5. **Per-client query rate limiter** — three tiers (Anonymous / ValidatedCookie /
   AuthenticatedIdentity), burst-window accounting (THREAT-051..053).

Per-stage `AtomicU64` telemetry counters are exposed at `/metrics`.

### DNSSEC and TSIG (Sprint 47)

- DNSSEC zone structural checks at load time (PROTO-101): NSEC/NSEC3 bitmap
  completeness, RRSIG coverage, DS/DNSKEY cross-check.
- TSIG algorithm negotiation E2E: SHA-256 (required), SHA-1 absent/rejected,
  BADSIG on bad MAC, BADKEY on unknown key name (task #589).
- Role enforcement: disabled-role REFUSED responses (ROLE-005/006); listener
  role unknown-key rejection (ROLE-021).

### Observability (Sprint 52)

- `/healthz` — always 200 OK (liveness probe).
- `/readyz` — 200 OK when ready; 503 Service Unavailable during drain (task #520).
- `/metrics` — OpenMetrics format; 18+ counters with `# HELP`/`# TYPE` lines
  and mandatory `# EOF` terminator; `application/openmetrics-text` content type
  (task #521).
- `/version` — JSON with all 10 build-info fields (task #522).
- `sd_notify(WATCHDOG=1)` sent every `WATCHDOG_USEC / 2` (task #523).

### Admin audit log (Sprint 52, task #524)

Every admin-RPC command is recorded in an HMAC-SHA256 chained log:

- Entries carry: sequence number, timestamp, identity, command, outcome,
  HMAC tag over `(seq ‖ prev_tag ‖ identity ‖ command ‖ outcome)`.
- `AuditLogger::verify_chain` validates the full chain offline.
- The sequence counter is held inside the same mutex as `prev_tag`,
  guaranteeing chain integrity under concurrent access.

### Stability and soak testing (Sprint 53)

Eight new integration-test modules validate long-running stability:

| Module | What it validates |
|--------|-------------------|
| `soak_sustained_load` | QPS stability algorithm; nightly 24h marker |
| `soak_memory_leak` | VmRSS growth bounded at 10 MiB; heaptrack guidance |
| `soak_fd_leak` | FD count stable within ±5 after 1 000 reload cycles |
| `soak_cache_eviction` | Hit-rate ≥ 25% under 4× over-capacity pressure |
| `soak_tek_rotation` | TEK generation counter monotonic under 8-thread concurrency |
| `soak_crash_recovery` | `SharedStore` persists across generation swaps; Redis reconnect |
| `soak_ddos` | RRL + per-client RL mitigate UDP flood / NXDOMAIN flood / NXNSAttack |
| `soak_reload_under_load` | 144 SIGHUP reload cycles with 0 concurrent-read failures |

Full-duration soak tests (24h, 1 000-cycle) are gated behind `HEIMDALL_SOAK_TESTS=1`.

### HTTP/2 and QUIC hardening (Sprint 42)

- HTTP/2: `MAX_CONCURRENT_STREAMS` enforced; header-field-count cap; server
  `SETTINGS` frame validated on connection open.
- QUIC: stream-count limit per connection; `CRYPTO` frame size cap to prevent
  amplification during the TLS handshake.

## Upgrade path from v1.0.0

v1.0.0 was non-functional.  Upgrade by replacing the binary; no configuration
migration is required.  All TOML schema, API surfaces, and wire formats are
stable as of v1.0.0 and unchanged in v1.1.0.

## Known limitations

- External security audit sign-off is pending publication.
- SLSA provenance hash binding is a stub (ENG-080).
- `cargo install --check-config` not yet implemented.
- crates.io publication for library crates is part of this release pipeline
  (task #568).
