# Network transports

**Purpose.** This document defines the set of DNS network transports that Heimdall supports, the invariant that governs when each transport listener is instantiated, and the matrix of (role, transport) combinations that Heimdall supports at the software level for incoming traffic. It also fixes, for the forwarder role, the set of outbound transports Heimdall uses when contacting configured upstream resolvers, and, for the recursive resolver role, the set of outbound transports Heimdall uses when contacting authoritative servers.

**Status.** Stable.

**Requirement category.** `NET`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md).

## 1. Supported transports

Heimdall supports exactly the following four DNS transports. Any given deployment combines a subset of them through configuration; no transport is always on and none is a default.

### 1.1 DNS classic over UDP and TCP on port 53

Traditional DNS carried over UDP and TCP on port 53, including the TCP requirements applicable to DNS servers and the EDNS(0) extension mechanism. Anchor RFCs: [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034), [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), [RFC 7766](https://www.rfc-editor.org/rfc/rfc7766), [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891).

### 1.2 DNS-over-TLS (DoT) on port 853

DNS messages carried over a TLS-protected TCP connection on port 853. Anchor RFCs: [RFC 7858](https://www.rfc-editor.org/rfc/rfc7858), [RFC 8310](https://www.rfc-editor.org/rfc/rfc8310).

### 1.3 DNS-over-HTTPS (DoH)

DNS messages carried as HTTPS requests and responses. Anchor RFC: [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484).

### 1.4 DNS-over-QUIC (DoQ) on port 853

DNS messages carried over QUIC on port 853. Anchor RFC: [RFC 9250](https://www.rfc-editor.org/rfc/rfc9250).

## 2. Normative requirements

- **NET-001.** Heimdall MUST support the four transports defined in section 1: DNS classic over UDP and TCP on port 53, DNS-over-TLS on port 853, DNS-over-HTTPS, and DNS-over-QUIC on port 853.
- **NET-002.** Heimdall MUST NOT implement DNSCrypt. DNSCrypt is explicitly out of scope.
- **NET-003.** DNS classic transport MUST conform to [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034), [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), [RFC 7766](https://www.rfc-editor.org/rfc/rfc7766), and [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891).
- **NET-004.** DNS-over-TLS transport MUST conform to [RFC 7858](https://www.rfc-editor.org/rfc/rfc7858) and [RFC 8310](https://www.rfc-editor.org/rfc/rfc8310) and MUST listen on port 853 when enabled.
- **NET-005.** DNS-over-HTTPS transport MUST conform to [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484).
- **NET-006.** DNS-over-HTTPS MUST be served over both HTTP/2 ([RFC 9113](https://www.rfc-editor.org/rfc/rfc9113)) and HTTP/3 ([RFC 9114](https://www.rfc-editor.org/rfc/rfc9114)).
- **NET-007.** HTTP version negotiation for DNS-over-HTTPS MUST use ALPN, and the server MUST advertise HTTP/3 via Alt-Svc where applicable.
- **NET-008.** DNS-over-QUIC transport MUST conform to [RFC 9250](https://www.rfc-editor.org/rfc/rfc9250) and MUST listen on port 853 when enabled.
- **NET-009.** A transport listener MUST be instantiated only when a configured role requires it. No transport listener is always on, and none is a default. This is the subordination of transports to role activation as defined by `ROLE-003`, `ROLE-004`, `ROLE-005`, `ROLE-006`, and `ROLE-007` in [`001-server-roles.md`](001-server-roles.md).
- **NET-010.** The set of active transports in an instance is determined exclusively by the roles enabled in its configuration and by the per-role transport bindings. There is no implicit default transport.
- **NET-011.** Heimdall MUST support, at the software level, every combination of (role, transport) for incoming traffic, where role is any of the three roles defined in [`001-server-roles.md`](001-server-roles.md) (authoritative server, recursive resolver, forwarder) and transport is any of the four transports defined in section 1 of this document (DNS classic over UDP and TCP on port 53, DNS-over-TLS on port 853, DNS-over-HTTPS, DNS-over-QUIC on port 853). Whether a given combination is active in a running instance is determined exclusively by configuration and remains subordinated to `NET-009` and to `ROLE-003` through `ROLE-007` in [`001-server-roles.md`](001-server-roles.md); no listener is instantiated for a combination whose role is not enabled.
- **NET-012.** Heimdall MUST support, on the outbound client-side of the forwarder role, every transport defined in section 1 of this document: DNS classic over UDP and TCP on port 53, DNS-over-TLS on port 853, DNS-over-HTTPS over both HTTP/2 and HTTP/3, and DNS-over-QUIC on port 853. This obligation is restricted to the forwarder role; outbound transport selection for the recursive resolver is tracked as open in section 5.
- **NET-013.** The outbound transport used to contact each configured upstream resolver MUST be declared on a per-upstream basis in the deployment configuration. There is no implicit default outbound transport, and no outbound transport is always on.
- **NET-014.** Outbound client code paths, sockets, connection pools, and TLS or QUIC client state for a given transport MUST NOT be instantiated unless at least one configured upstream in the running instance declares that transport. This is the outbound counterpart of `NET-009` and applies the same structural gating rule to the client side of the forwarder.
- **NET-015.** TLS-over-TCP outbound client connections used by the forwarder (DoT and DoH over HTTP/2) MUST negotiate TLS 1.3 only, consistent with `SEC-001`, `SEC-002`, and `SEC-003` in [`003-crypto-policy.md`](003-crypto-policy.md). Heimdall MUST NOT initiate, accept, or complete an outbound TLS handshake that negotiates any TLS version earlier than TLS 1.3 on these transports.
- **NET-016.** QUIC-based outbound client connections used by the forwarder (DoQ and DoH over HTTP/3) MUST use QUIC version 1 or QUIC version 2, and MUST NOT use QUIC draft versions, Google QUIC (gQUIC), or any other non-standard QUIC variant. This is consistent with `SEC-017`, `SEC-018`, and `SEC-019` in [`003-crypto-policy.md`](003-crypto-policy.md) and extends those requirements to the outbound direction.
- **NET-017.** Heimdall MUST NOT send TLS 1.3 early data (0-RTT) on TLS-over-TCP outbound client connections and MUST NOT send QUIC 0-RTT application data on QUIC-based outbound client connections on any outbound transport used by the forwarder. This extends to the outbound direction the server-side prohibitions established by `SEC-005` through `SEC-007` in [`003-crypto-policy.md`](003-crypto-policy.md) for TLS-over-TCP transports and by `SEC-022` through `SEC-024` in [`003-crypto-policy.md`](003-crypto-policy.md) for QUIC-based transports.
- **NET-018.** On every outbound TLS-protected or QUIC-based client connection used by the forwarder, Heimdall MUST validate the upstream server certificate. A handshake in which certificate validation fails MUST be aborted and MUST NOT be used to carry any DNS query or response. The precise validation policy (trust store scope, chain validation rules, hostname or SNI verification, optional SPKI pinning, revocation checking) is tracked as open in section 5.
- **NET-019.** Heimdall MUST support, on the outbound client-side of the recursive resolver role, DNS classic over UDP and TCP on port 53 as the mandatory baseline transport for contacting authoritative servers. Every authoritative contact attempt MUST be capable of falling back to DNS classic over UDP and TCP on port 53 when an encrypted alternative is unavailable or fails. The concrete fallback policy on handshake failure of an encrypted alternative is tracked as open in section 5.
- **NET-020.** Heimdall MUST support, on the outbound client-side of the recursive resolver role, Authoritative DNS-over-TLS (ADoT) as specified by [RFC 9539](https://www.rfc-editor.org/rfc/rfc9539) as an opt-in capability for contacting authoritative servers. ADoT MUST NOT be forced on any authoritative server. ADoT MUST be attempted against a specific authoritative server only when the resolver has positive, configured, or discovered evidence that that authoritative server supports it. The capability-discovery mechanism that produces such evidence is tracked as open in section 5.
- **NET-021.** Heimdall MUST NOT implement DNS-over-QUIC for authoritative contact ([RFC 9250, section 5.4](https://www.rfc-editor.org/rfc/rfc9250#section-5.4)) on the outbound client-side of the recursive resolver role at this stage. Heimdall MUST NOT implement Authoritative DNS-over-HTTPS (ADoH) on the outbound client-side of the recursive resolver role at this stage. Both transports are deliberately deferred and MAY be reconsidered in a future revision of this document; they are not rejected outright.
- **NET-022.** The TLS version policy applied to ADoT connections on the outbound client-side of the recursive resolver role is inherited from the outbound forwarder policy. Heimdall MUST negotiate TLS 1.3 only on ADoT and MUST NOT initiate, accept, or complete an outbound TLS handshake that negotiates any TLS version earlier than TLS 1.3. This obligation mirrors `NET-015` and is consistent with `SEC-001`, `SEC-002`, and `SEC-003` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **NET-023.** Heimdall MUST NOT send TLS 1.3 early data (0-RTT) on ADoT connections on the outbound client-side of the recursive resolver role. This obligation mirrors the client 0-RTT prohibition already established by `NET-017` for the outbound forwarder and extends to the outbound direction the server-side prohibitions in `SEC-005` through `SEC-007` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **NET-024.** On every ADoT connection on the outbound client-side of the recursive resolver role, Heimdall MUST validate the authoritative server certificate. A handshake in which certificate validation fails MUST be aborted and MUST NOT be used to carry any DNS query or response. The precise validation policy (trust store scope, chain validation rules, hostname or SNI verification, optional SPKI pinning, revocation checking) is a cross-transport question shared with the outbound forwarder and is tracked as open in section 5, jointly with the analogous item under `NET-018`.
- **NET-025.** The DoH server MUST implement the HTTP GET method and the HTTP POST method for DoH requests, as required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484). Heimdall MUST accept DoH requests on both methods and MUST NOT expose a configuration option that disables either method on a DoH listener. This applies to DoH over HTTP/2 and to DoH over HTTP/3.
- **NET-026.** The DoH server MUST use the media type `application/dns-message` for DoH request bodies and for DoH response bodies, as required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484). Heimdall MUST reject any DoH request whose declared media type is not `application/dns-message`, and MUST NOT emit any DoH response under a different media type. This applies to DoH over HTTP/2 and to DoH over HTTP/3.
- **NET-027.** The DoH URI path MUST be configurable per listener, and the default value MUST be `/dns-query`. A request whose path does not match the configured DoH URI path on a DoH listener MUST NOT be processed as a DoH request. This applies to DoH over HTTP/2 and to DoH over HTTP/3.
- **NET-028.** Heimdall MUST NOT implement any HTTP/2 or HTTP/3 feature or extension that is not required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484) for DoH at this stage. Features and extensions that are explicitly out of scope under this requirement include, but are not limited to, HTTP/2 server push, HTTP/2 and HTTP/3 priorities ([RFC 9218](https://www.rfc-editor.org/rfc/rfc9218)), the HTTP CONNECT method, WebTransport extensions, and HTTP/3 datagrams ([RFC 9297](https://www.rfc-editor.org/rfc/rfc9297)). ALPN-based HTTP version negotiation and Alt-Svc advertisement for HTTP/3 are governed by `NET-006` and `NET-007` and are not restated here. This applies to DoH over HTTP/2 ([RFC 9113](https://www.rfc-editor.org/rfc/rfc9113)) and to DoH over HTTP/3 ([RFC 9114](https://www.rfc-editor.org/rfc/rfc9114)).

The following table enumerates the (role, transport) combinations that Heimdall supports at the software level for incoming traffic, as required by `NET-011`. ## 4. Rationale

Each transport has a single implementation shared across all three roles; restricting which roles may bind which transports would not remove any code from the binary, while it would introduce configuration-level exceptions that must be specified, implemented, tested, and documented. The runtime attack surface is already constrained by the role-activation invariant (`ROLE-003` through `ROLE-007`) and by the listener-subordination rule (`NET-009`), so a disabled role exposes no listener regardless of the matrix. Authoritative service over encrypted transports is active IETF practice, including Authoritative DNS-over-TLS ([RFC 9539](https://www.rfc-editor.org/rfc/rfc9539)) and Authoritative DNS-over-QUIC ([RFC 9250, section 8](https://www.rfc-editor.org/rfc/rfc9250#section-8)); excluding those combinations would diverge from current practice without a corresponding security benefit. A uniform matrix is therefore simpler to specify, implement, test, and document than one carrying exceptions.

Requiring the forwarder to support all four outbound transports (`NET-012`) reflects the operational coverage the role must provide in 2026. Privacy-preserving and zero-trust deployments require encrypted upstream DNS, and the major upstream resolvers that those deployments typically target — Cloudflare, Google, Quad9, NextDNS, AdGuard — primarily expose DoT and DoH, with DoQ emerging. A forwarder limited to DNS classic would exclude the most relevant contemporary use cases and would push operators back onto unencrypted upstream paths. The marginal cost of supporting the full set is zero with respect to dependencies: every protocol stack is already present in the binary to satisfy the inbound-transport decisions in section 2, and the outbound client side reuses the same cryptographic and protocol primitives. Declaring the outbound transport per upstream in configuration (`NET-013`) keeps the choice explicit and local to each upstream rather than implied by a global default, which reduces the likelihood of an unintended downgrade from an encrypted to an unencrypted upstream path. Gating outbound client state structurally on configuration (`NET-014`) extends to the outbound direction the surface-reduction approach already applied to role activation in [`001-server-roles.md`](001-server-roles.md) and to inbound transport listeners by `NET-009`: client sockets, connection pools, and TLS or QUIC state for transports that no configured upstream actually uses are absent from the reachable binary rather than guarded by runtime flags. Inheriting the inbound cryptographic policy onto the outbound side (`NET-015`, `NET-016`, `NET-017`) avoids divergent inbound and outbound semantics for the same protocols, which would be a latent source of incidents; the same reasoning that fixed TLS 1.3 only, QUIC v1 and v2 only, and no 0-RTT on the server side applies unchanged when Heimdall acts as a client to an upstream DNS server. Requiring outbound server-certificate validation (`NET-018`) at the specification level fixes the non-negotiable property now and defers only the calibration of the validation policy, so no outbound flow can ever be established against an unauthenticated upstream even before the detailed validation rules are agreed.

Restricting the DoH server to the minimum HTTP feature surface required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484) (`NET-025`, `NET-026`, `NET-027`, `NET-028`) follows the security and performance principles of [`../CLAUDE.md`](../CLAUDE.md). DoH is a simple request/response protocol; no HTTP extension adds functional value beyond what RFC 8484 specifies. HTTP/2 server push is formally deprecated in major clients and is outside the DoH interaction model. HTTP/2 and HTTP/3 priorities ([RFC 9218](https://www.rfc-editor.org/rfc/rfc9218)) offer marginal benefit for a DoH workload in which queries are small, independent, and fast; implementing them would introduce parser, state, and ordering cost without measurable benefit. The HTTP CONNECT method, WebTransport extensions, and HTTP/3 datagrams ([RFC 9297](https://www.rfc-editor.org/rfc/rfc9297)) are outside the DoH model; implementing them would introduce attack surface and expand the test matrix without a legitimate use case. A minimal feature surface reduces fingerprintability, shrinks the test matrix, and lowers the risk of feature-interaction defects. Cross-referencing `NET-006` and `NET-007` for ALPN and Alt-Svc rather than restating them keeps the DoH HTTP-conformance rules localised to a single decision and avoids duplication drift between requirements.

The outbound transport set fixed for the recursive resolver role in `NET-019` through `NET-024` reflects the deployment reality of the authoritative ecosystem in 2026 and the asymmetry between resolver-to-upstream-resolver traffic and resolver-to-authoritative traffic. Requiring DNS classic over UDP and TCP on port 53 as the mandatory baseline (`NET-019`) is non-negotiable for the same reason the role itself is: the overwhelming majority of authoritative servers on the public Internet in 2026 speak only classical DNS, and a recursive resolver without this baseline cannot complete iterative resolution through the zone hierarchy. Supporting ADoT as an opt-in, never-forced capability (`NET-020`) is the specification-level commitment to follow [RFC 9539](https://www.rfc-editor.org/rfc/rfc9539), which is the official IETF direction for resolver-to-authoritative privacy and which is actively being adopted by large public resolvers; tying ADoT attempts to positive evidence of support avoids introducing query-level failures or latency regressions against the (still-dominant) set of authoritative servers that do not speak ADoT. Deferring DoQ for authoritative ([RFC 9250, section 5.4](https://www.rfc-editor.org/rfc/rfc9250#section-5.4)) and Authoritative DoH (ADoH) (`NET-021`) is a scope decision, not a rejection: their residual deployment among authoritative servers in 2026 does not justify the additional discovery, fallback, and hardening surface they would require on the resolver outbound side, and the TLS 1.3 client required for ADoT is already present in the binary from the outbound forwarder decisions, whereas a DoQ or ADoH authoritative client would expand the outbound surface further without a corresponding return today. Inheriting the outbound cryptographic policy already fixed for the forwarder onto the recursive-resolver outbound side (`NET-022`, `NET-023`) avoids divergent outbound semantics for the same TLS 1.3 stack: the reasoning that fixed TLS 1.3 only and no client 0-RTT for the outbound forwarder applies unchanged when Heimdall, as a recursive resolver, initiates ADoT to an authoritative server. Requiring certificate validation on every ADoT connection (`NET-024`) at the specification level fixes the non-negotiable property now and defers only the calibration of the validation policy, jointly with the analogous forwarder item under `NET-018`, so no outbound ADoT flow can ever be established against an unauthenticated authoritative even before the detailed validation rules are agreed.

## 5. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will need to be specified before the implementation of these transports can begin.

- **Upstream certificate validation policy (forwarder).** The validation policy applied to upstream server certificates under `NET-018` — strict chain validation, hostname or SNI verification, optional SPKI pinning, revocation checking via OCSP stapling or CRLs — is **to be specified**.
- **ADoT capability discovery for authoritative servers (recursive resolver).** The mechanism by which the recursive resolver obtains positive evidence that a specific authoritative server supports ADoT, as required by `NET-020`, is **to be specified**. Candidate mechanisms include SVCB and HTTPS resource records ([RFC 9460](https://www.rfc-editor.org/rfc/rfc9460) and [draft-ietf-dnsop-svcb-dns](https://datatracker.ietf.org/doc/draft-ietf-dnsop-svcb-dns/)), Discovery of Network-designated Resolvers ([RFC 9463](https://www.rfc-editor.org/rfc/rfc9463)), passive probing, static per-zone configuration, or a combination of these. The selected mechanism and its triggering rules are to be specified.
- **ADoT fallback policy on handshake failure (recursive resolver).** The fallback behaviour when an ADoT attempt against an authoritative server fails during or after the TLS handshake, including whether the resolver accepts a degradation to plain DNS over UDP or TCP on port 53 against the same authoritative, attempts another authoritative name server of the same zone, or fails the query outright, and the operator-visible signal that governs this choice, is **to be specified**. This item interacts with `NET-019` and `NET-020`.
- **Downgrade-protection strategy for ADoT capability signalling (recursive resolver).** The strategy that protects the ADoT capability-discovery signal from on-path spoofing is **to be specified**. Capability carriers such as SVCB, HTTPS, and DNR records can themselves be forged by an on-path attacker, so their integrity requires DNSSEC validation on the resolver side; this item therefore cross-references the DNSSEC requirements consolidated in [`005-dnssec-policy.md`](005-dnssec-policy.md) and will be resolved jointly with the open questions tracked there.
- **ADoT session-ticket cache policy for authoritative servers (recursive resolver).** The per-authoritative-server-IP session-ticket cache policy used by the ADoT client, including cache scoping (per server IP, per zone, or per name server), ticket TTL, invalidation triggers, and cache-size bounds, is **to be specified**.
- **Outbound connection pool policy for ADoT and for plain TCP on port 53 (recursive resolver).** The per-authoritative outbound connection pool policy for ADoT and for plain TCP on port 53 on the recursive resolver outbound side, including pool size per authoritative, idle-keepalive lifetime, and eviction strategy, is **to be specified**.
- **Upstream fallback policy on failure (forwarder).** The fallback behaviour between configured upstreams on failure, including whether an encrypted-to-plain downgrade is ever permitted and, if so, under what signal, is **to be specified**.
- **Upstream load-balancing, failover, and health-checking (forwarder).** The strategy for distributing queries across multiple configured upstreams, detecting upstream unavailability, and restoring traffic after recovery is **to be specified**.
- **Outbound connection pooling, keepalive, and multiplexing (forwarder).** The per-transport strategy for reusing outbound connections, keeping them alive, and multiplexing queries over them is **to be specified**.
- **EDNS(0) Client Subnet (ECS) behaviour on the upstream path (forwarder).** Whether ECS options received from downstream clients are propagated verbatim, stripped, rewritten, or controlled by operator configuration on outbound queries to upstream resolvers is **to be specified**.
- **TLS policy.** The minimum TLS version is resolved in [`003-crypto-policy.md`](003-crypto-policy.md) (TLS 1.3 only), 0-RTT is refused on DoT and on DoH over HTTP/2, session resumption on DoT and DoH over HTTP/2 uses stateless TLS 1.3 session tickets, and TLS client-certificate authentication (mTLS) is supported as an optional, default-OFF, per-listener feature on DoT and on DoH over HTTP/2, all as resolved in the same document. The remaining TLS policy items, namely the session-ticket operational defaults (TEK rotation frequency, acceptance window for retired TEKs, and maximum ticket lifetime) and the cross-transport mTLS validation policy (trust store scope, chain validation rules, pinning, revocation checking, certificate-to-identity mapping) applying uniformly to TLS-over-TCP and TLS-inside-QUIC mTLS, are **to be specified** and are tracked in [`003-crypto-policy.md`](003-crypto-policy.md).
- **QUIC policy.** The QUIC version support, QUIC version negotiation, the exclusion of QUIC draft versions and non-standard variants, the refusal of QUIC 0-RTT application data for DoQ and for DoH over HTTP/3, the amplification-mitigation and address-validation policy for DoQ and for DoH over HTTP/3 (amplification-limit enforcement, Retry on new flows without a valid server-issued `NEW_TOKEN`, single-use `NEW_TOKEN` with anti-replay enforcement, and rotatable `NEW_TOKEN` key), and the optional, default-OFF, per-listener QUIC-layer client-certificate authentication (TLS-inside-QUIC mTLS) for DoQ and for DoH over HTTP/3 are resolved in [`003-crypto-policy.md`](003-crypto-policy.md). The remaining QUIC policy items — the `NEW_TOKEN` operational defaults (token-encryption-key rotation frequency, key retention window, strike-register persistence strategy) — are **to be specified** and are tracked in [`003-crypto-policy.md`](003-crypto-policy.md).
- **HTTP/2 and HTTP/3 hardening numeric defaults.** The minimum HTTP/2 and HTTP/3 conformance scope for DoH is resolved by `NET-025`, `NET-026`, `NET-027`, and `NET-028` (GET and POST with `application/dns-message`, a configurable DoH URI path with `/dns-query` as default, and no HTTP extensions beyond those required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484); ALPN and Alt-Svc are governed by `NET-006` and `NET-007`). The HTTP/2 and HTTP/3 hardening posture is resolved in [`003-crypto-policy.md`](003-crypto-policy.md) (explicit server-layer mitigations covering total header-block size, per-connection stream-concurrency cap, HPACK and QPACK dynamic-table upper bounds, rapid-reset detection, CONTINUATION-frame cap, control-frame rate limits, incomplete-header-block timeout, and flow-control window bounds). The remaining HTTP/2 and HTTP/3 hardening item — the numeric defaults (threshold values, size limits, rate-limit parameters, window sizes, and timeouts) for those mitigations — is **to be specified** and is tracked in [`003-crypto-policy.md`](003-crypto-policy.md).

No implementation activity may proceed on the basis of assumptions about any of the items above.
