# Threat model

**Purpose.** This document defines Heimdall's threat model. It enumerates the classes of attacker that the server is structurally obliged to defend against (in scope), the classes of attacker that are delegated to the operator and to the deployment environment (out of scope), and the mapping between the in-scope classes and the mitigations already decided elsewhere in the specification. It also fixes the mandatory structural measures that apply to the supply-chain attacker class.

**Status.** Stable.

**Requirement category.** `THREAT`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). This document consolidates, at the threat-model level, decisions already made in [`001-server-roles.md`](001-server-roles.md), [`002-transports.md`](002-transports.md), [`003-crypto-policy.md`](003-crypto-policy.md), [`004-cache-policy.md`](004-cache-policy.md), [`005-dnssec-policy.md`](005-dnssec-policy.md), and [`006-protocol-conformance.md`](006-protocol-conformance.md).

## 1. Scope

This threat model is project-wide. It applies across all three roles (authoritative server, recursive resolver, forwarder), across all four transports (DNS over UDP and TCP on port 53, DoT, DoH over HTTP/2 and HTTP/3, DoQ), and across every deployment mode supported by Heimdall. It does not alter the decisions already made in the specification files cross-referenced above; it classifies them under attacker classes and identifies residual gaps to be closed by subsequent threat-model clarifications.

A class being "in scope" means that Heimdall, as a running process, MUST provide structural mitigation against that class. A class being "out of scope" means that mitigation is delegated to the operator and to the deployment environment; Heimdall MUST NOT be described, documented, or configured as if it structurally mitigated an out-of-scope class.

## 2. Normative requirements

### 2.1 Attacker classes in scope

- **THREAT-001.** The **off-path attacker** — an adversary who can inject packets into the network without observing the legitimate traffic — MUST be treated as in scope. Representative techniques include cache poisoning (including Kaminsky and its variants), spoofing of source addresses, and reflection amplification that abuses Heimdall's own responses or targets other victims through Heimdall.
- **THREAT-002.** The **on-path attacker** — an adversary who can observe and modify packets in flight between Heimdall and another party — MUST be treated as in scope. Representative techniques include interception of cleartext DNS, TLS downgrade, and session manipulation.
- **THREAT-003.** The **upstream peer compromise** attacker — an authoritative server reached by the recursive resolver role, or an upstream resolver reached by the forwarder role, that is malicious or has been compromised — MUST be treated as in scope.
- **THREAT-004.** The **adversarial zone operator** — the operator of a zone that Heimdall legitimately resolves, where the zone itself has been crafted to exploit the validator — MUST be treated as in scope. Representative techniques include published DNSSEC records constructed to stress or defeat the validating path.
- **THREAT-005.** The **volumetric denial-of-service and distributed denial-of-service attacker** — an adversary who attempts to exhaust Heimdall's capacity through query floods, reflection and amplification abuse, NXDOMAIN flooding, random-subdomain attacks, or analogous techniques — MUST be treated as in scope.
- **THREAT-006.** The **supply-chain attacker** — an adversary who compromises an external dependency of Heimdall, or the build pipeline that produces the Heimdall binary, in order to inject malicious code into the artefact distributed to operators — MUST be treated as in scope.

### 2.2 Attacker classes out of scope

- **THREAT-007.** The **local insider or operator compromise** attacker — an actor who already holds root or equivalent privilege on the host running Heimdall — is explicitly OUT OF SCOPE for structural mitigation by the Heimdall process. A process MUST NOT be documented as if it could structurally defend against an attacker who already controls the environment in which it runs. Operational hardening guidance for this class (privilege drop, sandboxing, systemd hardening, platform-equivalent mechanisms) is provided separately as guidance, not as structural mitigation of this class, and is tracked as an open question in section 5.
- **THREAT-008.** The **physical and side-channel** attacker — physical access to the host, timing side-channels, Rowhammer, hardware Trojans, and analogous hardware-level techniques — is explicitly OUT OF SCOPE. Side-channel resistance for cryptographic operations is inherited from the cryptographic libraries on which Heimdall depends; physical attacks against the host are delegated to the deployment environment. Specific primitive choices that influence the side-channel profile are tracked as an open question in section 5.
- **THREAT-009.** No class listed as out of scope in `THREAT-007` or `THREAT-008` MAY be reinterpreted as structurally mitigated by the server itself. Documentation, configuration, and communication about Heimdall MUST preserve the distinction between structural mitigation (in scope) and operational or platform delegation (out of scope).

### 2.3 Supply-chain structural measures

The requirements in this sub-section operationalise `THREAT-006` into concrete, non-negotiable measures on the Heimdall project.

- **THREAT-010.** External dependencies of Heimdall MUST be kept to the strict minimum justified on a case-by-case basis, consistent with the obligation already stated in [`../CLAUDE.md`](../CLAUDE.md). Each dependency addition is an explicit engineering decision, not a default convenience.
- **THREAT-011.** The build pipeline MUST integrate an audited-dependency gate in continuous integration. `cargo-audit` and `cargo-vet`, or equivalent audited-dependency tooling that provides the same guarantees, MUST be enforced as a gate; a build whose dependencies fail the gate MUST NOT be promotable to an official release.
- **THREAT-012.** Official release builds of Heimdall MUST be reproducible. The build configuration and the build steps MUST be documented and MUST be verifiable by an independent party using the documented inputs.
- **THREAT-013.** Official release artefacts MUST be signed. A cryptographic signing process (for example, sigstore / `cosign`) MUST be used, and the signing process — including the identity of the signing key, the verification procedure, and the revocation procedure — MUST be documented.
- **THREAT-014.** A Software Bill of Materials (SBOM) MUST be produced for each official release and MUST be distributed alongside the signed artefact.

### 2.4 Mapping of in-scope classes to mitigations already decided

The following mapping consolidates, for each in-scope attacker class, the requirements elsewhere in the specification that provide structural mitigation. The mapping enumerates existing requirement identifiers rather than restating them; the authoritative text for each requirement lives in the file where it is defined. Gaps in this mapping are the subject of the open questions in section 5.

- **THREAT-015.** The **off-path attacker** (`THREAT-001`) is structurally mitigated by, at minimum, the following requirements: 0x20 case randomisation and case-match verification on recursive-to-authoritative queries (`PROTO-*` in [`006-protocol-conformance.md`](006-protocol-conformance.md)); DNS Cookies as part of the mandatory EDNS(0) extension set (`PROTO-*` in [`006-protocol-conformance.md`](006-protocol-conformance.md)); DNSSEC validation on both validating roles, including aggressive NSEC / NSEC3 use, the KeyTrap cap, and the NSEC3 iteration cap (`DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md)); RFC 9471 glue-record handling on the recursive resolver role (`PROTO-*` in [`006-protocol-conformance.md`](006-protocol-conformance.md)); QUIC address validation with single-use `NEW_TOKEN` and the RFC 9000 amplification limit (`SEC-025` through `SEC-030` in [`003-crypto-policy.md`](003-crypto-policy.md)); and segregated recursive and forwarder caches (`CACHE-*` in [`004-cache-policy.md`](004-cache-policy.md)).
- **THREAT-016.** The **on-path attacker** (`THREAT-002`) is structurally mitigated by, at minimum, the following requirements: the encrypted transports DoT, DoH over HTTP/2, DoH over HTTP/3, and DoQ (`NET-*` in [`002-transports.md`](002-transports.md)); the TLS 1.3-only policy, including disabling the TLS 1.2 code path at build or configuration time (`SEC-001` through `SEC-004` in [`003-crypto-policy.md`](003-crypto-policy.md)); the QUIC version policy restricted to QUIC v1 and v2 (`SEC-017` through `SEC-021` in [`003-crypto-policy.md`](003-crypto-policy.md)); the prohibition of TLS 1.3 early data and of QUIC 0-RTT (`SEC-005` through `SEC-007`, `SEC-022` through `SEC-024` in [`003-crypto-policy.md`](003-crypto-policy.md)); the stateless TLS 1.3 session-ticket policy with TEK rotation (`SEC-008` through `SEC-011` in [`003-crypto-policy.md`](003-crypto-policy.md)); and the optional per-listener mTLS support on all encrypted transports (`SEC-012` through `SEC-016`, `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md)).
- **THREAT-017.** The **upstream peer compromise** attacker (`THREAT-003`) is structurally mitigated by, at minimum, the following requirements: independent DNSSEC validation by the recursive resolver role and by the forwarder role on every upstream response (`DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md)); segregated recursive and forwarder caches so that a compromised upstream cannot pollute the recursive cache (`CACHE-*` in [`004-cache-policy.md`](004-cache-policy.md)); and the encrypted outbound transport obligations cross-referenced by `NET-015`, `NET-016`, `NET-017`, `NET-022`, and `NET-023` in [`002-transports.md`](002-transports.md), which apply the TLS 1.3-only and QUIC version policies of [`003-crypto-policy.md`](003-crypto-policy.md) to Heimdall-initiated connections.
- **THREAT-018.** The **adversarial zone operator** (`THREAT-004`) is structurally mitigated by, at minimum, the following requirements: the mandatory KeyTrap cap on DNSKEY × RRSIG combinations per RRset, with `bogus` outcome on exceedance (`DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md)); the mandatory RFC 9276 cap of 150 NSEC3 iterations per query, with `insecure` outcome on exceedance (`DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md)); and the signature-algorithm and DS-digest acceptance policies that structurally reject obsolete or non-validating constructions (`DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md)).
- **THREAT-019.** The **volumetric denial-of-service and distributed denial-of-service attacker** (`THREAT-005`) is structurally mitigated, to the extent decided so far, by the following requirements: the QUIC amplification limit and address validation, including single-use `NEW_TOKEN` (`SEC-025` through `SEC-030` in [`003-crypto-policy.md`](003-crypto-policy.md)); the HTTP/2 and HTTP/3 hardening posture, covering header-block size limits, concurrent-stream caps, HPACK / QPACK dynamic-table caps, rapid-reset detection, CONTINUATION-flood caps, control-frame rate limits, header-completion timeouts, and flow-control window bounds (`SEC-036` through `SEC-046` in [`003-crypto-policy.md`](003-crypto-policy.md)); aggressive NSEC / NSEC3 use (`DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md)), which reduces the load imposed by random-subdomain and NXDOMAIN-flooding patterns on the validating roles; the rejection of QUIC 0-RTT application data (`SEC-022` through `SEC-024` in [`003-crypto-policy.md`](003-crypto-policy.md)); the multi-axis access-control-list matrix that admits traffic at the boundary of every listener (`THREAT-033` through `THREAT-047` in sub-section 2.6 of this document); the two orthogonal rate-limiting mechanisms, Response Rate Limiting (RRL) on the authoritative server role and per-client query rate limiting on the recursive resolver and forwarder roles, both active by default (`THREAT-048` through `THREAT-060` in sub-section 2.7 of this document); and the resource-limits and admission-control family, covering connection caps per `(transport, listener)`, per-connection and per-source in-flight caps, a global pending-query cap with explicit backpressure, per-cache memory caps, per-query allocation caps, slow-client disconnection, DNS-Cookie-based admission with eviction preference, transport-specific backpressure semantics, and the five-stage evaluation order that ensures a denied request does not consume budget of later stages (`THREAT-061` through `THREAT-078` in sub-section 2.8 of this document).
- **THREAT-020.** The **supply-chain attacker** (`THREAT-006`) is structurally mitigated by the requirements `THREAT-010` through `THREAT-014` defined in sub-section 2.3 of this document.

### 2.5 Operational hardening of the Heimdall process

The requirements in this sub-section fix a strict operational-hardening profile that the Heimdall binary and its reference deployment artefacts MUST implement and configure. The profile is not optional. It applies to all deployments and all roles, regardless of whether the process runs as authoritative, recursive, forwarder, or any combination of these. These requirements do not reclassify `THREAT-007` as in scope: local insider or operator compromise remains out of scope for structural mitigation by the Heimdall process. The purpose of the profile is to limit the blast radius of partial exploits of in-scope classes, consistent with the "fail-safe / fail-closed" principle in [`../CLAUDE.md`](../CLAUDE.md). Process separation per role (OpenBSD-style privsep) is explicitly out of scope at this stage; see `THREAT-032`.

- **THREAT-022.** On Linux, Heimdall MUST drop privileges after binding its listening sockets. The process MUST run as a dedicated unprivileged user for the remainder of its lifetime.
- **THREAT-023.** On Linux, the Heimdall process MUST retain the minimum set of Linux capabilities required to operate. That minimum set is `CAP_NET_BIND_SERVICE` alone, retained to allow re-binding to privileged ports across reloads. All other capabilities MUST be dropped.
- **THREAT-024.** On Linux, the Heimdall process MUST install a seccomp-bpf filter that allow-lists only the syscalls it actually uses. The structural categorisation of admitted and denied syscall classes is fixed by `THREAT-088` through `THREAT-092`; the concrete enumeration of admitted syscalls lives in the source-tree filter described by `THREAT-088`. The rejection action applied to any syscall not in the allow-list is fixed by `THREAT-094`.
- **THREAT-025.** The project MUST provide a reference systemd unit file for Linux deployments. The reference unit file MUST apply at minimum the following directives: `NoNewPrivileges=yes`, `ProtectSystem=strict`, `ProtectHome=yes`, `PrivateTmp=yes`, `PrivateDevices=yes`, `RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX`, `RestrictNamespaces=yes`, `RestrictSUIDSGID=yes`, `MemoryDenyWriteExecute=yes`, `LockPersonality=yes`, `SystemCallArchitectures=native`, `SystemCallFilter=@system-service ~@privileged ~@resources`, and an explicit minimal `ReadWritePaths=` and `ReadOnlyPaths=` appropriate to the role set.
- **THREAT-026.** On Linux, the Heimdall process MUST isolate its filesystem view to the minimum necessary directory tree. The isolation MUST be applied either through the systemd `RootDirectory=` directive, or through `chroot`, `pivot_root`, or a mount namespace.
- **THREAT-027.** The Heimdall binary MUST enforce W^X (writable or executable, never both) on all mapped regions. No mapped region may simultaneously carry write and execute permissions for the lifetime of the process.
- **THREAT-028.** On BSD and macOS, Heimdall MUST apply mechanisms equivalent to those required on Linux where the platform provides them. Where a Linux directive has no direct equivalent on the target platform, the closest equivalent available on that platform MUST be applied.
- **THREAT-029.** On OpenBSD, the Heimdall process MUST use `pledge(2)` to restrict the system-call surface and `unveil(2)` to restrict filesystem visibility to the minimum required by the active role set.
- **THREAT-030.** On macOS, a sandbox profile (App Sandbox or `sandbox-exec`) SHOULD be applied to the Heimdall process. The project MUST provide a reference macOS sandbox profile.
- **THREAT-031.** The project MUST ship, alongside the Heimdall binary, a reference systemd unit file for Linux, an OpenBSD `rc.d` script that invokes `pledge` and `unveil`, and a macOS sandbox profile. These reference artefacts MUST be kept under version control and MUST be updated in the same change that modifies the hardening profile they implement.
- **THREAT-032.** Process separation per role (OpenBSD-style privsep) is explicitly OUT OF SCOPE at this stage. The Heimdall process MUST NOT be described, documented, or configured as if its roles were isolated from each other in separate operating-system processes. Privsep MAY be reconsidered in a later iteration if the threat profile changes or if empirical incident data justifies the architectural cost.

### 2.6 Access control lists (ACLs)

The requirements in this sub-section fix the access-control surface applied at the boundary of every listener and on every outbound client path. They operationalise the "fail-safe / fail-closed" principle stated in [`../CLAUDE.md`](../CLAUDE.md) across all three roles, all four transports defined in [`002-transports.md`](002-transports.md), and every combination thereof permitted by `NET-011` in [`002-transports.md`](002-transports.md). They apply uniformly to inbound traffic and, where applicable, to outbound acceptance.

- **THREAT-033.** Heimdall MUST provide access control at the boundary of every listener and on every outbound client path through configurable access-control lists (ACLs). The ACL evaluation MUST be performed on every ingress and on every outbound acceptance as appropriate; no hot path MUST be permitted to bypass the ACL evaluation. This requirement applies to all three roles defined in [`001-server-roles.md`](001-server-roles.md) and to all four transports defined in [`002-transports.md`](002-transports.md).
- **THREAT-034.** ACL rules MUST be expressible as a combination of matchers across multiple orthogonal axes. Every ACL evaluation MUST be able to take into account any subset of the axes enumerated in `THREAT-035` through `THREAT-041` that applies to the traffic being evaluated.
- **THREAT-035.** Heimdall MUST support matching on source IP address or CIDR, with both IPv4 and IPv6 literal addresses and prefix-length notation.
- **THREAT-036.** Heimdall MUST support matching on client certificate identity when mTLS is enabled on the listener. The identity used for matching MUST be derived from the validated certificate's Subject field or from its Subject Alternative Name entries, consistently with the mTLS policy fixed by `SEC-012` through `SEC-016` and `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **THREAT-037.** Heimdall MUST support matching on TSIG key identity for authenticated operations that carry a TSIG signature, consistently with `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
- **THREAT-038.** Heimdall MUST support matching on transport. The transport axis MUST distinguish DNS over UDP on port 53, DNS over TCP on port 53, DoT, DoH over HTTP/2, DoH over HTTP/3, and DoQ, as defined in [`002-transports.md`](002-transports.md).
- **THREAT-039.** Heimdall MUST support matching on the role that would serve the query, as determined by the four-step precedence defined in `ROLE-008` through `ROLE-012` in [`001-server-roles.md`](001-server-roles.md).
- **THREAT-040.** Heimdall MUST support matching on the operation type carried by the request. The operation-type axis MUST distinguish at least standard query, AXFR, IXFR, and NOTIFY. RFC 2136 UPDATE is not included in this axis, consistent with the out-of-scope decision fixed by `PROTO-032` through `PROTO-035` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
- **THREAT-041.** Heimdall MUST support matching on QNAME or zone pattern, covering exact-name match, suffix match, and bounded wildcard forms.
- **THREAT-042.** On every listener, AXFR and IXFR requests MUST be default-deny. An AXFR or IXFR request MUST NOT be served unless an explicit ACL rule authorises that specific flow. This requirement is consistent with, and strengthens, `PROTO-045` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
- **THREAT-043.** On listeners configured for authoritative service, standard queries served by the authoritative role MUST be default-allow. An ACL rule MAY narrow this default, but no additional explicit allow is required for authoritative-role queries to be served on such listeners.
- **THREAT-044.** Standard queries that would be served by the recursive resolver role or by the forwarder role MUST be served only to clients permitted by an explicit ACL rule on the listener. No implicit default-allow MAY be applied to queries served by either of these roles.
- **THREAT-045.** On the outbound client path used when Heimdall operates as secondary for a zone, responses that do not match the configured primary MUST be default-deny. No broad acceptance of responses from unconfigured sources MAY occur on the outbound client path.
- **THREAT-046.** Views — that is, responses that differ for the same query name depending on client attributes such as source IP address or authenticated identity — MUST be implemented as ACL-driven response selection on top of the ACL matrix defined by `THREAT-034` through `THREAT-041`. Views MUST NOT be implemented as a separate subsystem. The ACL matrix MUST be the sole mechanism by which a query is routed to a specific zone dataset or to a specific forwarder destination on the basis of client attributes.
- **THREAT-047.** The ACL evaluation performed under `THREAT-033` MUST NOT materially degrade the query hot path. The implementation SHOULD compile ACL rules into an efficient representation suitable for hot-path evaluation. The concrete representation is an implementation detail and is tracked as an open question in section 5.

### 2.7 Rate limiting

The requirements in this sub-section fix two orthogonal rate-limiting mechanisms that Heimdall MUST implement and activate by default. Response Rate Limiting (RRL) under [RFC 8906](https://www.rfc-editor.org/rfc/rfc8906) targets the reflection and amplification vector on the authoritative server role. Per-client query rate limiting (query RL) targets direct abuse by individual clients against the recursive resolver role and the forwarder role, and MAY additionally be applied on the authoritative server role as a complementary layer. Both mechanisms MUST be active by default on the roles to which they apply. They apply uniformly to all four transports defined in [`002-transports.md`](002-transports.md) on the roles to which they apply. The concrete threshold values are operational parameters and are tracked as open in section 5.

- **THREAT-048.** Heimdall MUST implement Response Rate Limiting (RRL) in accordance with [RFC 8906](https://www.rfc-editor.org/rfc/rfc8906) on the authoritative server role defined in [`001-server-roles.md`](001-server-roles.md). RRL MUST be active by default on every authoritative listener and MUST apply uniformly to all four transports defined in [`002-transports.md`](002-transports.md) on that role.
- **THREAT-049.** RRL under `THREAT-048` MUST implement the "slip" behaviour specified by [RFC 8906](https://www.rfc-editor.org/rfc/rfc8906). When a client's response budget is exhausted, Heimdall MUST periodically return a truncated response with the `TC` bit set, at the configured slip ratio, so that legitimate clients whose source addresses are being spoofed by a reflection attacker remain able to recover via TCP fallback.
- **THREAT-050.** RRL under `THREAT-048` MUST be keyed by default on the tuple `(client-prefix, qname, qtype)`, where `client-prefix` is the source IP address truncated to a configured prefix length. Alternative keying dimensions are operational tuning and are tracked as open in section 5.
- **THREAT-051.** Heimdall MUST implement per-client query rate limiting (query RL) on the recursive resolver role and on the forwarder role defined in [`001-server-roles.md`](001-server-roles.md). Query RL MUST be active by default on every listener of those roles and MUST apply uniformly to all four transports defined in [`002-transports.md`](002-transports.md) on those roles. Query RL MAY be enabled on the authoritative server role as an additional layer; when enabled on the authoritative role, query RL complements RRL rather than replacing it.
- **THREAT-052.** Query RL under `THREAT-051` MUST support keying on the source IP address or CIDR, on the mTLS client-certificate identity when mTLS is enabled on the listener, on the TSIG key identity when the request carries a valid TSIG signature, and on the DNS Cookie bucket derived from the cookie state of the exchange. The mTLS-identity keying MUST be consistent with `SEC-012` through `SEC-016` and `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md) and with `THREAT-036`. The TSIG-identity keying MUST be consistent with `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md) and with `THREAT-037`. The DNS Cookie keying MUST be consistent with `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
- **THREAT-053.** Query RL under `THREAT-051` MUST honour the DNS Cookie decision fixed by `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md). A client that presents a validated DNS Cookie MAY be placed in a looser rate-limit bucket than an anonymous client; a client that presents no DNS Cookie, or an invalid DNS Cookie, MUST be placed in a stricter rate-limit bucket than a client with a validated DNS Cookie.
- **THREAT-054.** ACL evaluation under `THREAT-033` through `THREAT-047` MUST precede rate-limiting evaluation under `THREAT-048` through `THREAT-053`. Traffic denied by ACL MUST NOT consume any rate-limit budget, whether RRL or query RL. Rate-limiting evaluation MUST take place after ACL admission and before query processing.
- **THREAT-055.** The action applied when a rate-limit budget is exhausted MUST be one of: silent drop, `REFUSED` (RCODE 5) as defined in [RFC 1035, section 4.1.1](https://www.rfc-editor.org/rfc/rfc1035#section-4.1.1), a truncated response with the `TC` bit set forcing TCP fallback, or a deferred response with a bounded delay. The exact action MUST be configurable per rule. Under no configuration MAY a rate-limit overrun cause an unbounded resource commitment by Heimdall.
- **THREAT-056.** Rate limiting under `THREAT-048` through `THREAT-055` MUST NOT be exposed with a global off-switch on the roles to which each mechanism applies by default. RRL MUST be active by default on every authoritative listener under `THREAT-048`. Query RL MUST be active by default on every recursive-role and forwarder-role listener under `THREAT-051`. Operational tuning of thresholds and per-rule action selection remains available through configuration.
- **THREAT-057.** Heimdall MUST emit structured telemetry for each rate-limiting mechanism under `THREAT-048` through `THREAT-053`. The telemetry MUST report, per keying dimension, counters for admitted requests, counters for rate-limited requests, counters for slipped responses under `THREAT-049`, and counters for each of the overrun actions enumerated in `THREAT-055`. The granularity MUST be sufficient to detect an ongoing attack and to distinguish between RRL and query RL activity. The emission obligations of this requirement are subsumed by the observability surface fixed by `THREAT-079` through `THREAT-087` in sub-section 2.9; the exact schema of the resulting structured events is tracked as open in section 5.
- **THREAT-058.** Static configured thresholds are the initial calibration model for rate limiting under `THREAT-048` through `THREAT-055`. Adaptive or closed-loop threshold adjustment driven by live telemetry is deliberately deferred at this stage and MUST NOT be implemented on the default path. A future revision of this document MAY reconsider adaptive thresholds as an opt-in extension once the observability surface required by `THREAT-057` is mature and stability guards have been specified.
- **THREAT-059.** Delegation of rate limiting to an L4 firewall or to an external DDoS-protection front-end MUST NOT be treated as a substitute for the mechanisms required by `THREAT-048` through `THREAT-057`. Such external layers operate below the DNS semantics and cannot reason about the signals that drive the decisions fixed by `THREAT-048` through `THREAT-053` — including repeated queries for the same name versus diverse queries, cache-hit ratio, DNSSEC validation cost, and DNS Cookie validity. External layers MAY complement Heimdall's rate limiting but MUST NOT replace it.
- **THREAT-060.** The rate-limiting requirements in `THREAT-048` through `THREAT-059` cover inbound traffic on all three roles defined in [`001-server-roles.md`](001-server-roles.md), on all four transports defined in [`002-transports.md`](002-transports.md). They do not govern Heimdall-initiated outbound traffic, which is governed by the transport-level and protocol-conformance requirements cross-referenced by `NET-015`, `NET-016`, `NET-017`, `NET-022`, and `NET-023` in [`002-transports.md`](002-transports.md).

### 2.8 Resource limits and admission control

The requirements in this sub-section fix a complete set of application-layer resource limits and admission controls. These mechanisms are orthogonal to access control (sub-section 2.6) and to rate limiting (sub-section 2.7), and MUST be enforced independently of them. They apply to inbound traffic on all three roles defined in [`001-server-roles.md`](001-server-roles.md) and on all four transports defined in [`002-transports.md`](002-transports.md). They complement but do not replace operating-system-level limits such as `ulimit`, cgroups, or systemd resource directives. The concrete numeric values of each limit are operational parameters and are tracked as open in section 5.

- **THREAT-061.** Heimdall MUST enforce the resource-limits and admission-control family fixed by `THREAT-062` through `THREAT-078`. Every limit in this family MUST be active by default with a conservative value, on every role and on every transport to which it applies. Operators MAY tune the numeric values of individual limits through configuration to match a specific deployment profile, but MUST NOT disable any limit in this family entirely. There is no global off-switch for the enforcement family.
- **THREAT-062.** Heimdall MUST enforce a maximum concurrent-connection limit per `(transport, listener)` tuple on every connection-oriented transport. The limit MUST be independently configurable per tuple. When the limit is reached on a given `(transport, listener)`, new connection attempts on that tuple MUST be refused in accordance with the transport-specific backpressure semantics fixed by `THREAT-071` through `THREAT-075`, without affecting traffic on other `(transport, listener)` tuples.
- **THREAT-063.** Heimdall MUST enforce a maximum in-flight-queries-per-connection limit on every connection-oriented transport. The limit MUST cap the number of queries a single connection may have simultaneously in a non-terminal state (accepted, being parsed, being resolved, awaiting response). When the per-connection limit is reached, further queries on that connection MUST be rejected in accordance with the transport-specific backpressure semantics fixed by `THREAT-071` through `THREAT-075`.
- **THREAT-064.** Heimdall MUST enforce a maximum in-flight-queries limit keyed on source IP address, or on authenticated identity when mTLS is enabled under `SEC-012` through `SEC-016` and `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md) or when a valid TSIG signature is present under `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md). The limit MUST cap the aggregate number of queries a given source may have simultaneously in a non-terminal state across all of its connections. When the per-source limit is reached, further queries from that source MUST be rejected in accordance with the transport-specific backpressure semantics fixed by `THREAT-071` through `THREAT-075`.
- **THREAT-065.** Heimdall MUST enforce a maximum global pending-queries limit, server-wide, that caps the aggregate number of queries simultaneously in a non-terminal state across all roles, all transports, all listeners, and all sources. When the global limit is reached, Heimdall MUST apply explicit backpressure to new queries in accordance with the transport-specific semantics fixed by `THREAT-071` through `THREAT-075` and MUST NOT queue new queries unboundedly above the limit.
- **THREAT-066.** Heimdall MUST enforce a maximum memory budget per cache instance, applied consistently with the segregated-cache model fixed by `CACHE-001` through `CACHE-007` in [`004-cache-policy.md`](004-cache-policy.md). The recursive resolver cache and the forwarder cache MUST each have their own independent memory budget; no shared budget across the two caches MUST be applied. When a cache instance reaches its memory budget, admission of new entries into that cache MUST be governed by the per-cache eviction policy that is an open question in section 5, without cross-cache spill.
- **THREAT-067.** Heimdall MUST enforce a maximum allocation size per query for each of the following buffers: the parse buffer used to accept the inbound message, the response buffer used to compose the outbound message, and the decompression buffer used to expand compressed header blocks on DoH over HTTP/2 and on DoH over HTTP/3. Any allocation request that would exceed the configured cap MUST cause the query to be rejected without being processed further. On DoH over HTTP/2 and DoH over HTTP/3, this requirement reinforces, and MUST be enforced in addition to, the HTTP-layer bounds fixed by `SEC-039`, `SEC-040`, and `SEC-045` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **THREAT-068.** Heimdall MUST detect and disconnect slow clients on every connection-oriented transport. Detection MUST cover idle timeouts on established connections, stall detection on TCP flows that stop making forward progress, and cleanup of half-open connections whose handshake does not complete within a configured window. The concrete timeout values are operational parameters and are tracked as open in section 5. Disconnection MUST follow the transport-specific backpressure semantics fixed by `THREAT-071` through `THREAT-075`.
- **THREAT-069.** When Heimdall is under load, it MUST use DNS Cookies as specified by `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md) as an admission-control signal. Under load, Heimdall MUST prefer admission of requests that present a validated DNS Cookie over requests that do not, and MUST preferentially drop or reject requests without a valid cookie before dropping or rejecting requests with a valid cookie. The signal that classifies Heimdall as "under load" is an open question in section 5.
- **THREAT-070.** When Heimdall evicts in-flight work or cached state under the saturation of any of the limits fixed by `THREAT-062` through `THREAT-068`, it MUST preferentially preserve authenticated traffic — under mTLS as fixed by `SEC-012` through `SEC-016` and `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md), under TSIG as fixed by `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md), or bearing a validated DNS Cookie as fixed by `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md) — over anonymous traffic. The concrete eviction strategy within each limit is an open question in section 5.
- **THREAT-071.** On DNS classic over UDP on port 53, the backpressure behaviour at a saturation boundary fixed by `THREAT-062` through `THREAT-068` MUST be a silent drop of the incoming datagram at the boundary. Heimdall MUST NOT emit any DNS response under these saturation conditions on this transport.
- **THREAT-072.** On DNS classic over TCP on port 53 and on DNS-over-TLS on port 853, the backpressure behaviour at a saturation boundary fixed by `THREAT-062` through `THREAT-068` MUST be a connection close. Heimdall MUST emit a FIN on the affected connection when the close is clean, and MUST emit a RST on the affected connection when the close cannot proceed cleanly. No further queries on the closed connection MUST be processed.
- **THREAT-073.** On DNS-over-HTTPS over HTTP/2 and on DNS-over-HTTPS over HTTP/3, the backpressure behaviour at a stream-level saturation boundary fixed by `THREAT-062` through `THREAT-068` MUST be a stream reset with an error code appropriate to the HTTP version. The connection MUST remain usable for unaffected streams unless the saturation boundary is reached at the connection level, in which case the connection MUST be closed.
- **THREAT-074.** On DNS-over-QUIC on port 853, the backpressure behaviour at a stream-level saturation boundary fixed by `THREAT-062` through `THREAT-068` MUST be a stream reset. The connection MUST remain usable for unaffected streams unless the saturation boundary is reached at the connection level, in which case the connection MUST be closed in accordance with the QUIC connection-close mechanism.
- **THREAT-075.** Where a truncated response with the `TC` bit set is applicable in DNS terms under [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), Heimdall MAY emit such a response as a backpressure mechanism to force TCP fallback by the client, in lieu of the silent drop fixed by `THREAT-071` or of the stream reset fixed by `THREAT-073` and `THREAT-074`. This option MUST NOT relax the transport-specific semantics fixed by `THREAT-071` through `THREAT-074`; it is an alternative that an operator MAY select where the protocol admits it.
- **THREAT-076.** The composite admission pipeline applied on every inbound request MUST evaluate, in the following order: (1) ACL check as fixed by `THREAT-033` through `THREAT-047`; (2) connection-level admission as fixed by `THREAT-062`, `THREAT-063`, `THREAT-064`, `THREAT-065`, and `THREAT-068`; (3) DNS-Cookie-based admission as fixed by `THREAT-069` and `THREAT-070`; (4) rate limiting as fixed by `THREAT-048` through `THREAT-060`; (5) query processing. A request denied at an earlier stage MUST NOT consume the budget of any later stage.
- **THREAT-077.** Each limit fixed by `THREAT-062` through `THREAT-068` and each admission mechanism fixed by `THREAT-069` and `THREAT-070` MUST emit structured telemetry. The telemetry MUST include counters, and where appropriate histograms, sufficient to detect approach-to-limit conditions, to detect limit fires when they occur, and to observe eviction events driven by the eviction preference fixed by `THREAT-070`. The granularity MUST allow operators to distinguish between the individual limits and between the admission mechanisms. The emission obligations of this requirement are subsumed by the observability surface fixed by `THREAT-079` through `THREAT-087` in sub-section 2.9; the exact schema of the resulting structured events is tracked as open in section 5.
- **THREAT-078.** The enforcement family fixed by `THREAT-061` through `THREAT-077` MUST NOT be exposed with a global off-switch. Individual limits MAY be tuned, and the DNS-Cookie-based admission preference fixed by `THREAT-069` is active only under load, but no configuration option MUST permit the family as a whole, or any individual limit within it, to be disabled entirely.

### 2.9 Observability for security detection

The requirements in this sub-section fix the mandatory observability surface targeted at security: structured events for anomalous conditions and metrics for trend and alerting signal. They apply uniformly to all three roles defined in [`001-server-roles.md`](001-server-roles.md), to all four transports defined in [`002-transports.md`](002-transports.md), and to every deployment mode supported by Heimdall. They complement, and do not replace, the per-mechanism telemetry already required by `THREAT-057` for rate limiting and by `THREAT-077` for the resource-limits and admission-control family; this sub-section extends that obligation uniformly across the other mitigations in the specification.

- **THREAT-079.** Heimdall MUST emit structured events and MUST expose metrics sufficient for operators to detect attacks in progress and to perform post-incident forensic analysis. Per-query distributed tracing is explicitly opt-in at this stage and MUST NOT be mandated by default.
- **THREAT-080.** Heimdall MUST emit a structured event for each of the following conditions, at minimum: DNSSEC `bogus` validation outcomes, with zone, upstream, and algorithm context, consistent with `DNSSEC-012` and `DNSSEC-042` in [`005-dnssec-policy.md`](005-dnssec-policy.md); KeyTrap mitigation fires, consistent with `DNSSEC-041` in [`005-dnssec-policy.md`](005-dnssec-policy.md); NSEC3 iteration cap fires, consistent with `DNSSEC-044` in [`005-dnssec-policy.md`](005-dnssec-policy.md); rapid-reset detection fires, consistent with `SEC-041` in [`003-crypto-policy.md`](003-crypto-policy.md); HPACK decompression-bomb attempts, consistent with `SEC-039` in [`003-crypto-policy.md`](003-crypto-policy.md); QPACK decompression-bomb attempts, consistent with `SEC-040` in [`003-crypto-policy.md`](003-crypto-policy.md); CONTINUATION-flood detection fires, consistent with `SEC-042` in [`003-crypto-policy.md`](003-crypto-policy.md); 0x20 case mismatches, classified as adversarial versus non-adversarial, consistent with `PROTO-027` and `PROTO-028` in [`006-protocol-conformance.md`](006-protocol-conformance.md); QUIC `NEW_TOKEN` replay attempts, consistent with `SEC-028` in [`003-crypto-policy.md`](003-crypto-policy.md); TLS and certificate validation failures on inbound and outbound paths; DNS Cookie validation failures, consistent with `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md); ACL denies per rule, consistent with `THREAT-033` through `THREAT-047`; rate-limit fires per mechanism and per keying dimension, consistent with `THREAT-057`; resource-limit fires, including the connection cap, the memory cap, and the cookie-admission transition, consistent with `THREAT-061` through `THREAT-078`; Negative Trust Anchor registration, expiry, and manual revocation, consistent with `DNSSEC-017` in [`005-dnssec-policy.md`](005-dnssec-policy.md); and QNAME minimisation fallback events when the recursive resolver is in relaxed mode, consistent with `PROTO-024` in [`006-protocol-conformance.md`](006-protocol-conformance.md). The list is non-exhaustive; each additional mitigation introduced by a subsequent clarification MUST extend this catalogue in the same change.
- **THREAT-081.** Heimdall MUST expose metrics for at least the following dimensions: queries per second per `(role, transport, qtype)`; cache hit rate per role; DNSSEC validation outcome distribution across the values `secure`, `insecure`, `bogus`, and `indeterminate`; TLS and QUIC handshake success rate; counters per rate-limit mechanism and per resource-limit mechanism, consistent with `THREAT-057` and `THREAT-077`; ongoing-connections count per transport; and memory usage per cache, consistent with the segregated-cache model fixed by `CACHE-001` through `CACHE-007` in [`004-cache-policy.md`](004-cache-policy.md) and with the per-cache memory budget fixed by `THREAT-066`.
- **THREAT-082.** Structured events under `THREAT-080` MUST be emitted in a JSON-based format suitable for ingestion by common log pipelines, including but not limited to Loki, ELK, Splunk, and Datadog. Field naming SHOULD follow the OpenTelemetry semantic conventions where an applicable convention exists.
- **THREAT-083.** Metrics under `THREAT-081` MUST be exposable in the OpenMetrics format, compatible with Prometheus scraping.
- **THREAT-084.** When opt-in per-query distributed tracing is enabled under `THREAT-079`, the trace data MUST follow the OpenTelemetry specifications.
- **THREAT-085.** Structured events under `THREAT-080` MUST be emitted only on anomalous conditions. The normal-path query flow MUST NOT incur structured-event emission cost. Per-query logging on the normal path is not required by this sub-section and MUST NOT be enabled by default.
- **THREAT-086.** Metrics updates on the normal-path query flow MUST be performed with lock-free counters or with per-thread counters aggregated outside the hot path. No metrics-update mechanism applied on the normal path MUST introduce contention among worker threads, consistent with the performance principle fixed by [`../CLAUDE.md`](../CLAUDE.md).
- **THREAT-087.** The observability surface fixed by `THREAT-079` through `THREAT-086` MUST NOT be exposed with a global off-switch. Individual event categories and individual metrics MAY be gated by verbosity-level configuration, which is tracked as an open question in section 5, but the surface as a whole MUST remain available on every deployment so that operators can distinguish quiescence from a running attack already mitigated by the defaults.
- **THREAT-088.** The seccomp-bpf filter required by `THREAT-024` MUST be implemented as a per-architecture filter source maintained in the Heimdall source tree at the relative path `crates/heimdall-runtime/src/security/seccomp/`, with one source file per supported Linux architecture (`linux_x86_64`, `linux_aarch64`, `linux_riscv64`, in conformance with the architecture matrix fixed by `ENV-006` through `ENV-008` in [`009-target-environment.md`](009-target-environment.md)). The source files MUST emit BPF programs at process start through a maintained library binding (`libseccomp` or an equivalent). The structural categorisation of admitted and denied syscall classes is fixed normatively by `THREAT-089` and `THREAT-090`; the concrete enumeration of the syscalls under each category is implementation content (not normative spec content) and lives in the source-tree filter, in line with the spec-versus-implementation discipline of `ENG-125` in [`010-engineering-policies.md`](010-engineering-policies.md).
- **THREAT-089.** The seccomp-bpf filter required by `THREAT-024` MUST admit syscalls only from the following structural categories. Each category MUST be admitted at the minimum granularity required by the runtime libraries Heimdall depends on (Tokio, rustls, quinn, hyper, the Redis client, and the `io_uring` or `epoll` I/O backend per `ENV-002`):
  - **Network I/O for inbound and outbound communication.** Socket creation, bind, listen, accept, connect, send, receive, shutdown, getsockname, getsockopt, setsockopt, read, write, readv, writev, close, fcntl, pipe2, dup2, dup3, eventfd2.
  - **`io_uring` primary I/O and `epoll` fallback.** `io_uring_setup`, `io_uring_enter`, `io_uring_register` for the primary path; `epoll_create1`, `epoll_ctl`, `epoll_wait`, `epoll_pwait` for the fallback path defined in `ENV-002`. Only one of the two paths is admitted per running configuration; the unused path's syscalls MUST be denied at runtime.
  - **Path-restricted file I/O.** `openat`, `read`, `write`, `pread64`, `pwrite64`, `fstat`, `fstatat`, `getdents64`, `lseek`, `close`. Path-based filtering MUST be applied in user space against the explicit allow-list of paths fixed by the per-listener configuration and by the systemd `ReadWritePaths=` / `ReadOnlyPaths=` directives of `THREAT-025`; seccomp-bpf cannot filter paths directly.
  - **Memory management.** `mmap`, `munmap`, `mremap`, `madvise`, `brk`, `mprotect` constrained against `PROT_EXEC` toggling on previously non-executable regions (W^X enforcement of `THREAT-027`).
  - **Time and clock.** `clock_gettime`, `clock_nanosleep`, `nanosleep`, `gettimeofday`. Clock-mutating syscalls (`clock_settime`, `settimeofday`, `adjtimex`, `clock_adjtime`) MUST NOT be admitted.
  - **Synchronisation.** `futex` (including TIME64 variants), `set_robust_list`.
  - **Signals.** `rt_sigaction`, `rt_sigprocmask`, `rt_sigreturn`, `sigaltstack`.
  - **Threading (Tokio worker pool).** `clone3` and `clone` constrained to `CLONE_THREAD` semantics only (no `CLONE_FORK`); `gettid`, `tgkill`, `set_tid_address`; `prctl` constrained to `PR_SET_NAME`.
  - **Process information (read-only).** `getpid`, `getuid`, `geteuid`, `getgid`, `getegid`, `getresuid`, `getresgid`.
  - **Privilege drop and capability management (boot-time only).** `setuid`, `setgid`, `setgroups`, `prctl` constrained to `PR_SET_NO_NEW_PRIVS`, `PR_CAP_AMBIENT`, and `PR_GET_KEEPCAPS`; `capset`, `capget`. The filter MUST distinguish boot-time from steady-state and MUST deny these syscalls after privilege drop has completed (a second seccomp filter MAY be installed post-drop to enforce this; alternatively a one-shot use of `prctl(PR_SET_SECCOMP, ...)` after drop applies the steady-state filter).
  - **Random.** `getrandom`.
  - **Process lifecycle (terminal only).** `arch_prctl`, `exit_group`, `restart_syscall`.
- **THREAT-090.** The seccomp-bpf filter required by `THREAT-024` MUST deny syscalls in the following structural categories. The denial MUST be the rejection action fixed by the open question in section 5, applied uniformly to every category enumerated below.
  - **Process creation.** `fork`, `vfork`, `clone` with `CLONE_FORK` flag (i.e., not `CLONE_THREAD`), `execve`, `execveat`. Heimdall is a single-process application post-boot and MUST NOT spawn child processes.
  - **Kernel module operations.** `init_module`, `delete_module`, `finit_module`, `query_module`.
  - **Privileged kernel operations.** `kexec_load`, `kexec_file_load`, `bpf` (denied except for the narrow subset that `io_uring` may require, which MUST be specifically allow-listed by syscall argument when admitted), `unshare`, `setns`, `pivot_root`, `chroot` (admitted only during boot if used for filesystem isolation under `THREAT-026`).
  - **Privileged network device manipulation.** `socket(AF_PACKET, ...)`, `socket(AF_NETLINK, ...)` (except for the narrow subset that `io_uring` control plane may require), `iopl`, `ioperm`.
  - **Anti-debugging.** `ptrace`, `process_vm_readv`, `process_vm_writev`. Heimdall MUST NOT be debuggable from inside its own process post-drop.
  - **Memory-execution toggling.** `mprotect` MUST NOT be admitted for transitions from non-`PROT_EXEC` to `PROT_EXEC` on a previously-mapped region; this is the seccomp-layer reinforcement of `THREAT-027` (W^X).
  - **Privilege re-acquisition (post-drop).** `setuid`, `setgid`, `setgroups`, `setresuid`, `setresgid`, `capset` (admitted only during boot under `THREAT-089`'s privilege-drop category).
- **THREAT-091.** The seccomp-bpf filter required by `THREAT-024` MUST be a single allow-list applied uniformly across all role combinations supported by `ROLE-001` through `ROLE-007` in [`001-server-roles.md`](001-server-roles.md). The marginal attack-surface reduction of per-role variants (an authoritative-only deployment never initiating outbound TLS, a forwarder-only deployment never serving zone files) is not pursued at this stage; the implementation simplicity of a single filter is preferred. Per-role variants MAY be reconsidered in a future revision if measurement justifies the architectural complexity.
- **THREAT-092.** The seccomp-bpf filter required by `THREAT-024` MUST be aware of the running kernel version at process start and MUST select the per-architecture filter source under `THREAT-088` accordingly. Syscalls introduced in kernels newer than the minimum supported version (`ENV-001`) MUST NOT be admitted unless they are required by the implementation and the runtime check confirms availability; absent syscalls in older kernels MUST cause the filter installation to fall back to the corresponding alternative path (typically `epoll` instead of `io_uring`). Cross-architecture syscall-number portability MUST be handled by per-architecture filter source files; filter binaries are NOT portable across architectures.
- **THREAT-093.** The seccomp-bpf filter required by `THREAT-024` MUST be validated by continuous-integration testing in three tiers, in line with the testing pipeline fixed by `ENG-049` through `ENG-079` in [`010-engineering-policies.md`](010-engineering-policies.md). **Tier 1** (per-pull-request fast gate) MUST validate the filter source files for syntactic correctness — every emitted BPF program MUST compile and load on a target kernel-version mock. **Tier 2** (per-pull-request slow gate) MUST run Heimdall under the active filter against a synthetic workload exercising every active role (authoritative, recursive, forwarder), every transport (DNS classic UDP/TCP, DoT, DoH over HTTP/2 and HTTP/3, DoQ), and every privilege-drop transition; the run MUST complete without any syscall being denied during legitimate operation. **Tier 3** (nightly) MUST run Heimdall under a `SECCOMP_RET_LOG` variant of the filter (logs denied syscalls without killing the process) against a longer-duration realistic workload; any newly-introduced syscall (caused by a library upgrade, a code change, or a kernel-feature reliance) MUST appear in the Tier 3 log and MUST trigger a CI alert. The drift-detection property of Tier 3 is the operational mechanism by which library upgrades cannot silently introduce new syscalls without triggering the spec maintenance cycle.
- **THREAT-094.** The rejection action applied by the seccomp-bpf filter required by `THREAT-024` to any syscall not admitted under `THREAT-089` MUST be `SECCOMP_RET_KILL_PROCESS`. The denied syscall MUST cause the entire Heimdall process to be terminated immediately by the kernel with signal `SIGSYS` (signal number 31 on every supported architecture); no further syscalls MUST be reachable from the terminated process; no application-level recovery from a denied syscall MUST be admissible. The rationale is the fail-safe / fail-closed principle of [`../CLAUDE.md`](../CLAUDE.md): a denied syscall implies either (a) an unanticipated code path in a Heimdall library dependency (which is a defect to be investigated, not silently handled), or (b) an attempted post-exploitation primitive after a memory-safety bug or protocol-level compromise (which must be terminated before any blast-radius expansion is possible). `SECCOMP_RET_KILL_THREAD` is rejected because it leaves the Tokio worker pool in a partially-degraded state; `SECCOMP_RET_ERRNO(EPERM)` is rejected because it allows the application to continue past the violation, which both masks the underlying defect and admits the possibility of an adversary inducing repeated violations to probe the syscall surface; `SECCOMP_RET_TRAP` is rejected because the userspace signal handler runs in the security-critical path and adds complexity for an end-effect equivalent to `SECCOMP_RET_KILL_PROCESS`. The `SECCOMP_RET_LOG` mode is reserved for the Tier 3 drift-detection workload required by `THREAT-093` and MUST NOT be used in production.
- **THREAT-095.** The observability expectations on a `SECCOMP_RET_KILL_PROCESS` event are: (a) the systemd journal (or the analogous platform journal) MUST record the process termination by SIGSYS with exit code 159 (128 + 31, the standard convention for kernel-signalled termination); (b) the kernel audit subsystem (`auditd`, with `audit=1` enabled at boot or with the corresponding runtime configuration) SHOULD record the offending syscall number, the offending architecture, and the program counter at the time of the violation; (c) the systemd reference unit file delivered under `THREAT-031` MUST configure `Restart=on-failure` (or `Restart=on-abnormal`) so that the systemd-supervised Heimdall instance is automatically restarted after a SIGSYS termination, preserving service availability while ensuring that the violation is not silently recovered from; (d) the systemd reference unit file SHOULD additionally configure `OnFailure=` to a maintainer-specified alerting service so that operators are notified of the termination; (e) Heimdall MUST NOT attempt to emit a structured event under `THREAT-080` from inside the SIGSYS path, because the structured-event emission would itself require syscalls that the filter is in the process of denying; the post-mortem observability is delivered exclusively through the kernel-side mechanisms in (a) and (b) and through the systemd-side mechanism in (c) and (d).
- **THREAT-096.** The reference systemd unit file mandated by `THREAT-025` and `THREAT-031` MUST be delivered under version control at the relative path `contrib/systemd/heimdall.service`, accompanied by per-role drop-in snippets at `contrib/systemd/heimdall.service.d/`. The drop-in pattern MUST follow the standard systemd convention for service overrides: an operator deploys the base unit file plus the drop-in snippets corresponding to the active role combination (an authoritative-only deployment uses `01-authoritative.conf`; a forwarder + recursive hybrid uses `01-forwarder.conf` plus `02-recursive.conf`). The base unit file MUST contain the directives common to every role; the drop-in snippets MUST contain the role-specific `ReadOnlyPaths=`, `ReadWritePaths=`, and any role-specific resource limits.
- **THREAT-097.** The reference systemd unit file under `THREAT-096` MUST apply, at minimum, the following operationally-tuned directives in addition to the hardening directives already fixed by `THREAT-025`: `User=heimdall`, `Group=heimdall` (dedicated unprivileged user, conventional for the distribution); `RuntimeDirectory=heimdall` and `RuntimeDirectoryMode=0750` (managing `/run/heimdall/` lifecycle for the admin-RPC Unix domain socket and the PID artefacts); `StateDirectory=heimdall` and `StateDirectoryMode=0750` where on-disk state is required; `Restart=on-failure` (consistent with `THREAT-095`); `RestartSec=2` (minimum wait before restart, to avoid tight restart loops on persistent failure); `StartLimitBurst=5` and `StartLimitIntervalSec=30` (cap restart attempts within a window); `AmbientCapabilities=CAP_NET_BIND_SERVICE` and `CapabilityBoundingSet=CAP_NET_BIND_SERVICE` (consistent with `THREAT-023`); `LimitNOFILE=65536` and `LimitNPROC=512` (sized for the connection pools of `NET-048` through `NET-061` and the Tokio worker pool); `OnFailure=` left as a commented placeholder for operator-supplied alerting integration.
- **THREAT-098.** The default path mapping applied by the reference systemd unit file under `THREAT-096` MUST be: configuration directory `/etc/heimdall/` mounted read-only; TLS certificate directory `/etc/heimdall/tls/` mounted read-only when DoT, DoH, or DoQ listeners are enabled (drop-in pattern); zone-files directory `/var/lib/heimdall/zones/` mounted read-only by default and read-write only when an operator opts into in-process zone editing via the admin-RPC; Redis Unix domain socket `/run/heimdall/redis.sock` mounted read-write under `RuntimeDirectory=`; runtime state directory `/run/heimdall/` mounted read-write under `RuntimeDirectory=`; trust-anchor file `/var/lib/heimdall/trust-anchors.xml` mounted read-write only on the recursive resolver role (for [RFC 5011](https://www.rfc-editor.org/rfc/rfc5011) rollover writes), read-only on the authoritative-only or forwarder-only roles. Operators MAY override every default path through systemd drop-in snippets following the standard `[Service]` directive override semantics.
- **THREAT-099.** The reference systemd unit file under `THREAT-096` MUST be validated by continuous-integration testing in two tiers, in line with the testing pipeline of [`010-engineering-policies.md`](010-engineering-policies.md). **Tier 1** (per-pull-request fast gate) MUST run `systemd-analyze verify` against the base unit file and against every role drop-in snippet; the verification MUST pass without warnings. **Tier 1** MUST additionally run `systemd-analyze security` against the unit file with the snippets applied; the resulting exposure score MUST be `1.5` or lower (the systemd-analyze "highly hardened" band); a higher score MUST cause the gate to fail. The verification and the score-gate together prevent drift in the hardening posture: a removed directive or a relaxed `ReadWritePaths=` raises the exposure score and trips the gate. Drift detection on systemd directive support across distribution-supplied systemd versions is the operator's responsibility; the project's reference unit file targets the systemd version shipped with the distributions enumerated by `ENV-018` through `ENV-021` in [`009-target-environment.md`](009-target-environment.md).
- **THREAT-100.** On OpenBSD, the Heimdall process MUST apply [`pledge(2)`](https://man.openbsd.org/pledge.2) in lifecycle stages, narrowing the system-call surface as the process transitions from boot to steady state. The stages and their permitted promises are: **(a) Boot stage** (the process running with privileges sufficient to bind privileged ports and to drop to the unprivileged user), promises `stdio rpath wpath cpath inet unix proc id`. **(b) Pre-bind stage** (after configuration has been loaded and the unprivileged-user privilege drop has happened), promises `stdio rpath wpath cpath inet unix`. **(c) Steady-state stage** (post-bind, post-drop, serving traffic), promises fixed by `THREAT-101` per active role combination. The transitions between stages MUST be one-directional (each `pledge()` call narrows the surface; promises cannot be re-acquired). The promises `dns`, `exec`, `prot_exec`, `id` (post-drop), `proc` (post-boot), `tty`, `audio`, `pf`, `chown` MUST NOT appear in any stage's promise set: Heimdall does not use the OpenBSD system DNS resolver (it uses raw `inet` sockets directly), is single-process post-boot, enforces W^X under `THREAT-027`, must not re-acquire privileges after dropping, and does not interact with terminals, audio devices, the PF firewall, or filesystem ownership.
- **THREAT-101.** The steady-state pledge promise set under `THREAT-100` (c) per role combination MUST be: **(authoritative-only) and (forwarder-only) and (authoritative + forwarder)**: `stdio inet unix rpath`. **(recursive-only) and (any combination including the recursive role)**: `stdio inet unix rpath wpath cpath`, where the additional `wpath cpath` promises are required for [RFC 5011](https://www.rfc-editor.org/rfc/rfc5011) trust-anchor rollover to write the updated trust-anchor file. No other role combination MUST require a different promise set; the four combinations of {authoritative, recursive, forwarder} produce two pledge sets at steady state, distinguished only by whether the recursive role is active.
- **THREAT-102.** On OpenBSD, the Heimdall process MUST apply [`unveil(2)`](https://man.openbsd.org/unveil.2) at boot to restrict the filesystem-visible surface to the minimum required by the active role combination. The default unveil set MUST be:
  - `/etc/heimdall/`: mode `r` (read), all roles.
  - `/etc/heimdall/tls/`: mode `r`, when DoT, DoH, or DoQ listeners are enabled.
  - `/etc/ssl/cert.pem`: mode `r`, on the forwarder role (the OpenBSD system trust bundle is a fallback when no per-upstream `trust_anchor` is configured under `SEC-047`).
  - `/var/heimdall/zones/`: mode `r`, on the authoritative role.
  - `/var/heimdall/trust-anchors.xml`: mode `rwc` (read, write, create), on the recursive role only; mode `r` elsewhere.
  - `/var/run/heimdall/`: mode `rwc`, all roles (admin-RPC Unix domain socket, Redis Unix domain socket, PID file).
  After the last unveil call, Heimdall MUST call `unveil(NULL, NULL)` to lock the unveil set. Subsequent attempts to access paths outside the unveiled set MUST fail with `ENOENT` regardless of whether the path exists; this is the OpenBSD-native equivalent of the systemd `ProtectSystem=strict` posture.

- **THREAT-108.** ACL rules required by `THREAT-033` through `THREAT-047` MUST be declared in TOML in conformance with `ROLE-016` in [`001-server-roles.md`](001-server-roles.md). Each `[[<role>.listener.acl]]` array entry declares one ACL rule, scoped to the listener under which it appears. A listener that declares no `[[<role>.listener.acl]]` array MUST inherit the global default action declared in the top-level `[acl]` table via the key `acl.default_action` (string enum `"allow"` or `"deny"`); when the global default is also absent, the per-class defaults of `THREAT-042` through `THREAT-045` apply (default-deny for AXFR / IXFR; default-deny for recursive-role and forwarder-role queries; default-allow for authoritative-role queries on listeners configured for authoritative service; default-deny for outbound acceptance of responses not matching the configured primary).
- **THREAT-109.** Each `[[<role>.listener.acl]]` rule entry MUST carry the string key `action`, whose value is exactly one of `"allow"` or `"deny"`. The entry MAY carry any subset of the matcher fields enumerated by `THREAT-110`; a rule with no matcher fields matches every request reaching the listener. The configuration loader MUST reject any rule carrying an unknown key, per `ROLE-021` in [`001-server-roles.md`](001-server-roles.md).
- **THREAT-110.** The matcher fields admissible in an ACL rule MUST be exactly the seven listed below, each corresponding to one axis of the ACL matrix fixed by `THREAT-035` through `THREAT-041`:
  - `source_cidr` (string): IPv4 or IPv6 address with prefix length, in conformance with `THREAT-035`.
  - `mtls_identity` (string): exact-match string against the identity derived under `SEC-067` in [`003-crypto-policy.md`](003-crypto-policy.md), in conformance with `THREAT-036`.
  - `tsig_key_name` (string): exact-match TSIG key identity, in conformance with `THREAT-037` and `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
  - `transport` (string enum): one of `"udp"`, `"tcp"`, `"dot"`, `"doh"`, `"doq"`, in conformance with `THREAT-038` and the listener-transport enum of `ROLE-020` in [`001-server-roles.md`](001-server-roles.md).
  - `role` (string enum): one of `"authoritative"`, `"recursive"`, `"forwarder"`, in conformance with `THREAT-039`.
  - `operation` (string enum): one of `"query"`, `"axfr"`, `"ixfr"`, `"notify"`, in conformance with `THREAT-040`.
  - `qname_pattern` (string): a DNS name in either exact, suffix (`*.example.com.`), or wildcard (RFC 1034 single-label) form, with the syntax fixed by `FWD-003` through `FWD-008` in [`014-forward-zones.md`](014-forward-zones.md), in conformance with `THREAT-041`.
- **THREAT-111.** Any matcher field's value MAY be prefixed with the negation character `!` to invert the match. A rule with `source_cidr = "!10.0.0.0/8"` matches every source address EXCEPT those in `10.0.0.0/8`. Negation applies to the matcher field individually; it does not invert the rule as a whole. The `action` field of `THREAT-109` MUST NOT carry the negation prefix.
- **THREAT-112.** ACL rule combination semantics MUST be: **(a)** within a single rule, all declared matcher fields are combined with logical AND — every field MUST match for the rule to apply; **(b)** across rules within the same listener's `[[<role>.listener.acl]]` array, evaluation MUST be in declaration order, and the first rule whose matcher set evaluates to TRUE for the request MUST determine the outcome (first-match-wins); **(c)** when no rule on the listener matches, the listener-level default action of `THREAT-108` MUST apply, falling back through the global `acl.default_action` and finally through the per-class defaults of `THREAT-042` through `THREAT-045`. There MUST NOT be an explicit numeric `priority` field; rule ordering is by declaration order alone, in line with the operator-intent-as-array discipline already established by `FWD-025` in [`014-forward-zones.md`](014-forward-zones.md).
- **THREAT-113.** The configuration loader MUST validate every ACL rule at load time. The validation MUST verify: `action` is `"allow"` or `"deny"`; `source_cidr`, when present, parses as a valid IPv4 or IPv6 address-with-prefix; `qname_pattern`, when present, conforms to the syntax of `FWD-003` through `FWD-008`; `transport`, `role`, and `operation`, when present, lie within their respective enumerations; every matcher field name is admissible under `THREAT-110`. A configuration containing any rule that fails any of these validations MUST be rejected at load and the instance MUST refuse to start.
- **THREAT-114.** The action applied by Heimdall to a request denied by the ACL evaluation under `THREAT-033` through `THREAT-047` MUST take the following defaults, with optional per-rule override under `THREAT-115`. **(a) DNS-layer ACL deny** (a request matched a rule with `action = "deny"`): MUST respond with RCODE `REFUSED` (RCODE 5) per [RFC 1035 §4.1.1](https://www.rfc-editor.org/rfc/rfc1035#section-4.1.1), accompanied by an EDNS(0) Extended DNS Errors option carrying INFO-CODE `18` ("Prohibited") per [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914). The choice of `EDE 18` (rather than `EDE 20` "Not Authoritative" used by `ROLE-024` in [`001-server-roles.md`](001-server-roles.md) for step-4 outcomes) reflects the active-prohibition semantics of an ACL deny: the operator's policy explicitly disallows the request, in contrast to the "no path to serve" semantics of step-4. **(b) Rate-limit fire** under `THREAT-048` through `THREAT-060`: MUST be silent drop on UDP transports (anti-amplification: a response would feed the same exhaustion vector the rate-limit is mitigating); MUST be TCP-level connection close on TCP, DoT, DoH (HTTP/2 and HTTP/3 connection-level close), and DoQ. **(c) Resource cap reached** under `THREAT-061` through `THREAT-078`: same actions as (b). **(d) Transport-level abuse** (malformed DNS messages, protocol violations beyond DNS layer): MUST be silent drop on UDP; MUST be TCP-level connection close on TCP-based transports. The DNS message bytes that constitute the `REFUSED` + EDE 18 response under (a) MUST be byte-identical across every transport on which the response is delivered, parallel to the per-transport invariance of `ROLE-024`.
- **THREAT-115.** Each `[[<role>.listener.acl]]` rule with `action = "deny"` MAY carry the optional sub-key `deny_action`, whose value is exactly one of `"refused"`, `"drop"`, or `"tcp_close"`. When the sub-key is present, it overrides the default of `THREAT-114` (a) for that specific rule's denial outcome. The values map as follows: `"refused"` produces RCODE `REFUSED` + EDE 18 (the default of `THREAT-114` (a)); `"drop"` produces silent drop on every transport (the operator-specified-equivalent of the saturation default of `THREAT-114` (b)); `"tcp_close"` produces TCP-level connection close on TCP-based transports and silent drop on UDP (the per-transport hybrid of (b) and (d)). The `deny_action` sub-key is admissible only on rules with `action = "deny"`; when applied to an `action = "allow"` rule, the configuration MUST be rejected at load. The per-rule override exists primarily for the case of a known abusive source for which the operator prefers to suppress all diagnostic feedback (`"drop"`) rather than emit `REFUSED` + EDE 18 that would acknowledge Heimdall's presence.
- **THREAT-116.** Every ACL deny event under `THREAT-114` and every per-rule override under `THREAT-115` MUST emit a structured event under `THREAT-080`, regardless of whether the corresponding response is delivered to the client or silently dropped. - **THREAT-120.** Dynamic ACL reload MUST be supported through two mechanisms acting on the same atomic-swap primitive: **(a) SIGHUP full reload**, governed by `OPS-001` through `OPS-006` in [`012-runtime-operations.md`](012-runtime-operations.md), in which the entire configuration (including the full ACL rule set) is parsed, validated, and atomically swapped into the running state; **(b) admin-RPC fine-grained ACL edits**, governed by `OPS-007` through `OPS-014` in [`012-runtime-operations.md`](012-runtime-operations.md), in which the admin-RPC capability set adds, removes, lists, and reloads individual ACL rules without requiring a full SIGHUP. - **THREAT-123.** The compiled internal representation of ACL rules required by `THREAT-047` MUST be a per-axis index structure that the rule-evaluation hot path consults via constant-time-or-near-constant-time lookups, in conformance with the performance principle of [`../CLAUDE.md`](../CLAUDE.md). The per-axis indexes MUST be: **(a)** a radix or PATRICIA trie for `source_cidr`, with separate trees for IPv4 and IPv6, with leaf entries pointing to the rule-ID set of every rule whose `source_cidr` matches the leaf; **(b)** a hash map (`HashMap<String, RuleSet>` or equivalent, with a fast non-cryptographic hash such as FxHash for hot-path use) for `mtls_identity`, `tsig_key_name`, and exact-mode `qname_pattern`; **(c)** a suffix trie keyed by reversed-label form for suffix-mode `qname_pattern`; **(d)** a labelled trie with a single-label wildcard slot for wildcard-mode `qname_pattern` per RFC 1034; **(e)** a per-axis bitset for the small enums `transport`, `role`, and `operation` (each rule contributes one bit per axis value to its membership bitset).
- **THREAT-124.** The cross-axis evaluation of ACL rules MUST proceed by querying each per-axis index under `THREAT-123` for the request's attributes, producing a candidate-rule-ID bitset per axis, then computing the bitwise AND of the per-axis bitsets to produce the final candidate set, then selecting the first rule by declaration order (the lowest-numbered set bit) as the matched rule per `THREAT-112`. The AND-of-bitsets MUST be implemented as a tight loop over `u64` words; for a 10000-rule ACL set, the bitset width is approximately 156 `u64` words (~1.2 kB), and the per-query AND operation runs in sub-microsecond time on every supported architecture. Negated matchers under `THREAT-111` MUST be incorporated by complementing the relevant axis bitset before the AND step.
- **THREAT-125.** The compiled ACL set MUST support a small-rule-set fast path: when the total rule count for a listener is at most 64, the per-axis bitsets MUST collapse to single `u64` values, and the cross-axis AND MUST be a single 64-bit AND operation; the trie indexes are still constructed but the hot-path lookup degenerates to a constant-time bitset operation rather than a trie descent. The fast-path threshold (64 rules) is the natural alignment of the bitset implementation; rule sets above the threshold use the multi-word AND of `THREAT-124` without further branching.
- **THREAT-126.** The compiled ACL set MUST be built synchronously at configuration load and at every reload under `THREAT-120`; the compilation MUST be stored behind the `ArcSwap` pattern of `OPS-016` through `OPS-018` in [`012-runtime-operations.md`](012-runtime-operations.md). - **THREAT-131.** The operational tuning procedure for the RRL parameters of `THREAT-128` MUST be documented in the operational runbook delivered under the documentation policy. The procedure MUST cover at least: a 7-day baseline observation of `rrl-fired` structured events (per `THREAT-057`) before adjusting any parameter; identification of legitimate high-rate clients to be added to `exempt_cidrs` with operator authorisation; iterative adjustment of `responses_per_second` upward when false-positive throttling is observed against legitimate traffic; iterative adjustment downward when amplification ratio remains higher than operator policy admits; re-running the dnsperf benchmark of `THREAT-130` after each parameter change.
- **THREAT-132.** The operational parameters of per-client query rate limiting fixed by `THREAT-051` through `THREAT-053` MUST take the following defaults, each operator-configurable within bounded ranges enforced at configuration load. The defaults differentiate four authentication-state buckets (consistent with `THREAT-053`) and apply a per-transport multiplier to reflect the natural amplification asymmetry between UDP (amplification-prone) and TCP-based transports (TCP is itself a speed cap on legitimate clients).

| Bucket | Configuration key | Default (qps) | Inclusive range |
|---|---|---|---|
| Anonymous (no DNS Cookie or invalid DNS Cookie) | `recursive.query_rl.anonymous_qps`, `forwarder.query_rl.anonymous_qps` | `50` | `[1, 10000]` |
| Validated DNS Cookie bearer | `recursive.query_rl.validated_cookie_qps`, `forwarder.query_rl.validated_cookie_qps` | `200` | `[1, 100000]` |
| mTLS-authenticated identity | `recursive.query_rl.mtls_identity_qps`, `forwarder.query_rl.mtls_identity_qps` | `500` | `[1, 100000]` |
| TSIG-authenticated identity | `recursive.query_rl.tsig_identity_qps`, `forwarder.query_rl.tsig_identity_qps` | `500` | `[1, 100000]` |
| Burst window (seconds) | `*.query_rl.burst_window_seconds` | `10` | `[1, 60]` |
| TCP-based transport multiplier | `*.query_rl.tcp_multiplier` | `2` | `[1, 10]` |
| IPv4 client prefix length (anonymous keying) | `*.query_rl.client_prefix_v4` | `24` | `[16, 32]` |
| IPv6 client prefix length (anonymous keying) | `*.query_rl.client_prefix_v6` | `56` | `[32, 64]` |
| Exempt identities (no rate limit) | `*.query_rl.exempt_identities` | `[]` | array of strings |

The peak-burst capacity within the burst window is the product of the sustained qps and the window: under defaults, an anonymous client can transmit `50 × 10 = 500` queries within a 10-second window before being throttled, which absorbs typical legitimate burstiness without admitting sustained abuse.

- **THREAT-133.** The configuration loader MUST enforce, at instance start, the following invariants on the values fixed by `THREAT-132`: `validated_cookie_qps >= anonymous_qps` (validated cookies are a looser bucket per `THREAT-053`); `mtls_identity_qps >= validated_cookie_qps` (mTLS identity is a stronger authentication signal); `tsig_identity_qps >= validated_cookie_qps`; every numeric parameter is a positive integer within its permitted range; every entry of `exempt_identities` is a non-empty string. A configuration violating any invariant MUST be rejected at load and the instance MUST refuse to start.
- **THREAT-134.** The TCP-based transport multiplier of `THREAT-132` MUST be applied uniformly to every TCP-based outbound bucket (TCP, DoT, DoH over HTTP/2, DoH over HTTP/3, DoQ): the effective rate budget for a connection on a TCP-based transport is `bucket_qps × tcp_multiplier`, where `bucket_qps` is the bucket's per-source rate. UDP transports MUST use the unmodified bucket rate. The rationale is that TCP imposes a per-connection speed cap by design (round-trip-bound query/response cadence on a single connection), whereas UDP is connectionless and per-packet; an attacker that completes a TCP handshake has already paid the legitimate cost of the connection establishment, so the per-source rate budget can safely be relaxed without admitting reflection amplification.
- **THREAT-135.** The fair-sharing property under the defaults of `THREAT-132` MUST be empirically validated by a `dnsperf`-driven benchmark exercising all four authentication-state buckets concurrently at saturation, with a fair distribution of authenticated and anonymous clients on each transport. The benchmark MUST measure the per-bucket admitted-query rate and MUST assert that each bucket's throttling rate is approximately proportional to its configured `qps` value (within ±10% over a one-minute steady-state observation window). The benchmark MUST be delivered under `crates/heimdall-runtime/benches/query_rl.rs` and MUST be exercised in the Tier 3 nightly performance run.
- **THREAT-136.** Adaptive rate limiting (a feedback control loop that mutates the static thresholds of `THREAT-128` and `THREAT-132` in response to runtime signals such as queue depth, CPU utilisation, memory pressure, or observed attack patterns) is explicitly **deferred to post-1.0** and MUST NOT be implemented at this stage. The pre-1.0 posture is: the static parameters of `THREAT-128` and `THREAT-132` are authoritative; operators tune them through configuration per `THREAT-131` and per the operator runbook; no runtime feedback mutates them. Reconsideration of adaptive rate limiting in a post-1.0 revision MUST be conditioned on the following pre-conditions, which together establish that the project has the operational data, the observability infrastructure, and the stability discipline required to add a feedback loop without introducing new failure modes: **(a)** the observability surface fixed by `THREAT-079` through `THREAT-087` is mature and exposes structured events for every static-RL fire and metrics for queue depth, CPU utilisation, and memory pressure; **(b)** static-RL operational data has been accumulated from real deployments for at least 12 months, with documented incidents that adaptive RL would have mitigated; **(c)** the stability-guard design (controller mathematics, e.g., AIMD with min / max bounds, hysteresis, oscillation detection) is specified in a dedicated specification revision; **(d)** the opt-in activation model is specified, with the default for the feature being OFF; **(e)** the rollback trigger (criteria under which the controller is detected as misbehaving and the system reverts to static thresholds) is specified. Until all five pre-conditions are met, any proposal to implement adaptive rate limiting MUST be rejected. The deferral decision is tracked at the project's roadmap level for post-1.0 reconsideration.
- **THREAT-137.** The resource-limit and admission-control parameters required by `THREAT-061` through `THREAT-078` MUST take the following defaults, each operator-configurable within bounded ranges enforced at configuration load. - **THREAT-138.** The configuration loader MUST enforce, at instance start, the following invariants on the values fixed by `THREAT-137`: every numeric parameter is a positive integer within its permitted range; `max_inflight_per_source <= global_pending_queries_cap` (a single source cannot reserve more in-flight slots than the global pool admits); `slow_client_stall_timeout_seconds <= slow_client_idle_timeout_seconds` (the stall timeout is the tighter discipline applied once data has begun flowing). A configuration violating any invariant MUST be rejected at load and the instance MUST refuse to start.
- **THREAT-139.** When any cap fixed by `THREAT-137` is reached and admission of a new request would exceed the cap, eviction MUST follow a single priority function applied uniformly across every cap. The priority function MUST assign each in-flight or pending request to one of the following five priority classes, in strict descending order (higher value retained, lower value evicted first):
  - **Priority 4**: mTLS-authenticated identity per `SEC-067` in [`003-crypto-policy.md`](003-crypto-policy.md).
  - **Priority 3**: TSIG-authenticated identity per `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
  - **Priority 2**: Validated DNS Cookie bearer per `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md).
  - **Priority 1**: Anonymous (no DNS Cookie offered).
  - **Priority 0**: Anonymous with invalid DNS Cookie.
  Within the same priority class, the eviction strategy MUST be drop-oldest (longest-idle for connections, oldest-enqueued for pending queries). The priority function MUST be a single shared implementation used by every cap; per-cap variant priority functions MUST NOT be implemented. The function applies uniformly to connection-cap, in-flight-cap, and pending-queue-cap evictions.
- **THREAT-140.** The "under load" signal that activates DNS-Cookie-based admission bias under `THREAT-069` MUST be a composite OR signal: Heimdall MUST enter the "under load" state when ANY of the following four conditions is true.
  - **CPU utilisation** of the Heimdall process exceeds `cpu_threshold_pct` (default `80`, range `[10, 100]`) over a 5-second moving average.
  - **Memory utilisation** of the Heimdall process exceeds `memory_threshold_pct` (default `85`, range `[10, 100]`) of the process's resident-set-size limit.
  - **Pending-queries count** exceeds `pending_queue_threshold_pct` (default `75`, range `[10, 100]`) of `global_pending_queries_cap` from `THREAT-137`.
  - **Rate-limit fires per second** (aggregated across RRL and query RL per `THREAT-057`) exceeds `rate_limit_fires_per_second` (default `1000`, range `[10, 1000000]`).
  Each threshold is operator-configurable under `[under_load_signal]` in the configuration file. The numeric defaults are calibrated against typical deployment profiles and may require local adjustment for specific workloads.
- **THREAT-141.** The "under load" state MUST be guarded by hysteresis to avoid flapping between under-load and steady states under transient load. Heimdall MUST enter the "under load" state immediately when any threshold of `THREAT-140` is exceeded; Heimdall MUST exit the "under load" state only when ALL of the following hold for at least `hysteresis_exit_duration_seconds` (default `30`, range `[5, 300]`) consecutive seconds: every threshold of `THREAT-140` is below the threshold reduced by `hysteresis_exit_margin_pct` (default `10`, range `[1, 50]`) percent. The asymmetric entry-exit policy ensures that brief drops below threshold do not reset the under-load state prematurely; only sustained recovery exits the state.
- **THREAT-142.** Transitions of the "under load" state MUST emit structured events under `THREAT-080`: `under-load-entered` (carrying the triggering threshold and its observed value) and `under-load-exited` (carrying the duration of the under-load state and the metric values at exit). Eviction events under `THREAT-139` MUST emit a `eviction-fired` structured event carrying the cap that was reached, the count of evicted entries, the priority class of the evicted entries, and the priority class of the admitted entry that triggered the eviction. - **THREAT-143.** The structured-event schema required by `THREAT-080` MUST follow a common base envelope plus event-specific field sets. The base envelope MUST contain, on every event without exception, the following fields: `schema_version` (string, initial value `"1.0"`); `timestamp` (string, RFC 3339 UTC with millisecond precision); `level` (string enum from `THREAT-144`); `event_type` (string, the canonical event-type identifier such as `"acl-deny"`, `"rrl-fired"`, `"dnssec-bogus"`, `"rapid-reset"`, etc.); `instance` (object with sub-fields `nsid` — the NSID per `PROTO-013` — `node` — the operating-system hostname or container identity — and `version` — the Heimdall release version); and `correlation_id` (string, a per-event unique identifier suitable for cross-event correlation, encoded as a [ULID](https://github.com/ulid/spec) or equivalent monotonic identifier). ## 4. Rationale

Attacker classes `THREAT-001` through `THREAT-005` map directly onto decisions already made in the specification. Encrypted transports (DoT, DoH over HTTP/2 and HTTP/3, DoQ) defend the wire against observation and modification. DNSSEC validation — mandatory on both validating roles, with aggressive NSEC / NSEC3, the KeyTrap cap, the NSEC3 iteration cap, and strict algorithm and digest acceptance policies — defends the data against off-path injection, upstream compromise, and adversarial zones. 0x20 case randomisation with case-match verification, DNS Cookies, RFC 9471 glue-record handling, segregated recursive and forwarder caches, QUIC address validation with single-use `NEW_TOKEN`, and the HTTP/2 / HTTP/3 hardening posture close off further protocol-level and volumetric vectors. Formalising the threat model consolidates this work, makes the coverage visible, and exposes the remaining gaps as concrete open questions rather than as implicit assumptions.

Supply chain (`THREAT-006`) is already obliged by the "SEGURANÇA" section of [`../CLAUDE.md`](../CLAUDE.md), which lists supply-chain risks among the attack vectors that every change must account for and caps external dependencies at the strict minimum. Making that obligation structural — `cargo-audit` and `cargo-vet` as CI gates, reproducible builds, signed release artefacts, and an SBOM per release — places the pipeline on the non-negotiable side of the project's engineering policy. None of these measures depends on runtime configuration, so they cannot be weakened by operator choice; they are properties of the artefact itself.

Local insider or operator compromise (`THREAT-007`) is delegated honestly. A process cannot structurally defend against an attacker who already holds root on its host, and claiming otherwise would be security theatre. Placing `THREAT-007` out of scope for structural mitigation does not, however, remove the obligation to apply operational hardening to the Heimdall process: the "fail-safe / fail-closed" principle in [`../CLAUDE.md`](../CLAUDE.md) is architectural, not only operational, and the hardening profile in sub-section 2.5 (requirements `THREAT-022` through `THREAT-032`) limits the blast radius of partial exploits of in-scope classes — in particular the rare memory-safety bugs that Rust does not eliminate entirely, and protocol-level bugs in the parsing and validating paths. Privilege drop paired with minimum capability retention closes the largest local blast-radius door at near-zero cost. Seccomp-bpf blocks entire classes of post-exploitation primitives — `execve`, privileged socket creation, `ptrace`, module loading — even when a memory-safety bug is present. Systemd sandboxing is effectively free: it requires only configuration and cuts off lateral movement, filesystem exfiltration, and privilege re-escalation paths. Filesystem restriction prevents reads of unrelated sensitive content (for example, `/etc/shadow` or user home directories) even under a bug that permits arbitrary path operations inside the allowed surface. On BSD and macOS, `pledge` and `unveil` on OpenBSD and a sandbox profile on macOS provide equivalent coverage where the platform supplies it. Process separation per role (OpenBSD-style privsep) is deliberately deferred rather than adopted: it adds substantial architectural complexity — serialisation of state across processes, inter-process communication on the hot path, context-switch overhead, and harder debugging — and in a Rust codebase the class of bugs it primarily defends against (C-era memory-safety exploitation) is already dramatically reduced by the language itself. Privsep may be re-considered later if the threat profile changes or if empirical incident data justifies the architectural cost.

Physical and side-channel attacks (`THREAT-008`) are delegated for the same structural reason. Constant-time and side-channel-resistant cryptographic primitives are inherited from the TLS and signature libraries Heimdall depends on; those choices remain a downstream open question, but they are not a property that Heimdall could add on top. Physical attacks against the host sit entirely in the deployment environment.

This document is deliberately the first of several threat-model decisions. It establishes the classification and the existing coverage, and it fixes the operational-hardening profile applied to the Heimdall process on every supported platform, the multi-axis ACL matrix, the two orthogonal rate-limiting mechanisms, the resource-limits and admission-control family, and the observability surface targeted at security detection; subsequent clarifications will specify per-class mitigation mapping in more depth, the concrete parameters of the operational-hardening profile left open in section 5 (seccomp-bpf allow-list and rejection action, systemd directive values where latitude exists, OpenBSD `pledge` promises and `unveil` paths, and the macOS sandbox profile and its review cadence), the concrete values of the rate-limiting and resource-limits parameters left open in section 5, the definition of the "under load" signal that activates the DNS-Cookie-based admission preference fixed by `THREAT-069`, the concrete schema and operational policies of the observability surface (event schema, verbosity levels, log rotation and retention, redaction and privacy, alerting baseline, tracing sampling, and SIEM integration), cryptographic primitive choices that affect the side-channel profile, and the residual risks that the project accepts and documents.

The access-control-list policy fixed by `THREAT-033` through `THREAT-047` in sub-section 2.6 is required at the specification level for five reinforcing reasons. First, deployment coverage: the representative deployment scenarios that Heimdall is expected to serve — public resolver, corporate DNS with views and TSIG-authenticated zone transfer, managed DNS with mTLS-based tenant isolation, and hybrid authoritative-plus-recursive-plus-forwarder enterprise setups — each require a different cross-section of the multi-axis matrix defined by `THREAT-034` through `THREAT-041`. A less expressive matrix would restrict Heimdall to a narrow subset of these scenarios and would force operators to either deploy a second product alongside Heimdall for access control or to rely on ad-hoc filtering at an outer network element, both of which fragment the security posture and reintroduce the class of misconfigurations that a native, integrated ACL surface exists to prevent. Second, alignment with prior decisions: the mTLS-identity axis under `THREAT-036` is the operational lever that makes `SEC-012` through `SEC-016` and `SEC-031` through `SEC-035` in [`003-crypto-policy.md`](003-crypto-policy.md) useful beyond handshake authentication, because without an ACL axis that consumes the validated client-certificate identity the mTLS decision would stop at "the connection was authenticated" and not propagate identity into authorisation; the TSIG-identity axis under `THREAT-037` similarly operationalises `PROTO-044` in [`006-protocol-conformance.md`](006-protocol-conformance.md) by turning the TSIG key identity into an authorisation discriminator rather than only an authentication check. Without these axes, both the mTLS and the TSIG decisions would remain under-used. Third, fail-safe defaults: the default-deny posture for AXFR and IXFR under `THREAT-042`, the default-deny posture for recursive-role and forwarder-role queries under `THREAT-044`, and the default-deny posture for outbound acceptance under `THREAT-045` embody the "fail-safe / fail-closed" principle fixed by [`../CLAUDE.md`](../CLAUDE.md). The default-allow posture for authoritative-role queries on listeners configured for authoritative service under `THREAT-043` is the single exception and is consistent with the observation that publishing authoritative data is the explicit purpose of an authoritative listener; narrowing that default is always available to the operator through an explicit ACL rule, whereas broadening the default-deny postures would require reopening this decision. Fourth, views consolidated: the open question on views tracked in [`001-server-roles.md`](001-server-roles.md) is now resolved by `THREAT-046`. Views are ACL-driven response selection built on top of the same matrix that governs admission; they are not a separate architectural subsystem with its own matching logic, its own configuration surface, and its own failure modes. This consolidation removes a latent source of drift between admission rules and response-selection rules, because both are expressed in the same vocabulary and evaluated by the same engine. Fifth, performance discipline: the expressiveness of the matrix is only valuable if it can be evaluated on the hot path without measurable cost, consistent with the performance principle fixed by [`../CLAUDE.md`](../CLAUDE.md). `THREAT-047` therefore requires an efficient compiled representation for hot-path evaluation, while leaving the concrete representation (for example, a radix or PATRICIA trie for CIDRs combined with hash tables for identity-based axes and a compiled decision tree across axes) as an implementation detail so that the chosen data structure can be selected and revised against measurement.

The resource-limits and admission-control policy fixed by `THREAT-061` through `THREAT-078` in sub-section 2.8 is required at the specification level for five reinforcing reasons. First, structural defence in depth: each mechanism in the family closes a distinct denial-of-service vector that the other mechanisms do not address on their own. Connection caps fixed by `THREAT-062` close file-descriptor exhaustion and connection-table exhaustion, which neither ACLs nor rate limiting address. Per-connection and per-source in-flight caps fixed by `THREAT-063` and `THREAT-064` close slowloris-style starvation in which a modest number of sources each hold an abusive number of in-flight queries open. Per-cache memory caps fixed by `THREAT-066` close the leak-to-OOM vector in which an adversarial zone with millions of distinct QNAMEs would otherwise grow the validating cache until the process is killed by the operating system, which — because a single Heimdall process hosts every enabled role — would impact every role on that instance, not only the role that received the abusive queries. Per-query allocation caps fixed by `THREAT-067` close the pathological-input class in which a single crafted message forces an unbounded buffer allocation, reinforcing the HTTP-layer bounds already fixed by `SEC-039`, `SEC-040`, and `SEC-045` in [`003-crypto-policy.md`](003-crypto-policy.md). Slow-client disconnection fixed by `THREAT-068` closes the class in which idle or stalled clients pin server-side connection state indefinitely. The DNS-Cookie-based admission preference fixed by `THREAT-069` and `THREAT-070` provides graceful degradation against amplification abuse that has leaked past RRL under `THREAT-048` through `THREAT-050` and past query RL under `THREAT-051` through `THREAT-053`, by preferentially shedding anonymous traffic rather than authenticated traffic. Second, operating-system-level limits such as `ulimit`, cgroups, and systemd resource directives are backstops, not substitutes: they kill resources without application-layer context and cannot distinguish authenticated traffic from anonymous traffic, so relying on them as the primary defence would convert an overload into the loss of legitimate flows indistinguishable from abusive flows. Third, DNS-Cookie-based admission consumes no additional machinery: DNS Cookies are already mandatory across every applicable role under `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md), so using cookie validity as an admission-control signal under `THREAT-069` converts an existing cryptographic fact into an authorisation discriminator at effectively zero incremental cost, in the same spirit as the cookie-driven bucketing in query RL under `THREAT-053`. Fourth, per-cache memory governance aligns with the segregated-cache decision fixed by `CACHE-001` through `CACHE-007` in [`004-cache-policy.md`](004-cache-policy.md): without an independent memory cap per cache, the cache-segregation property would hold in the data-flow dimension while still allowing one role's abusive growth to starve another role's cache of memory, because both caches live in the same process. Enforcing per-cache memory budgets preserves the role-level isolation invariant in the memory dimension as well as in the data-flow dimension. Fifth, explicit transport-specific backpressure — silent drop on UDP under `THREAT-071`, connection close on TCP and DoT under `THREAT-072`, stream reset on DoH and DoQ under `THREAT-073` and `THREAT-074`, the optional TC-flag fallback under `THREAT-075`, and the five-stage evaluation order under `THREAT-076` — is the correct behaviour under saturation: it signals the client clearly in the vocabulary of the transport it is using, rather than crashing, silently failing, or committing resources the server cannot honour. The "fail-safe / fail-closed" principle fixed by [`../CLAUDE.md`](../CLAUDE.md) maps directly to preserving partial availability over total failure: a server that closes a connection or resets a stream under saturation leaves the rest of its capacity available to legitimate traffic, whereas a server that attempts to service every arrival unconditionally converts an overload into a full outage. The no-global-off-switch posture fixed by `THREAT-078` applies the same structural-gating rationale already stated for DNSSEC validation under `DNSSEC-009` and `DNSSEC-020` in [`005-dnssec-policy.md`](005-dnssec-policy.md), for QNAME minimisation under `PROTO-023` in [`006-protocol-conformance.md`](006-protocol-conformance.md), for 0x20 case randomisation under `PROTO-030` in [`006-protocol-conformance.md`](006-protocol-conformance.md), and for the rate-limiting family under `THREAT-056`: the protection is part of the system's structure, not a configuration-dependent property that can be weakened by accident or by template inheritance. Operators tune values; they do not toggle the enforcement family. The observability requirement fixed by `THREAT-077` is a direct consequence of the fact that limits whose approach and whose firing are not visible to the operator are indistinguishable from limits that never fire; under the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md), approach-to-limit conditions are a leading indicator of abusive traffic and MUST be surfaced in the same structured-telemetry channel used by the rate-limiting mechanisms under `THREAT-057`.

The observability policy fixed by `THREAT-079` through `THREAT-087` in sub-section 2.9 is required at the specification level for five reinforcing reasons. First, operational security requires detection signal. Every mitigation decided previously in this specification — rate limiting under `THREAT-048` through `THREAT-060`, the KeyTrap cap under `DNSSEC-041` in [`005-dnssec-policy.md`](005-dnssec-policy.md), the NSEC3 iteration cap under `DNSSEC-044` in [`005-dnssec-policy.md`](005-dnssec-policy.md), 0x20 adaptive disable under `PROTO-027` and `PROTO-028` in [`006-protocol-conformance.md`](006-protocol-conformance.md), Negative Trust Anchor management under `DNSSEC-017` in [`005-dnssec-policy.md`](005-dnssec-policy.md), and DNS-Cookie-based admission under `THREAT-069` — presupposes that the operator can observe when the mechanism fires; without observability, there is no way to distinguish quiescence from a running attack already mitigated by the defaults, and a silently-mitigated attack is indistinguishable from no attack at all. Second, events and metrics are complementary rather than redundant: structured events under `THREAT-080` provide the forensic context needed for post-incident analysis — which client, which QNAME, which algorithm, which upstream — while metrics under `THREAT-081` provide the trend and alerting signal needed to detect an attack in progress and to observe steady-state behaviour, in particular aggregate QPS, cache hit rate, handshake success rate, and the aggregate counts of `bogus` validation outcomes or rate-limit fires. Neither surface replaces the other; dropping either would leave a gap that the remaining surface could not cover. Third, standard formats protect against vendor lock-in and integrate with the existing monitoring ecosystem: the JSON-based event format required by `THREAT-082` is ingestible by Loki, ELK, Splunk, Datadog, and equivalent pipelines with no bespoke adapter work, and OpenTelemetry semantic conventions give field naming a shared vocabulary with other observability producers; the OpenMetrics format required by `THREAT-083` is scrapable by Prometheus and compatible with every monitoring system that consumes Prometheus metrics. Fourth, per-query distributed tracing is deferred to opt-in by `THREAT-079` and `THREAT-084` because per-query tracing at the "EXTREMELY HIGH LOAD AND CONCURRENCY" target stated in [`../CLAUDE.md`](../CLAUDE.md) can be costly and is rarely warranted on the default path; operators enable it where the diagnostic value outweighs the per-query overhead, and OpenTelemetry then gives them a standard way to consume it. Fifth, the hot-path discipline fixed by `THREAT-085` and `THREAT-086` preserves the performance principle of [`../CLAUDE.md`](../CLAUDE.md): events are emitted only on anomalous conditions so that the normal-path query flow pays no event-emission cost, and metrics updates on the normal path MUST use lock-free counters or per-thread aggregation so that observability does not become a new contention point between worker threads. The specification already requires structured telemetry per rate-limiting mechanism under `THREAT-057` and per resource-limit mechanism under `THREAT-077`; sub-section 2.9 extends that obligation uniformly across the other mitigations so that every decision made elsewhere in the specification has a corresponding detection surface. The no-global-off-switch posture fixed by `THREAT-087` applies the same structural-gating rationale already adopted for DNSSEC validation under `DNSSEC-009` and `DNSSEC-020` in [`005-dnssec-policy.md`](005-dnssec-policy.md), for QNAME minimisation under `PROTO-023` in [`006-protocol-conformance.md`](006-protocol-conformance.md), for 0x20 case randomisation under `PROTO-030` in [`006-protocol-conformance.md`](006-protocol-conformance.md), for the rate-limiting family under `THREAT-056`, and for the resource-limits family under `THREAT-078`: verbosity levels tune what is emitted, but the detection surface as a whole MUST remain available on every deployment.

The rate-limiting policy fixed by `THREAT-048` through `THREAT-060` in sub-section 2.7 is required at the specification level for five reinforcing reasons. First, RRL is the baseline for any public authoritative server in 2026: BIND, NSD, Knot Authoritative, and PowerDNS Authoritative all implement it. Without RRL, a public authoritative server is a ready-made amplifier in DRDoS reflection attacks — the response is routinely larger than the query, especially for DNSSEC-signed answers as required by `DNSSEC-*` in [`005-dnssec-policy.md`](005-dnssec-policy.md), and the transport is UDP with spoofable source addresses. That directly contradicts the volumetric-DoS attacker class treated as in scope by `THREAT-005`. The "slip" behaviour required by `THREAT-049` is the feature that prevents RRL from becoming a denial-of-service vector against the very clients whose source addresses an attacker is spoofing: by periodically returning a truncated response instead of silently dropping every over-budget response, Heimdall preserves a TCP fallback path for legitimate clients while still denying the reflection path for the attacker. Second, query rate limiting closes the gap that RRL does not cover: RRL is keyed on the response shape `(client-prefix, qname, qtype)` and is tuned for reflection scenarios, whereas direct abuse against a recursive resolver or forwarder by a single client flooding diverse queries is a workload that slips under RRL's aggregation. Unbound, Knot Resolver, and PowerDNS Recursor all implement per-client rate limiting for exactly this reason; omitting it on the recursive and forwarder roles would be regressive relative to the deployed validating-resolver ecosystem. Third, cross-reference with prior decisions: the DNS Cookie decision fixed by `PROTO-010` in [`006-protocol-conformance.md`](006-protocol-conformance.md) provides a natural bucketing signal for query RL, which `THREAT-053` consumes by requiring stricter buckets for clients that present no cookie or an invalid cookie and allowing looser buckets for clients that present a validated cookie; without that bucketing, the DNS Cookie mechanism would authenticate source addresses without being used as an authorisation discriminator on the volumetric path. The single-use `NEW_TOKEN` requirement fixed by `SEC-028` and `SEC-029` in [`003-crypto-policy.md`](003-crypto-policy.md) and the QUIC amplification limit fixed by `SEC-025` in [`003-crypto-policy.md`](003-crypto-policy.md) mitigate DoS at the transport layer, but they do not substitute for application-layer rate limiting, because an attacker who completes a legitimate QUIC handshake can still issue application-layer queries at an abusive rate; the two layers are complementary. Fourth, delegation to L4 firewalls or external DDoS-protection front-ends is insufficient, as fixed by `THREAT-059`: those layers cannot reason about the DNS semantics that drive the decisions in `THREAT-048` through `THREAT-053` — repeated queries for the same name versus diverse queries across a zone, cache-hit ratio, the cost of DNSSEC validation on bespoke zones, and the validity of a DNS Cookie on a specific exchange. A purely L4 filter cannot make the distinction between a client hammering a single name (which RRL handles efficiently) and a client issuing diverse queries at abusive rate (which query RL handles), and it cannot increase the rate budget granted to a client that presents a validated DNS Cookie. Fifth, adaptive or closed-loop thresholds are deliberately deferred by `THREAT-058`: reliable adaptive thresholding requires a mature observability surface, of the kind that is still tracked as open in section 5, and a poorly calibrated adaptive controller can introduce stability problems worse than the attack it is meant to mitigate — oscillation between thresholds, feedback loops with legitimate traffic spikes, and convergence on thresholds that effectively deny a subset of well-behaved clients. Fixing static configured thresholds first and treating adaptive behaviour as a later, opt-in extension matches the cautious-first trajectory already adopted for TEK rotation in [`003-crypto-policy.md`](003-crypto-policy.md) and for NEW_TOKEN rotation in the same document. Rate limiting is fail-safe under `THREAT-055`: the enumerated overrun actions (silent drop, REFUSED, truncation with TC=1, bounded-delay deferral) each cap the resource cost of an abusive client, and under no configuration may an overrun result in an unbounded resource commitment. The no-global-off-switch posture fixed by `THREAT-056` applies the same structural-gating rationale already stated for DNSSEC validation under `DNSSEC-009` and `DNSSEC-020` in [`005-dnssec-policy.md`](005-dnssec-policy.md), for QNAME minimisation under `PROTO-023` in [`006-protocol-conformance.md`](006-protocol-conformance.md), and for 0x20 case randomisation under `PROTO-030` in [`006-protocol-conformance.md`](006-protocol-conformance.md): the protection is part of the system's structure on the roles to which it applies by default, not a configuration-dependent property that can be weakened by accident or by template inheritance. Operators tune thresholds and select per-rule overrun actions; they do not toggle the mechanism as a whole.
