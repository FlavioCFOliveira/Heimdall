# Forward-zone rules for the forwarder role

**Purpose.** This document defines the syntax, matching semantics, and precedence of forward-zone rules, which are the operator-declared input to step 2 of the four-step query-resolution precedence defined in [`001-server-roles.md`](001-server-roles.md).

**Status.** Stable.

**Requirement category.** `FWD`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model and the four-step precedence this document depends on, see [`001-server-roles.md`](001-server-roles.md). For the ACL matrix that composes with forward-zone rules and for the views mechanism that emerges from that composition, see [`007-threat-model.md`](007-threat-model.md).

## 1. Scope

A forward-zone rule declares that queries matching a zone pattern MUST be forwarded to the upstream resolver or resolvers declared by that rule, in accordance with `ROLE-010` in [`001-server-roles.md`](001-server-roles.md). The present document fixes:

- The TOML shape of forward-zone rules under the `[[forwarder.forward_zone]]` array introduced by `ROLE-022` in [`001-server-roles.md`](001-server-roles.md).
- The three matching modes (`"exact"`, `"suffix"`, `"wildcard"`) applied to the query name against the rule's `zone` pattern.
- The deterministic rule-precedence algorithm applied when more than one rule matches a query.
- The fallback behaviour of step 2 of the four-step query-resolution precedence when all upstreams of a matching rule have been exhausted with every contact attempt resulting in upstream failure.
- The interaction with the multi-axis ACL matrix fixed by `THREAT-033` through `THREAT-047` in [`007-threat-model.md`](007-threat-model.md), and in particular with the views mechanism fixed by `THREAT-046`.

The present document does NOT fix:

- The upstream-side cryptographic and transport validation policy (certificate validation, mTLS client authentication, SPKI pinning, session-ticket cache, connection pooling, health-checking, load-balancing, EDNS Client Subnet). These items are tracked as open questions in [`002-transports.md §5`](002-transports.md) and in [`003-crypto-policy.md`](003-crypto-policy.md).
- The syntax of the optional ACL binding on a forward-zone rule. The ACL configuration syntax is tracked as an open question in [`007-threat-model.md §5`](007-threat-model.md); the ACL binding on a forward-zone rule reuses that syntax once fixed.
- The HTTP version (HTTP/2 versus HTTP/3) selection policy on outbound DoH connections to upstreams. This is tracked as an open question in section 5 of this document.

## 2. Normative requirements

- **FWD-001.** Forward-zone rules MUST be declared as an array of tables at the TOML path `[[forwarder.forward_zone]]`, nested under the `[forwarder]` role-activation table in accordance with `ROLE-022` in [`001-server-roles.md`](001-server-roles.md).
- **FWD-002.** Each `[[forwarder.forward_zone]]` entry MUST carry, at minimum, the string key `zone` and the array-of-tables key `upstream`. The `upstream` array MUST contain at least one entry. A `[[forwarder.forward_zone]]` entry lacking `zone`, lacking `upstream`, or declaring an empty `upstream` array MUST be rejected at configuration load and MUST cause the instance to refuse to start.
- **FWD-003.** The value of the `zone` key MUST be a syntactically valid DNS name in canonical form with a trailing dot, as defined by [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034), [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343), and [RFC 8499](https://www.rfc-editor.org/rfc/rfc8499). The root zone MUST be representable as the single-character value `"."`. A `zone` value that is not a syntactically valid DNS name MUST be rejected at configuration load.
- **FWD-004.** Each `[[forwarder.forward_zone]]` entry MAY carry the string key `match` whose value is exactly one of the three enumerated strings `"exact"`, `"suffix"`, or `"wildcard"`. When the `match` key is absent, the implicit value MUST be `"suffix"`. Any other value for `match` MUST be rejected at configuration load and MUST cause the instance to refuse to start.
- **FWD-005.** When `match = "wildcard"`, the `zone` value MUST begin with the label `*` (i.e., the `zone` value MUST start with the three-character prefix `*.`), and the portion of `zone` following the leading `*.` MUST itself be a syntactically valid DNS name in canonical form with a trailing dot. When `match = "exact"` or `match = "suffix"`, the `zone` value MUST NOT contain the label `*` at any position.
- **FWD-006.** In `"exact"` mode, a rule matches a query QNAME if and only if QNAME is equal to `zone` label-for-label. Name comparison MUST be ASCII-case-insensitive as defined by [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343).
- **FWD-007.** In `"suffix"` mode, a rule matches a query QNAME if and only if QNAME is equal to `zone` OR QNAME is a proper descendant of `zone` (i.e., QNAME ends with `zone` as a label-aligned suffix and has at least one additional label to the left of `zone`). Name comparison MUST be ASCII-case-insensitive as defined by [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343).
- **FWD-008.** In `"wildcard"` mode, let `zone_suffix` denote the value of `zone` with the leading `*.` removed. A rule matches a query QNAME if and only if QNAME consists of exactly one label immediately followed by `zone_suffix` (label-for-label), regardless of the content of that leftmost label. The leftmost label MAY be any syntactically valid DNS label, including the literal label `*`. Name comparison MUST be ASCII-case-insensitive as defined by [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343). A QNAME equal to `zone_suffix` (no additional leftmost label) MUST NOT be considered to match in `"wildcard"` mode, and a QNAME with two or more additional labels to the left of `zone_suffix` MUST NOT be considered to match in `"wildcard"` mode.
- **FWD-009.** Multiple forward-zone rules MAY match the same query QNAME. When the set of matching rules for a given query is non-empty, after ACL filtering under `FWD-015` has been applied, the rule of highest rank MUST be selected to determine the upstreams. The rank of a rule against a given query MUST be computed as the ordered tuple `(mode_priority, zone_label_count, negative_declaration_order)`, and rules MUST be compared by lexicographic ordering on that tuple, greater-wins. When every component of the tuple is equal between two rules, the rules are identical in effect and the declaration_order tiebreak in `FWD-012` selects the earliest-declared.
- **FWD-010.** The mode_priority component of the rank tuple MUST take the following values, and no other: `"exact"` → `3`; `"wildcard"` → `2`; `"suffix"` → `1`.
- **FWD-011.** The zone_label_count component of the rank tuple MUST be the number of dot-separated labels in the `zone` value of the rule, counting the leading `*` label in `"wildcard"` mode as one label and counting the trailing root as zero labels. Under this definition: the root zone `"."` has zone_label_count 0; `"example.com."` has zone_label_count 2; `"*.example.com."` has zone_label_count 3.
- **FWD-012.** The declaration_order component of the rank tuple MUST be the zero-indexed position of the rule in the configuration file, with earlier-declared rules having smaller declaration_order values. In the rank tuple, declaration_order MUST be negated so that the greater-wins lexicographic comparison selects the earliest-declared rule when all other components are equal.
- **FWD-013.** Two or more `[[forwarder.forward_zone]]` entries that share the same `zone` value and the same `match` mode MUST be permitted by the configuration loader, in order to support the views mechanism fixed by `THREAT-046` in [`007-threat-model.md`](007-threat-model.md). When such rules coexist and all pass ACL filtering under `FWD-015`, the declaration_order tiebreak in `FWD-009` selects the earliest-declared among them.
- **FWD-014.** Each entry in the `upstream` array of a forward-zone rule MUST carry, at minimum, the three keys `address`, `port`, and `transport`. The `transport` value MUST be exactly one of the five enumerated strings `"udp"`, `"tcp"`, `"dot"`, `"doh"`, `"doq"`, matching the listener-transport enum of `ROLE-020` in [`001-server-roles.md`](001-server-roles.md). The declaration of `transport` on every upstream is mandatory in accordance with `NET-013` in [`002-transports.md`](002-transports.md); there is no implicit default outbound transport. Additional per-upstream sub-keys — including, but not limited to, TLS client certificate and private-key material paths, the optional outbound mTLS sub-table, SPKI pinning material, TSIG key binding, connection-pool parameters, and DoH HTTP version selection — are admitted under each upstream entry and are governed by the specification documents applicable to each. Any upstream sub-key not defined by this specification or by the specifications it cross-references MUST be rejected at configuration load per `ROLE-021` in [`001-server-roles.md`](001-server-roles.md).
- **FWD-015.** The multi-axis ACL matrix fixed by `THREAT-033` through `THREAT-047` in [`007-threat-model.md`](007-threat-model.md) MUST be evaluated against the client of a query before forward-zone rule ranking under `FWD-009`. A forward-zone rule whose optional ACL binding is present and whose ACL does not match the client under evaluation MUST be excluded from the set of matching rules considered by `FWD-009`. A forward-zone rule with no ACL binding MUST be treated as matching the ACL filter for every client. The exact syntax of the ACL binding on a forward-zone rule entry is governed by the "ACL configuration syntax" open question in [`007-threat-model.md §5`](007-threat-model.md) and is NOT fixed by this document; the composition rule stated in the present requirement is however fixed.
- **FWD-016.** When no forward-zone rule matches a given query after `FWD-015` and `FWD-009` have been applied, step 2 of the four-step query-resolution precedence fixed by `ROLE-008` through `ROLE-012` in [`001-server-roles.md`](001-server-roles.md) MUST NOT match, and evaluation of the query MUST proceed to step 3 (recursive resolver, if active under `ROLE-011`) or to step 4 (error response under `ROLE-012`), as applicable. The forward-zone rule evaluation itself MUST NOT produce an error response to the client; the error-response decision under `ROLE-012` is the sole authority on the absence-of-matching-role outcome.
- **FWD-017.** The configuration loader MUST reject any `[[forwarder.forward_zone]]` entry that carries a key not defined by this specification or by the specifications it cross-references. This rule inherits from `ROLE-021` in [`001-server-roles.md`](001-server-roles.md) and enforces the "Assume Nothing" principle of [`../CLAUDE.md`](../CLAUDE.md) at the forward-zone rule boundary.
- **FWD-018.** Each `[[forwarder.forward_zone]]` entry MAY carry the string key `fallback` whose value is exactly one of the two enumerated strings `"error"` or `"recursive"`. When the `fallback` key is absent, the implicit value MUST be `"error"`. Any other value for `fallback` MUST be rejected at configuration load and MUST cause the instance to refuse to start. The default value `"error"` realises the fail-closed posture required by [`../CLAUDE.md`](../CLAUDE.md) at the forwarder boundary, and the per-rule `"recursive"` opt-in is the operator's explicit declaration that resilience to upstream failure outweighs the fail-closed default for the zone the rule covers.
- **FWD-019.** The set of upstream-resolver outcomes that constitute a **successful response** for the purposes of forward-zone rule resolution MUST consist of, and only of: a NOERROR DNS response carrying one or more answer records in the Answer section; a NODATA DNS response (NOERROR with an empty Answer section and a SOA record in the Authority section, as defined by [RFC 2308](https://www.rfc-editor.org/rfc/rfc2308)); and an NXDOMAIN DNS response. A successful response MUST be propagated to the client unchanged in its DNS Header RCODE, and MUST NOT cause `fallback` handling to be invoked, regardless of the rule's `fallback` value.
- **FWD-020.** The set of upstream-resolver outcomes that constitute an **upstream failure** for the purposes of forward-zone rule resolution MUST be every outcome that is not a successful response under `FWD-019`. This set MUST include, but is not limited to: every transport-level failure (connection refused, connection reset, connection timeout, TLS handshake failure, QUIC handshake failure, ICMP unreachable, network error, DNS protocol query timeout); every DNS-protocol RCODE other than NOERROR and NXDOMAIN, including SERVFAIL (RCODE 2), FORMERR (RCODE 1), NOTIMP (RCODE 4), REFUSED (RCODE 5), and any reserved or extended RCODE; and a NOERROR response carrying neither answer records nor a NODATA-conforming SOA in the Authority section (a malformed empty response). The conditions under which the upstreams of a single rule are considered "exhausted" — including the order in which upstreams are tried within a rule, the per-upstream timeouts, and the inter-upstream fallback policy — are governed by the open questions tracked in [`002-transports.md §5`](002-transports.md) and are NOT fixed by this document.
- **FWD-021.** When all upstreams of a matching rule have been exhausted with every contact attempt resulting in an upstream failure under `FWD-020`, and the rule's `fallback` value is `"error"`, Heimdall MUST generate the client-visible response according to the following rules. If the last contact attempt resulted in a transport-level failure, Heimdall MUST synthesise a SERVFAIL (RCODE 2) DNS response and return it to the client; the Answer, Authority, and Additional sections of the synthesised response MUST be empty. If the last contact attempt resulted in an upstream-returned RCODE, Heimdall MUST propagate that RCODE to the client unchanged in the response Header; the Answer, Authority, and Additional sections of the propagated response MUST carry the upstream's content if and only if the upstream returned content in those sections. Step 3 of the four-step query-resolution precedence fixed by `ROLE-008` through `ROLE-012` in [`001-server-roles.md`](001-server-roles.md) MUST NOT be evaluated for that query.
- **FWD-022.** When all upstreams of a matching rule have been exhausted with every contact attempt resulting in an upstream failure under `FWD-020`, and the rule's `fallback` value is `"recursive"`, evaluation of the query MUST proceed to step 3 of the four-step query-resolution precedence fixed by `ROLE-008` through `ROLE-012` in [`001-server-roles.md`](001-server-roles.md), subject to `ROLE-011` (the recursive resolver role MUST be active for step 3 to apply). The recursive resolver MUST treat the query as if it had arrived directly at step 3, unaffected by the prior step-2 attempt and unaffected by the failed upstream contacts. The outcome produced by the recursive resolver — whether a successful response or a SERVFAIL — MUST be returned to the client.
- **FWD-023.** The configuration loader MUST reject any configuration file that contains at least one `[[forwarder.forward_zone]]` entry with `fallback = "recursive"` while `[recursive].enabled` evaluates to `false` or while the `[recursive]` role-activation table is absent from the configuration file. The instance MUST refuse to start in either case. This rule enforces a structural consistency property at the configuration boundary: a rule that opts into recursive fallback presumes the existence of an active recursive role, and the absence of one MUST be a configuration error rather than a runtime degradation. The rejection MUST occur regardless of whether any other rule in the configuration uses `fallback = "error"`.
- **FWD-024.** Heimdall MUST emit a structured event under `THREAT-080` in [`007-threat-model.md`](007-threat-model.md) on every distinct invocation of fallback handling triggered by `FWD-021` or `FWD-022`. The event MUST carry, at minimum: the matched rule's `zone` value and `match` mode; the ordered set of upstreams attempted; the failure outcome of each upstream attempt classified as either a transport-level failure type or a DNS-protocol RCODE; the rule's `fallback` value; and the client-visible outcome (the synthesised or propagated RCODE under `FWD-021`, or the indication that step 3 was invoked under `FWD-022`). This obligation extends the catalogue of structured events maintained by `THREAT-080`.
- **FWD-025.** Within a `[[forwarder.forward_zone]]` entry, the upstreams declared in the `upstream` array MUST be attempted in **declaration order** for the present query. Heimdall MUST attempt upstream entry `U_i` (the entry at index `i` in the `upstream` array) only after upstream entry `U_{i-1}` has either failed under `FWD-020` for the present query or has been recorded as broken in the upstream-broken cache under `FWD-026`. The first attempted upstream that returns a successful response under `FWD-019` terminates the failover sequence; the response is returned to the client per the regular forwarder pipeline.
- **FWD-026.** Heimdall MUST maintain an **upstream-broken cache** keyed by the position of the upstream entry within the rule (the index `i` of the `upstream` array entry). When an upstream attempt under `FWD-025` fails under `FWD-020`, the corresponding cache entry MUST be marked broken with a bounded Time-To-Live during which subsequent queries through the same rule MUST skip that upstream entry under `FWD-025`. The TTL MUST be a small finite value sufficient to cover transient failures (brief network glitches, certificate-rotation windows, transient upstream load) without persisting the skip across reasonable recovery windows; an operational default in the order of tens of seconds is appropriate. After the TTL expires, the next query through the same rule MUST re-attempt the upstream entry, allowing automatic recovery without operator intervention. The numeric default of the TTL is tracked as an open question in section 5.
- **FWD-027.** Each `[[forwarder.forward_zone]]` entry MAY carry the Boolean key `allow_transport_downgrade` whose value defaults to `false` when the key is absent. This key controls whether inter-upstream failover under `FWD-025` MAY transition from an encrypted-transport upstream (an upstream whose `transport` value is one of `"dot"`, `"doh"`, or `"doq"`) to a plain-transport upstream (an upstream whose `transport` value is one of `"udp"` or `"tcp"`) within the same query's failover sequence. The key has effect only on the inter-upstream sequence; it has no effect on the rule-level fallback decision under `FWD-021` and `FWD-022`, which is governed by the separate `fallback` key.
- **FWD-028.** When `allow_transport_downgrade = false`, failover under `FWD-025` MUST maintain a transport-class floor for the duration of a single query's failover sequence. Once any upstream attempted in the sequence has had `transport` value in the encrypted set (`"dot"`, `"doh"`, `"doq"`), every subsequent attempt in the same sequence MUST also have `transport` in the encrypted set. Any plain-transport upstream entry (`"udp"`, `"tcp"`) declared at a position later in the `upstream` array than an encrypted-transport entry that has been attempted MUST be skipped silently for the failover purpose, and the skip MUST emit a structured event under `FWD-031` indicating a blocked transport downgrade. The transport-class floor is per-query state, reset at the start of each new query's failover sequence; the floor MUST NOT be persisted across queries.
- **FWD-029.** When `allow_transport_downgrade = true`, failover under `FWD-025` proceeds without regard to transport class. Plain-transport upstream entries declared at positions later in the `upstream` array than encrypted-transport entries that have been attempted MAY be attempted as part of the failover sequence. The choice to enable transport downgrade is the operator's explicit declaration that availability is preferred to the privacy guarantee of the encrypted upstream attempt for the zones the rule covers; the trade-off is recorded by `FWD-031` for operator visibility on every actual downgrade transition.
- **FWD-030.** When the failover sequence under `FWD-025` has exhausted every eligible upstream entry of the rule — every entry has either been attempted and failed under `FWD-020`, has been skipped under `FWD-026` because of the upstream-broken cache, or has been skipped under `FWD-028` because of the transport-class floor — the rule-level fallback fixed by `FWD-021` (under `fallback = "error"`) or by `FWD-022` (under `fallback = "recursive"`) MUST be invoked. The two failover layers are sequential and ordered: inter-upstream failover under `FWD-025` runs first; the rule-level fallback under `FWD-021`/`FWD-022` runs only if the inter-upstream failover fails to produce a successful response.
- **FWD-031.** Every inter-upstream transition under `FWD-025` (every attempt of an upstream entry that was preceded by a failed or skipped earlier upstream entry in the same query), every entry created in the upstream-broken cache under `FWD-026`, every blocked transport downgrade under `FWD-028`, and every actual transport downgrade performed under `FWD-029` MUST emit a structured event under `THREAT-080` in [`007-threat-model.md`](007-threat-model.md) carrying, at minimum: the matched rule's `zone` value and `match` mode; the position of the upstream entry in the rule's `upstream` array; the `transport` value of the upstream entry; the failure category (transport-level failure type, DNS RCODE, broken-cache skip, blocked-downgrade skip, downgrade-permitted); the configured value of `allow_transport_downgrade` for the rule. This obligation extends the catalogue of structured events maintained by `THREAT-080`.
- **FWD-032.** Each `[[forwarder.forward_zone]]` entry MAY carry the string key `balance` whose value is exactly one of the three enumerated strings `"sequential"`, `"round-robin"`, or `"weighted"`. When the `balance` key is absent, the implicit value MUST be `"sequential"`. Any other value for `balance` MUST be rejected at configuration load. The `balance` value selects the load-distribution strategy applied to the initial upstream selection for each query that matches the rule; it does NOT affect the failover order under `FWD-025`, which remains sequential by declaration order regardless of the `balance` value once failover has begun (per `FWD-036`).
- **FWD-033.** In `"sequential"` mode (the default), the initial upstream for the present query MUST be the first **eligible** upstream of the rule, where an upstream entry `U_i` at position `i` in the `upstream` array is eligible if and only if `U_i` is not currently recorded as broken in the upstream-broken cache under `FWD-026` AND `U_i` satisfies the transport-class floor under `FWD-028` (when `allow_transport_downgrade = false`) for the present query. Eligibility is evaluated in declaration order; the first eligible upstream is the initial attempt. Queries through the rule do not rotate across upstreams while the first eligible upstream remains eligible.
- **FWD-034.** In `"round-robin"` mode, the initial upstream for each query MUST be selected from the eligible set (defined as in `FWD-033`) in round-robin order: a per-rule rotation index advances by one (modulo the cardinality of the eligible set) for each query, distributing queries uniformly across the eligible upstreams in the long run. The rotation index MUST be reset to zero whenever the cardinality or composition of the eligible set changes (an upstream entering or leaving the broken cache under `FWD-026`, or the transport-class floor changing across queries with different prior failover state).
- **FWD-035.** In `"weighted"` mode, each upstream entry in the rule's `upstream` array MAY carry an additional integer key `weight` whose value MUST be a positive integer (i.e., greater than or equal to 1; default value 1 if the key is absent). The initial upstream for each query MUST be selected from the eligible set (defined as in `FWD-033`) such that the long-run distribution of selections across upstreams converges to the proportions `weight_i / sum(weight_j over j eligible)`. Implementations MAY use deterministic weighted round-robin or randomised proportional selection; the long-run distribution is the normative invariant. A `weight` value of zero or a negative value MUST be rejected at configuration load.
- **FWD-036.** The `balance` value of a rule influences only the **initial** upstream selection for each query, not the inter-upstream failover that follows under `FWD-025`. Once an initial upstream has been selected under `FWD-033`, `FWD-034`, or `FWD-035`, and that upstream has failed under `FWD-020`, subsequent failover within the same query MUST proceed sequentially through the remaining eligible upstreams of the rule in declaration order, regardless of the rule's `balance` value. This rule preserves the deterministic-exhaustion property of `FWD-025` while allowing operators to distribute the **first** attempt across upstreams.
- **FWD-037.** Health-checking on the forwarder upstream side MUST be passive: upstream health state is derived exclusively from the outcomes of regular DNS queries that traverse the failover sequence under `FWD-025` and that are recorded in the upstream-broken cache under `FWD-026`. Heimdall MUST NOT issue active health-probe queries — synthetic DNS queries whose purpose is to test upstream availability outside the regular resolution path. Restoration of an upstream from the broken cache to the eligible set MUST happen automatically when the broken-cache TTL under `FWD-026` expires; the next query through the rule re-attempts the upstream, and a successful response removes the entry from the broken cache. Active probing is rejected for the same security and operational reasons as passive ADoT probing under `NET-032` in [`002-transports.md`](002-transports.md): probing traffic is outside the resolution path it serves, exposes a behavioural fingerprint to network observers, and creates a low-rate DoS pattern against upstreams that have not consented to probe traffic.
- **FWD-038.** Selection events under `FWD-033`, `FWD-034`, and `FWD-035` MUST NOT emit a per-query structured event under `THREAT-080`; emitting one event per query on the hot path would saturate the structured-event subsystem under realistic load. Selection statistics (per-upstream query count, per-upstream success and failure counts, current eligible-set cardinality, current rotation index for `"round-robin"`, current weighted-distribution snapshot for `"weighted"`) MUST be exposed as metrics under `THREAT-083` in [`007-threat-model.md`](007-threat-model.md) for operator visibility, sampled at the same cadence as the broader Heimdall metrics surface. The structured-event obligation under `FWD-031` (transitions, broken cache, blocked downgrades, permitted downgrades) is unaffected by the present requirement and continues to apply on every relevant non-hot-path event.

## 3. Rationale

The separation of forward-zone rules into a dedicated specification document under the `FWD-*` category keeps [`001-server-roles.md`](001-server-roles.md) focused on the role-activation model and the four-step precedence. Forward-zone rules are a substantial sub-surface with their own syntax, matching semantics, precedence algorithm, and ACL-composition rules; collapsing them into [`001-server-roles.md`](001-server-roles.md) would conflate two concerns that evolve independently and would increase the size of the role-model document beyond what is useful for the reader whose question is solely about role activation.

The three matching modes fixed by `FWD-004` cover the operator use cases on a specificity-to-breadth continuum. Exact mode is the most specific and matches only the zone apex; it is the right choice when an operator wants to forward a single name (for example, a delegation-only test domain). Wildcard mode follows the [RFC 1034 §4.3.3](https://www.rfc-editor.org/rfc/rfc1034#section-4.3.3) wildcard semantics of "exactly one label below the anchor" and is narrower than suffix but wider than exact. Suffix mode is the most common choice, matches the zone apex and all its descendants, and is therefore the default under `FWD-004` because it aligns with the operator expectation of "forward this whole zone". Making `"suffix"` the implicit default allows the simplest and most common rule to be expressed with the minimum number of keys.

The matching semantics in `FWD-006` through `FWD-008` are ASCII-case-insensitive per [RFC 4343](https://www.rfc-editor.org/rfc/rfc4343), consistent with DNS name-comparison semantics across the rest of the specification. The strict restriction on wildcard mode in `FWD-008` — exactly one additional label to the left, never zero and never two or more — preserves the bounded nature of the wildcard and distinguishes it from suffix mode in a way that makes the rank algorithm in `FWD-009` through `FWD-012` well-defined.

The rank-based precedence algorithm fixed by `FWD-009` through `FWD-012` produces a total ordering on the set of matching rules and therefore a deterministic selection. Mode priority as the lexicographically most-significant component of the rank tuple ensures that mode-based specificity dominates label-count-based specificity when the two conflict: an exact-mode rule for `example.com.` is the most specific possible rule for queries equal to that name, regardless of how many labels any concurrent suffix-mode rule might carry. Within a mode, zone_label_count expresses the longest-match-wins principle that is the operator experience on widely-deployed DNS software (Unbound's `forward-zone`, BIND's `forward`, dnsmasq's `--server=/domain/`). Declaration order as the tiebreak of last resort gives the operator a deterministic handle over residual ambiguity without introducing a priority field that would have to be manually maintained and whose divergence from declaration order would be a silent source of misconfiguration.

The uniform TOML shape fixed by `FWD-002` and `FWD-004` — three core keys (`zone`, `match`, `upstream`) on every entry, with a single enum `match` rather than sigil prefixes in the `zone` string, mutually exclusive `exact` / `suffix` / `wildcard` top-level keys, or tagged-union constructions — yields a homogeneous configuration surface across all rules. The loader validates the same keys on every entry, the unknown-key rejection of [`ROLE-021`](001-server-roles.md) applies uniformly, and the rule shape remains extensible in the future (for example, by the addition of an ACL-binding sub-key) without reshaping the existing rules.

The composition with the ACL matrix fixed by `FWD-015` makes views (`THREAT-046` in [`007-threat-model.md`](007-threat-model.md)) an emergent property rather than a separate subsystem. When multiple forward-zone rules share the same `zone` and `match` mode but bind to different ACLs, the ACL filter selects the subset of rules whose client-attribute constraints are satisfied by the query's client; the rank algorithm under `FWD-009` then selects among the remaining rules, with declaration order as the final tiebreak. This composition preserves the invariant established by `THREAT-046` that ACLs are the sole mechanism by which client attributes influence routing.

The fallback semantics fixed by `FWD-018` through `FWD-024` apply the fail-safe / fail-closed posture of [`../CLAUDE.md`](../CLAUDE.md) to the rule-to-recursive boundary. The default `fallback = "error"` ensures that an operator who declares a forward-zone rule with explicit upstreams gets exactly the routing outcome the rule expresses: queries for the zone go to the declared upstreams, and if the upstreams fail the failure is visible to the client rather than being silently re-routed through the recursive resolver. This default protects internal-only zones from inadvertent data exfiltration to public recursion when the internal upstream becomes unavailable. The opt-in `fallback = "recursive"` is the operator's explicit declaration that resilience to upstream failure is preferred over the fail-closed default for the zones the rule covers; this opt-in carries the structural pre-condition fixed by `FWD-023`, so that a rule presuming an active recursive role cannot be configured against an instance whose recursive role is disabled. The successful-outcome set fixed by `FWD-019` (NOERROR with records, NODATA, NXDOMAIN) and the upstream-failure set fixed by `FWD-020` (every other outcome, including SERVFAIL, REFUSED, transport-level failures) align with the operator semantics established by Unbound's `forward-first` and BIND's `forward first / forward only` directives, so that operators migrating from those products encounter no surprise in the rule-to-recursive failover behaviour. The transparent-forwarder propagation rule under `FWD-021` (transport failures synthesise SERVFAIL, RCODE-level failures propagate the upstream's RCODE unchanged) preserves the upstream's policy signal to the client where one exists (a REFUSED from the upstream is communicated as REFUSED to the client, not laundered into SERVFAIL), while ensuring that transport failures, where no upstream signal exists, produce a well-defined synthesised SERVFAIL.

The detailed rationale for the location, syntax, and precedence decisions, including the alternatives considered and the rejection grounds for each, is recorded in [`../docs/adr/0003-forward-zone-rule-syntax-and-precedence.md`](../docs/adr/0003-forward-zone-rule-syntax-and-precedence.md). The detailed rationale for the fallback decisions, including the alternatives considered and the rejection grounds for each, is recorded in [`../docs/adr/0004-forward-zone-fallback-on-upstream-failure.md`](../docs/adr/0004-forward-zone-fallback-on-upstream-failure.md).

## 5. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here to make the gap visible; several are tracked authoritatively in other specification files and are cross-referenced only for convenience.

- **ACL binding syntax on forward-zone rules.** The exact TOML shape of the ACL binding on a `[[forwarder.forward_zone]]` entry is bound to the "ACL configuration syntax" open question in [`007-threat-model.md §5`](007-threat-model.md) and will be fixed jointly with it.
- **DoH outbound HTTP version selection (HTTP/2 versus HTTP/3).** When `transport = "doh"` is declared on an upstream, the mechanism by which Heimdall selects HTTP/2 or HTTP/3 for the outbound connection — operator-declared per-upstream preference, ALPN-driven negotiation on a per-connection basis, or a fallback policy — is **to be specified**. `NET-012` in [`002-transports.md`](002-transports.md) requires that both HTTP versions be supported on the outbound DoH path, but the selection policy among them is not fixed.

Items previously tracked here that have been closed during sprint 1:

- **Upstream certificate validation policy** is closed by `SEC-047` through `SEC-059` in [`003-crypto-policy.md`](003-crypto-policy.md), with cross-references from `NET-018` and `NET-024` in [`002-transports.md`](002-transports.md).
- **Upstream fallback policy between configured upstreams** is closed by `FWD-025` through `FWD-031` of this document (inter-upstream failover, transport-class floor, broken cache).
- **Upstream load-balancing, failover, and health-checking** is closed by `FWD-032` through `FWD-038` of this document (`balance` enum: sequential / round-robin / weighted; passive health via `FWD-026`).
- **Outbound connection pooling, keepalive, and multiplexing** is closed by `NET-054` through `NET-061` in [`002-transports.md`](002-transports.md).
- **EDNS(0) Client Subnet on the upstream path** is closed by `NET-062` in [`002-transports.md`](002-transports.md), which formalises the strip-and-don't-forward behaviour required by `PROTO-017` and `PROTO-018` in [`006-protocol-conformance.md`](006-protocol-conformance.md).

No implementation activity may proceed on the basis of assumptions about any of the items above.
