# Network transports

**Purpose.** This document defines the set of DNS network transports that Heimdall supports, the invariant that governs when each transport listener is instantiated, and the matrix of (role, transport) combinations that Heimdall supports at the software level for incoming traffic. It also fixes, for the forwarder role, the set of outbound transports Heimdall uses when contacting configured upstream resolvers, and, for the recursive resolver role, the set of outbound transports Heimdall uses when contacting authoritative servers.

**Status.** Stable.

**Requirement category.** `NET`.

For the project-wide principles that frame these requirements (security non-negotiable, performance as the primary guide, "Assume Nothing"), see [`../CLAUDE.md`](../CLAUDE.md). For specification-wide conventions, see [`README.md`](README.md). For the role model this document depends on, see [`001-server-roles.md`](001-server-roles.md).

## 1. Supported transports

Heimdall supports exactly the following four DNS transports. Any given deployment combines a subset of them through configuration; no transport is always on and none is a default.

### 1.1 DNS classic over UDP and TCP on port 53

Traditional DNS carried over UDP and TCP on port 53, including the TCP requirements applicable to DNS servers and the EDNS(0) extension mechanism. Anchor RFCs: [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034), [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), [RFC 7766](https://www.rfc-editor.org/rfc/rfc7766), [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891).

### 1.2 DNS-over-TLS (DoT) on port 853

DNS messages carried over a TLS-protected TCP connection on port 853. Anchor RFCs: [RFC 7858](https://www.rfc-editor.org/rfc/rfc7858), [RFC 8310](https://www.rfc-editor.org/rfc/rfc8310).

### 1.3 DNS-over-HTTPS (DoH)

DNS messages carried as HTTPS requests and responses. Anchor RFC: [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484).

### 1.4 DNS-over-QUIC (DoQ) on port 853

DNS messages carried over QUIC on port 853. Anchor RFC: [RFC 9250](https://www.rfc-editor.org/rfc/rfc9250).

## 2. Normative requirements

- **NET-001.** Heimdall MUST support the four transports defined in section 1: DNS classic over UDP and TCP on port 53, DNS-over-TLS on port 853, DNS-over-HTTPS, and DNS-over-QUIC on port 853.
- **NET-002.** Heimdall MUST NOT implement DNSCrypt. DNSCrypt is explicitly out of scope.
- **NET-003.** DNS classic transport MUST conform to [RFC 1034](https://www.rfc-editor.org/rfc/rfc1034), [RFC 1035](https://www.rfc-editor.org/rfc/rfc1035), [RFC 7766](https://www.rfc-editor.org/rfc/rfc7766), and [RFC 6891](https://www.rfc-editor.org/rfc/rfc6891).
- **NET-004.** DNS-over-TLS transport MUST conform to [RFC 7858](https://www.rfc-editor.org/rfc/rfc7858) and [RFC 8310](https://www.rfc-editor.org/rfc/rfc8310) and MUST listen on port 853 when enabled.
- **NET-005.** DNS-over-HTTPS transport MUST conform to [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484).
- **NET-006.** DNS-over-HTTPS MUST be served over both HTTP/2 ([RFC 9113](https://www.rfc-editor.org/rfc/rfc9113)) and HTTP/3 ([RFC 9114](https://www.rfc-editor.org/rfc/rfc9114)).
- **NET-007.** HTTP version negotiation for DNS-over-HTTPS MUST use ALPN, and the server MUST advertise HTTP/3 via Alt-Svc where applicable.
- **NET-008.** DNS-over-QUIC transport MUST conform to [RFC 9250](https://www.rfc-editor.org/rfc/rfc9250) and MUST listen on port 853 when enabled.
- **NET-009.** A transport listener MUST be instantiated only when a configured role requires it. No transport listener is always on, and none is a default. This is the subordination of transports to role activation as defined by `ROLE-003`, `ROLE-004`, `ROLE-005`, `ROLE-006`, and `ROLE-007` in [`001-server-roles.md`](001-server-roles.md).
- **NET-010.** The set of active transports in an instance is determined exclusively by the roles enabled in its configuration and by the per-role transport bindings. There is no implicit default transport.
- **NET-011.** Heimdall MUST support, at the software level, every combination of (role, transport) for incoming traffic, where role is any of the three roles defined in [`001-server-roles.md`](001-server-roles.md) (authoritative server, recursive resolver, forwarder) and transport is any of the four transports defined in section 1 of this document (DNS classic over UDP and TCP on port 53, DNS-over-TLS on port 853, DNS-over-HTTPS, DNS-over-QUIC on port 853). Whether a given combination is active in a running instance is determined exclusively by configuration and remains subordinated to `NET-009` and to `ROLE-003` through `ROLE-007` in [`001-server-roles.md`](001-server-roles.md); no listener is instantiated for a combination whose role is not enabled.
- **NET-012.** Heimdall MUST support, on the outbound client-side of the forwarder role, every transport defined in section 1 of this document: DNS classic over UDP and TCP on port 53, DNS-over-TLS on port 853, DNS-over-HTTPS over both HTTP/2 and HTTP/3, and DNS-over-QUIC on port 853. This obligation is restricted to the forwarder role; outbound transport selection for the recursive resolver is tracked as open in section 5.
- **NET-013.** The outbound transport used to contact each configured upstream resolver MUST be declared on a per-upstream basis in the deployment configuration. There is no implicit default outbound transport, and no outbound transport is always on.
- **NET-014.** Outbound client code paths, sockets, connection pools, and TLS or QUIC client state for a given transport MUST NOT be instantiated unless at least one configured upstream in the running instance declares that transport. This is the outbound counterpart of `NET-009` and applies the same structural gating rule to the client side of the forwarder.
- **NET-015.** TLS-over-TCP outbound client connections used by the forwarder (DoT and DoH over HTTP/2) MUST negotiate TLS 1.3 only, consistent with `SEC-001`, `SEC-002`, and `SEC-003` in [`003-crypto-policy.md`](003-crypto-policy.md). Heimdall MUST NOT initiate, accept, or complete an outbound TLS handshake that negotiates any TLS version earlier than TLS 1.3 on these transports.
- **NET-016.** QUIC-based outbound client connections used by the forwarder (DoQ and DoH over HTTP/3) MUST use QUIC version 1 or QUIC version 2, and MUST NOT use QUIC draft versions, Google QUIC (gQUIC), or any other non-standard QUIC variant. This is consistent with `SEC-017`, `SEC-018`, and `SEC-019` in [`003-crypto-policy.md`](003-crypto-policy.md) and extends those requirements to the outbound direction.
- **NET-017.** Heimdall MUST NOT send TLS 1.3 early data (0-RTT) on TLS-over-TCP outbound client connections and MUST NOT send QUIC 0-RTT application data on QUIC-based outbound client connections on any outbound transport used by the forwarder. This extends to the outbound direction the server-side prohibitions established by `SEC-005` through `SEC-007` in [`003-crypto-policy.md`](003-crypto-policy.md) for TLS-over-TCP transports and by `SEC-022` through `SEC-024` in [`003-crypto-policy.md`](003-crypto-policy.md) for QUIC-based transports.
- **NET-018.** On every outbound TLS-protected or QUIC-based client connection used by the forwarder, Heimdall MUST validate the upstream server certificate. A handshake in which certificate validation fails MUST be aborted and MUST NOT be used to carry any DNS query or response. The precise validation policy — trust store scope, chain validation rules, hostname or SNI verification, optional SPKI pinning, revocation checking — is fixed by `SEC-047` through `SEC-059` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **NET-019.** Heimdall MUST support, on the outbound client-side of the recursive resolver role, DNS classic over UDP and TCP on port 53 as the mandatory baseline transport for contacting authoritative servers. Every authoritative contact attempt MUST be capable of falling back to DNS classic over UDP and TCP on port 53 when an encrypted alternative is unavailable or fails. The concrete fallback policy on handshake failure of an encrypted alternative is tracked as open in section 5.
- **NET-020.** Heimdall MUST support, on the outbound client-side of the recursive resolver role, Authoritative DNS-over-TLS (ADoT) as specified by [RFC 9539](https://www.rfc-editor.org/rfc/rfc9539) as an opt-in capability for contacting authoritative servers. ADoT MUST NOT be forced on any authoritative server. ADoT MUST be attempted against a specific authoritative server only when the resolver has positive, configured, or discovered evidence that that authoritative server supports it. The capability-discovery mechanism that produces such evidence is tracked as open in section 5.
- **NET-021.** Heimdall MUST NOT implement DNS-over-QUIC for authoritative contact ([RFC 9250, section 5.4](https://www.rfc-editor.org/rfc/rfc9250#section-5.4)) on the outbound client-side of the recursive resolver role at this stage. Heimdall MUST NOT implement Authoritative DNS-over-HTTPS (ADoH) on the outbound client-side of the recursive resolver role at this stage. Both transports are deliberately deferred and MAY be reconsidered in a future revision of this document; they are not rejected outright.
- **NET-022.** The TLS version policy applied to ADoT connections on the outbound client-side of the recursive resolver role is inherited from the outbound forwarder policy. Heimdall MUST negotiate TLS 1.3 only on ADoT and MUST NOT initiate, accept, or complete an outbound TLS handshake that negotiates any TLS version earlier than TLS 1.3. This obligation mirrors `NET-015` and is consistent with `SEC-001`, `SEC-002`, and `SEC-003` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **NET-023.** Heimdall MUST NOT send TLS 1.3 early data (0-RTT) on ADoT connections on the outbound client-side of the recursive resolver role. This obligation mirrors the client 0-RTT prohibition already established by `NET-017` for the outbound forwarder and extends to the outbound direction the server-side prohibitions in `SEC-005` through `SEC-007` in [`003-crypto-policy.md`](003-crypto-policy.md).
- **NET-024.** On every ADoT connection on the outbound client-side of the recursive resolver role, Heimdall MUST validate the authoritative server certificate. A handshake in which certificate validation fails MUST be aborted and MUST NOT be used to carry any DNS query or response. The precise validation policy is fixed by `SEC-047` through `SEC-059` in [`003-crypto-policy.md`](003-crypto-policy.md), as a single cross-role, cross-transport policy that applies uniformly to ADoT outbound connections from the recursive resolver and to outbound encrypted connections from the forwarder under `NET-018`.
- **NET-025.** The DoH server MUST implement the HTTP GET method and the HTTP POST method for DoH requests, as required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484). Heimdall MUST accept DoH requests on both methods and MUST NOT expose a configuration option that disables either method on a DoH listener. This applies to DoH over HTTP/2 and to DoH over HTTP/3.
- **NET-026.** The DoH server MUST use the media type `application/dns-message` for DoH request bodies and for DoH response bodies, as required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484). Heimdall MUST reject any DoH request whose declared media type is not `application/dns-message`, and MUST NOT emit any DoH response under a different media type. This applies to DoH over HTTP/2 and to DoH over HTTP/3.
- **NET-027.** The DoH URI path MUST be configurable per listener, and the default value MUST be `/dns-query`. A request whose path does not match the configured DoH URI path on a DoH listener MUST NOT be processed as a DoH request. This applies to DoH over HTTP/2 and to DoH over HTTP/3.
- **NET-028.** Heimdall MUST NOT implement any HTTP/2 or HTTP/3 feature or extension that is not required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484) for DoH at this stage.
- **NET-029.** ADoT capability discovery for an authoritative server, in support of `NET-020`, MUST be performed by querying the SVCB resource record at `_dns.<NS>.`, as defined by [draft-ietf-dnsop-svcb-dns](https://datatracker.ietf.org/doc/draft-ietf-dnsop-svcb-dns/), where `<NS>` is the canonical DNS name of the authoritative name server. The SVCB record's `TargetName`, the `alpn` SvcParam, and the `port` SvcParam together determine the encrypted-DNS transport profile that the authoritative server supports. When the SVCB record's `alpn` SvcParam carries the value `"dot"`, the authoritative server is considered to support ADoT for the purposes of `NET-020`.
- **NET-030.** An SVCB resource record discovered under `NET-029` MUST be DNSSEC-validated to the `secure` outcome (as defined by `DNSSEC-010` in [`005-dnssec-policy.md`](005-dnssec-policy.md), and per the validation pipeline fixed by `DNSSEC-008` through `DNSSEC-014` in [`005-dnssec-policy.md`](005-dnssec-policy.md)) before it is treated as the "positive, configured, or discovered evidence" of ADoT support required by `NET-020`. An SVCB record that validates as `bogus` (per `DNSSEC-013`), as `insecure` (per `DNSSEC-012`), or as `indeterminate` (per `DNSSEC-014`) MUST NOT be treated as evidence of ADoT support, MUST be discarded for the purposes of `NET-029`, and the authoritative server MUST be contacted via plain DNS over UDP and TCP per `NET-019`. The resolver-side verification pipeline applicable to the SVCB capability record is identical to the pipeline applicable to any other DNSSEC-protected RRset on the recursive resolver role: trust-anchor lookup, chain-of-trust construction from the trust anchor down to the SVCB owner name, signature verification at every step using the algorithm acceptance policy fixed by `DNSSEC-031` through `DNSSEC-040` in [`005-dnssec-policy.md`](005-dnssec-policy.md), and the validation-hardening mitigations fixed by `DNSSEC-041` through `DNSSEC-047` in [`005-dnssec-policy.md`](005-dnssec-policy.md) (KeyTrap signature-combination cap, RFC 9276 NSEC3 iteration cap). When the SVCB capability record's chain of trust does not lead to a configured trust anchor, the outcome is `indeterminate` and the record MUST NOT be treated as evidence; this case is operationally indistinguishable from `insecure` for the purposes of `NET-020`. This requirement closes the downgrade-protection question by binding ADoT discovery to mandatory DNSSEC validation: an on-path attacker that injects a forged SVCB record advertising ADoT support cannot influence Heimdall's outbound transport choice without also producing a valid DNSSEC signature chain rooted at a trust anchor that Heimdall has accepted, which is computationally infeasible without the corresponding zone signing key.
- **NET-031.** The recursive-role configuration MAY declare an operator-static override that designates specific authoritative name servers as supporting ADoT, independent of any SVCB record discovered under `NET-029`. The override MUST be expressed as an array of tables `[[recursive.adot_static]]` nested under the `[recursive]` role-activation table, in conformance with `ROLE-022` and `ROLE-023` in [`001-server-roles.md`](001-server-roles.md). Each entry MUST carry, at minimum, the string keys `ns_name` (the canonical DNS name of the authoritative name server, in the form required by `FWD-003` for DNS names) and `port` (TCP port, integer, default `853` if the key is absent). Each entry MAY carry the additional optional keys allowed by `ROLE-023` (TLS server-name override, SPKI pinning, trust anchor) under `SEC-047` through `SEC-059` in [`003-crypto-policy.md`](003-crypto-policy.md). An operator-static override entry is treated equivalently to a DNSSEC-validated SVCB record under `NET-029` and `NET-030` for the purposes of `NET-020`. Operator-static evidence does NOT require an associated DNSSEC chain because the configuration boundary is itself an authenticated, operator-controlled input gated by `ROLE-016` and `ROLE-021`.
- **NET-032.** When no DNSSEC-validated SVCB record under `NET-029` and `NET-030`, and no operator-static override under `NET-031`, declares ADoT support for a given authoritative name server, Heimdall MUST NOT attempt ADoT against that server. Contact MUST proceed via DNS over UDP and TCP per `NET-019`. Heimdall MUST NOT issue speculative ADoT connection attempts ("passive probing") in the absence of validated capability evidence; an ADoT connection attempt without prior evidence is prohibited because it both fails the "positive evidence" requirement of `NET-020` and exposes a behavioural fingerprint and a potential downgrade vector that mandatory-evidence discovery is designed to remove.
- **NET-033.** SVCB resource records used as ADoT capability evidence under `NET-029` MUST be cached according to the TTL semantics fixed by [RFC 1035 §4.2](https://www.rfc-editor.org/rfc/rfc1035#section-4.2). Negative answers (the absence of an SVCB resource record at `_dns.<NS>.`) MUST be cached according to the negative-caching rules of [RFC 2308](https://www.rfc-editor.org/rfc/rfc2308). Both positive and negative cache entries MUST honour the segregated-cache and bounded-TTL discipline fixed by [`004-cache-policy.md`](004-cache-policy.md). When the cache entry for an authoritative name server's SVCB record expires, Heimdall MUST re-perform discovery before the next ADoT attempt against that authoritative.
- **NET-034.** ADoT capability discovery under `NET-029` MUST be triggered on-demand when the recursive resolver needs to contact an authoritative name server for which neither a current SVCB cache entry (positive or negative) nor an operator-static override under `NET-031` is available. Discovery MUST NOT be performed eagerly (pre-fetched outside the resolution path), MUST NOT be performed for authoritative servers the resolver has no current intent to contact, and MUST NOT generate query traffic outside the authoritative-server resolution path it serves.
- **NET-035.** ADoT capability evidence under `NET-029` and `NET-031` is scoped per authoritative name server canonical name, not per zone. ADoT capability validated for the name server `<NS>` applies to every zone served by `<NS>` for as long as the underlying SVCB cache entry under `NET-033` remains valid or the operator-static override under `NET-031` remains configured. Discovery MUST NOT be repeated per-zone for the same authoritative name server.
- **NET-036.** An ADoT contact attempt against an authoritative name server `<NS>` is considered an **ADoT handshake failure** when any of the following occurs: a TCP-level failure preceding TLS (connection refused, connection reset, ICMP unreachable, network error, connection timeout); a TLS handshake timeout; a TLS protocol error during the handshake; an outbound server-certificate validation failure under any of `SEC-047` through `SEC-059` in [`003-crypto-policy.md`](003-crypto-policy.md); or a TLS handshake completion that yields a TLS state from which DNS data cannot be exchanged. A successful TLS handshake followed by a DNS-protocol-level failure (timeout on the DNS query / response, malformed DNS response, etc.) is NOT an ADoT handshake failure under this requirement; such failures are governed by the regular DNS-protocol resolution semantics.
- **NET-037.** On an ADoT handshake failure under `NET-036` against authoritative name server `<NS>`, Heimdall MUST proceed by attempting contact with the next authoritative name server `<Y>` in the zone's NS-set, in the order chosen by the recursive resolver's NS-selection algorithm. The next attempt against `<Y>` MUST be made via ADoT if `<Y>` has positive ADoT capability evidence under `NET-029` through `NET-031`, and via plain DNS over UDP and TCP per `NET-019` otherwise. Heimdall MUST NOT immediately retry plain DNS against the same `<NS>` that just failed ADoT; doing so would convert every ADoT handshake failure into an immediate downgrade of the encrypted-DNS contact for that authoritative.
- **NET-038.** An ADoT handshake failure under `NET-036` against `<NS>` MUST be recorded in an **ADoT-broken cache**: a per-`(<NS>, transport=ADoT)` negative-evidence entry with a bounded Time-To-Live during which no further ADoT attempt is made against `<NS>`. The cache TTL MUST be a small finite value sufficient to cover transient handshake failures (certificate-rotation windows, brief network glitches) without persisting the downgrade across reasonable recovery windows; an operational default in the order of a few minutes is appropriate. After the cache TTL expires, the next contact attempt against `<NS>` MUST re-attempt ADoT, allowing automatic recovery without operator intervention. The exact default TTL value is an operational parameter and is tracked as an open question in section 5; the existence of the cache and the bounded-TTL semantics are normative.
- **NET-039.** When every authoritative name server in the NS-set of a zone has either no positive ADoT capability evidence (and was therefore contacted via plain DNS per `NET-019` and `NET-037`) or has been recorded in the ADoT-broken cache under `NET-038`, **and** the recursive role's strict-ADoT mode under `NET-040` is not active, Heimdall MAY contact the zone's authoritative name servers via plain DNS over UDP and TCP per `NET-019` as a last-resort transport degradation. This last-resort plain contact MUST be recorded as a structured event under `NET-042`, and the response returned to the client MAY carry an EDNS(0) Extended DNS Errors INFO-CODE under `NET-041` if the response itself is a failure. The last-resort plain contact preserves availability for deployments that have not opted into strict-ADoT enforcement, in line with the operator's evident intent to forward through ADoT-capable authoritatives where possible without rendering the zone unreachable when every authoritative's encrypted endpoint is broken.
- **NET-040.** The recursive-role configuration MAY enable strict-ADoT mode through the Boolean key `strict_adot` nested under the `[recursive]` role-activation table (default `false` if the key is absent). When `strict_adot = true`, the last-resort plain contact authorised by `NET-039` MUST NOT be performed; instead, when every authoritative name server in the NS-set of the zone has either no positive ADoT capability evidence or has been recorded in the ADoT-broken cache, Heimdall MUST return SERVFAIL (RCODE 2) to the client with an Extended DNS Errors INFO-CODE under `NET-041`. Strict-ADoT mode is the operator's explicit declaration that encrypted-DNS contact is preferred over availability for the zones the recursive resolver serves; the trade-off is intentional.
- **NET-041.** When Heimdall returns a SERVFAIL response to the client because the resolution of a query has been blocked by ADoT-handshake failures (every authoritative name server in the NS-set ADoT-capable but ADoT-broken, with strict-ADoT mode active under `NET-040`; or every authoritative name server unreachable, with strict-ADoT mode either active or inactive), Heimdall MUST include an EDNS(0) Extended DNS Errors option per `PROTO-012` in [`006-protocol-conformance.md`](006-protocol-conformance.md) carrying INFO-CODE `22` ("No Reachable Authority") as defined by the IANA Extended DNS Error Codes registry under [RFC 8914](https://www.rfc-editor.org/rfc/rfc8914), regardless of which transport (encrypted or plain) was last attempted. When the SERVFAIL is the consequence of strict-ADoT mode preventing a plain-DNS fall-through that would otherwise have been performed, Heimdall MAY additionally include INFO-CODE `23` ("Network Error") on the same response, in conformance with the IANA registry semantics for that code; the dual-EDE encoding informs the client that the failure is operator-configured at the strict-ADoT boundary rather than a true unreachability event. A successful response that arrives via the last-resort plain contact under `NET-039` MUST NOT carry an EDE INFO-CODE indicating failure; the response is propagated to the client unchanged, and the operator-visible signal of the transport degradation is delivered through the structured event under `NET-042`.
- **NET-042.** Every ADoT handshake failure under `NET-036`, every ADoT-broken cache entry created under `NET-038`, every last-resort plain contact under `NET-039`, and every strict-ADoT-mode SERVFAIL produced under `NET-040` MUST emit a structured event under `THREAT-080` in [`007-threat-model.md`](007-threat-model.md) carrying, at minimum: the authoritative name server's canonical name and address; the zone the contact was made for; the failure category (transport-level, TLS protocol error, certificate validation failure under one of the categories enumerated in `SEC-059`, or strict-ADoT-mode policy denial); the resolver's chosen next action (next-NS attempt under `NET-037`, plain-DNS last-resort under `NET-039`, or SERVFAIL response under `NET-040`); and the configured value of `strict_adot` for the running instance. This obligation extends the catalogue of structured events maintained by `THREAT-080`.
- **NET-043.** Heimdall MAY maintain a client-side TLS 1.3 session-ticket cache for outbound ADoT connections, to enable session resumption per [RFC 8446 §2.2](https://www.rfc-editor.org/rfc/rfc8446#section-2.2) and to reduce the asymmetric-cryptography cost and the additional round trip of a full handshake on subsequent contacts to the same authoritative. The cache MUST be scoped per `(<NS>, <IP>)` pair, where `<NS>` is the authoritative name server's canonical DNS name and `<IP>` is the IPv4 or IPv6 address that Heimdall connects to for the cached session. A ticket cached for the pair `(N_a, I_x)` MUST NOT be presented on an outbound TLS handshake to a different pair (`(N_a, I_y)` with a different IP, or `(N_b, I_x)` with a different name server). This per-`(NS, IP)` scoping is conservative against authoritative deployments in which different IPs of the same NS may be served by independent TLS terminations with non-shared Ticket-Encryption-Keys.
- **NET-044.** A cached session ticket's effective TTL MUST be the smaller of: the `ticket_lifetime` field of the NewSessionTicket message that delivered it ([RFC 8446 §4.6.1](https://www.rfc-editor.org/rfc/rfc8446#section-4.6.1), upper-bounded by the protocol at 7 days); and an operator-configurable maximum ticket-reuse window. The numeric default of the operator-configurable maximum is tracked as an open question in section 5 and MUST NOT be assumed before it is specified. The cap exists to bound the maximum reuse window of any single ticket independently of the server-issued lifetime, in line with the bounded-state discipline of section 3.
- **NET-045.** The client-side session-ticket cache MUST be bounded in size by two enforced caps: a per-`(NS, IP)`-pair cap on the number of tickets stored for that pair (which permits an authoritative to issue several tickets to support rotation and parallel sessions); and a global cap on the number of `(NS, IP)` pairs the cache may simultaneously hold. Eviction across either cap MUST follow Least-Recently-Used (LRU) semantics. The numeric defaults of the per-pair and global caps are tracked as open questions in section 5 and MUST NOT be assumed before they are specified.
- **NET-046.** A cached session ticket MUST be discarded — and therefore MUST NOT be presented on any subsequent outbound TLS handshake — under any of the following invalidation triggers: (a) on an outbound TLS handshake against the cached `(NS, IP)` pair, the SHA-256 digest of the leaf certificate's SubjectPublicKeyInfo as fixed by `SEC-054` in [`003-crypto-policy.md`](003-crypto-policy.md) differs from the digest recorded when the ticket was cached; (b) the authoritative server returns a TLS `decrypt_error` alert ([RFC 8446 §6.2](https://www.rfc-editor.org/rfc/rfc8446#section-6.2)) attributable to the presented ticket, indicating that the server has refused the ticket (most commonly because its Ticket-Encryption-Key has rotated past the acceptance window); (c) an explicit operator-issued invalidation through the admin-RPC surface fixed by the runtime-operations policy in [`012-runtime-operations.md`](012-runtime-operations.md). On invalidation under (a) or (b), Heimdall MUST proceed with a full TLS 1.3 handshake; the structured-event obligation under `NET-042` MUST be evaluated for any handshake outcome under that fallback path.
- **NET-047.** The client-side session-ticket cache fixed by `NET-043` through `NET-046` is independent from the server-side session-ticket and Ticket-Encryption-Key policy fixed by `SEC-008` through `SEC-011` in [`003-crypto-policy.md`](003-crypto-policy.md). The server-side policy governs the encryption keys Heimdall uses when issuing tickets to clients on its inbound listeners; the client-side cache stores opaque tickets received from authoritatives that Heimdall as a client cannot decrypt. A rotation of an authoritative's TEK is observable to Heimdall only through the `decrypt_error` invalidation trigger under `NET-046` (b), not through any shared state. The two policies MUST NOT be combined, and the same cache surface MUST NOT serve both directions.
- **NET-048.** Heimdall MUST maintain a per-`(<NS>, <IP>, <transport>)` outbound connection pool for TCP-based DNS contact from the recursive resolver role to authoritative name servers, where `<transport>` is exactly one of the two values `"tcp"` (DNS classic on port 53) or `"dot"` (DNS-over-TLS on port 853, used for ADoT). The pool MUST be bounded by two enforced caps: a per-pool cap on the number of concurrent connections held to a given `(NS, IP, transport)` triple; and a global cap on the total number of outbound connections held across all pools. Eviction MUST follow oldest-idle-first semantics. When a query needs an outbound connection and the pool for the relevant triple is at its cap with no idle connection available, the query MUST wait in a per-pool FIFO queue for an existing connection to free up or for a new connection to become available below the cap; the queue MUST itself be bounded against unbounded growth. The numeric defaults for the per-pool cap, the global cap, and the queue depth are operational defaults and are tracked as an open question in section 5.
- **NET-049.** Every connection held in the outbound pool fixed by `NET-048` MUST be subject to an idle-keepalive timeout. A connection that has been idle (no in-flight queries and no pending queries enqueued for transmission) for longer than the timeout MUST be closed and removed from the pool. The default idle-keepalive timeout is an operational default and is tracked as an open question in section 5; the existence of the timeout, the bounded-idle-state property, and the closure-on-expiry behaviour are normative.
- **NET-050.** Heimdall MUST issue every TCP-based outbound query (plain TCP and DoT) carrying the EDNS(0) edns-tcp-keepalive option as fixed by `PROTO-014` in [`006-protocol-conformance.md`](006-protocol-conformance.md), in conformance with [RFC 7828](https://www.rfc-editor.org/rfc/rfc7828), to signal Heimdall's support for connection reuse. When a response received on a pooled connection carries the edns-tcp-keepalive option, the `TIMEOUT` field of the option (in units of 100 ms per RFC 7828) MUST replace the default idle-keepalive timeout under `NET-049` for the remainder of that connection's lifetime. A `TIMEOUT` value of `0` MUST be interpreted as the server's request to close the connection immediately after the in-flight query completes; Heimdall MUST honour the request and remove the connection from the pool after the response is delivered to the resolver pipeline.
- **NET-051.** Connection reuse on pooled connections MUST permit pipelining: multiple outstanding DNS queries on a single TCP or DoT connection, each correlated to its response via the DNS Header `ID` field per [RFC 7766 §6.2.1](https://www.rfc-editor.org/rfc/rfc7766#section-6.2.1) for TCP and the equivalent framing for DoT. The number of in-flight queries permitted per pooled outbound connection MUST be bounded by a configurable per-connection cap. This cap is the outbound counterpart of the inbound per-connection in-flight cap fixed by `THREAT-063` in [`007-threat-model.md`](007-threat-model.md); its numeric default is an operational default and is tracked as an open question in section 5. A query that would push the per-connection cap above the bound MUST instead be queued under `NET-048` until an in-flight slot becomes available on the connection or another connection becomes available.
- **NET-052.** A pooled connection MUST be removed from the pool and closed under any of the following conditions: TCP RST or FIN received from the authoritative on the connection; TLS protocol error on a DoT connection (the same trigger class as the ticket-invalidation case under `NET-046` (b)); idle-keepalive timeout expiry under `NET-049`, including expiry under the timeout signalled via edns-tcp-keepalive under `NET-050`; explicit operator-issued connection drain via the admin-RPC surface fixed by [`012-runtime-operations.md`](012-runtime-operations.md). Queries that were in-flight on a connection closed under any of these conditions MUST be re-issued on a freshly opened connection from the same pool, subject to the recursive resolver's standard retry budget; queries that had already received a response on the closed connection MUST NOT be re-issued.
- **NET-053.** The outbound connection pool fixed by `NET-048` through `NET-052` is the outbound counterpart of the inbound resource-limit and admission-control family fixed by `THREAT-061` through `THREAT-078` in [`007-threat-model.md`](007-threat-model.md). The per-pool cap of `NET-048` mirrors `THREAT-062` (maximum concurrent connections per `(transport, listener)`) on the client side; the per-connection in-flight cap of `NET-051` mirrors `THREAT-063` (maximum in-flight queries per connection); the slow-server discipline implicit in `NET-049` and `NET-050` mirrors the slow-client disconnection requirement of `THREAT-068`. The two families share the structured-event taxonomy of `THREAT-080`: pool-overflow events (a query queued because the global cap is reached), idle-timeout closures, slow-server disconnections, and admin-RPC drains MUST emit structured events for operator visibility.
- **NET-054.** The forwarder's outbound connection pool MUST be scoped per `[[forwarder.forward_zone.upstream]]` entry: each upstream entry has its own pool, independent from the pool of any other entry, even when two entries declare the same `address`, `port`, and `transport`. The chosen scope is more granular than the recursive resolver's per-`(NS, IP, transport)` scope under `NET-048` because the forwarder operates on operator-declared upstream entries each potentially carrying distinct trust anchor, mTLS material, SPKI pinning, server name override, weight, and other sub-keys; the transport state of each entry is consequently distinct from sibling entries.
- **NET-055.** The connection-pool model for a forwarder upstream entry depends on the entry's `transport` value as follows. (a) `"udp"`: NO connection pool is maintained; UDP is connectionless under DNS classic semantics, and each query is a fire-and-forget UDP datagram subject to the standard DNS retry and timeout semantics. (b) `"tcp"` and `"dot"`: a connection pool with TCP-level pipelining per [RFC 7766](https://www.rfc-editor.org/rfc/rfc7766) MUST be maintained; the semantics of `NET-048` through `NET-053` apply with the per-`(NS, IP, transport)` scope of `NET-048` replaced by the per-upstream-entry scope of `NET-054`. (c) `"doh"`: an HTTP/2 or HTTP/3 connection pool with HTTP-stream multiplexing MUST be maintained, where the HTTP version is the one negotiated per `NET-006`. (d) `"doq"`: a QUIC connection pool with QUIC-stream multiplexing MUST be maintained, where each DoQ query / response uses its own bi-directional QUIC stream as fixed by [RFC 9250 §4.2](https://www.rfc-editor.org/rfc/rfc9250#section-4.2).
- **NET-056.** The forwarder's outbound connection pool under `NET-055` MUST be bounded by enforced caps: a per-pool cap on the number of concurrent connections held to a single upstream entry; a per-connection cap on the number of concurrent in-flight streams (for HTTP/2, HTTP/3, and DoQ transports) or in-flight queries (for `"tcp"` and `"dot"` pipelining); and a global cap on the total number of outbound forwarder connections held across all pools. For HTTP/2 connections, the per-connection in-flight stream cap MUST additionally respect the server's `SETTINGS_MAX_CONCURRENT_STREAMS` setting per [RFC 9113 §6.5.2](https://www.rfc-editor.org/rfc/rfc9113#section-6.5.2): the effective cap MUST be the smaller of the server-advertised value and the operator-configured client-side cap. A bounded FIFO queue for queries waiting on connection or stream availability MUST be maintained per pool. Eviction MUST follow oldest-idle-first semantics across pools. The numeric defaults of all caps and the queue depth are operational parameters and are tracked as an open question in section 5.
- **NET-057.** Idle keepalive for forwarder outbound connections is governed per transport. For `"tcp"` and `"dot"`: the policy of `NET-049` and `NET-050` (default operational TTL with edns-tcp-keepalive override under [RFC 7828](https://www.rfc-editor.org/rfc/rfc7828)) applies. For `"doh"` over HTTP/2 and HTTP/3 and for `"doq"`: a default operational idle timeout MUST apply, and Heimdall MAY use HTTP/2 PING frames ([RFC 9113 §6.7](https://www.rfc-editor.org/rfc/rfc9113#section-6.7)), the HTTP/3 PING-equivalent control mechanism, or QUIC PING frames ([RFC 9000 §19.2](https://www.rfc-editor.org/rfc/rfc9000#section-19.2)) to detect connection liveness during long idle periods. A connection that exceeds its idle timeout without producing liveness evidence MUST be closed and removed from the pool. The numeric default of the idle timeout for HTTP/2, HTTP/3, and QUIC outbound connections is an operational parameter and is tracked as an open question in section 5.
- **NET-058.** Multiplexing on stream-based forwarder outbound transports MUST be bounded as follows. For `"doh"` over HTTP/2: the number of concurrent HTTP/2 streams per connection MUST be bounded by `SETTINGS_MAX_CONCURRENT_STREAMS` advertised by the server AND by an operator-configurable client-side cap, whichever is smaller. For `"doh"` over HTTP/3: the number of concurrent HTTP/3 bidirectional streams per QUIC connection MUST be bounded by the QUIC `initial_max_streams_bidi` transport parameter advertised by the server AND by an operator-configurable client-side cap. For `"doq"`: the number of concurrent QUIC bidirectional streams per connection MUST be bounded by `initial_max_streams_bidi` AND by an operator-configurable client-side cap. The client-side cap numeric defaults are operational parameters and are tracked as an open question in section 5.
- **NET-059.** The client-side TLS / QUIC session-ticket cache for forwarder outbound encrypted connections (`"dot"`, `"doh"`, `"doq"`) MUST be maintained per the policy of `NET-043` through `NET-047`, with the per-`(NS, IP)` scope of `NET-043` replaced by the per-`(upstream-entry, IP)` scope appropriate to the forwarder's operator-declared upstream model. The ticket cache for one upstream entry MUST NOT serve another upstream entry, even when both entries declare the same `address` and `port`. The independence from the server-side TEK policy fixed by `SEC-008` through `SEC-011` in [`003-crypto-policy.md`](003-crypto-policy.md), as fixed by `NET-047`, applies to the forwarder ticket cache uniformly.
- **NET-060.** Invalidation triggers for pooled forwarder outbound connections, per transport: for `"tcp"` and `"dot"`, TCP RST or FIN, TLS protocol error, idle-keepalive expiry, and admin-RPC drain (per `NET-052`); for `"doh"` over HTTP/2, any of the preceding triggers, plus an HTTP/2 GOAWAY frame from the server ([RFC 9113 §6.8](https://www.rfc-editor.org/rfc/rfc9113#section-6.8)) — on GOAWAY, the connection MUST be removed from the pool after currently-in-flight streams complete; for `"doh"` over HTTP/3 and for `"doq"`, the QUIC analogues of TCP-level triggers (CONNECTION_CLOSE per [RFC 9000 §19.19](https://www.rfc-editor.org/rfc/rfc9000#section-19.19), abrupt application-level close), TLS protocol error within the QUIC TLS handshake, idle-keepalive expiry, and admin-RPC drain. Pending queries on a connection that is closed under any of these triggers MUST be re-issued on a freshly opened connection from the same pool, subject to the standard retry budget.
- **NET-061.** The structured-event obligation under `NET-053` (pool overflow, idle-timeout closure, slow-server disconnection, admin-RPC drain) applies uniformly to forwarder outbound connections, with the following additional event types specific to stream-based transports: HTTP/2 GOAWAY received from the server; QUIC CONNECTION_CLOSE received from the server; per-connection multiplexing cap reached (a query queued because the per-connection stream cap is at the limit). All such events MUST emit structured events under `THREAT-080` in [`007-threat-model.md`](007-threat-model.md) carrying, at minimum, the upstream-entry identifier, the transport, the failure or saturation category, and the resolver's chosen next action. Features and extensions that are explicitly out of scope under this requirement include, but are not limited to, HTTP/2 server push, HTTP/2 and HTTP/3 priorities ([RFC 9218](https://www.rfc-editor.org/rfc/rfc9218)), the HTTP CONNECT method, WebTransport extensions, and HTTP/3 datagrams ([RFC 9297](https://www.rfc-editor.org/rfc/rfc9297)). ALPN-based HTTP version negotiation and Alt-Svc advertisement for HTTP/3 are governed by `NET-006` and `NET-007` and are not restated here. This applies to DoH over HTTP/2 ([RFC 9113](https://www.rfc-editor.org/rfc/rfc9113)) and to DoH over HTTP/3 ([RFC 9114](https://www.rfc-editor.org/rfc/rfc9114)).

## 4. Rationale

Each transport has a single implementation shared across all three roles; restricting which roles may bind which transports would not remove any code from the binary, while it would introduce configuration-level exceptions that must be specified, implemented, tested, and documented. The runtime attack surface is already constrained by the role-activation invariant (`ROLE-003` through `ROLE-007`) and by the listener-subordination rule (`NET-009`), so a disabled role exposes no listener regardless of the matrix. Authoritative service over encrypted transports is active IETF practice, including Authoritative DNS-over-TLS ([RFC 9539](https://www.rfc-editor.org/rfc/rfc9539)) and Authoritative DNS-over-QUIC ([RFC 9250, section 8](https://www.rfc-editor.org/rfc/rfc9250#section-8)); excluding those combinations would diverge from current practice without a corresponding security benefit. A uniform matrix is therefore simpler to specify, implement, test, and document than one carrying exceptions.

Requiring the forwarder to support all four outbound transports (`NET-012`) reflects the operational coverage the role must provide in 2026. Privacy-preserving and zero-trust deployments require encrypted upstream DNS, and the major upstream resolvers that those deployments typically target — Cloudflare, Google, Quad9, NextDNS, AdGuard — primarily expose DoT and DoH, with DoQ emerging. A forwarder limited to DNS classic would exclude the most relevant contemporary use cases and would push operators back onto unencrypted upstream paths. The marginal cost of supporting the full set is zero with respect to dependencies: every protocol stack is already present in the binary to satisfy the inbound-transport decisions in section 2, and the outbound client side reuses the same cryptographic and protocol primitives. Declaring the outbound transport per upstream in configuration (`NET-013`) keeps the choice explicit and local to each upstream rather than implied by a global default, which reduces the likelihood of an unintended downgrade from an encrypted to an unencrypted upstream path. Gating outbound client state structurally on configuration (`NET-014`) extends to the outbound direction the surface-reduction approach already applied to role activation in [`001-server-roles.md`](001-server-roles.md) and to inbound transport listeners by `NET-009`: client sockets, connection pools, and TLS or QUIC state for transports that no configured upstream actually uses are absent from the reachable binary rather than guarded by runtime flags. Inheriting the inbound cryptographic policy onto the outbound side (`NET-015`, `NET-016`, `NET-017`) avoids divergent inbound and outbound semantics for the same protocols, which would be a latent source of incidents; the same reasoning that fixed TLS 1.3 only, QUIC v1 and v2 only, and no 0-RTT on the server side applies unchanged when Heimdall acts as a client to an upstream DNS server. Requiring outbound server-certificate validation (`NET-018`) at the specification level fixes the non-negotiable property now and defers only the calibration of the validation policy, so no outbound flow can ever be established against an unauthenticated upstream even before the detailed validation rules are agreed.

Restricting the DoH server to the minimum HTTP feature surface required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484) (`NET-025`, `NET-026`, `NET-027`, `NET-028`) follows the security and performance principles of [`../CLAUDE.md`](../CLAUDE.md). DoH is a simple request/response protocol; no HTTP extension adds functional value beyond what RFC 8484 specifies. HTTP/2 server push is formally deprecated in major clients and is outside the DoH interaction model. HTTP/2 and HTTP/3 priorities ([RFC 9218](https://www.rfc-editor.org/rfc/rfc9218)) offer marginal benefit for a DoH workload in which queries are small, independent, and fast; implementing them would introduce parser, state, and ordering cost without measurable benefit. The HTTP CONNECT method, WebTransport extensions, and HTTP/3 datagrams ([RFC 9297](https://www.rfc-editor.org/rfc/rfc9297)) are outside the DoH model; implementing them would introduce attack surface and expand the test matrix without a legitimate use case. A minimal feature surface reduces fingerprintability, shrinks the test matrix, and lowers the risk of feature-interaction defects. Cross-referencing `NET-006` and `NET-007` for ALPN and Alt-Svc rather than restating them keeps the DoH HTTP-conformance rules localised to a single decision and avoids duplication drift between requirements.

The ADoT fallback policy on handshake failure fixed by `NET-036` through `NET-042` resolves the question of what Heimdall does when an ADoT attempt against an authoritative — chosen because the resolver had positive evidence of capability under `NET-029` through `NET-031` — fails at the transport or TLS layer. The chosen design preserves the encrypted-DNS contact intent as long as practicable (NS-iteration before plain fallback under `NET-037`), bounds the downgrade window through a short ADoT-broken cache under `NET-038` that auto-recovers without operator intervention, admits a last-resort plain-DNS contact under `NET-039` for deployments that have not opted into strict-ADoT enforcement, and offers operators an explicit strict-ADoT mode under `NET-040` for deployments that prefer encrypted-DNS-or-failure to encrypted-or-degraded. The structural-event obligation under `NET-042` ensures every transport degradation and every strict-mode denial is observable, so that operators detect persistent ADoT failures (broken authoritative TLS configuration, mass certificate-validation issues, downgrade-attack patterns) without relying on client-side complaints. The EDE encoding under `NET-041` distinguishes a true "no reachable authority" outcome (every NS unreachable across both transports) from a strict-ADoT-mode policy denial (encrypted unavailable but plain might have worked); the dual-EDE option for the strict-mode-denial case informs the client that the failure is configurable at the operator boundary, which improves diagnosability without leaking strict-mode policy state to adversaries (a strict-mode-denial response already requires the adversary to have suppressed every encrypted endpoint, which presumes transport-layer capability, so the EDE distinction does not aid further attack). Rejecting an immediate same-NS plain-DNS retry on the failed `<NS>` (the option that `NET-037` explicitly denies) is the linchpin of the policy: an immediate plain retry on the same `<NS>` would convert every transient ADoT failure into a silent transport downgrade against that authoritative, which is precisely the failure mode the entire ADoT-discovery + capability-evidence + DNSSEC-validation chain (`NET-020`, `NET-029`, `NET-030`) was designed to prevent. The detailed rationale for the choice and the alternatives considered is recorded in [`../docs/adr/0009-adot-fallback-policy.md`](../docs/adr/0009-adot-fallback-policy.md).

The ADoT capability-discovery mechanism fixed by `NET-029` through `NET-035` selects [draft-ietf-dnsop-svcb-dns](https://datatracker.ietf.org/doc/draft-ietf-dnsop-svcb-dns/) as the IETF-aligned discovery path for resolver-to-authoritative encrypted-DNS capabilities, supplemented by an operator-static override for authoritatives that have not yet published SVCB records but whose ADoT support is known to the operator. The selection rests on four axes acting together. First, alignment with IETF direction: SVCB-DNS is the standardisation track for resolver-to-authoritative encrypted-DNS discovery, and committing to it now lets Heimdall benefit from the growing publication base over the deployment lifetime of the binary, without paying a migration cost when adoption matures. Second, downgrade protection by construction: the mandatory DNSSEC validation under `NET-030` makes the discovered evidence cryptographically authenticated against the resolver's trust anchor, removing the on-path attacker's ability to spoof an SVCB advertisement and thereby coerce ADoT-capable resolvers into an attacker-chosen transport profile. Third, refusal of passive probing under `NET-032`: speculative ADoT connection attempts without prior evidence both fail the positive-evidence requirement of `NET-020` and expose a behavioural fingerprint (every cold contact attempts ADoT and falls back), introduce a downgrade vector against attackers who block port 853, and inflate cold-connection latency without compensating gain. Fourth, operator-static escape hatch under `NET-031`: where SVCB has not yet been published by an authoritative whose ADoT support is otherwise known, the operator-controlled configuration boundary fixed by `ROLE-016` and `ROLE-021` substitutes for the DNSSEC-authenticated discovery channel — the configuration boundary is itself an authenticated, operator-controlled input. The cache lifecycle under `NET-033` and the on-demand triggering under `NET-034` reuse the resolver-cache infrastructure of [`004-cache-policy.md`](004-cache-policy.md) and follow the standard DNS-cache TTL discipline, so the spec introduces no new cache-management concept for capability evidence. The per-NS scoping under `NET-035` matches the unit at which encrypted-DNS endpoints are configured at authoritative deployments — an authoritative name server, not a zone — and avoids duplicate discovery work for every zone served by a name server. The detailed rationale for the choice and the alternatives considered is recorded in [`../docs/adr/0008-adot-capability-discovery.md`](../docs/adr/0008-adot-capability-discovery.md).

The outbound transport set fixed for the recursive resolver role in `NET-019` through `NET-024` reflects the deployment reality of the authoritative ecosystem in 2026 and the asymmetry between resolver-to-upstream-resolver traffic and resolver-to-authoritative traffic. Requiring DNS classic over UDP and TCP on port 53 as the mandatory baseline (`NET-019`) is non-negotiable for the same reason the role itself is: the overwhelming majority of authoritative servers on the public Internet in 2026 speak only classical DNS, and a recursive resolver without this baseline cannot complete iterative resolution through the zone hierarchy. Supporting ADoT as an opt-in, never-forced capability (`NET-020`) is the specification-level commitment to follow [RFC 9539](https://www.rfc-editor.org/rfc/rfc9539), which is the official IETF direction for resolver-to-authoritative privacy and which is actively being adopted by large public resolvers; tying ADoT attempts to positive evidence of support avoids introducing query-level failures or latency regressions against the (still-dominant) set of authoritative servers that do not speak ADoT. Deferring DoQ for authoritative ([RFC 9250, section 5.4](https://www.rfc-editor.org/rfc/rfc9250#section-5.4)) and Authoritative DoH (ADoH) (`NET-021`) is a scope decision, not a rejection: their residual deployment among authoritative servers in 2026 does not justify the additional discovery, fallback, and hardening surface they would require on the resolver outbound side, and the TLS 1.3 client required for ADoT is already present in the binary from the outbound forwarder decisions, whereas a DoQ or ADoH authoritative client would expand the outbound surface further without a corresponding return today. Inheriting the outbound cryptographic policy already fixed for the forwarder onto the recursive-resolver outbound side (`NET-022`, `NET-023`) avoids divergent outbound semantics for the same TLS 1.3 stack: the reasoning that fixed TLS 1.3 only and no client 0-RTT for the outbound forwarder applies unchanged when Heimdall, as a recursive resolver, initiates ADoT to an authoritative server. Requiring certificate validation on every ADoT connection (`NET-024`) at the specification level fixes the non-negotiable property now and defers only the calibration of the validation policy, jointly with the analogous forwarder item under `NET-018`, so no outbound ADoT flow can ever be established against an unauthenticated authoritative even before the detailed validation rules are agreed.

## 5. Open questions

The following items are **not yet decided** and MUST NOT be assumed. They are listed here because they are directly downstream of the decisions in this file and will need to be specified before the implementation of these transports can begin.

- **ADoT-broken cache TTL default.** The exact default Time-To-Live of the per-`(<NS>, ADoT)` ADoT-broken cache entries fixed by `NET-038` is **to be specified**. The current normative content fixes the existence of the cache and the bounded-TTL property; the numeric default (in the order of a few minutes) and the configuration knob to override it remain to be specified.
- **ADoT session-ticket cache numeric defaults.** The maximum ticket-reuse window per `NET-044`, the per-`(NS, IP)`-pair ticket cap per `NET-045`, and the global cap on `(NS, IP)` pairs in the cache per `NET-045` are **to be specified**. The current normative content fixes the existence of the cache, its scoping (per `(NS, IP)` pair), its TTL behaviour (smaller of `ticket_lifetime` and operator-configurable cap), its bounded LRU eviction, and its invalidation triggers; only the numeric defaults of the operator-configurable parameters remain to be specified.
- **Outbound connection pool numeric defaults (recursive resolver outbound).** The numeric defaults of the per-pool cap, the global cap, the per-pool queue depth, the idle-keepalive timeout, and the per-connection in-flight cap fixed by `NET-048`, `NET-049`, and `NET-051` are **to be specified**. The current normative content fixes the structure of the pool (per-`(NS, IP, transport)` scoping, bounded sizes, oldest-idle eviction, FIFO queue), the idle-keepalive timeout's existence and its edns-tcp-keepalive override under `NET-050`, the pipelining semantics, and the invalidation triggers; only the numeric defaults of the operational parameters remain to be specified.
- **Inter-upstream broken-cache TTL default (forwarder).** The numeric default of the per-upstream-entry broken-cache TTL fixed by `FWD-026` in [`014-forward-zones.md`](014-forward-zones.md) is **to be specified**. The current normative content fixes the existence of the cache, its keying (by upstream-entry index in the rule), its bounded-TTL property, and its automatic-recovery semantics; only the numeric default and the operator-tunable knob remain to be specified.
- **Forwarder outbound connection pool numeric defaults.** The numeric defaults of the per-pool cap, the per-connection in-flight or stream cap, the global cap, the queue depth, the idle-keepalive timeout for HTTP/2, HTTP/3, and QUIC outbound connections, and the per-transport client-side stream caps fixed by `NET-056`, `NET-057`, and `NET-058` are **to be specified**. The current normative content fixes the existence of the pool, its scoping, its bounded-state property, the per-transport pool model, the multiplexing semantics, the idle-keepalive policy, and the invalidation triggers; only the numeric defaults of the operational parameters remain to be specified.
- **EDNS(0) Client Subnet (ECS) behaviour on the upstream path (forwarder).** Whether ECS options received from downstream clients are propagated verbatim, stripped, rewritten, or controlled by operator configuration on outbound queries to upstream resolvers is **to be specified**.
- **TLS policy.** The minimum TLS version is resolved in [`003-crypto-policy.md`](003-crypto-policy.md) (TLS 1.3 only), 0-RTT is refused on DoT and on DoH over HTTP/2, session resumption on DoT and DoH over HTTP/2 uses stateless TLS 1.3 session tickets, and TLS client-certificate authentication (mTLS) is supported as an optional, default-OFF, per-listener feature on DoT and on DoH over HTTP/2, all as resolved in the same document. The remaining TLS policy items, namely the session-ticket operational defaults (TEK rotation frequency, acceptance window for retired TEKs, and maximum ticket lifetime) and the cross-transport mTLS validation policy (trust store scope, chain validation rules, pinning, revocation checking, certificate-to-identity mapping) applying uniformly to TLS-over-TCP and TLS-inside-QUIC mTLS, are **to be specified** and are tracked in [`003-crypto-policy.md`](003-crypto-policy.md).
- **QUIC policy.** The QUIC version support, QUIC version negotiation, the exclusion of QUIC draft versions and non-standard variants, the refusal of QUIC 0-RTT application data for DoQ and for DoH over HTTP/3, the amplification-mitigation and address-validation policy for DoQ and for DoH over HTTP/3 (amplification-limit enforcement, Retry on new flows without a valid server-issued `NEW_TOKEN`, single-use `NEW_TOKEN` with anti-replay enforcement, and rotatable `NEW_TOKEN` key), and the optional, default-OFF, per-listener QUIC-layer client-certificate authentication (TLS-inside-QUIC mTLS) for DoQ and for DoH over HTTP/3 are resolved in [`003-crypto-policy.md`](003-crypto-policy.md). The remaining QUIC policy items — the `NEW_TOKEN` operational defaults (token-encryption-key rotation frequency, key retention window, strike-register persistence strategy) — are **to be specified** and are tracked in [`003-crypto-policy.md`](003-crypto-policy.md).
- **HTTP/2 and HTTP/3 hardening numeric defaults.** The minimum HTTP/2 and HTTP/3 conformance scope for DoH is resolved by `NET-025`, `NET-026`, `NET-027`, and `NET-028` (GET and POST with `application/dns-message`, a configurable DoH URI path with `/dns-query` as default, and no HTTP extensions beyond those required by [RFC 8484](https://www.rfc-editor.org/rfc/rfc8484); ALPN and Alt-Svc are governed by `NET-006` and `NET-007`). The HTTP/2 and HTTP/3 hardening posture is resolved in [`003-crypto-policy.md`](003-crypto-policy.md) (explicit server-layer mitigations covering total header-block size, per-connection stream-concurrency cap, HPACK and QPACK dynamic-table upper bounds, rapid-reset detection, CONTINUATION-frame cap, control-frame rate limits, incomplete-header-block timeout, and flow-control window bounds). The remaining HTTP/2 and HTTP/3 hardening item — the numeric defaults (threshold values, size limits, rate-limit parameters, window sizes, and timeouts) for those mitigations — is **to be specified** and is tracked in [`003-crypto-policy.md`](003-crypto-policy.md).

No implementation activity may proceed on the basis of assumptions about any of the items above.
