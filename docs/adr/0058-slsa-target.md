# ADR-0058: SLSA Build Level Target — Level 3

**Status:** Accepted  
**Date:** 2026-04-27  
**References:** ENG-073, THREAT-013  
**Deciders:** Lead maintainer

---

## Context

SLSA (Supply-chain Levels for Software Artefacts) is a framework for assessing and
communicating the integrity of a software build pipeline.  Without an explicit SLSA
target, Heimdall's supply-chain posture cannot be objectively measured or compared
against the claims made in SECURITY.md.

SLSA defines four build levels, each with progressively stronger guarantees:

| Level | Key property |
|---|---|
| SLSA 1 | Provenance exists (build script / CI log) |
| SLSA 2 | Provenance is signed by the build platform |
| SLSA 3 | Provenance is generated by a hardened, isolated build platform; not forgeable by the developer |
| SLSA 4 | Two-party review; hermetic, reproducible builds |

---

## Decision

**Target SLSA Build Level 3 for the v1.0 release.**

SLSA 3 is the highest level achievable with the current infrastructure (GitHub Actions +
sigstore/slsa-github-generator) without requiring hermetic build isolation (SLSA 4).
It provides:

- Platform-generated, unforgeable provenance (the CI platform signs the provenance, not
  the developer).
- Isolated builds (each job runs in a fresh ephemeral runner with no persistent state).
- Signed provenance bundled as an in-toto attestation attached to the release artefact.

SLSA 4 (hermetic + two-party review) is recorded as a post-v1.0 aspiration.

### Provenance Generation

Provenance is generated by the official
[`slsa-github-generator`](https://github.com/slsa-framework/slsa-github-generator)
action, which runs as a reusable workflow on GitHub Actions:

```yaml
# .github/workflows/release.yml (Tier 4 release pipeline)
jobs:
  build:
    outputs:
      hashes: ${{ steps.hash.outputs.hashes }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build release binary
        run: |
          export SOURCE_DATE_EPOCH=$(git log -1 --format=%ct HEAD)
          cargo build --release --locked
          sha256sum target/release/heimdall > SHA256SUMS
      - name: Output hash
        id: hash
        run: |
          echo "hashes=$(cat SHA256SUMS | base64 -w0)" >> "$GITHUB_OUTPUT"

  provenance:
    needs: [build]
    permissions:
      actions: read
      id-token: write
      contents: write
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@v1
    with:
      base64-subjects: "${{ needs.build.outputs.hashes }}"
      upload-assets: true
```

The workflow produces a `multiple.intoto.jsonl` provenance file and uploads it as a
release asset alongside the binary.

### Verification (Downstream Consumer)

```sh
# Install slsa-verifier
go install github.com/slsa-framework/slsa-verifier/v2/cli/slsa-verifier@latest

# Verify a downloaded artefact
slsa-verifier verify-artifact \
  heimdall-linux-amd64 \
  --provenance-path multiple.intoto.jsonl \
  --source-uri github.com/FlavioCFOliveira/Heimdall \
  --source-tag v1.0.0
```

A successful verification confirms:

- The artefact was built from the declared source commit.
- The build was performed by GitHub Actions (not a developer's workstation).
- The provenance was signed by GitHub's OIDC-based Sigstore identity.

### Evidence Collection (Per Release)

Each Tier 4 release must collect and publish the following SLSA evidence:

| Evidence | Location | Produced by |
|---|---|---|
| Signed provenance (SLSA 3) | `multiple.intoto.jsonl` attached to GitHub release | slsa-github-generator |
| Reproducible build confirmation | Tier 3 nightly run log + sha256 diff | repro-build CI job (ADR-0057) |
| SBOM (CycloneDX) | `heimdall-VERSION.cdx.json` attached to release | cargo-cyclonedx (ADR-0056) |
| Signing key provenance | Rekor transparency log URL | cosign (runbook/signing.md) |

## Gap Analysis (Current State → SLSA 3)

| Requirement | Current state | Gap |
|---|---|---|
| Build script exists | Yes (CI workflow) | None |
| Provenance generated | No | Add slsa-github-generator step to release.yml |
| Provenance signed by platform | No | Covered by slsa-github-generator (OIDC) |
| Isolated build environment | Partial (GitHub-hosted runner) | Use ephemeral runners; confirm no cross-job state |
| Source integrity | Yes (Cargo.lock + --locked) | None |
| Reproducible build | Partial (SOURCE_DATE_EPOCH added, ADR-0057) | Confirm byte-for-byte match in Tier 3 |
| SBOM published | No | Add release step (ADR-0056) |

Outstanding gaps are tracked as follow-up engineering items:

- **ENG-080**: Wire slsa-github-generator into release.yml.
- **ENG-081**: Confirm ephemeral runner isolation (no cross-job state).
- **ENG-082**: Publish SBOM as release asset at Tier 4.

## Consequences

### Positive

- A third party can independently verify that a downloaded Heimdall binary was built
  from the declared source on GitHub Actions, using only the provenance file and
  `slsa-verifier`.
- SLSA 3 satisfies the supply-chain integrity requirements of most enterprise
  environments and government procurement guidelines.
- Provenance is machine-readable (in-toto format) and can be ingested by policy
  engines (OPA, Kyverno, Sigstore Policy Controller).

### Negative

- SLSA 3 does not prevent a malicious or compromised GitHub Actions runner from
  producing a forged binary (SLSA 4 requirement).  This is an accepted risk for v1.0.
- slsa-github-generator relies on GitHub's OIDC endpoint; if GitHub's Fulcio CA is
  compromised, the provenance chain breaks.  Mitigation: publish Rekor transparency
  log entries for independent auditability.

### Neutral

- SLSA 4 (hermetic builds + two-party review) is a post-v1.0 goal.  It requires
  a fully reproducible build (confirmed by ADR-0057) and a two-reviewer merge policy
  (already enforced by ENG-156 branch protection).

---

## Alternatives Rejected

| Alternative | Reason for rejection |
|---|---|
| SLSA 1 | Provenance not signed; insufficient for a security-critical server |
| SLSA 2 | Does not require platform-generated provenance; developer can forge it |
| SLSA 4 target for v1.0 | Hermetic build isolation not yet achievable; deferred post-v1.0 |
| No SLSA target | Cannot make credible supply-chain integrity claims |
